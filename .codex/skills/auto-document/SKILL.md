# 自动文档维护规范

本技能由原 `.claude/skills/auto-document` 迁移而来。底层逻辑保持不变：任何重要代码或配置变更完成后，都要同步维护项目文档，避免代码和说明脱节。

## 1. 触发标准

满足任一条件时，应执行文档维护：

- 新功能落地：实现业务逻辑、Agent 流程、工具调用、前端页面等。
- 架构或配置变更：修改系统配置、部署方式、模型路径、协议、状态结构等。
- Bug 修复或重构：修复系统级/业务级问题，调整核心代码结构。
- 测试用例编写或执行：新增测试、执行自动化测试、完成手工验证。
- 用户明确要求：用户表示阶段完成、要求整理文档、要求记录问题处理过程。

## 2. 文档分流规则

必须遵守当前项目的中文文档结构，禁止在根目录或 `docs/` 中随意创建零散说明文件。

| 文档或目录 | 唯一职责 | 触发更新的变化 |
|---|---|---|
| `docs/README.md` | 文档总导航 | 阅读顺序、目录职责或事实优先级变化 |
| `docs/项目总览.md` | 当前项目事实 | 项目定位、模块状态、能力边界和下一步变化 |
| `docs/接口说明.md` | 接口参考 | HTTP、SSE、WebSocket、请求响应和错误协议变化 |
| `docs/运行与排错.md` | 操作手册 | 安装、配置、测试、部署和故障处理变化 |
| `docs/项目证据.md` | 验证证据 | 自动化测试、专项评测和真实链路结果变化 |
| `docs/interview/` | 面试学习材料 | 简历口径、项目话术、技术问答和模拟面试变化 |
| `docs/technical/` | 技术专题 | 路由、动作、工具系统等深入设计变化 |
| `docs/optimization/` | 优化演进 | 记忆、上下文和生产化方案变化 |
| `docs/miniprogram/` | 微信小程序专项 | 页面、组件、接口封装、流式解析和联调状态变化 |
| `docs/superpowers/` | 早期归档 | 原则上只保留，不作为当前状态修改 |

## 3. 记录原则

- 当前事实、设计方案和历史归档必须明确区分。
- 测试结果只保留面试和维护需要的高价值证据，不恢复日期流水账目录。
- 修改面试材料时，第一次出现的关键技术要解释“是什么、解决什么、项目怎么用、替代方案、为什么选择”。
- 文档与代码冲突时，优先修正文档，并在当前边界中准确说明。

## 4. 执行方式

1. 盘点本次实际修改了哪些代码、配置、测试或文档。
2. 判断是否影响项目说明、进度、接口、运行方式或问题记录。
3. 按分流规则更新对应文档。
4. 在最终回复中简要说明已同步哪些文档。
