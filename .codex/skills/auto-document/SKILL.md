# 自动文档维护规范

本技能由原 `.claude/skills/auto-document` 迁移而来。底层逻辑保持不变：任何重要代码或配置变更完成后，都要同步维护项目文档，避免代码和说明脱节。

## 1. 触发标准

满足任一条件时，应执行文档维护：

- 新功能落地：实现业务逻辑、Agent 流程、工具调用、前端页面等。
- 架构或配置变更：修改系统配置、部署方式、模型路径、协议、状态结构等。
- Bug 修复或重构：修复系统级/业务级问题，调整核心代码结构。
- 测试用例编写或执行：新增测试、执行自动化测试、完成手工验证。
- 用户明确要求：用户表示阶段完成、要求整理文档、要求记录问题处理过程。

## 2. 文档分流规则

必须遵守当前项目的 `docs/` 目录结构，禁止在根目录随意创建零散说明文件。

### `docs/README.md`

用于维护当前项目主状态：

- 项目是做什么的。
- 当前模块进度。
- 已知问题和处理方式。
- 运维命令和运行方式。
- 配置说明。
- 下一步计划。

### `docs/API.md`

用于维护接口细节：

- 新增、删除或修改 HTTP/WebSocket/SSE 接口。
- 请求体、响应体、状态码变化。
- 意图路由规则变化。
- API 调用示例变化。

### `docs/miniprogram/`

用于维护微信小程序专项文档：

- 页面、组件、API 封装、SSE 解析器变化。
- 小程序联调方式变化。
- 小程序实现状态变化。

### `docs/progress/`

用于记录阶段性开发、重构和 Bug 修复过程。Bug 记录应包含：

- Bug 表现。
- 根因分析。
- 修复方案。
- 影响范围。
- 防御性改进。
- 后续计划。

### `docs/tests/`

用于记录测试和验收：

- 新增测试用例。
- 自动化测试执行结果。
- 手工冒烟测试。
- 分级验收。
- 遗留风险。

### `docs/superpowers/`

用于保留早期设计方案和规格。一般不随意修改或删除，除非用户明确要求迁移、归档或整理。

## 3. 记录格式

写入 `docs/progress/` 或 `docs/tests/` 的记录建议包含：

- 时间戳。
- 操作类型：功能新增、代码重构、Bug 修复、测试执行、配置变更。
- 变更概述。
- 影响范围。
- 测试结果或遗留问题。
- Next Steps。

## 4. 执行方式

1. 盘点本次实际修改了哪些代码、配置、测试或文档。
2. 判断是否影响项目说明、进度、接口、运行方式或问题记录。
3. 按分流规则更新对应文档。
4. 在最终回复中简要说明已同步哪些文档。
