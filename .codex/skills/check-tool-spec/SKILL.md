# 工具设计规范检查器

本技能由原 `.claude/skills/check-tool-spec` 迁移而来。底层逻辑保持不变：任何工具或函数接口设计，都需要从职责、输入、输出、权限和错误处理五个维度审查。

## 1. 职责清晰

要求：

- 工具边界明确。
- 保持单一职责。
- 描述准确说明用途和适用边界。
- 避免“万能工具”。

检查点：

- 这个工具到底解决什么问题？
- 是否混入了不相关能力？
- 调用者是否能从名称和描述理解何时使用它？

## 2. 输入清晰

要求：

- 参数结构明确、稳定、可校验。
- 优先使用结构化字段，而不是大段模糊字符串。
- 合理使用类型、必填字段、枚举、范围约束。

检查点：

- 参数类型是否完整声明？
- 是否有必填字段？
- 是否需要枚举值或数值范围？
- 是否避免让模型猜测隐式格式？

## 3. 输出清晰

要求：

- 输出格式稳定可预测。
- 推荐返回结构化 JSON 或对象。
- 输出包含后续处理所需的状态、ID、错误信息和元数据。

检查点：

- 调用方能否稳定解析返回值？
- 成功和失败输出是否区分清楚？
- 是否包含必要的状态码或错误码？

## 4. 权限清晰

要求：

- 明确工具可以做什么，不能做什么。
- 涉及写操作、敏感数据、高风险操作时，应有确认或权限边界。
- 对边界模糊的操作，应请求用户确认。

检查点：

- 是否涉及文件写入、删除、外部命令、网络、凭据、个人数据？
- 是否存在不可逆风险？
- 是否需要用户确认？

## 5. 错误可处理

要求：

工具失败时，错误类型应足够明确，便于模型或程序采取不同处理策略。

错误分类：

- 参数错误：可修正参数后重试。
- 权限错误：应停止并告知用户。
- 网络错误：可提示用户或有限重试。
- 数据为空：正常业务状态，不应伪装成未知异常。
- 内部错误：记录上下文，避免吞错。

## 6. 输出审查结果的建议格式

```text
结论：通过 / 需修改 / 高风险

职责清晰：...
输入清晰：...
输出清晰：...
权限清晰：...
错误可处理：...

主要风险：
- ...

建议修改：
- ...
```

如果用户要求给出重构后的工具定义，应直接提供完整 schema 或类型定义。
