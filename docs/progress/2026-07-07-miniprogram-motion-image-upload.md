# 黄金演示闭环阶段 2：小程序 Motion 图片上传

## 阶段目标

让项目最具辨识度的 Motion 能力进入小程序主流程，形成用户可见的“主动选择图片—真实姿态提取—结果与边界展示”闭环，而不是只在 Web UI 或接口文档中存在。

## 已完成内容

- 聊天输入区新增动作图片按钮，支持相册和相机；旧基础库降级到 `chooseImage`。
- 客户端在上传前检查 10MB 限制，用户取消选择时不制造错误消息。
- `wx.uploadFile` 接入 `/motion/analyze-image`，统一解析成功结果、HTTP 错误和网络错误。
- 用户消息显示本地图片缩略图，点击可调用系统图片预览。
- 分析结果展示帧数、关键点数、姿态模型、关键点规范、平均置信度和静态图片边界。
- 后端图片响应增加 `execution`，成功时明确标记 `motion · mediapipe_image`，不存在 mock 图片分析。

## 接口设计检查

- 职责：上传函数只负责单张动作图片，不混入视频或聊天请求。
- 输入：必须由用户主动选择本地临时文件；格式由选择器和后端共同校验，最大 10MB。
- 输出：复用后端结构化姿态摘要，并增加稳定执行标签。
- 权限：不后台读取相册，不持久化上传图片；服务端处理完成后关闭上传文件。
- 错误处理：取消选择静默结束；超大图片前置提示；422、503、网络错误分别保留可理解原因。

## 当前验证

- 图片接口定向测试：`2 passed, 1 warning`。
- 全量自动化回归：`129 passed, 2 skipped, 1 warning`。
- Python 编译和小程序 JavaScript 语法检查通过。

## 当前边界

- 单张图片只能证明姿态关键点提取，不能判断动作节奏、轨迹、重复稳定性或完整动作质量。
- 尚未在微信开发者工具和真机验证相册/相机授权、`uploadFile` 合法域名和缩略图布局。
- 上传过程目前只有等待状态，没有百分比进度；视频阶段统一补充。
