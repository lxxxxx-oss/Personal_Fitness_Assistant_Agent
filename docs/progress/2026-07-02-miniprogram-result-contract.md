# 小程序阶段 1：回答元数据闭环

## 阶段目标

统一后端非流式、SSE 和 WebSocket 的回答元数据，并让小程序能够展示来源与非致命运行提示，为后续结构化卡片和媒体能力建立稳定协议。

## 本阶段完成内容

- `/chat`、`/chat/stream`、`/chat/ws` 统一返回 `sources` 与 `warnings`。
- 后端对来源和提示进行顺序去重，避免组合路由重复展示同一信息。
- 小程序 WebSocket 主链路和 HTTP 降级链路均保存回答元数据。
- 消息气泡新增“参考来源”和“运行提示”卡片；数据为空时不渲染。
- 增加 API 自动化测试，覆盖非流式去重、SSE 字段和 WebSocket meta 消息。

## 工程取舍

本阶段只透传链路级来源，不伪造“正文句子—来源”逐条对应关系。这样先解决结果可追溯和降级可见性，同时把严格 citation 校验保留为后续独立能力。

## 影响范围

- 后端公开响应协议新增可选数组字段，但不破坏旧客户端。
- 小程序消息本地结构新增 `sources`、`warnings`。
- Search、组合路由和工具降级的运行信息可以在端侧被用户看到。

## 验证与边界

- API 定向测试 `6 passed, 1 warning`；全量回归 `124 passed, 2 skipped, 1 warning`。
- Python 编译与小程序 JavaScript 语法检查已通过。
- 尚未在微信开发者工具和真机验证卡片布局、URL 交互及弱网降级。
- 来源目前以可选择文本展示，点击复制或打开能力留到体验优化阶段。

## 下一阶段

增加后端地址设置、连接检测和环境切换入口，减少小程序联调时修改源码的成本。
