# 2026-06-30 Interview 简历口径修正

## 背景

另一台电脑根据简历更新了 `docs/interview/`，但出现了三类问题：

1. 默认面试官能直接看到代码仓库，材料围绕“当前代码 / 当前仓库”展开。
2. 频繁主动暴露原型差异，容易削弱简历主线。
3. 没有完全按“项目思路、技术取舍、问题解决、能力边界”的面试回答方式组织。

## 本次调整

- 重写 `docs/interview/README.md`，明确默认场景是面试官基于简历追问，不直接看本地实现。
- 新增 `docs/interview/00_RESUME_TECH_INDEX.md`，把简历上的项目、技能、实习和论文技术点汇总成面试学习索引。
- 重写 `docs/interview/01_MUST_MASTER_PROJECT_STORY.md`，按简历主线组织项目介绍、个人贡献、核心亮点和高压问题。
- 重写 `docs/interview/02_SHOULD_MASTER_TECH_QA.md`，按简历技术点准备问答，强调选型理由、替代方案、问题解决和边界表达。
- 重写 `docs/interview/03_GOOD_TO_KNOW_DEEP_DIVE.md`，从文件路径背诵改为白板级实现链路、算法解释和生产化路线。

## 口径原则

1. 先按简历讲项目价值和个人贡献。
2. 技术选型要讲为什么选、替代方案和取舍。
3. 能力边界只在被追问时讲，用“原型链路 + 生产化补齐”承接。
4. 不主动把简历亮点拆成“当前仓库没做完”。
5. 不虚构低级事实；Milvus、MCP、Motion 等高压点要能讲清架构位置、当前边界和补齐路径。
6. 第一次提到技术点时必须讲清“是什么、解决什么、怎么用、类似方案、为什么选它”。
7. 简历上出现过的技术点要集中维护到 `docs/interview/00_RESUME_TECH_INDEX.md`，避免学习材料漏掉简历关键词。

## 测试说明

本次仅修改面试学习文档，没有修改运行时代码，因此未执行自动化测试。
