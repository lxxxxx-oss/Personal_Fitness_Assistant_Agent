# 项目文档总导航

这个项目的首要目标，是帮助你通过 Agent 相关岗位面试。文档不是写给已经会编程的人看的技术手册，而是一套从“听懂项目”到“能独立回答追问”的学习材料。

如果你完全没有代码基础，不要从接口、配置或源码开始。按下面的顺序学习即可。

## 正确的学习顺序

```text
先理解项目故事
  -> 再理解 Agent 和各项技术为什么存在
  -> 再学习面试表达
  -> 最后用代码与测试作为回答证据
```

| 阶段 | 阅读材料 | 学完后的目标 |
|---|---|---|
| 1. 听懂 | [零基础项目地图](./interview/01_零基础项目地图.md) | 能用自己的话解释项目解决什么问题、为什么算 Agent |
| 2. 讲清 | [项目主线](./interview/02_项目讲解与面试话术.md) | 能完成 30 秒和 3 分钟项目介绍 |
| 3. 补技术 | [简历技术点总表](./interview/03_简历技术点总表.md) | 理解简历上的技术名词，不只是背缩写 |
| 4. 扛追问 | [技术问答](./interview/04_高频技术问答.md) | 能回答“为什么这样设计”“还有什么替代方案” |
| 5. 做模拟 | [面试模拟](./interview/08_模拟面试.md) | 练习连续追问，不依赖照稿念答案 |
| 6. 临场复习 | [一页速记](./interview/07_面试前速记.md) | 面试前十分钟快速唤醒记忆 |

## 只记住项目的一个核心

> 这不是为了把健身产品做到多完整，而是借助健身场景，证明自己理解如何设计一个可路由、会使用知识和工具、能记住上下文、可以评测和降级的 Agent 系统。

健身只是承载技术的业务场景。真正要展示的是以下能力：

- 能把复杂需求拆成不同任务，而不是把所有问题塞给一次大模型回答。
- 能让大模型与知识库、联网搜索、动作算法和外部工具协作。
- 能管理对话上下文和长期用户信息，而不是每轮都“失忆”。
- 能解释系统为什么这样设计、失败时怎么办、如何判断效果。
- 能区分演示原型和生产系统，并讲清楚后续升级路线。

## 文档分成四个区域

| 区域 | 内容 | 你怎么使用 |
|---|---|---|
| 面试学习区 | `interview/` | 主要学习区；理解、背诵和模拟都在这里完成 |
| 项目事实区 | `项目总览.md`、`接口说明.md`、`运行与排错.md` | 面试官追问实现时，用来核对当前代码事实 |
| 深入设计区 | `optimization/`、`technical/`、`miniprogram/` | 技术深挖时查阅，不要求一开始学习 |
| 证据与历史区 | `项目证据.md`、`superpowers/` | 前者证明做过测试；后者保存早期方案，不作为当前口径 |

## 每个技术点都按同一方法学习

遇到 LangGraph、RAG、Milvus、MCP 等陌生词时，不要先背定义。按下面五个问题理解：

1. 它是什么？先用生活化语言解释。
2. 它解决什么问题？没有它会发生什么？
3. 项目里怎么用？输入、处理、输出分别是什么？
4. 还有哪些方案？为什么没有选择它们？
5. 面试官继续追问时，当前边界和升级方向是什么？

所有面试材料都应逐步统一到这个结构。

## 不需要现在阅读的内容

- `接口说明.md`：它像产品的“插座说明书”，需要联调时再查。
- `运行与排错.md`：它解决项目怎么启动，不帮助你理解 Agent 主线。
- `technical/`：适合已经掌握项目故事后深挖。
- `superpowers/`：属于早期设计记录，可能与当前代码不同。

## 文档与代码冲突时怎么办

当前事实以代码和自动化测试为准；面试表达以 `interview/` 为主，但必须保留可以应对追问的解释。优先级是：

```text
代码与测试事实
  > 当前项目说明
  > 面试学习材料
  > 设计规划与历史材料
```

这不意味着面试开场要主动罗列不足。正确方法是先讲已经验证的核心思路和价值；只有被问到具体实现深度时，再说明当前原型做到哪里，以及生产化会如何补齐。
