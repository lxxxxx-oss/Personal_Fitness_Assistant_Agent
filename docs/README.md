# 项目文档总导航

这套文档只服务一个最终目标：帮助你理解项目、讲清项目，并通过智能体相关岗位面试。

项目以个人健身助手为业务外壳，真正展示的是智能体工程能力：任务路由、工作流编排、知识检索、工具调用、记忆管理、失败降级和效果评测。

## 第一次阅读只看这四篇

如果你完全没有代码基础，请严格按照顺序阅读：

| 顺序 | 文档 | 解决的问题 | 学完的标准 |
|---:|---|---|---|
| 1 | [零基础项目地图](./interview/01_零基础项目地图.md) | 这个项目为什么算智能体 | 能不用技术名词讲清完整流程 |
| 2 | [项目讲解与面试话术](./interview/02_项目讲解与面试话术.md) | 面试开场怎么讲 | 能完成 30 秒和 3 分钟介绍 |
| 3 | [简历技术点总表](./interview/03_简历技术点总表.md) | 每项技术为什么存在 | 能解释核心技术的作用和取舍 |
| 4 | [高频技术问答](./interview/04_高频技术问答.md) | 面试官会怎样追问 | 能连续回答两到三轮追问 |

其他文档都属于查阅资料，不应该在第一轮学习时从头阅读。

## 项目最重要的一句话

> 这个项目不是为了证明健身功能有多完整，而是借健身场景证明我能把大模型、知识、搜索、算法、工具和记忆组织成一个受控、可解释、可评测的智能体系统。

所有文档都围绕这句话展开。学习时不要被大量技术名词带偏。

## 遇到陌生技术名词怎么办

不要先背定义。依次问五个问题：

1. 它可以用什么生活化角色来类比？
2. 项目没有它会遇到什么问题？
3. 它接收什么，处理什么，最后输出什么？
4. 还有哪些替代方案？
5. 为什么这个项目选择当前方案？

例如，路由器可以类比医院分诊台；向量数据库可以类比按“意思”而不是按“原词”找资料的档案系统；工具注册表可以类比公司内部工具台账。

## 事实与面试表达如何区分

同一项能力分为三层：

| 层次 | 含义 | 使用位置 |
|---|---|---|
| 项目主线 | 希望向面试官展示的完整设计 | 开场介绍和架构讲解 |
| 当前证据 | 代码、接口、测试能够支撑的部分 | 被追问实现细节时 |
| 演进方向 | 真正生产化还要补充的能力 | 回答边界和下一步时 |

面试开场先讲项目价值，不需要主动罗列所有不足；被问到实现深度时，再准确说明当前证据和升级路线。
