# 个人健身助手智能体项目总览

本文档是当前项目事实的唯一总入口。它负责说明项目是什么、代码已经做到哪里、有哪些证据、还存在哪些边界。

如果你的目标是准备面试，不要从本文开始背诵，请先阅读[项目文档总导航](./README.md)；如果需要学习项目话术，请阅读[项目讲解与面试话术](./interview/02_项目讲解与面试话术.md)。

## 1. 项目定位

项目的第一目标是服务智能体相关岗位面试。健身、营养和动作分析是业务场景，真正要展示的是：

- 怎样判断用户意图并选择流程。
- 怎样让大模型与知识库、搜索、算法和外部工具协作。
- 怎样管理状态、上下文和长期记忆。
- 怎样处理工具失败、依赖降级和结果追踪。
- 怎样用测试和评测验证系统。

一句话描述：

> 这是一个基于 LangGraph 的多任务智能体原型，通过受控路由把领域知识检索、实时搜索、动作分析、外部工具和记忆系统组织成可解释、可评测、可降级的执行流程。

## 2. 总体架构

系统分为四层：

```text
接入层
  FastAPI / HTTP / SSE / WebSocket / Web UI / 微信小程序
      ↓
控制层
  Router / LangGraph / 路由计划 / 子工作流
      ↓
能力层
  Knowledge / Search / Motion / MCP / ToolRegistry
      ↓
保障层
  Memory / PromptBuilder / ToolResult / fallback / 测试与评测
```

各层职责：

| 层次 | 负责什么 | 不负责什么 |
|---|---|---|
| 接入层 | 接收请求和返回结果 | 决定任务应该怎样执行 |
| 控制层 | 判断意图、选择流程、维护共享状态 | 完成具体检索或计算 |
| 能力层 | 检索知识、搜索网络、计算动作、调用工具 | 决定整个业务流程 |
| 保障层 | 管理上下文、错误、降级和质量证据 | 替代业务能力本身 |

## 3. 用户请求怎样流转

### 普通文本请求

```text
用户问题
  -> FastAPI 接收请求
  -> Router 识别主意图、次意图和置信度
  -> 生成单路由或白名单组合计划
  -> LangGraph 调用对应子工作流
  -> 子工作流调用知识、搜索、工具或模型
  -> 写回回答、来源、警告和执行轨迹
  -> 返回最终结果
```

### 图片或视频动作请求

```text
图片 / 视频 / .npz
  -> 独立动作分析接口
  -> 姿态估计或数值文件解析
  -> 统一 PoseSequence
  -> 可选标准动作比较
  -> 结构化动作结果与提示
```

媒体上传通过独立接口执行；对话 Router 仍可以识别动作相关文本，并进入 Motion 子工作流进行规划和解释。

## 4. 当前能力状态

### 4.1 路由与图编排

当前实现：

- 顶层使用 LangGraph StateGraph 组织路由和子工作流。
- Router 支持 `search`、`motion`、`diet`、`chat`、`mcp` 兼容意图。
- Chat 与 Diet 在内部统一到 Knowledge 能力域，再区分普通问答和饮食建议。
- 支持主意图、次意图、置信度和受控两步组合。
- 当前白名单组合为 Search + Diet、Search + Chat、Motion + Chat、Motion + Diet。
- 低置信路径已接入可选 embedding 样例匹配，默认关闭；需要通过语义改写集和延迟评测证明收益后再启用。
- 本地 Qwen 分类器已完成对比验证，但默认关闭。

设计取舍：明确问题优先使用可解释、低延迟的确定性策略；复杂组合不开放任意规划，只执行经过测试的白名单。

### 4.2 领域知识与饮食建议

当前实现：

- 本地健身和营养知识支持文档分块、向量编码和检索。
- 分块支持中英文句子边界、超长句硬切分，以及可配置 chunk size / overlap。
- Retriever 可以选择内存后端或 Milvus。
- 默认 embedding 已统一为 512 维 `BAAI/bge-small-zh-v1.5`；RAG 与可选长期记忆索引使用模型专属默认 Collection 名，避免模型迁移时误用旧维度向量。
- Milvus 支持版本化稳定主键、幂等写入、来源字段、向量索引和同源旧片段清理。
- 入库结果会返回轻量 manifest，记录知识版本、来源、chunk 数、chunk id 和内容哈希，便于面试时说明知识更新如何追踪。
- Chat 和 Diet 共用统一的 `RETRIEVER_TOP_K` 与 `RETRIEVER_THRESHOLD` 配置，检索结果经过阈值过滤、去重和编号后进入 Prompt。
- 已建立第一版 15 条 RAG 检索黄金集，同时检查正确来源、正确片段、无答案拒绝和查询延迟；评测结果表明阈值提高会改善拒答但损失部分召回。
- Knowledge 和 Diet 的检索通过 ToolRegistry 中的 `knowledge.retrieve` 执行。
- 饮食画像经过 JSON 解析和 Pydantic 范围、枚举校验。
- 非法画像不会直接进入推荐链路，而是降级为空画像并产生警告。

当前边界：来源已经能够透传，轻量 manifest 能支撑原型级知识更新追踪，第一版黄金集能做参数回归；embedding 模型加载失败和 Milvus 维度冲突都有可观察降级，但新模型首次使用仍需下载或预置缓存。样本只有 15 条，逐句引用验证、生产级批次回滚、重排序和大规模问答评测仍可增强。

### 4.3 联网搜索

当前实现：

- 先理解并改写查询，再调用 Tavily，最后合成回答。
- 搜索结果保留标题、摘要和网址。
- 未配置密钥时可以使用演示结果。
- Search 已通过 ToolRegistry 中的 `search.tavily` 调用。
- 真实调用失败会返回可处理错误和降级信息。

当前边界：现有来源约束不等于逐句引用完全正确，后续仍需来源可信度和引用校验。

### 4.4 动作分析

当前实现：

- 支持 `.npz`、图片和短视频输入。
- MediaPipe 可将图片和视频帧转换为 33 点人体姿态。
- OpenCV 负责视频读取和抽帧。
- PoseSequence 统一保存关键点、帧率、模型、关节结构和置信度。
- 支持与兼容标准样本进行 FastDTW、余弦、对齐后逐关节平均距离、最大偏差关节/帧定位和 MediaPipe 膝/髋角度误差计算。
- 支持用关键点置信度过滤低质量关节；当有效对齐比例不足时返回低置信 warning 和 `quality.accepted=false`，全部关键点都不可用时才拒绝输出相似度结论。
- 支持从标准视频离线构建标准动作文件。
- 不兼容的姿态模型或关节结构会被拒绝。
- `motion.compare_pose` 已登记到 ToolRegistry，但媒体接口仍由 API 和 LangGraph 直接控制。

当前边界：相似度和角度误差表示与参考样本的统计接近程度，不等同于专业教练或医疗诊断；正式动作库、周期切分、阶段级专项规则和教练标签仍需扩充。

### 4.5 工具系统与 MCP

当前实现：

- 使用统一 `ToolResult` 和 `ErrorCode` 表达成功、失败和错误类型。
- ToolSpec 描述工具名称、说明、输入结构、权限、执行器、超时策略字段、重试和 fallback。
- ToolRegistry 支持注册、列出、参数校验、权限检查、有限重试、fallback 和审计。
- 每次执行记录 `execution_id`、`duration_ms`、尝试次数和降级来源。
- 默认登记知识检索、Tavily 搜索、动作比较和 MCP 调用四类代表工具。
- Search、Knowledge/RAG 和 MCP 执行路径已经接入注册表。
- MCP Client 支持子进程、标准输入输出、JSON-RPC 初始化、工具列表和工具调用主链路。
- 默认 MCP 使用 mock，保证本地演示稳定。

当前边界：注册表的 `timeout_seconds` 目前是策略元数据，不会强制中断长时间执行；MCP 的响应编号匹配、深层参数校验、进程隔离和真实服务兼容性仍需增强。

### 4.6 记忆与上下文

当前实现：

- 滑动窗口保存近期对话。
- 会话消息和长期记忆写入本地 SQLite。
- 会话摘要存储基础已接通：完整问答按单事务写入，active summary 可替换，`last_compacted_message_id` 记录摘要覆盖边界，并按用户与会话隔离读取。
- 已实现独立的确定性抽取式摘要器：只使用原始消息片段，按字符阈值触发、限制摘要长度、复用上一版 active summary，并保留最近三轮原文。
- HTTP、SSE 和 WebSocket 共用同一个 turn 持久化与摘要更新入口；下一次请求会加载 active summary，并在 Chat、Diet、Search 和 MCP 最终 Prompt 中与最近对话、长期记忆分区注入。
- 长期记忆支持创建、读取、修改、逻辑删除和检索。
- 用户明确说“记住……”时可以写入长期记忆。
- 健康、伤病和过敏等敏感内容先进入候选确认。
- SQLite 是事实来源，Milvus 是可选语义召回增强。
- PromptBuilder 集中组织近期对话、长期记忆、知识证据和任务信息。
- Prompt 超过阈值时执行确定性 compact，并记录长度和压缩轨迹。

当前边界：第一版会话摘要是确定性原文摘录，不具备模型级语义归纳能力；使用字符阈值而非精确 token，摘要更新也没有多实例并发锁。摘要失败不会阻塞聊天，系统继续使用最近滑动窗口和本轮临时 compact。完整候选确认界面、常驻后台同步任务、多实例共享、用户授权和生产级隐私治理尚未完成。

### 4.7 接口与客户端

当前实现：

- FastAPI 提供非流式 HTTP、SSE 和 WebSocket 对话接口。
- 同步图执行和模型生成通过线程桥接进入异步服务。
- 模型生成仍使用进程内锁，保证同进程串行推理。
- `/ui` 提供本地 Web 演示界面。
- 微信小程序代码已覆盖对话和动作媒体上传主链路。

当前边界：微信开发者工具、真机、HTTPS 和弱网环境尚未完成完整验收。

| 服务 | Python、FastAPI、Pydantic、uvicorn | 接口、校验和服务运行 |

| 客户端 | Web UI、微信小程序 | 演示与用户交互 |
| 质量 | pytest、Router 评测、专项 benchmark | 防回归和效果验证 |

这组测试主要证明接口、状态流、工具契约、错误处理和算法可以回归。测试默认替换真实本地模型和部分 embedding，不等同于真实回答质量。

其他证据：

- Router 困难样本集 36 条。

- RAG 第一版黄金集包含 12 条可回答问题和 3 条无答案问题；BGE、Top-K=5、阈值 0.50 下 Recall@5 为 100%、MRR 为 0.892，但无答案拒绝率只有 33.3%，说明仍需校准阈值而不能只报召回率。

## 8. 本地运行

```powershell
conda activate fitness-agent
pip install -r requirements.txt
$env:LLM_MOCK="true"
$env:RETRIEVER_BACKEND="memory"
$env:MCP_SERVER_COMMAND="mock"
python -m uvicorn app.main:app --host 127.0.0.1 --port 8000
```

启动后：

- Web 界面：`http://127.0.0.1:8000/ui`
- 存活检查：`http://127.0.0.1:8000/health`
- OpenAPI：`http://127.0.0.1:8000/docs`

动作图片和视频分析需要额外安装 `requirements-motion.txt`，并准备 MediaPipe 姿态模型。完整说明见[项目运行与排错](./运行与排错.md)。

## 9. 当前系统边界

当前版本只适合本地开发和面试演示，不能直接暴露到公网，原因包括：

- 没有登录鉴权、租户隔离和请求限流。
- `user_id` 是客户端会话键，不是安全身份凭证。
- CORS 允许任意来源。
- `/health` 只证明进程存活，不检查外部依赖。
- 本地 SQLite 和进程内缓存不支持完整多实例治理。
- 模型、Milvus、MediaPipe、Tavily 和 MCP 都需要独立就绪检查。
- Docker 尚未完成包含动作依赖和模型资产的跨机器完整验证。

## 10. 生产化演进顺序

1. 建立认证授权、用户级数据隔离、限流和审计。
2. 把模型推理拆成独立服务，增加超时、队列和资源监控。
3. 扩大现有 RAG 检索黄金集，并补充生成答案忠实度、更多困难负样本和 Milvus 端到端延迟。
4. 扩充标准动作、动作周期和教练标签，建立 Motion 质量评测。
5. 增强工具权限、硬超时、熔断和 MCP 进程隔离。
6. 完善长期记忆授权、删除同步、加密和多实例存储。
7. 完成小程序真机、弱网、HTTPS 和跨机器容器验收。
