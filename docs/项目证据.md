## 证据强弱与面试防守清单

这一节专门回答“哪些已经证明得比较实，哪些只是原型链路或仍需验收”。面试时不要把弱证据说成生产级能力。

| 链路 | 当前证据强度 | 已证明什么 | 不要过度声称什么 |
|---|---|---|---|
| 后端核心 API | 强 | 全量 pytest、专项评测和服务级 E2E 已覆盖 `/health`、`/chat`、Search/MCP mock、`/memory`、`/motion/analyze`、`/chat/stream` | 不等于所有真实外部服务、所有用户输入和线上并发都已经验证 |
| Router | 强 | 常规集、challenge set、语义改写集均为当前确定性配置下 100% | 不等于线上自然语言分布永远 100%，也不等于 embedding/Qwen 路由默认启用 |
| RAG / Milvus | 中强 | 有 Retriever/Milvus 回归、15 条黄金集和已缓存 BGE 环境下的 embedding 基线 | 不等于任意新机器都能无下载复现；15 条样本不能代表生产级知识问答质量 |
| Web UI | 中 | `/ui/` 可访问，前端 JavaScript 语法检查通过，后端 SSE 已服务级验证 | 尚未完成浏览器点击、输入、流式渲染和异常态的完整手工/自动化验收 |
| 微信小程序 | 中弱 | 代码层面已覆盖对话和动作媒体上传主链路，文档已说明协议设计 | 微信开发者工具、真机、HTTPS、弱网和跨机器后端尚未完整验收 |
| Motion | 中 | 图片/视频姿态链路、结构化 PoseSequence、专项回归和 `.npz` 服务级 E2E 已覆盖 | 不等于专业教练级纠错；正式动作库、周期切分、阶段规则和教练标签仍需扩充 |
| Tavily / 真实 MCP / 本地 Qwen | 中弱 | 配置入口、mock 降级和错误轨迹可验证；真实依赖可替换接入 | 稳定演示环境不证明真实联网、真实 MCP Server 或真实本地模型回答质量 |

被追问“项目哪里还最弱”时，可以这样回答：

> 当前后端核心链路和工程回归证据最完整；薄弱点主要在真实端侧验收和真实外部依赖质量评估。比如 Web UI 目前证明了可访问和 SSE 后端契约，但还没补完整浏览器交互自动化；小程序还需要真机、HTTPS 和弱网验收；Motion 已打通姿态算法链路，但专业级动作质量需要标准动作库和教练标签。这些不是架构问题，而是生产化验证阶段要继续补的证据。

## 怎么回答“你怎么证明项目不是玩具项目”

可以这样说：

> 我没有只做功能演示，而是给关键 Agent 能力补了可回归证据。Router 有离线 eval、challenge set 和语义改写集；RAG 除了 Milvus 真实链路验证，还有包含正确来源、正确片段和无答案问题的黄金集，用 Recall@K、MRR、拒绝率和延迟观察参数取舍；Motion 有真实图片/视频姿态链路和可解释指标回归；工具系统记录 execution_id、耗时、重试和 fallback；Memory + Context + RAG 还有统一 benchmark，验证多个模块组合后没有回归。

## RAG 基线怎样解读

- 基线模型：在已缓存 `BAAI/bge-small-zh-v1.5` 的评测环境中，使用内存后端验证与 Milvus 路径相同的 Sentence-Transformers 编码、余弦相似度、Top-K 和阈值逻辑。
- 默认模型可用性：在上述有缓存的评测环境中，清除 `EMBEDDING_MODEL` 环境变量后，脚本仍使用默认 BGE 完成 15/15 次 embedding 检索；这不代表任意新机器都预装了模型。
- 索引迁移防护：模型输出维度与已有 Milvus Collection 不一致时，新增 `CONFIG_CONFLICT` 和结构化维度/模型/建议 Collection 元数据；系统不自动删除旧索引。
- 阈值 0.50：Recall@5 100%、MRR 0.892、来源命中率 100%、无答案拒绝率 33.3%，查询平均/P95 延迟约 5.81/7.02 ms。
- 阈值 0.56：Recall@5 91.7%、MRR 0.875、来源命中率 100%、无答案拒绝率 100%。
- 结论：提高阈值能减少知识库外问题的误召回，但会漏掉低分的正确片段；当前数据更适合证明评测方法和参数取舍，不能外推为生产质量。
- 边界：延迟只统计模型已加载、知识已入库后的单次本地查询，不包含模型冷启动、文档编码、Milvus 网络和生成回答耗时；15 条人工样本仍需继续扩充。
