## 1. 项目概述

为健身智能助手 Agent 开发微信小程序前端，复刻并增强现有 Web UI (`/ui`) 的聊天交互体验。当前代码通过 WebSocket 与后端 FastAPI 进行 token 级流式对话，失败时降级到普通 HTTP；同时支持五种意图展示、历史管理和 Motion 图片/视频上传。开发者工具与真机尚未完成端到端验收。

### 1.1 与后端仓库关系

小程序代码放在同一仓库的 `miniprogram/` 目录下，与后端 `app/` 并列：

```
fitness-assistant/
├── app/                 # FastAPI 后端
├── miniprogram/         # 🆕 微信小程序前端
├── docs/                # 文档（含本设计规范）
├── data/                # 数据
└── tests/               # 测试
```

**互不影响：** 微信小程序运行在微信客户端内，只通过 HTTPS/WSS 请求后端 API，与后端代码在同一个 git 仓库中不产生任何运行时冲突。小程序开发工具仅关注 `miniprogram/` 目录下的文件。

## 2. 技术栈

| 组件 | 技术选型 | 说明 |
|------|----------|------|
| 框架 | 微信小程序原生框架 | 无需第三方框架，降低体积和复杂度 |
| 语言 | JavaScript (ES6+) + WXML + WXSS | 小程序标准技术栈 |
| 对话流式 | `wx.connectSocket` | 当前主路径连接 `/chat/ws`，接收 meta/token/done JSON 消息 |
| HTTP 与上传 | `wx.request` + `wx.uploadFile` | `/chat` 作为非流式降级，Motion 使用独立上传接口 |
| SSE 实验代码 | 自定义 `SseParser` | 无引用的实验文件已删除，不作为现行架构 |
| 本地存储 | `wx.setStorageSync` / `wx.getStorageSync` | 持久化 userId 和服务地址 |
| 样式方案 | WXSS (CSS 子集) + rpx 单位 | 响应式适配不同屏幕 |
| 状态管理 | `app.globalData` + Page `data` | 轻量级，无需引入 Redux/MobX |

### 2.1 为什么不用第三方框架

- **体积限制：** 微信小程序主包限制 2MB，总包限制 20MB。不使用框架可以最大化有效代码空间
- **功能匹配：** 当前小程序功能聚焦于聊天交互，原生框架完全满足需求
- **维护成本：** 原生框架的 API 稳定，不依赖第三方库的更新

## 5. 组件设计

### 5.1 message-bubble — 消息气泡

**功能：** 渲染单条聊天消息，区分用户消息与助手消息，支持流式打字机效果。

**属性：**

| 属性 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| role | String | `"user"` | `"user"` \| `"assistant"` |
| intent | String | `""` | 意图标签，仅 assistant 显示 |
| content | String | `""` | 消息文本内容 |
| isStreaming | Boolean | `false` | 是否处于流式渲染中（显示光标动画） |
| timestamp | Number | `0` | 消息时间戳 |

**视觉状态：**
- **用户消息：** 蓝色背景，右对齐，无意图标签
- **助手消息（已完成）：** 深色背景，左对齐，顶部显示意图标签
- **助手消息（流式中）：** 末尾显示闪烁光标 `▊`，内容实时增长
- **错误消息：** 红色边框，显示错误信息

**内部方法：**
- `stripThinkTags(content)` — 过滤 Qwen3 模型的 `<think>...</think>` 标签
- `formatContent(content)` — 处理换行和特殊字符

### 5.2 intent-badge — 意图标签

**功能：** 显示当前对话的意图分类，高亮活跃意图。

**属性：**

| 属性 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| intent | String | `"chat"` | 意图标识 |
| active | Boolean | `false` | 是否高亮 |
| label | String | `""` | 显示文本（不传则自动映射） |
| size | String | `"small"` | `"small"` \| `"large"` |

**意图映射表：**

| intent | 标签 | 颜色 | 图标 |
|--------|------|------|------|
| chat | 知识问答 | #5eead4 | 💬 |
| search | 联网搜索 | #93c5fd | 🔍 |
| motion | 动作分析 | #c4b5fd | 🏃 |
| diet | 饮食推荐 | #fde68a | 🥗 |

## 6. 数据流设计

### 6.1 用户 ID 管理

```
App 启动
  │
  ├── 检查 wx.getStorageSync('user_id')
  │     ├── 存在 → 使用已有 userId
  │     └── 不存在 → 生成新 userId，写入 Storage
  │
  └── 健康检查 GET /health → 更新连接状态
```

userId 格式: `wx_user_<timestamp>_<random6>` (如 `wx_user_1623456789_a3f2e1`)

### 6.2 对话流程

```
用户输入消息
  │
  ├── 1. 本地立即渲染用户消息气泡
  ├── 2. 创建空的助手消息气泡 (isStreaming=true)
  ├── 3. wx.connectSocket 连接 /chat/ws
  │     ├── 收到 type=meta → 更新 intent / sources / warnings / execution
  │     ├── 收到 type=token → 追加 text → setData 更新气泡
  │     ├── 收到 type=done → isStreaming=false → 关闭连接
  │     └── 建连或执行失败 → 降级 POST /chat
  └── 4. 滚动到最新消息
```

### 6.3 非流式降级

当 WebSocket 不受当前基础库支持，或者连接/执行失败时，页面降级为普通 POST `/chat`：

```
POST /chat (非流式)
  │
  ├── 1. 发送请求（含 loading 状态）
  ├── 2. 等待完整响应
  ├── 3. 一次性渲染完整回复
  └── 4. 更新 intent 标签
```

### 6.4 数据流图

```
┌──────────────┐      setData      ┌──────────────────┐
│   WXML View  │ ◄── setData ─── │   Page/Component │
│  (消息列表)   │                  │   (chat.js)      │
└──────────────┘                  └──────┬───────────┘
                                         │
                              ┌──────────┴───────────┐
                              │    utils/api.js       │
                              │  ┌─────────────────┐  │
                              │  │ wsChat()          │  │  WebSocket /chat/ws
                              │  │ chat()            │  │  POST /chat fallback
                              │  │ analyzeMotion*()  │  │  wx.uploadFile
                              │  │ getHistory()      │  │  GET /chat/{user_id}/history
                              │  │ clearHistory()    │  │  DELETE /chat/{user_id}/history
                              │  └─────────────────┘  │
                              └──────────────────────┘
```

## 7. 当前 WebSocket 协议与 SSE 历史代码

### 7.1 当前主协议

当前小程序使用 `wx.connectSocket` 连接 `/chat/ws`。一次连接处理一条消息，服务端按 JSON 消息返回：

```text
client -> {"user_id":"...","message":"..."}
server -> {"type":"meta","intent":"...","sources":[],"warnings":[],"execution":[]}
server -> {"type":"token","text":"..."}
server -> {"type":"done"}
```

### 7.2 失败降级

`chat.js` 在不支持 WebSocket、建连失败或流式执行失败时调用 `/chat`，一次性渲染完整回复。这个降级提高可用性，但当前尚未经过真机弱网验收，不能宣称生产级自动重连已经完成。

### 7.3 SSE 历史方案的当前地位

早期 `wx.request + enableChunked` 方案对应的无引用 `utils/sse-parser.js` 已删除。除非后续重新设计 SSE 小程序适配，否则面试中应讲 WebSocket 主路径，而不是把两个方案都说成当前运行链路；早期设计过程仍可在归档性质的《实施计划》中查看。

## 10. 与后端接口的适配

### 10.1 请求域名配置

小程序要求所有网络请求的目标域名必须在「小程序管理后台 → 开发 → 开发设置 → 服务器域名」中配置：

| 类型 | 开发环境 | 生产环境 |
|------|----------|----------|
| request 合法域名 | `http://127.0.0.1:8000` (开发工具不校验) | `https://your-domain.com` |
| socket 合法域名 | 开发工具可临时不校验 | `wss://your-domain.com` |

**开发阶段：** 微信开发者工具中勾选「不校验合法域名」可绕过此限制。

### 10.2 WebSocket 消息格式对接

后端 `/chat/ws` 返回格式回顾：

```
{"type":"meta","intent":"diet","sources":[],"warnings":[],"execution":[]}
{"type":"token","text":"减脂"}
{"type":"token","text":"期间"}
{"type":"done"}
```

小程序需要处理 `meta`、`token`、`done` 和 `error` 四种消息；`meta` 同时承载 intent、来源、warning 和实际执行模式。后端当前一次连接只处理一条用户消息，完成后关闭。

### 10.3 响应模型映射

| 后端字段 | 小程序用途 |
|----------|-----------|
| `ChatResponse.user_id` | 回显验证（不展示） |
| `ChatResponse.intent` | 更新 intent-badge 高亮状态 |
| `ChatResponse.reply` | 消息气泡的 content |
| `ChatResponse.sources` | 显示 Search URL 或 RAG 来源标识 |
| `ChatResponse.warnings` | 展示非致命降级和边界提示 |
| `ChatResponse.execution` | 区分真实路径、mock 与 fallback |
| `HistoryResponse.history` | history 页的消息列表 |
| `HistoryResponse.user_id` | 回显验证 |

## 11. 特殊处理

本节记录需要考虑的客户端优化。其中 Think 标签过滤、token 节流和长列表虚拟化目前没有接入当前 `chat.js` 主链，不能在面试中说成已经完成。

### 11.1 Think 标签过滤

Qwen3 模型如果输出 `<think>...</think>` 内部推理文本，应该优先由模型配置或后端输出层抑制/过滤，客户端可以再做防御性清洗。当前小程序没有显式接入下面的过滤函数，因此真实模型演示前必须单独验证，不能把这段设计代码说成现状：

```javascript
function stripThinkTags(text) {
  // 方法1: 正则移除完整的 think 块
  return text.replace(/<think>[\s\S]*?<\/think>/g, '');
  // 方法2: 流式场景下，实时去除未闭合的 <think> 起始标签
  // 在流式渲染中，维护一个状态标记 inThinkBlock
}
```

### 11.2 流式渲染性能

微信小程序的 `setData` 调用有频率限制。当前页面按 token 增量更新，下面是高频输出时的后续节流方案：

```javascript
// 节流策略: 每 50ms 最多更新一次 UI
const THROTTLE_MS = 50;
let pendingContent = '';
let throttleTimer = null;

function onToken(token) {
  pendingContent += token;
  if (!throttleTimer) {
    throttleTimer = setTimeout(() => {
      this.setData({ content: pendingContent });
      pendingContent = '';
      throttleTimer = null;
    }, THROTTLE_MS);
  }
}
```

### 11.3 消息列表性能

长对话场景（>50条消息）可以使用虚拟列表或限制渲染条数。当前页面尚未实现下面的虚拟化方案：

```xml
<scroll-view scroll-y enhanced show-scrollbar="{{false}}" 
             scroll-into-view="{{lastMsgId}}">
  <view wx:for="{{visibleMessages}}" wx:key="id">
    <message-bubble ... />
  </view>
</scroll-view>
```

目标方案只渲染最近 N 条消息并用 `scroll-into-view` 自动滚动；具体 N 值应通过真机性能验收决定，不把示例中的 100 当成当前参数。
