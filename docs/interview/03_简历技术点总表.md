# 简历技术点总表

这篇文档负责解释简历上的技术关键词。它不是技术词典，而是告诉你：这个词为什么会出现在项目里，面试时应该怎样解释。

每个核心技术都使用同一结构：

1. 用普通话解释它是什么。
2. 说明它解决什么问题。
3. 说明项目怎样使用。
4. 列出替代方案。
5. 给出面试表达。

## 1. 先建立三层技术地图

| 层次 | 核心问题 | 代表技术 |
|---|---|---|
| 控制层 | 用户的问题应该进入哪条流程 | 智能体、LangGraph、StateGraph、Router、子工作流 |
| 能力层 | 具体任务由谁完成 | RAG、Milvus、Tavily、Motion、ToolRegistry、MCP |
| 保障层 | 怎样管理上下文、失败和质量 | Memory、PromptBuilder、ToolResult、fallback、测试与评测、Docker |

面试时先讲三层结构，再展开具体技术。不要把技术栈念成一串互不相关的名词。

## 2. 大模型与智能体

### 它是什么

大语言模型擅长理解和生成自然语言。智能体是在大模型外增加任务判断、状态、工具和执行流程的系统。

生活化类比：大模型像一位擅长沟通的员工，智能体像一家公司。员工会说话，公司还要有经理、流程、资料室、工具和质量检查。

### 解决什么问题

普通大模型调用通常是“输入一句话，输出一段话”。健身助手需要查资料、搜互联网、分析动作和调用外部工具，仅靠一次生成无法稳定完成。

### 项目怎样使用

大模型负责：

- 理解用户语言。
- 辅助判断意图。
- 改写搜索词。
- 整理检索和工具结果。
- 把数值指标解释成自然建议。

大模型不负责：

- 凭空代替向量检索。
- 精确计算动作序列相似度。
- 不受限制地执行外部命令。
- 决定所有确定性业务规则。

### 替代方案

- 单 Prompt 应用：简单，但流程和工具难控制。
- 传统规则系统：稳定，但语言理解和生成能力弱。
- 完全自主智能体：更自由，但延迟、成本和风险更高。

### 面试表达

> 我把大模型当成语言理解和结果表达层，而不是让它包办所有任务。智能体的价值在于根据问题选择知识、搜索、算法和工具，并管理执行状态。

## 3. LangGraph、StateGraph 与 Router

### 它们分别是什么

- **LangGraph**：用图结构组织智能体流程的框架。
- **StateGraph**：LangGraph 中围绕共享状态运行的状态图。
- **Router**：读取用户问题并决定下一条执行路径的路由器。
- **子工作流**：专门完成某一类任务的小流程。

生活化类比：LangGraph 是公司流程图，StateGraph 是在部门之间流转的任务单，Router 是分配任务的值班经理，子工作流是不同专业部门。

### 解决什么问题

项目有知识、搜索、动作、饮食和外部工具等多条流程。如果全部写进一个 Prompt 或大量散乱的条件判断，难以解释、测试和处理失败。

### 项目怎样使用

```text
用户问题
  -> Router 计算意图和置信度
  -> 生成单任务或白名单组合计划
  -> LangGraph 把共享状态交给对应子工作流
  -> 子工作流写回结果、来源、警告和执行轨迹
  -> 合成最终回答
```

Router 使用确定性规则、语义样例和可选本地模型。明确问题优先使用稳定策略，低置信问题才考虑模型；复合任务只执行经过设计的组合。

### 替代方案

- 手写条件判断：轻量，但流程增多后难维护。
- 纯大模型路由：语义能力强，但延迟和稳定性更难控制。
- embedding 分类：适合语义匹配，但需要样本和阈值。
- AutoGen、CrewAI：更偏多智能体协作，本项目更需要受控工作流。

### 面试表达

> LangGraph 管整个业务流程，Router 负责选择路径，StateGraph 保存执行过程中共享的信息。三者共同解决“任务怎样受控流转”，但不负责具体工具内部怎样执行。

## 4. RAG、向量表示与 Milvus

### 它们分别是什么

- **RAG**：先检索资料，再让大模型基于资料回答。
- **embedding**：把文本转换成能表示语义的数字向量。
- **Sentence-Transformers**：生成文本向量的模型工具。
- **Milvus**：存储向量并进行相似度搜索的向量数据库。

生活化类比：RAG 是“先查资料再回答”的工作方式；Milvus 是一个能按意思查找资料，而不只按原词查找的档案系统。

### 解决什么问题

大模型可能记错、知识过时或产生幻觉。健身和营养问题需要更稳定的领域依据，因此回答前先检索本地知识。

### 项目怎样使用

```text
知识文档
  -> 按句子边界切成资料片段
  -> 转换为向量
  -> 写入 Milvus，并保存内容和来源

用户问题
  -> 转换为查询向量
  -> 找出语义最相近的资料
  -> 经过阈值过滤、去重和来源编号
  -> 注入 Prompt
  -> 大模型基于资料回答
```

项目使用余弦相似度衡量语义方向，Milvus 侧使用 IVF_FLAT 索引，并通过 Top-K、`nlist` 和 `nprobe` 调节召回与速度。

### 替代方案

- BM25 关键词检索：关键词准确，但同义表达能力较弱。
- FAISS：本地向量检索方便，但服务化和元数据治理较弱。
- Qdrant、Chroma、Elasticsearch：都可以作为向量存储方案。
- 直接把资料放进 Prompt：资料少时可用，规模增大后不可持续。

### 面试表达

> RAG 解决的是回答依据问题，Milvus 解决的是向量怎样持久化和高效检索。接入向量库并不等于效果一定好，还要用 Recall@K、MRR、回答忠实度和来源正确性评测。

## 5. Tavily 联网搜索

### 它是什么

Tavily 是面向大模型应用的联网搜索服务，可以返回标题、摘要和网址等结构化结果。

生活化类比：本地 RAG 是公司内部资料室，Tavily 是外部信息调查员。

### 解决什么问题

本地知识库适合相对稳定的知识，但无法及时覆盖最新研究、新闻和变化中的信息。

### 项目怎样使用

```text
用户原始问题
  -> 查询理解与改写
  -> Tavily 搜索
  -> 整理标题、摘要和网址
  -> 基于来源合成回答
```

如果没有配置密钥或外部调用失败，系统可以进入演示降级，并在执行轨迹中说明结果来源。

### 替代方案

- Bing Search、SerpAPI：通用搜索服务。
- 直接爬取网页：灵活，但解析、安全和维护成本更高。
- 只使用本地 RAG：更稳定，但缺少时效性。

### 面试表达

> RAG 和联网搜索不是重复能力。RAG 负责稳定领域知识，Tavily 负责最新外部信息。Search 子工作流还负责查询改写、来源整理和失败降级。

## 6. 动作分析、姿态序列与 FastDTW

### 它们分别是什么

- **MediaPipe Pose**：从图片或视频帧中估计人体关键点。
- **PoseSequence**：项目内部统一的人体姿态序列结构。
- **FastDTW**：比较两段速度不同、长度不同的时间序列。
- **NumPy**：处理关键点数组和相似度计算的数值工具。

生活化类比：MediaPipe 像把视频翻译成骨骼坐标的记录员；FastDTW 像允许两个人动作快慢不同，再对齐比较动作过程的裁判。

### 解决什么问题

动作标准性涉及人体轨迹和时间变化。大模型擅长解释，但不应该凭语言直觉计算两段动作是否相似。

### 项目怎样使用

```text
图片 / 视频 / 数值文件
  -> 提取人体关键点
  -> 统一成 PoseSequence
  -> 中心和尺度归一化
  -> 与兼容的标准动作进行时间对齐
  -> 计算 DTW、余弦相似度和逐关节平均距离
  -> 输出结构化指标和建议
```

PoseSequence 保存关键点、帧率、来源类型、姿态模型、关节结构和置信度，使下游算法不必绑定某一种输入格式。

### 替代方案

- 只让多模态大模型看视频：表达能力强，但时序精度和可解释指标不足。
- 固定关节角规则：稳定，但难覆盖速度和个体差异。
- 训练动作分类或评分模型：效果上限高，但需要大量标注样本。
- 标准 DTW、Soft-DTW：可以替代 FastDTW，但性能和训练方式不同。

### 面试表达

> Motion 模块最能体现大模型和确定性算法协作：算法负责姿态提取、归一化和相似度计算，大模型负责理解需求和解释指标。当前相似度表示与参考样本的统计接近程度，专业级纠错仍需要更多标准样本、动作周期和关节级阈值。

## 7. 工具系统、工具注册表与 MCP

### 它们分别是什么

- **工具**：搜索、检索、动作计算等可以被系统调用的确定性能力。
- **ToolResult / ErrorCode**：统一描述工具成功、失败和错误类型的返回结构。
- **ToolRegistry**：集中登记工具说明、参数、权限、执行器和失败策略。
- **MCP**：模型应用连接外部工具的一种标准协议。
- **JSON-RPC**：MCP 通信使用的请求与响应格式之一。

生活化类比：ToolRegistry 是公司内部工具台账；MCP 是对外统一插头；ToolResult 是每次使用工具后填写的执行报告。

### 解决什么问题

工具数量增加后，会出现参数错误、权限、失败重试、降级、审计和结果格式不一致等问题。直接调用函数无法完整表达这些治理要求。

### 项目怎样使用

默认工具注册表登记知识检索、Tavily 搜索、动作比较和 MCP 调用等代表工具。执行时依次进行：

```text
查找工具说明
  -> 校验参数
  -> 检查权限
  -> 调用执行器
  -> 有限重试或 fallback
  -> 返回统一 ToolResult
  -> 记录执行编号、耗时和降级来源
```

MCP Client 通过子进程和标准输入输出发送 JSON-RPC，覆盖初始化、工具发现和工具调用主链路。

### 替代方案

- 直接调用 Python 函数：简单，但治理信息分散。
- LangChain Tools：生态成熟，但仍要设计项目自己的执行边界。
- OpenAI Function Calling：适合模型选择工具，但不等于完整工具运行治理。
- 直接 HTTP API：适合单个外部服务，但每种工具需要单独适配。

### 面试表达

> LangGraph、ToolRegistry 和 MCP 解决三个不同问题：LangGraph 管业务流程，ToolRegistry 管内部工具执行，MCP 管外部工具协议。当前注册表是最小可运行治理层，不是让模型自由执行任意工具的生产平台。

## 8. Memory 与上下文工程

### 它们分别是什么

- **短期记忆**：保留最近若干轮对话。
- **长期记忆**：保存用户目标、偏好和约束。
- **Context Engineering**：决定本轮模型真正看到哪些信息。
- **PromptBuilder**：集中组织近期对话、长期记忆、检索证据和任务信息。
- **compact**：上下文过长时保留高价值摘要，减少无关内容。

生活化类比：短期记忆是当前会议记录，长期记忆是用户档案，PromptBuilder 是每次会议前准备材料的人。

### 解决什么问题

如果每轮都没有历史，系统无法连续对话；如果把所有历史无限放入 Prompt，又会出现长度、成本、噪声和隐私问题。

### 项目怎样使用

- 滑动窗口保存近期对话。
- SQLite 保存完整消息；旧消息超过字符阈值后生成有覆盖边界的 active 抽取式摘要，只处理新增消息并保留最近原文。
- SQLite 作为长期记忆事实存储。
- FTS/LIKE 提供基础检索，Milvus 可提供语义增强。
- 健康、伤病和过敏等敏感内容先进入候选确认。
- PromptBuilder 按预算注入近期对话、长期记忆和 RAG 证据。
- 超过阈值时执行确定性 compact，并记录 Prompt 元数据。

### 替代方案

- 全量对话历史：简单，但不可持续。
- 只做滑动窗口：可控，但容易丢失长期偏好。
- 只做向量记忆：语义召回方便，但写入和隐私治理更复杂。
- Redis、PostgreSQL：生产环境可承担共享会话和长期存储。

### 面试表达

> 记忆不是把全部聊天记录塞给模型，而是分层保存、按需召回、按预算注入。SQLite 保存完整会话，active 抽取式摘要承接较早内容，最近原文保留细节，长期记忆只保存跨会话仍有价值的信息。摘要失败时仍走滑动窗口；Milvus 只是可选语义增强，敏感信息先确认，避免模型自动写入健康隐私。

当前边界：会话摘要是确定性原文摘录，按字符阈值而非精确 token 触发；现有专项回归只覆盖 1 个摘要事实保持场景，因此不宣称模型级总结质量或生产级并发治理。

## 9. FastAPI、流式输出与 Docker

### FastAPI

FastAPI 是 Python Web 框架，负责把智能体能力提供为 HTTP、SSE 和 WebSocket 接口。Pydantic 用来校验请求和响应数据。

项目中同步的图执行和模型生成通过线程桥接进入异步服务，避免直接阻塞事件循环；模型仍使用进程内锁控制并发推理。

### 流式输出

- **HTTP**：一次返回完整结果。
- **SSE**：服务器通过 HTTP 持续向客户端推送文本事件。
- **WebSocket**：客户端和服务器保持双向连接。

三种方式解决不同客户端和交互体验问题，不代表模型本身发生变化。

### Docker

Docker 把依赖、启动命令和运行环境封装成容器。它体现可交付意识，但当前容器仍需要处理模型资产、动作可选依赖和跨机器验证。

### 面试表达

> FastAPI 负责服务接入，SSE 和 WebSocket 提供流式体验，Docker 负责环境交付。它们属于工程层，不是智能体决策层。

## 10. 测试与评测

### 两者有什么区别

- **自动化测试**：检查代码和契约有没有被改坏。
- **效果评测**：检查策略和回答质量是否达到目标。

生活化类比：测试检查汽车零件能不能正常工作；评测检查汽车开得是否安全、快速、舒适。

### 项目怎样使用

- pytest 检查接口、状态流、工具结果、错误处理和算法。
- Router 数据集检查主意图、次意图和路由计划。
- RAG 评测关注召回、排序、来源和回答忠实度。
- Memory 与 Context benchmark 检查记忆召回、Prompt 注入和压缩链路。
- Motion 后续需要标准动作和教练标签评测专业准确性。

### 面试表达

> 测试数字只能证明代码契约可回归，不能直接证明真实回答质量。策略效果要靠专项评测，泛化能力还需要真实用户数据和线上反馈。

## 11. 支撑性技术速查

这些词需要认识，但不必在项目开场主动展开。

| 技术 | 普通话解释 | 项目中的位置 |
|---|---|---|
| Python | 人工智能生态成熟的通用编程语言 | 后端、智能体、检索、算法和测试 |
| Pydantic | 按规则检查结构化数据 | 接口模型、饮食画像、参数校验 |
| Sentence-aware 分块 | 尽量沿句子边界切资料 | RAG 文档预处理 |
| COSINE | 比较两个向量方向是否相近 | 文本检索和姿态特征比较 |
| IVF_FLAT | 先把向量分桶再搜索附近桶 | Milvus 向量索引 |
| `.npz` | 适合保存 NumPy 数组的压缩文件 | 姿态序列和标准动作 |
| OpenCV | 视频读取和抽帧工具 | 动作视频输入 |
| `deque` | 两端高效进出的固定队列 | 滑动窗口短期记忆 |
| subprocess / stdio | 启动子进程并通过标准输入输出通信 | 本地 MCP Server |
| fallback | 主方案失败后使用备用方案 | Search、Retriever、MCP 等依赖 |
| audit log | 记录工具何时怎样被调用 | 工具注册表执行审计 |

## 13. 论文与了解项

| 技术 | 应该知道什么 | 不要混淆什么 |
|---|---|---|
| RT-DETR | 实时目标检测模型，可讨论轻量化和精度 | 它不是健身动作姿态模型 |
| mAP50 | 目标检测在指定 IoU 阈值下的精度指标 | 它与 Router 准确率不是同类指标 |
| 无人机航拍图像 | 论文和实习中的视觉数据来源 | 不要与 Motion 的人体姿态序列混为一谈 |
| Claude Code 等编程智能体 | 可以讨论工具调用和上下文管理思路 | 不要说成自己实现了这些完整产品 |

## 14. 最终归纳话术

> 我的技术栈可以分成三层。控制层用 LangGraph、StateGraph 和 Router 决定任务怎样流转；能力层用 RAG、Milvus、Tavily、动作算法和 MCP 解决大模型不能稳定完成的问题；保障层用 Memory、上下文管理、统一错误、测试评测和 Docker 保证系统可运行、可观察、可演进。项目最想证明的不是我调用了多少框架，而是我能根据问题性质选择合适的执行者，并把它们组织成一个受控系统。

## 16. 同一知识点怎样逐层回答

以“为什么使用 Milvus”为例：

- 基础级：它负责持久化向量和相似度检索，支持按语义寻找资料。
- 项目级：Retriever 抽象隔离上层；当前使用 `IVF_FLAT`、`COSINE`，通过 `nlist` 和 `nprobe` 平衡候选范围与延迟，并保留内存降级。
- 工程级：索引选择必须由数据规模和 Recall@K/延迟实验决定；小数据未必比 FLAT 更有优势，生产环境还要做版本化入库、影子索引、灰度切换和回滚。

每次追问都应增加一层证据、取舍或风险，而不是换一种说法重复定义。

## 17. 工程问题的固定分析顺序

```text
定义用户可感知的失败
  -> 定位失败发生在哪一层
  -> 用日志、指标和 trace 找证据
  -> 选择超时、重试、降级或拒绝
  -> 验证恢复结果
  -> 把真实失败样本加入回归集
```

例如 Tavily 超时，不能只说“加重试”。要判断是否可安全重试，限制次数和总时间预算；仍失败则返回明确降级状态，不能把 mock 或旧内容伪装成最新结果；最后记录工具错误率、p95 延迟、fallback 比例和任务成功率。
