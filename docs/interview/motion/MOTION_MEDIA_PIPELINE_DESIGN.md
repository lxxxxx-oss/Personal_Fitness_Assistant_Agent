# Motion 媒体输入技术路线与需求分析

本文档用于面试讲解 Motion 子图的下一阶段演进：从当前 `.npz` 姿态序列分析，升级到普通用户可理解的图片/视频动作分析。重点不是宣称已经完成图片/视频分析，而是把需求、技术路线、工程边界和可落地顺序讲清楚。

后续执行进度、实际做法和路线偏差统一维护在 [MOTION_OPTIMIZATION_ROADMAP.md](./MOTION_OPTIMIZATION_ROADMAP.md)。

## 1. 面试定位

Motion 子图是本项目最容易体现“不是纯 LLM 套壳”的模块。

当前代码已经实现的是后半段能力：

```text
姿态序列 .npz -> 姿态归一化 -> 时序对齐 -> 相似度指标 -> LLM 解释报告
```

真实用户更自然的输入是：

```text
图片 / 视频 -> 姿态估计 -> 统一姿态序列 -> 现有 Motion 分析工具
```

所以后续技术路线的核心是补齐前半段 `media-to-pose` 适配层，而不是从零训练一个人体姿态估计模型。

面试时可以这样讲：

> Motion 当前已经完成“姿态数据进入系统之后怎么分析”的部分，包括 `.npz` 上传、姿态归一化、FastDTW 对齐和相似度指标。普通用户不会直接上传 `.npz`，所以下一步不是推翻现有设计，而是在前面加一个媒体转姿态的适配层：图片或视频先通过成熟姿态估计模型提取关键点，再统一成 PoseSequence / `.npz`，复用已有分析链路。

### 1.1 完整流程理解

可以先用这条主线理解后续完整 Motion 流程：

```text
用户上传图片 / 视频
  -> 开源姿态估计模型提取人体关键点
  -> 转成统一 PoseSequence
  -> 必要时保存为 .npz
  -> 进入现有 Motion 姿态分析工具
  -> 计算关节角度 / 动作轨迹 / 与标准动作库的相似度
  -> LLM 把指标解释成专业动作指导
```

这里有两个需要面试时讲严谨的点：

1. `.npz` 不一定是线上每次分析都必须生成的文件。
   - `.npz` 更适合作为内部持久化、调试、评测和标准动作库格式。
   - 线上接口也可以直接把 `PoseSequence.keypoints` 的 ndarray 传给分析函数。

2. “动作是否标准”不是直接硬比关键点坐标。
   - 关键点会受到身高、拍摄距离、站位和镜头角度影响。
   - 因此需要先做姿态归一化，再计算关节角度、身体姿态、轨迹、节奏和相似度。
   - 视频场景下还要用 DTW 对齐动作节奏，避免因为用户做得快或慢就误判。

更适合面试的表述是：

> 后续完整链路是：用户上传图片或视频，系统先用 MediaPipe Pose 这类开源姿态估计模型提取人体关键点，再转换成统一的 PoseSequence。`.npz` 是内部持久化和标准动作库格式，不一定是用户直接接触的输入。Motion 分析层会对关键点做归一化，计算关节角度、动作轨迹，并在视频场景下和标准动作库做 DTW 对齐与相似度评估，最后由 LLM 把这些指标解释成用户能理解的动作指导。

## 2. 当前真实状态

### 已实现

- Motion 子图：`think -> parse -> tool -> check`。
- `/motion/analyze`：支持上传 `.npz` 姿态数据。
- `/motion/analyze-image`：支持上传 `.jpg`、`.jpeg`、`.png` 图片，提取单帧静态姿态摘要。
- `motion_tool.py`：支持加载 `(T, J, 3)` 姿态数组、姿态归一化、FastDTW、余弦相似度、形状差异和整体评价。
- 无数据时的降级回答：说明如何准备姿态数据，不让 LLM 凭空判断用户动作问题。

### 未实现

- 视频上传后的逐帧姿态估计。
- 标准动作库数据。
- 自定义训练的人体姿态估计模型。
- 小程序端图片/视频上传到 Motion 分析接口的端到端联调。

### 不应夸大的说法

不要说：

> 系统已经支持用户上传视频自动分析深蹲。

更准确的说法：

> 当前系统已经实现了姿态序列分析能力和图片单帧姿态提取能力，视频到姿态序列的适配层是下一阶段最明确的工程演进方向。

## 3. 需求分析

### 用户侧需求

普通用户不会理解 `.npz`，他们更可能提出以下需求：

- 上传一张深蹲底部照片，问膝盖、腰背、髋部位置是否明显异常。
- 上传一段深蹲、硬拉、卧推视频，想知道动作节奏、幅度、轨迹和标准动作差异。
- 不知道自己是什么动作，只希望系统先识别动作类型，再给出分析建议。
- 希望得到可解释反馈，而不是只有一个分数。

因此 Motion 的产品输入应该分三层：

| 输入类型 | 适合分析什么 | 不适合分析什么 |
|---|---|---|
| 单张图片 | 静态姿态、关节位置、明显体态问题 | 动作节奏、轨迹、完整发力过程 |
| 短视频 | 动作周期、幅度、节奏、轨迹、与标准动作对齐 | 复杂遮挡、多人场景、医疗诊断 |
| `.npz` | 算法测试、标准动作库、离线评测 | 普通用户直接使用 |

### 系统侧需求

- 将用户媒体文件转换为统一姿态数据结构。
- 保留姿态估计模型、关键点格式、帧率、置信度等元信息。
- 对低置信度、关键点缺失、多人入镜、画面遮挡等情况给出可解释失败原因。
- 保持现有 Motion 工具链稳定，优先通过适配层接入，而不是重写分析算法。
- 将输出区分为“算法指标”和“LLM 解释”，避免让面试官误以为是大模型主观猜测。

## 4. 技术路线

### 4.1 不从零训练模型

当前没有 `.npz` 数据，也没有训练模型，这是合理的原型阶段边界。

工程上更主流、也更适合面试解释的做法是复用成熟姿态估计模型：

- 第一阶段：MediaPipe Pose。本地运行、接入简单、适合快速做图片和短视频关键点提取。
- 后续增强：MoveNet、RTMPose、OpenPose 或 YOLO-Pose。用于更高精度、更复杂场景或服务端部署。

面试回答：

> 我不会在个人项目里从零训练人体姿态估计模型，因为那需要大量标注数据和算力。更工程化的路线是把姿态估计作为可替换模块，先用 MediaPipe Pose 快速打通 image/video-to-pose，再把输出转成统一 PoseSequence，后续需要精度时再替换成 MoveNet、RTMPose 或 OpenPose。

### 4.2 统一中间格式

当前已新增统一的 `PoseSequence` 概念，`.npz` 是它的落盘格式之一。

```text
PoseSequence
  keypoints: float array, shape = (T, J, C)
  fps: number
  source_type: image | video | npz
  pose_model: mediapipe_pose | movenet | rtmpose | openpose
  joint_schema: mediapipe_33 | coco_17 | custom
  confidence / visibility: optional array
  metadata: source filename, frame count, width, height
```

当前代码位置：

```text
app/tools/pose_sequence.py
app/tools/pose_estimator.py
```

说明：

- `PoseSequence` 允许 `C >= 2`，方便后续保存 x/y、x/y/z 或带 visibility 的关键点。
- 当前 Motion 分析算法仍使用前三维坐标，所以 `.npz` 进入 `load_npz_pose()` 后会转换为 `(T, J, 3)`。
- `.npz` 兼容 `keypoints`、`pose`、`positions` 三种 key；推荐新数据统一使用 `keypoints`。
- `pose_estimator.py` 已提供图片数组到 `PoseSequence` 的适配器骨架，MediaPipe 为可选依赖；当前尚未开放图片/视频上传 API。

与当前实现的关系：

```text
图片 / 视频
  -> pose_estimator 提取关键点
  -> PoseSequence
  -> 保存为内部 .npz 或直接传 ndarray
  -> motion_tool.compute_similarity()
```

这样可以解释清楚：`.npz` 不是让用户手动准备的最终产品输入，而是系统内部可测试、可缓存、可评测的姿态序列载体。

### 4.3 图片分析链路

图片只包含单帧，所以适合做静态姿态分析。

```text
POST /motion/analyze-image
  -> 校验图片类型和大小
  -> MediaPipe Pose 提取 33 个关键点
  -> 关键点置信度检查
  -> 转成 PoseSequence(T=1)
  -> 返回静态姿态摘要和边界提醒
```

图片阶段优先做：

- 深蹲底部姿态。
- 硬拉起始姿态。
- 平板支撑体线。
- 俯卧撑底部/顶部姿态。

图片阶段不应该承诺：

- 动作速度。
- 完整动作轨迹。
- 发力顺序。
- 多周期动作稳定性。

当前实现边界：

- `/motion/analyze-image` 已开放图片上传入口。
- 当前返回 PoseSequence 摘要、关键点置信度和静态分析 warning。
- 尚未实现动作专项关节角度规则，具体深蹲/硬拉/卧推动作质量规则放到后续 Step 6。

### 4.4 视频分析链路

视频可以生成多帧姿态序列，适合复用现有 `(T, J, 3)` 分析算法。

```text
POST /motion/analyze-media
  -> 校验视频类型、时长、大小
  -> 抽帧或按 fps 采样
  -> 每帧姿态估计
  -> 关键点平滑和缺失帧处理
  -> PoseSequence(T=N)
  -> 与标准动作库 PoseSequence 对齐
  -> FastDTW + 多指标相似度
  -> LLM 输出动作报告
```

视频阶段的关键工程点：

- 帧采样：避免每一帧都跑模型导致延迟过高。
- 关键点平滑：减少姿态估计抖动。
- 动作片段截取：从完整视频中找到一次动作周期。
- 置信度过滤：关键关节缺失时不给过度结论。
- 参考动作库：标准动作也要由同一套姿态估计模型生成，避免关键点 schema 不一致。

## 5. 接口需求草案

当前已实现接口仍是：

```text
POST /motion/analyze
```

后续建议新增接口，而不是改变现有接口语义：

```text
POST /motion/analyze-media
```

请求字段草案：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `file` | file | 是 | 图片或短视频 |
| `media_type` | string | 否 | `image` / `video`，不传时由后缀或 MIME 推断 |
| `action_type` | string | 否 | 用户声明的动作类型，如 `squat`、`deadlift` |
| `reference_name` | string | 否 | 标准动作库中的参考动作名称 |

响应字段草案：

| 字段 | 说明 |
|---|---|
| `status` | `success` / `failed` |
| `source_type` | `image` / `video` |
| `pose_model` | 使用的姿态估计模型 |
| `frames_used` | 实际用于分析的帧数 |
| `joint_schema` | 关键点格式 |
| `confidence_summary` | 关键点置信度摘要 |
| `metrics` | 关节角度、DTW、余弦相似度、形状差异等 |
| `warnings` | 画面遮挡、置信度低、只能静态分析等提醒 |
| `answer` | 面向用户的中文解释 |

注意：这是后续接口设计，不是当前已实现 API。

## 6. 分阶段实现顺序

### Step 1：定义姿态序列格式

- 明确 `.npz` 内部 key，例如 `keypoints`、`fps`、`joint_schema`、`source_type`。
- 兼容当前 `motion_tool.load_npz_pose()` 只读取第一个 ndarray 的实现。
- 文档中写清楚 `(T, J, 3)` 是分析层要求，不是用户输入要求。

### Step 2：新增姿态估计适配器

建议新增：

```text
app/tools/pose_estimator.py
```

职责：

- 接收图片路径或视频帧。
- 调用 MediaPipe Pose。
- 输出统一 `PoseSequence`。
- 给出置信度和失败原因。

### Step 3：打通图片上传

先实现图片比视频更稳：

- 支持 `.jpg`、`.jpeg`、`.png`。
- 单帧转 `PoseSequence(T=1)`。
- 先输出静态姿态分析和关键角度，不强行做时序相似度。

### Step 4：打通视频上传

- 支持短视频上传。
- 控制文件大小和时长。
- 抽帧生成 `PoseSequence(T=N)`。
- 与标准动作库做 FastDTW 对齐。

### Step 5：建设标准动作库

标准动作库不需要手写 `.npz`：

```text
标准动作视频 -> 同一 pose_estimator -> 标准 PoseSequence / .npz
```

这样用户视频和标准视频来自同一套关键点 schema，比较才有意义。

### Step 6：增加评测集

- 图片姿态估计成功率。
- 视频关键点连续性。
- 标准动作对比稳定性。
- 低质量视频的拒答/降级准确性。

## 7. 风险和边界

- 单张图片只能分析静态姿态，不能判断完整动作质量。
- 视频分析受拍摄角度、遮挡、光照、衣着、多人体影响明显。
- 当前没有标准动作库，所以“和标准动作对比”的演示需要先补数据。
- 当前没有自训练模型，姿态估计质量依赖外部成熟模型。
- 输出是健身参考建议，不是医疗诊断，也不能替代专业教练现场指导。

## 8. 面试问答准备

### Q1：用户不会上传 `.npz`，这个设计是不是不实用？

答：

> `.npz` 在当前阶段是内部中间格式，不是最终用户交互方式。真实用户会上传图片或视频，系统应该先通过姿态估计模型把媒体转成关键点序列，再复用已有的 Motion 分析算法。也就是说当前已经完成后半段分析链路，下一步补的是 media-to-pose 适配层。

### Q2：你没有数据也没有训练模型，怎么做动作分析？

答：

> 我不准备从零训练人体姿态估计模型。个人项目更合理的工程路线是复用 MediaPipe Pose 这类成熟模型先打通链路，把图片/视频转成关键点序列；标准动作库也可以用标准动作视频离线生成 `.npz`。训练自定义模型是后续精度优化，不是第一阶段必要条件。

### Q3：为什么这仍然是 Agent 亮点，而不是普通算法接口？

答：

> 单独的姿态算法只能给指标，Agent 的价值在于把用户自然语言、媒体输入、动作库检索、数值计算和解释报告串起来。LLM 不直接判断动作好坏，而是负责理解用户意图和解释工具结果；关键判断来自姿态估计和相似度算法，这能降低幻觉。

### Q4：如果面试官问现在能演示什么？

答：

> 当前可以演示 `.npz` 上传后的姿态数据加载和相似度分析链路，也可以演示没有数据时系统不会乱猜，而是提示如何准备姿态数据。图片/视频上传是下一阶段路线，文档已经拆成了 PoseSequence 格式、MediaPipe 适配器、媒体接口、标准动作库和评测集几个步骤。
