# 零基础项目地图：先真正理解智能体

这篇文档不要求你会写代码。它只解决一件事：让你先形成一幅完整的“项目地图”，以后再看到 LangGraph、RAG、MCP 等技术名词时，知道它们为什么会出现在这里。

## 1. 这个项目到底为了什么

项目表面上是个人健身助手，真正目的则是展示你具备 Agent 岗位需要的核心思维：

> 面对一个复杂需求，能够判断任务类型，选择合适的知识、工具或算法，组织执行过程，处理失败，并用评测证明方案有效。

健身场景很适合展示这种能力，因为用户的需求天然不同：

- “深蹲时膝盖应该朝哪里？”需要专业知识。
- “帮我查一下最近的蛋白质摄入研究。”需要联网搜索。
- “看看我这段深蹲视频哪里不标准。”需要视觉和动作算法。
- “我减脂期晚餐吃什么？”需要用户信息、营养知识和推荐逻辑。
- “番茄炒蛋怎么做？”可以调用外部菜谱工具。

这些问题虽然都和健身有关，但不能用同一种办法解决。项目的核心亮点，就是让系统先判断问题，再把任务交给合适的能力。

## 2. Agent 是什么

可以把普通聊天机器人想象成一个只会坐在前台回答问题的人。无论用户问什么，它都尝试凭自己的记忆回答。

Agent 更像一个懂得分工的项目经理：

1. 先听懂用户想做什么。
2. 判断应该查内部资料、搜索互联网、调用工具，还是执行计算。
3. 把任务交给合适的执行者。
4. 收集执行结果。
5. 最后用自然语言回答用户。

所以，Agent 不只是“大模型回答得更聪明”，而是“大模型参与了一个有流程、有工具、有状态的系统”。

## 3. 用一家公司理解整个项目

把系统想象成一家小型健身服务公司：

| 项目组件 | 公司里的角色 | 它负责什么 |
|---|---|---|
| FastAPI | 前台接待 | 接收网站、小程序或其他客户端的请求 |
| Router | 值班经理 | 判断用户需求，决定把任务交给谁 |
| LangGraph | 工作流程图 | 规定任务先做什么、后做什么、失败后走哪里 |
| LLM / Qwen | 沟通与分析人员 | 理解语言、整理信息、生成自然回答 |
| Knowledge / RAG | 资料室管理员 | 从健身和营养资料中寻找相关证据 |
| Milvus | 资料室的语义检索系统 | 即使用词不同，也能找到意思相近的资料 |
| Tavily Search | 外部信息调查员 | 查找本地资料里没有的最新信息 |
| Motion | 动作分析教练 | 把图片或视频变成人体关键点并计算动作差异 |
| ToolRegistry | 工具管理员 | 记录工具能做什么、需要什么参数、失败怎么办 |
| MCP Client | 对外工具翻译员 | 按统一协议和外部工具沟通 |
| Memory | 用户档案与会议记录 | 保存近期对话、摘要和长期偏好 |
| 测试与评测 | 质量检查员 | 判断流程有没有坏、策略是否真的有效 |

这张表是整个项目最重要的理解框架。技术名词可以暂时记不住，但必须知道每个角色为什么存在。

## 4. 一条请求是怎样完成的

假设用户说：

> 帮我查一下最近关于高蛋白饮食的研究，再给我一些减脂建议。

系统的思路可以拆成六步：

```text
第一步：前台收到问题
  -> 第二步：Router 识别出“最新搜索 + 饮食建议”两个意图
  -> 第三步：Search 查找最新资料
  -> 第四步：Knowledge 结合营养知识和用户信息生成建议
  -> 第五步：系统合并两个结果，并保留来源和执行信息
  -> 第六步：LLM 把结果整理成自然语言回答
```

这里最重要的不是具体代码，而是三个 Agent 思维：

- **任务拆分**：一句话里可能有多个目标。
- **能力选择**：最新研究交给搜索，饮食建议交给知识链路。
- **受控组合**：不是让模型随意执行任意步骤，而是只允许经过设计的组合。

## 5. 七条最值得面试展示的技术亮点

### 亮点一：用 Router 做任务分流

Router 就像分诊台。它判断问题属于知识问答、搜索、动作分析、饮食建议还是外部工具调用。

为什么这是亮点：真实 Agent 面对的不是单一问题。能否稳定地选择正确流程，往往比“模型能不能多说几句话”更重要。

项目的表达重点：Router 以明确规则和字符/词面样例为默认路径，低置信时预留可选 embedding 与本地模型判断；常见问题优先走可解释、可测试的路线，复杂组合只开放白名单。

### 亮点二：用 LangGraph 控制执行流程

LangGraph 可以理解成一张“带条件的流程图”。每个方框是一个处理步骤，每条线表示下一步可能去哪里。

为什么这是亮点：如果所有逻辑都写进一个巨大 Prompt，出错时很难知道问题发生在哪里。拆成节点后，可以分别测试路由、搜索、工具和结果合成。

项目的表达重点：LangGraph 管的是业务流程和状态，不负责每个具体工具内部怎么执行。

### 亮点三：用 RAG 给回答补充可靠知识

RAG 的简单理解是“先查资料，再回答”。Milvus 则帮助系统按照语义相似度查资料。

为什么这是亮点：大模型可能记错、过时或编造。把检索到的健身资料连同来源交给模型，可以减少完全凭记忆回答的问题。

项目的表达重点：资料经过切分和向量化后写入 Milvus；提问也转成向量，再找最相近的资料片段，最后注入 Prompt。

### 亮点四：把最新信息交给联网搜索

本地知识库适合稳定知识，联网搜索适合最新研究、新闻和实时资料。两者解决的问题不同。

为什么这是亮点：它体现了“根据知识时效性选择数据源”，而不是认为一个工具能解决所有问题。

项目的表达重点：Search 先改写搜索词，再调用 Tavily，最后根据标题、摘要和链接合成回答；失败时可以降级。

### 亮点五：动作分析交给确定性算法

LLM 擅长语言，但不应该凭感觉计算两段动作轨迹是否相似。Motion 链路把人体动作转换为关键点序列，再用 FastDTW 等算法比较。

为什么这是亮点：它展示了 Agent 的本质不是让 LLM 包办一切，而是让 LLM 与专业算法协作。

项目的表达重点：图片或视频经过 MediaPipe 形成 PoseSequence；标准动作和用户动作经过归一化、时序对齐与相似度计算，结构化指标在对话链路中还可以交给 LLM 解释成用户容易理解的建议。

### 亮点六：用 ToolRegistry 和 MCP 管理工具

ToolRegistry 像内部工具台账，记录工具名称、用途、参数、权限和失败策略。MCP 像一种对外通用插头，让不同外部工具用相对统一的方式接入。

为什么这是亮点：Agent 会调用工具后，安全、参数错误、超时、重试和审计都会成为工程问题，不能只停留在“调用一个函数”。

项目的表达重点：LangGraph 决定何时进入某个业务流程，ToolRegistry 管具体工具怎样被安全执行，MCP 负责一种外部工具协议；三者职责不同。

### 亮点七：Memory、Context 与评测形成闭环

Memory 让系统记住近期对话和用户长期偏好；Context Engineering 决定哪些信息应该放进本轮 Prompt；评测则检查这套组合有没有稳定工作。

为什么这是亮点：对话越长，并不是把全部历史无限塞给模型就越好。需要控制长度、保留重要信息，并谨慎处理健康、伤病和过敏等敏感内容。

项目的表达重点：近期原文使用滑动窗口，较早会话由抽取式 active summary 承接，长期记忆以 SQLite 为事实存储，Milvus 可以作为可选语义召回增强；本轮 Prompt 太长时再做临时 compact，敏感长期信息先进入候选确认。

## 6. 面试官真正想从这些亮点判断什么

| 面试官问的表面问题 | 他可能在判断什么 |
|---|---|
| 为什么用 LangGraph？ | 你是否理解工作流编排，而不是跟风选框架 |
| Router 怎么做？ | 你是否会任务拆分、处理歧义并建立评测集 |
| 为什么需要 RAG？ | 你是否理解模型幻觉、知识更新和检索质量 |
| MCP 和工具调用有什么区别？ | 你是否理解协议、工具治理与安全边界 |
| 动作分析准确吗？ | 你是否知道 LLM 和算法各自适合做什么 |
| Memory 为什么不全塞进 Prompt？ | 你是否理解上下文预算、隐私和信息价值 |
| 怎么证明项目有效？ | 你是否具备测试、评测和数据闭环意识 |

所以回答技术问题时，最终要落回“我如何分析问题并作出工程选择”，而不只是解释名词。

## 7. 三种说法必须分清

为了既突出项目又经得住追问，所有材料都区分三层：

| 层次 | 含义 | 面试中怎么用 |
|---|---|---|
| 核心主线 | 项目要展示的完整 Agent 设计 | 开场重点讲，突出价值和架构 |
| 当前证据 | 代码、接口、测试或演示能够支撑的部分 | 被问实现细节时用来证明可信度 |
| 生产化演进 | 真正上线还需要补的能力 | 展示工程视野，不在开场自我削弱 |

功能并不需要为了面试无限扩张，但核心说法必须能够解释。最有价值的代码不是数量最多的代码，而是能支撑 Router、RAG、工具治理、Memory 和评测等关键追问的证据。

## 8. 现在应该能回答的三个问题

读完后，先尝试不用技术名词回答：

### 这个项目为什么不是普通聊天机器人？

因为它不会把所有问题都交给模型直接回答。它会先判断任务，再选择知识库、联网搜索、动作算法或外部工具，最后组合结果。

### 这个项目最有价值的设计是什么？

把不同性质的能力放进一个受控流程：语言交给大模型，稳定知识交给 RAG，最新信息交给搜索，动作比较交给算法，外部能力交给工具系统。

### 这个项目对 Agent 岗位有什么证明作用？

它可以展示任务路由、工作流编排、工具调用、知识增强、记忆管理、失败降级和效果评测，而这些正是 Agent 工程岗位经常关注的能力。
