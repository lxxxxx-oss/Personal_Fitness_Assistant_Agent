# Agent 岗位面试学习中心

这个目录只服务一个目标：让你即使没有编程基础，也能理解“个人健身助手 Agent”项目，并在面试中把它讲得有价值、有逻辑、经得住追问。

## 先理解，不要先背

面试官真正判断的不是你能否背出 LangGraph、Milvus、MCP 等名词，而是：

- 你是否知道项目为什么需要 Agent。
- 你是否知道每项技术分别解决什么问题。
- 你能否解释自己的设计选择和替代方案。
- 面试官改变问法后，你是否仍然能回答。
- 你是否具备把原型升级成生产系统的思路。

因此，第一次学习必须从 [00_零基础项目地图.md](./00_零基础项目地图.md) 开始。

## 统一回答方法

任何问题都尽量按下面顺序回答：

```text
先给结论
  -> 说明遇到的业务问题
  -> 解释方案如何工作
  -> 说明为什么选择它
  -> 给出结果、边界或升级方向
```

例如，面试官问“为什么使用 LangGraph”，不要只回答“因为它支持 StateGraph”。更完整的逻辑是：

> 健身助手需要处理知识问答、最新信息搜索、动作分析和外部工具调用，这些任务的数据和执行方式不同。LangGraph 可以把它们画成明确的节点和连线，让 Router 决定下一步进入哪条流程。相比把所有逻辑写进一个 Prompt，它更容易控制、测试和定位失败；相比完全自主的多 Agent，它也更适合健康场景对稳定性的要求。

## 项目的技术主线

先用普通话理解，再记技术名称：

| 普通话 | 技术名称 | 面试价值 |
|---|---|---|
| 先判断用户到底要做什么 | Router + LangGraph | 展示任务拆分和流程编排 |
| 回答专业问题前先查资料 | RAG + Milvus | 展示知识增强和幻觉控制 |
| 本地资料过时时去网上找 | Tavily Search | 展示实时信息和来源约束 |
| 动作不能靠语言猜，要交给算法计算 | MediaPipe + PoseSequence + FastDTW | 展示 LLM 与确定性算法协作 |
| 外部工具要按统一协议接入 | ToolRegistry + MCP | 展示工具治理和协议意识 |
| 系统要记住对话和用户偏好 | Memory + Context Engineering | 展示长对话和个性化设计 |
| 失败不能让整个系统崩溃 | ToolResult + fallback + audit | 展示稳定性与可观测性 |
| 效果不能只靠主观感觉 | pytest + Router/RAG benchmark | 展示评测意识 |

## 面试表达的三个层次

同一项能力要准备三层说法：

1. **价值层**：它让用户或系统获得了什么价值，适合开场。
2. **设计层**：数据怎样流动、模块怎样协作，适合技术追问。
3. **防守层**：当前原型做到哪里、有什么限制、生产化怎样升级。

不要在开场主动用大量限制削弱项目，也不要把目标架构说成已经全面上线。最稳妥的方式是突出已经验证的核心思路，并准备好被追问时的分阶段解释。

## 学习检查

完成第一轮学习后，你应该能不用术语回答下面五个问题：

1. 为什么这个项目不是普通聊天机器人？
2. 用户问不同问题时，系统如何决定下一步？
3. 为什么有的问题交给大模型，有的问题交给知识库、搜索或算法？
4. 工具失败、资料不足或问题太复杂时，系统怎么办？
5. 如果再给你三个月，你会怎样把它升级成真正可用的产品？

如果这五个问题还答不顺，先回到零基础项目地图，不要继续背更多名词。
