# 项目讲解与面试话术

这篇文档只负责一件事：教你在面试中把项目讲成一个完整、有价值、能继续展开的故事。

技术名词的详细解释放在[简历技术点总表](./03_简历技术点总表.md)，高频追问放在[高频技术问答](./04_高频技术问答.md)。这里不再重复百科式定义。

## 1. 先记住讲项目的顺序

一个完整项目故事由五部分组成：

```text
为什么做
  -> 遇到什么核心问题
  -> 系统怎样设计
  -> 最有价值的技术亮点
  -> 结果、边界和下一步
```

不要一上来罗列技术栈。面试官先要知道你解决了什么问题，然后才关心为什么使用 LangGraph、Milvus 或 MCP。

## 2. 项目核心定位

### 给自己理解的版本

这个项目不是为了把健身功能做得特别多，而是用健身场景展示一套完整的智能体工程方法：

- 先判断用户要做什么。
- 再选择知识库、联网搜索、动作算法或外部工具。
- 用明确工作流组织执行过程。
- 保存必要的上下文和用户信息。
- 处理工具失败和结果降级。
- 用测试与评测检查系统质量。

### 面试中的一句话版本

> 我设计了一个面向个人健身场景的多任务智能体，用 LangGraph 组织路由和子工作流，把领域知识检索、实时搜索、动作分析、外部工具和记忆系统整合成可解释、可评测、可降级的执行流程。

这句话已经包含四个重点：

1. 不是单一聊天，而是多任务智能体。
2. LangGraph 负责流程控制。
3. 系统组合了不同性质的能力。
4. 项目强调工程质量，不只是功能演示。

## 3. 三十秒项目介绍

> 健身用户的问题差异很大：有些需要专业知识，有些需要最新资料，有些需要分析动作，还有些需要调用菜谱等外部工具。如果全部交给一次大模型回答，结果很难控制。因此我把系统拆成路由层和执行层：路由层判断任务类型和允许的组合，执行层分别使用 RAG、Tavily、动作算法和 MCP 工具完成任务。整个流程由 LangGraph 管理状态，同时补充记忆、失败降级和评测机制。这个项目主要展示我对智能体任务拆分、工具编排和工程边界的理解。

三十秒版本只需要完成三个任务：

- 说明为什么需要智能体。
- 说明系统怎样分层。
- 说明项目体现了什么能力。

不要主动展开索引参数、姿态坐标或接口字段。

## 4. 三分钟完整介绍

下面不是让你逐字背诵，而是提供五个逻辑锚点。每段记住第一句话即可。

### 第一段：为什么做

> 我选择个人健身场景，是因为它天然包含多种不同任务。稳定的训练知识适合查本地知识库，最新研究适合联网搜索，动作标准性需要视觉和数值算法，饮食和菜谱还可能涉及用户画像与外部工具。这种差异很适合展示智能体如何根据问题选择能力，而不是把所有任务塞进一个 Prompt。

这一段想证明：你选择场景有技术原因，不是随便套一个聊天界面。

### 第二段：总体架构

> 系统分为接入层、控制层、能力层和保障层。FastAPI 接收网页、小程序和流式请求；Router 判断主意图和可执行的组合；LangGraph 根据路由计划进入知识、搜索、动作或外部工具子工作流；Memory、统一工具结果、降级轨迹和测试评测负责保障上下文与稳定性。

可以画成：

```text
用户请求
  -> 接入层：接收请求
  -> 控制层：判断任务并选择流程
  -> 能力层：知识 / 搜索 / 动作 / 外部工具
  -> 保障层：记忆 / 错误处理 / 评测
  -> 最终回答
```

这一段想证明：你能从系统层面组织多个模块。

### 第三段：能力如何分工

> 我没有让大模型包办所有事情。语言理解和结果表达交给大模型；稳定领域知识交给 RAG 和 Milvus；需要时效性的内容交给 Tavily；动作相似度交给 MediaPipe、PoseSequence 和 FastDTW 等确定性算法；外部工具通过工具注册表和 MCP 接入。这样每类问题都由更合适的执行者处理。

这一段是项目最重要的技术思想：大模型是系统的一部分，不是整个系统。

### 第四段：工程取舍

> 路由没有完全交给大模型，而是采用确定性规则、语义样例和可选模型兜底，因为明确任务更需要稳定、低延迟和可解释。复合请求也只允许经过验证的白名单组合。工具返回统一结果和错误码，外部依赖失败时记录降级轨迹，避免一个工具拖垮整个请求。

这一段想证明：你考虑过稳定性、可观察性和失败处理。

### 第五段：验证和演进

> 项目通过自动化测试保护接口、状态流、工具契约和算法，通过 Router 评测检查路由策略，通过 RAG、Memory 和 Context benchmark 验证关键链路。当前版本定位为可运行、可演示、可解释的面试原型；如果生产化，我会继续补充真实用户数据、权限与隐私治理、依赖监控、动作标准样本和更完整的质量指标。

这一段想证明：你知道“功能跑通”和“效果可靠”不是一回事。

## 5. 面向不同面试官怎么调整

### 人力或业务面试官

重点讲：

- 为什么健身场景需要智能体。
- 系统能解决哪些不同问题。
- 项目最大的创新点是什么。
- 你从项目中学到了什么。

少讲：索引参数、数据结构、协议细节。

推荐说法：

> 这个项目最大的价值是让一个助手根据问题选择正确能力。例如专业知识先查资料，最新研究去联网搜索，动作问题交给算法计算。大模型主要负责理解和表达，因此比单纯聊天机器人更稳定、更容易解释。

### 普通技术面试官

重点讲：

- LangGraph 与 Router 的关系。
- RAG、Search、Motion、MCP 怎样分工。
- 为什么使用受控流程。
- 工具失败和上下文过长怎样处理。
- 使用什么测试和评测。

### 资深智能体面试官

重点讲：

- 状态怎样在顶层图和子图之间传递。
- 路由置信度、歧义和多意图怎样处理。
- LangGraph、工具注册表和 MCP 的职责边界。
- 检索质量、回答忠实度和任务成功怎样评测。
- 记忆写入、召回、隐私和 Prompt 预算怎样治理。

遇到这类追问时，再进入[技术深挖与白板](./06_技术深挖与白板.md)。

## 6. 三个最值得主打的亮点

### 亮点一：异构能力的受控编排

“异构能力”指工作方式不同的能力：知识库负责检索，搜索服务负责实时信息，数值算法负责动作计算，外部协议负责工具调用。

推荐说法：

> 项目不是把多个功能简单堆在一起，而是先判断问题性质，再选择更合适的执行者。LangGraph 负责流程，Router 负责分发，每个子工作流可以单独测试和降级。

面试价值：体现智能体架构和任务拆分能力。

### 亮点二：大模型与确定性系统协作

推荐说法：

> 我没有让大模型直接计算动作相似度，也没有让它凭记忆回答所有专业问题。动作交给数值算法，知识交给检索，大模型负责理解、规划和解释。这种分工降低了完全依赖模型产生幻觉或不稳定结果的风险。

面试价值：体现你知道大模型擅长什么、不擅长什么。

### 亮点三：用评测和降级约束智能体

推荐说法：

> 路由策略有独立评测集，工具调用有统一结果和错误码，真实依赖不可用时会记录 mock 或 fallback，RAG 和记忆链路也有专项 benchmark。这样项目不只展示正常路径，还展示如何发现问题和控制失败。

面试价值：体现工程意识、可观察性和质量闭环。

## 7. 个人贡献怎么讲

不要只说“我使用了很多框架”。贡献应该围绕问题和决策来讲：

> 我主要负责整个智能体架构的设计与整合。第一，我把健身需求拆成知识、搜索、动作和外部工具等能力域，并设计顶层路由和子工作流；第二，我明确大模型、检索、算法和工具各自的职责，避免所有任务都依赖模型；第三，我补充了统一错误结构、失败降级、上下文与长期记忆；第四，我建立自动化测试和专项评测，让路由、检索和关键链路可以持续验证。

如果面试官让你举一个最有代表性的决策，可以选择 Router：

> 我没有为了“更像智能体”就把路由完全交给大模型，而是先对真实问题做分类，使用确定性规则和语义样例覆盖高频场景，再让模型只处理低置信情况。因为面试项目不仅要有技术亮点，还要能稳定演示和解释。

## 8. 面试官可能怎样继续追问

### 为什么不是普通聊天机器人？

> 因为系统不只生成文字。它会判断任务、选择工作流、调用检索或工具、执行数值计算，并根据执行结果生成回答。大模型参与流程，但不是唯一执行者。

### 为什么使用 LangGraph？

> 因为项目包含路由、多个子工作流、工具失败和结果合成。用显式节点、状态和边表示后，流程更容易解释、测试和扩展，也能限制智能体的自由度。

### 为什么路由不全部交给大模型？

> 明确意图使用确定性方法更稳定、快速和可解释。只有规则难判断的低置信样本才值得使用模型，而且最终仍要由离线评测决定模型是否带来收益。

### 功能很多，会不会只是技术堆砌？

> 每项技术都对应一种不同问题：RAG 解决稳定知识，搜索解决时效性，动作算法解决确定性计算，MCP 解决外部工具协议，Memory 解决跨轮信息。它们由同一套路由、状态、错误和评测体系组织，所以重点是分工和治理，不是数量。

## 9. 当前版本和生产化怎么说

推荐结构是“先成果，后边界，再升级”：

> 当前版本已经打通多任务路由、知识检索、联网搜索、动作媒体输入、外部工具协议、短期与长期记忆，并通过自动化测试和专项评测验证核心链路。它定位为本地可运行的面试原型。生产化还需要补认证授权、隐私治理、依赖监控、多实例存储、真实用户评测集，以及更完整的动作标准样本。

不要说：

- “其实大部分都没有实现。”
- “这就是一个很简单的课程项目。”
- “用了 Milvus，所以 RAG 效果肯定很好。”
- “接了 MCP，所有工具就自动安全了。”
- “视频能上传，所以动作纠错已经非常准确。”

也不要说：

- “已经是生产级健身平台。”
- “Router 的离线评测等于线上准确率。”
- “所有工具都能由大模型自由发现和执行。”

## 10. 最短练习方法

第一轮：只练一句话定位和三十秒介绍。

第二轮：按照“为什么做、架构、能力分工、工程取舍、验证演进”讲三分钟。

第三轮：不看答案，回答本篇第八节的四个问题。

第四轮：进入[简历技术点总表](./03_简历技术点总表.md)，逐项补齐你在讲述中不理解的名词。

判断是否掌握的标准不是能否逐字背诵，而是被打断、被换一种问法后，仍然能沿着同一条逻辑主线回答。
