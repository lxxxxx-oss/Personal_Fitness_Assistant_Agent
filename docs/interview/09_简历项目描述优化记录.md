这篇文档专门记录简历中项目技术亮点的逐项审查与优化。目标不是把描述写得更夸张，而是同时满足三项要求：第一眼有技术亮点，深入追问有代码证据，谈到边界时能体现工程判断。

## 使用规则

每个技术亮点都按照同一顺序讨论和记录：

1. 保留简历原文。
2. 将原文拆成可验证的技术主张。
3. 对照代码、测试和评测判断实现程度。
4. 分析是否值得继续优化代码。
5. 给出优化后的简历版本及修改理由。
6. 标记“当前已实现”和“优化后才能使用”的表达，避免提前使用目标口径。

## 1. 混合意图路由

### 简历原文

> **构建可评测的混合意图路由**：基于 LangGraph StateGraph 组织顶层路由，结合加权规则、语义样例匹配和可选 Qwen 兜底处理模糊表达；记录路由分数与决策原因，并通过白名单执行复合任务，在 66 条常规样本和 36 条困难样本上建立持续回归基线。

### 代码审查结论

已经真实实现的部分包括：

- LangGraph StateGraph 顶层路由。
- `search`、`motion`、`diet`、`chat`、`mcp` 五类意图。
- 加权短语、组合规则和歧义信号。
- 路由分数、置信度、匹配依据、决策原因和执行计划。
- 四种经过约束的双任务白名单组合。
- 可选本地 Qwen 分类接口和严格 JSON、意图、置信度校验。
- 66 条常规集和 36 条困难集；当前分别达到 66/66 与 36/36。

需要修正的表述是“语义样例匹配”。当前实现使用中文二元、三元字符特征与 Jaccard 相似度，本质是轻量词面样例匹配，不是真正的 embedding 语义检索。Qwen 已接入但默认关闭，现有评测成绩主要来自确定性规则和词面样例，不应把成绩归因于 Qwen。

### 是否直接换成 embedding

结论：不直接替换，而是在现有路由中增加可选 embedding 语义层。

使用本地 `BAAI/bge-small-zh-v1.5` 对现有样例做纯 embedding 最近邻对照，结果如下：

| 评测集 | 当前混合路由 | 纯 embedding 最近邻 |
|---|---:|---:|
| 66 条常规集 | 66/66，100% | 50/66，75.8% |
| 36 条困难集 | 36/36，100% | 22/36，61.1% |

纯 embedding 更容易理解“问题谈论什么”，但不一定理解“用户要求系统执行什么”。例如“查一下深蹲标准动作”在主题上接近 Motion，但执行要求是 Search；“鸡胸肉怎么做不柴”在主题上接近 Diet，但实际需要菜谱工具。否定词、文件信号、时效要求、任务顺序和工具边界仍应由确定性规则保护。

### 确定的优化方案

```text
高置信加权规则
  -> 轻量字符样例匹配
  -> 低置信时使用 embedding 样例检索
  -> 相似度与 Top1-Top2 分差均达标才接受
  -> 仍不确定时尝试可选 Qwen
  -> 最后回退 chat 或要求澄清
```

工程要求：

- 样例向量在启动时预计算并缓存，不能每次重复编码。
- embedding 模型加载失败时退回原路由，不能阻塞项目启动。
- 同时使用绝对相似度和类别分差，不能只取最近样例。
- 记录模型版本、最相似样例、各类分数、分差、耗时和最终采用来源。
- 新增同义改写、隐式意图、错别字、中英混合和 hard negative 数据集。
- 做规则、词面、embedding、Qwen 四层消融，比较准确率、宏平均 F1、计划准确率、p95 延迟和各层调用率。

### 当前代码不变时的安全表述

> **构建可评测的混合意图路由**：基于 LangGraph StateGraph 组织顶层路由，结合加权规则、字符/词语特征样例匹配和可选本地 Qwen 分类兜底处理模糊表达；记录路由分数、置信度、歧义信号与决策原因，并通过白名单执行复合任务，在 66 条常规样本和 36 条困难样本上建立持续回归基线。

### embedding 优化完成后可使用的目标表述

> **构建可评测的混合意图路由**：基于 LangGraph StateGraph 组织顶层路由，采用“加权规则、轻量词面匹配、embedding 语义样例检索与可选 Qwen 分类兜底”的分层策略处理明确、隐式和模糊表达；通过相似度阈值、类别分差和白名单约束复合任务执行，记录路由分数、决策原因与各阶段耗时，并基于常规集、困难集和语义改写集进行消融评测与持续回归。

### 优化理由

这次优化不是为了多写一个模型名称，而是为了形成清晰的能力分工：规则保证业务行为与安全边界，embedding 处理词面不同的语义改写，Qwen 只处理少量复杂输入，低置信结果可以拒绝或澄清。这个方案既保留当前稳定基线，又能展示模型选型、阈值校准、故障降级、可观测性和消融评测等工程意识。

### 当前状态

方案已经讨论确定，但 embedding 路由层尚未写入代码。因此，在实现和重新评测完成前，只能使用“当前代码不变时的安全表述”，不能提前使用目标表述。

## 2. 3D 运动动作相似度分析

### 简历原文

> **实现 3D 运动动作相似度分析模块**：使用 MediaPipe 与 OpenCV 打通图片、短视频到统一 PoseSequence 的姿态提取链路，并针对 3D 关键点序列实现尺度归一化、关节角计算、FastDTW 时序对齐及余弦相似度、形状差异等多维动作对比，实现可解释的动作分析。

### 代码审查结论

已经真实实现的部分包括：

- 图片、短视频和 `.npz` 到统一 `PoseSequence` 的转换。
- MediaPipe 33 关键点、置信度、帧率、姿态模型、关节结构和坐标空间元数据。
- 基于髋部中心的平移归一化和基于平均关节距离的尺度归一化。
- FastDTW 时序对齐、余弦相似度和对齐后逐关节平均距离。
- 坐标空间、姿态模型和关节结构兼容性检查。
- 结构化指标、固定阈值标签和总体评价。

需要修正的地方是关节角和可解释性。`compute_joint_angles()` 已实现并有单元测试，但尚未接入最终 `compute_similarity()`，所以当前不能说关节角已经参与多维评分。现有解释主要是 DTW、余弦和形状差异三个全局指标，还不能定位具体错误关节和动作时刻。

“3D”可以保留，但必须解释为单目 MediaPipe 产生的 `x、y、z` 关键点序列；有 world landmarks 时使用世界坐标，否则使用带深度估计的归一化图像坐标。它不等同于多摄像头或专业动作捕捉精度。

### 确定的优化方案

保留当前全局指标，并增加关节级解释与输入质量控制：

1. 将左右膝、髋、肘、肩等关键关节角正式接入对齐后的比较链路。
2. 输出每个关键关节的平均角度误差和最大角度误差。
3. 定位偏差最大的关节、对齐帧和对应视频时刻。
4. 使用关键点置信度屏蔽或降低低质量关节对评分的影响。
5. 保留坐标空间、姿态模型和 joint schema 一致性校验。
6. 为新增指标补充平移、缩放、速度变化、遮挡和错误姿态测试。
7. 当前固定阈值继续作为原型解释，后续再用标准动作、常见错误动作和教练标签做分动作校准。

本轮不引入复杂动作周期模型、训练数据集或专业评分模型，避免为了简历过度扩大范围。动作阶段切分、关键点平滑、多参考样本和教练标注作为后续生产化方向。

### 当前代码不变时的安全表述

> **实现 3D 运动动作相似度分析模块**：使用 MediaPipe 与 OpenCV 打通图片、短视频到统一 PoseSequence 的姿态提取链路，针对三维关键点序列实现髋部中心与尺度归一化、FastDTW 时序对齐，并结合余弦相似度和对齐后逐关节形状差异进行多维动作对比；同时实现关节角基础计算能力，输出结构化、可解释的动作相似度指标。

### 代码优化完成后可使用的目标表述

> **实现 3D 运动动作相似度分析模块**：使用 MediaPipe 与 OpenCV 打通图片、短视频到统一 PoseSequence 的姿态提取链路，针对三维关键点序列实现髋部中心与尺度归一化、置信度过滤和 FastDTW 时序对齐，结合余弦相似度、对齐后逐关节形状差异及关键关节角度误差进行多维动作对比，并定位偏差最大的关节与动作时刻，输出结构化、可解释的动作分析结果。

### 优化理由

Motion 是本项目区别于普通 RAG 助手的核心亮点。将现有独立关节角函数接入最终链路，可以把“全局相似度”升级为“局部关节偏差”，使可解释性有具体证据；置信度过滤则说明系统考虑了遮挡和姿态估计噪声。这个范围能明显增强面试价值，又不需要虚构专业数据集或重做训练模型。

### 当前状态

描述优化方案已经确定，代码修改延后到四项技术亮点全部讨论完成后统一实施。在关节角集成、置信度过滤、局部定位和专项测试完成前，简历只能使用当前安全表述。

## 3. Milvus RAG 检索

### 简历原文

> **基于 Milvus 构建健身领域 RAG 检索模块**：完成中文资料分块、Sentence-Transformers 向量化及 IVF_FLAT + COSINE 检索，并通过幂等入库、Top-K、阈值过滤、结果去重和来源元数据管理，兼顾检索质量、数据一致性与答案可追溯性。

### 代码审查结论

已经真实实现的部分包括：

- 中英文标点与换行驱动的句子感知分块。
- Sentence-Transformers 文档/查询向量化和向量归一化。
- Milvus Collection、`IVF_FLAT`、`COSINE`、`nlist` 与 `nprobe`。
- 基于内容稳定哈希主键和 Milvus `upsert` 的片段级幂等写入。
- Top-K、相似度阈值、空内容过滤和精确内容去重。
- `content`、`score`、`source`、主键及 `[RefN]` 证据编号透传。
- Strong consistency 查询、可配置 Milvus 后端和内存降级。

需要收紧的地方有四点。第一，单个超长句目前不会真正按最大长度继续切分，也没有 overlap。第二，稳定主键只由内容生成，不包含来源、位置和版本；相同内容来自不同文件时可能覆盖来源，文档更新后旧 chunk 也不会自动清理。第三，Chat 子图硬编码 `top_k=5`、`threshold=0.3`，没有统一读取配置。第四，当前具备链路与来源透传测试，但没有正式 RAG 黄金集、Recall@K、MRR 和参数消融，因此“兼顾检索质量”尚未形成可量化证据。

当前追溯能力是来源级：能说明本次使用了哪些知识片段和文件，但没有逐句验证答案中的每个结论是否由对应 `[RefN]` 支持，不能表述为严格事实引用系统。

### 确定的优化方案

优先补齐数据一致性、可评测性与配置治理，暂不直接增加 BM25、RRF 或 Cross-Encoder Rerank：

1. 修复超长句切分，并增加可配置 chunk size 与 overlap。
2. 使用 `doc_id/source + chunk_index + content_hash + version` 标识 chunk。
3. 为每次知识入库保存 manifest，支持文档更新、旧 chunk 清理和失败回滚。
4. 统一从配置读取 Top-K、threshold、`nlist` 和 `nprobe`。
5. 建立包含正确片段、正确来源和无答案问题的 RAG 黄金集。
6. 评测 Recall@K、MRR、来源命中率、无答案识别率和检索延迟。
7. 对 chunk size、overlap、Top-K、threshold 和 `nprobe` 做单变量消融。
8. 记录 embedding 模型、知识库、索引和入库批次版本，便于复现与回滚。
9. 保留 Milvus 故障到内存检索的降级，并在执行元数据中明确真实 backend 和降级原因。

是否加入混合检索和 Rerank，要由黄金集中的错误归因决定。只有 dense retrieval 确实存在关键词漏召回或排序问题，并且增量收益高于延迟成本时再加入，避免技术堆叠。

### 当前代码不变时的安全表述

> **构建可配置的健身领域 Milvus RAG 模块**：实现面向中英文标点的句子感知分块、Sentence-Transformers 向量化及 IVF_FLAT + COSINE 检索；Milvus 后端通过稳定内容哈希与 upsert 实现片段级幂等写入，并结合 Top-K、阈值过滤、内容去重、证据编号和来源元数据透传，支持检索结果的来源级追溯与故障降级。

### 代码优化完成后可使用的目标表述

> **构建可评测、可降级的健身领域 RAG 检索模块**：完成句子感知分块与重叠控制、Sentence-Transformers 向量化及 Milvus IVF_FLAT + COSINE 检索；通过版本化 chunk 身份、幂等 upsert 和旧版本清理保证知识更新一致性，结合 Top-K、阈值过滤、结果去重、证据编号和来源元数据实现来源级追溯，并基于 Recall@K、MRR、来源命中率与检索延迟开展参数消融和持续回归。

### 优化理由

RAG 的面试价值不在于“接入了向量数据库”，而在于能否证明知识如何更新、错误如何定位、参数为什么这样选、检索是否真的变好。版本化入库与旧片段清理补足数据一致性，黄金集与 Recall@K/MRR 补足质量证据，来源透传与运行模式记录补足可追溯和降级能力。先建立这些工程闭环，比没有评测依据就加入 Rerank 更有说服力。

描述优化方案已经确定，代码修改延后到四项技术亮点全部讨论完成后统一实施。在重叠分块、版本化入库、旧片段清理、配置统一、RAG 黄金集和指标基线完成前，简历只能使用当前安全表述。
