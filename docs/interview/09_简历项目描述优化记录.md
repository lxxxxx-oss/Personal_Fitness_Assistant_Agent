# 简历项目描述优化记录

这篇文档专门记录简历中项目技术亮点的逐项审查与优化。目标不是把描述写得更夸张，而是同时满足三项要求：第一眼有技术亮点，深入追问有代码证据，谈到边界时能体现工程判断。

## 使用规则

每个技术亮点都按照同一顺序讨论和记录：

1. 保留简历原文。
2. 将原文拆成可验证的技术主张。
3. 对照代码、测试和评测判断实现程度。
4. 分析是否值得继续优化代码。
5. 给出优化后的简历版本及修改理由。
6. 标记“当前已实现”和“优化后才能使用”的表达，避免提前使用目标口径。

## 1. 混合意图路由

### 简历原文

> **构建可评测的混合意图路由**：基于 LangGraph StateGraph 组织顶层路由，结合加权规则、语义样例匹配和可选 Qwen 兜底处理模糊表达；记录路由分数与决策原因，并通过白名单执行复合任务，在 66 条常规样本和 36 条困难样本上建立持续回归基线。

### 代码审查结论

已经真实实现的部分包括：

- LangGraph StateGraph 顶层路由。
- `search`、`motion`、`diet`、`chat`、`mcp` 五类意图。
- 加权短语、组合规则和歧义信号。
- 路由分数、置信度、匹配依据、决策原因和执行计划。
- 四种经过约束的双任务白名单组合。
- 可选本地 Qwen 分类接口和严格 JSON、意图、置信度校验。
- 66 条常规集和 36 条困难集；当前分别达到 66/66 与 36/36。

需要修正的表述是“语义样例匹配”。当前实现使用中文二元、三元字符特征与 Jaccard 相似度，本质是轻量词面样例匹配，不是真正的 embedding 语义检索。Qwen 已接入但默认关闭，现有评测成绩主要来自确定性规则和词面样例，不应把成绩归因于 Qwen。

### 是否直接换成 embedding

结论：不直接替换，而是在现有路由中增加可选 embedding 语义层。

使用本地 `BAAI/bge-small-zh-v1.5` 对现有样例做纯 embedding 最近邻对照，结果如下：

| 评测集 | 当前混合路由 | 纯 embedding 最近邻 |
|---|---:|---:|
| 66 条常规集 | 66/66，100% | 50/66，75.8% |
| 36 条困难集 | 36/36，100% | 22/36，61.1% |

纯 embedding 更容易理解“问题谈论什么”，但不一定理解“用户要求系统执行什么”。例如“查一下深蹲标准动作”在主题上接近 Motion，但执行要求是 Search；“鸡胸肉怎么做不柴”在主题上接近 Diet，但实际需要菜谱工具。否定词、文件信号、时效要求、任务顺序和工具边界仍应由确定性规则保护。

### 确定的优化方案

```text
高置信加权规则
  -> 轻量字符样例匹配
  -> 低置信时使用 embedding 样例检索
  -> 相似度与 Top1-Top2 分差均达标才接受
  -> 仍不确定时尝试可选 Qwen
  -> 最后回退 chat 或要求澄清
```

工程要求：

- 样例向量在启动时预计算并缓存，不能每次重复编码。
- embedding 模型加载失败时退回原路由，不能阻塞项目启动。
- 同时使用绝对相似度和类别分差，不能只取最近样例。
- 记录模型版本、最相似样例、各类分数、分差、耗时和最终采用来源。
- 新增同义改写、隐式意图、错别字、中英混合和 hard negative 数据集。
- 做规则、词面、embedding、Qwen 四层消融，比较准确率、宏平均 F1、计划准确率、p95 延迟和各层调用率。

### 当前代码不变时的安全表述

> **构建可评测的混合意图路由**：基于 LangGraph StateGraph 组织顶层路由，结合加权规则、字符/词语特征样例匹配和可选本地 Qwen 分类兜底处理模糊表达；记录路由分数、置信度、歧义信号与决策原因，并通过白名单执行复合任务，在 66 条常规样本和 36 条困难样本上建立持续回归基线。

### 代码优化完成后可使用的目标表述

> **构建可评测的混合意图路由**：基于 LangGraph StateGraph 组织顶层任务流程，以加权规则和歧义检测稳定识别五类意图，并对低置信表达采用经语义改写集验证的 embedding 样例匹配；通过相似度阈值、类别分差和白名单约束复合任务执行，记录路由分数、决策原因与耗时，结合常规集、困难集和语义改写集开展持续回归。

### 优化理由

这次优化不是为了多写一个模型名称，而是为了形成清晰的能力分工：规则保证业务行为与安全边界，embedding 只处理规则低置信但语义相近的改写表达，低置信结果可以拒绝或澄清。这个方案既保留当前稳定基线，又能展示阈值校准、故障降级、可观测性和消融评测等工程意识。

### 当前状态

方案已经完成第一轮代码落地：新增语义改写评测集、默认关闭的 embedding 样例 fallback、样例向量缓存、置信度阈值、Top1-Top2 分差门控和失败降级。当前默认路由基线仍保持 66/66、36/36，语义改写集为 10/10。面试中可以说“embedding fallback 已接入但默认关闭，需要更大语义改写集和 hard negative 证明收益后再启用”，不要说它已经替代主路由。

## 2. 3D 运动动作相似度分析

### 简历原文

> **实现 3D 运动动作相似度分析模块**：使用 MediaPipe 与 OpenCV 打通图片、短视频到统一 PoseSequence 的姿态提取链路，并针对 3D 关键点序列实现尺度归一化、关节角计算、FastDTW 时序对齐及余弦相似度、形状差异等多维动作对比，实现可解释的动作分析。

### 代码审查结论

已经真实实现的部分包括：

- 图片、短视频和 `.npz` 到统一 `PoseSequence` 的转换。
- MediaPipe 33 关键点、置信度、帧率、姿态模型、关节结构和坐标空间元数据。
- 基于髋部中心的平移归一化和基于平均关节距离的尺度归一化。
- FastDTW 时序对齐、余弦相似度和对齐后逐关节平均距离。
- 坐标空间、姿态模型和关节结构兼容性检查。
- 结构化指标、固定阈值标签和总体评价。

此前需要修正的地方是关节角和可解释性：`compute_joint_angles()` 已实现但未接入最终比较链路。当前这一点已补齐到 `compute_pose_sequence_similarity()`：在 `mediapipe_33` 下会输出左右膝、左右髋的平均/最大角度误差，并结合 FastDTW 对齐路径定位偏差最大的关节、帧和可选视频时刻。现有解释已经从全局指标扩展到局部关节证据，但仍不是完整专业动作纠错模型。

“3D”可以保留，但必须解释为单目 MediaPipe 产生的 `x、y、z` 关键点序列；有 world landmarks 时使用世界坐标，否则使用带深度估计的归一化图像坐标。它不等同于多摄像头或专业动作捕捉精度。

### 确定的优化方案

保留当前全局指标，并增加关节级解释与输入质量控制：

1. 将左右膝、髋、肘、肩等关键关节角正式接入对齐后的比较链路。
2. 输出每个关键关节的平均角度误差和最大角度误差。
3. 定位偏差最大的关节、对齐帧和对应视频时刻。
4. 使用关键点置信度屏蔽或降低低质量关节对评分的影响。
5. 保留坐标空间、姿态模型和 joint schema 一致性校验。
6. 为新增指标补充平移、缩放、速度变化、遮挡和错误姿态测试。
7. 当前固定阈值继续作为原型解释，后续再用标准动作、常见错误动作和教练标签做分动作校准。

本轮不引入复杂动作周期模型、训练数据集或专业评分模型，避免为了简历过度扩大范围。动作阶段切分、关键点平滑、多参考样本和教练标注作为后续生产化方向。

### 当前代码不变时的安全表述

> **实现 3D 运动动作相似度分析模块**：使用 MediaPipe 与 OpenCV 打通图片、短视频到统一 PoseSequence 的姿态提取链路，针对三维关键点序列实现髋部中心与尺度归一化、FastDTW 时序对齐，并结合余弦相似度和对齐后逐关节形状差异进行多维动作对比；同时实现关节角基础计算能力，输出结构化、可解释的动作相似度指标。

### 代码优化完成后可使用的目标表述

> **实现可解释的 3D 运动动作相似度分析模块**：使用 MediaPipe 与 OpenCV 打通图片、短视频到统一 PoseSequence 的姿态提取链路；针对深蹲三维关键点序列实现髋部中心与尺度归一化、置信度过滤和 FastDTW 时序对齐，结合全局相似度、对齐后逐关节形状差异及膝/髋关键角度误差进行动作对比，并定位偏差最大的关节与动作时刻，输出可执行的结构化反馈。

### 优化理由

Motion 是本项目区别于普通 RAG 助手的核心亮点。将现有独立关节角函数接入最终链路，可以把“全局相似度”升级为“局部关节偏差”，使可解释性有具体证据；置信度过滤则说明系统考虑了遮挡和姿态估计噪声。这个范围能明显增强面试价值，又不需要虚构专业数据集或重做训练模型。

### 当前状态

代码已经完成第一轮落地：保留全局 DTW/余弦/形状差异，新增 `joint_distance`、`joint_angle_errors` 和 `quality`，覆盖最大偏差关节/帧/时间点、MediaPipe 膝/髋角度误差和低置信关键点过滤。Motion 专项测试、PoseSequence、MediaPipe 适配器和工具注册表/API 回归均通过。仍未完成的是正式深蹲标准/典型错误样本集、动作周期切分、阶段级规则和教练标签校准，因此简历可以使用目标表述，但面试时必须说明这是“可解释相似度原型”，不是专业评分模型。

## 3. Milvus RAG 检索

### 简历原文

> **基于 Milvus 构建健身领域 RAG 检索模块**：完成中文资料分块、Sentence-Transformers 向量化及 IVF_FLAT + COSINE 检索，并通过幂等入库、Top-K、阈值过滤、结果去重和来源元数据管理，兼顾检索质量、数据一致性与答案可追溯性。

### 代码审查结论

已经真实实现的部分包括：

- 中英文标点与换行驱动的句子感知分块，超长句会按最大 chunk 长度继续切分。
- Sentence-Transformers 文档/查询向量化和向量归一化。
- Milvus Collection、`IVF_FLAT`、`COSINE`、`nlist` 与 `nprobe`。
- 基于 `version + source + chunk_index + content_hash` 的稳定主键和 Milvus `upsert` 的片段级幂等写入。
- Chat/Diet 统一读取配置中的 Top-K、相似度阈值、chunk size 与 overlap，检索后执行空内容过滤和精确内容去重。
- `content`、`score`、`source`、主键及 `[RefN]` 证据编号透传。
- 入库返回轻量 manifest，记录知识版本、来源、chunk id 和内容哈希；同一进程内重复入库同 source 时会清理旧 chunk，避免文档更新后新旧片段同时命中。
- Strong consistency 查询、可配置 Milvus 后端和内存降级。

需要收紧的地方还有三点。第一，当前 manifest 是入库返回的轻量结构和进程内 source 记录，还不是独立持久化的批次账本，因此不能宣称支持生产级失败回滚和跨进程对账。第二，当前已有 15 条第一版 RAG 黄金集和阈值扫描，但样本规模仍小，也没有生成答案忠实度评测，因此只能说建立了检索回归基线，不能宣称已经证明大规模问答质量。第三，当前 overlap 只用于超长句硬切分，尚未做面向相邻自然句的滑动窗口重叠。

当前追溯能力是来源级：能说明本次使用了哪些知识片段和文件，但没有逐句验证答案中的每个结论是否由对应 `[RefN]` 支持，不能表述为严格事实引用系统。

### 确定的优化方案

优先补齐数据一致性、可评测性与配置治理，暂不直接增加 BM25、RRF 或 Cross-Encoder Rerank：

1. 修复超长句切分，并增加可配置 chunk size 与 overlap。已完成第一版。
2. 使用 `doc_id/source + chunk_index + content_hash + version` 标识 chunk。已完成第一版，当前使用 `source + chunk_index + content_hash + RETRIEVER_KNOWLEDGE_VERSION`。
3. 为每次知识入库保存 manifest，支持文档更新、旧 chunk 清理和失败回滚。已完成轻量 manifest 与同源旧 chunk 清理；独立持久化 manifest 和失败回滚仍未完成。
4. 统一从配置读取 Top-K、threshold、`nlist` 和 `nprobe`。Chat/Diet 的 Top-K 与 threshold 已完成，Milvus `nlist`/`nprobe` 原本已由配置控制。
5. 建立包含正确片段、正确来源和无答案问题的 RAG 黄金集。已完成第一版 15 条。
6. 评测 Recall@K、MRR、来源命中率、无答案识别率和检索延迟。已完成第一版，并支持回归门槛参数。
7. 对 chunk size、overlap、Top-K、threshold 和 `nprobe` 做单变量消融。
8. 记录 embedding 模型、知识库、索引和入库批次版本，便于复现与回滚。
9. 保留 Milvus 故障到内存检索的降级，并在执行元数据中明确真实 backend 和降级原因。

是否加入混合检索和 Rerank，要由黄金集中的错误归因决定。只有 dense retrieval 确实存在关键词漏召回或排序问题，并且增量收益高于延迟成本时再加入，避免技术堆叠。

### 当前推荐表述

> **构建可评测、可降级的健身领域 RAG 检索模块**：完成句子感知分块、Sentence-Transformers 向量化及 Milvus IVF_FLAT + COSINE 检索；通过版本化 chunk 身份、幂等 upsert 和旧片段清理保证知识更新一致性，结合 Top-K、阈值过滤、内容去重、证据编号和来源元数据实现来源级追溯，并以包含正确片段、来源和无答案问题的黄金集持续评测 Recall@K、MRR、拒绝率与检索延迟。

### 生产化增强后的目标表述

> **构建具备版本治理与端到端质量闭环的健身领域 RAG 模块**：在可评测 Milvus 检索链路上增加持久化入库批次、失败回滚和跨进程对账，结合扩大后的困难查询集、生成答案忠实度和端到端延迟评测，对分块、召回、阈值及可选重排序策略进行持续回归和灰度校准。

### 优化理由

RAG 的面试价值不在于“接入了向量数据库”，而在于能否证明知识如何更新、错误如何定位、参数为什么这样选、检索是否真的变好。版本化入库与旧片段清理补足数据一致性，黄金集与 Recall@K/MRR 补足质量证据，来源透传与运行模式记录补足可追溯和降级能力。先建立这些工程闭环，比没有评测依据就加入 Rerank 更有说服力。

### 当前状态

已完成第四轮代码落地：在分块、配置治理、版本化 chunk 身份、轻量 manifest、同源旧片段清理和 15 条黄金集之外，默认 embedding 已统一为评测过的 512 维 `BAAI/bge-small-zh-v1.5`。知识 RAG 与可选用户记忆分别使用新的模型专属默认 Collection 名，旧 text2vec Collection 保留；如果显式复用了不同维度旧索引，系统返回结构化 `CONFIG_CONFLICT` 并建议新 Collection，再按配置降级到内存检索。真实 BGE 基线显示 threshold=0.50 时 Recall@5 为 100%、MRR 0.892，但拒绝率仅 33.3%；threshold=0.56 时拒绝率 100%、Recall@5 为 91.7%，因此面试时应强调“用评测校准取舍”，不能只报最好看的单项数字。独立持久化 manifest、失败回滚、扩大黄金集和答案忠实度仍未完成。

## 4. 分层记忆与上下文压缩

### 简历原文

> **分层记忆与上下文压缩机制**：将短期对话状态、压缩摘要、长期用户偏好/约束解耦管理，结合候选确认、预算注入和可选 Milvus 语义召回，在控制 Prompt 长度的同时提升多轮问答的一致性、个性化与可追溯性。

### 代码审查结论

已经真实实现的部分包括：

- SQLite 持久化完整会话，并按 `user_id + conversation_id` 恢复。
- 运行时滑动窗口默认保留最近 6 轮，Chat Prompt 默认注入最近 6 条消息。
- 长期记忆独立保存，支持 CRUD、逻辑删除、稳定 `memory_key` 和来源元数据。
- 用户明确“记住”触发写入，伤病、过敏等敏感内容先进入候选确认。
- SQLite FTS5/LIKE 关键词检索和可选 Milvus 语义增强。
- 关键词、embedding、importance 和访问次数组合排序，最多召回 5 条。
- 长期记忆区按约 1200 字符预算注入。
- Prompt 超过阈值时，基于任务、画像、来源、工具摘要、决策和长期记忆执行确定性结构化压缩。
- `_prompt_meta` 和 execution 记录压缩前后字符数、触发状态和 Prompt 区域。

需要收紧的地方有五点。第一，跨请求摘要已经接入 HTTP、SSE、WebSocket 和最终回答 Prompt，但它是确定性原文摘录，不是模型语义总结。第二，预算使用字符数而不是真实 token。第三，长期记忆主要依赖用户显式“记住”，还没有普通信息候选提取、冲突替代和过期治理。第四，SQLite 删除或更新后，Milvus 中的旧向量没有形成完整的更新/删除同步闭环。第五，现有 9 条专项 benchmark 中只有 1 条持久化摘要用例，能证明基础链路没有回归，但不能证明已显著提高真实多轮一致性。

当前“可追溯”主要指记忆 ID、来源、检索分数、Prompt 注入区域和压缩轨迹可见，尚未验证最终回答中的每个结论具体使用了哪条记忆。

### 确定的优化方案

采用面试可解释的简化方案，不在第一版加入模型摘要、跨会话合并、摘要向量检索和复杂并发治理：

1. 使用现有 `summaries` 保存同一会话的 active 摘要，旧摘要只标记为 superseded，不直接删除，便于审计。存储层已完成。
2. 用 `last_compacted_message_id` 记录摘要覆盖边界，每次只处理边界之后的消息。存储层已完成。
3. 完整 user/assistant turn 使用一个 SQLite transaction 写入，避免只保存半轮。已完成。
4. 第一版使用确定性抽取式摘要，限制长度并保留最近三轮原文，不依赖额外模型。
5. Prompt 只组合“当前会话摘要 + 最近对话 + 长期记忆 + 当前问题”，不同 `conversation_id` 不混合。
6. 摘要失败时继续使用原滑动窗口，不阻塞聊天主链路；记录是否加载、更新和降级。
7. 建立小规模多轮回归，覆盖早期目标保持、边界不重复、跨用户隔离、归档失效和 Prompt 长度。

模型摘要、token 精确预算、乐观锁、记忆冲突/过期和 SQLite/Milvus 完整对账作为生产化追问回答，不纳入这轮必须实现范围。

### 当前基础表述

> **实现分层记忆与上下文管理机制**：将 SQLite 持久化会话、运行时滑动窗口和长期用户偏好/约束解耦管理；通过显式记忆写入、敏感信息候选确认、FTS5/可选 Milvus 召回和字符预算注入筛选相关记忆，并在 Prompt 超长时基于任务、来源、工具结果和长期记忆执行确定性结构化压缩，支持多轮个性化和注入过程追溯。

### 当前推荐表述

> **实现分层记忆与持久化会话压缩机制**：将 SQLite 完整会话、抽取式会话摘要、运行时滑动窗口和长期用户记忆分开管理；通过摘要覆盖边界只处理新增消息，并组合 active summary、最近原文和已确认长期记忆构造 Prompt，摘要失败时保留原滑动窗口路径，结合用户隔离、边界不重复和早期事实保持用例开展多轮回归。

### 优化理由

记忆系统的面试价值不在于堆叠复杂组件，而在于清楚区分完整历史、会话进度和长期用户事实。简化版用事务保证一轮问答完整写入，用摘要边界避免重复处理，用 active/superseded 状态保留审计能力，用滑动窗口 fallback 保证摘要失败不阻塞聊天；这些已经能体现数据一致性、隔离、降级和可测试性。模型摘要、精确 token 和并发乐观锁只作为生产化演进，不写成当前能力。

### 当前状态

简化版持久化摘要第一版已经完成：完整问答事务写入、active summary 替换、压缩边界、用户/会话隔离、确定性抽取和最近三轮保留均已落地；HTTP、SSE、WebSocket 共用保存入口，Chat、Diet、Search、MCP 最终回答 Prompt 都能加载摘要。加载或更新失败时保留原滑动窗口路径，不阻塞聊天；摘要与最近原文按边界去重，历史摘录也被明确标记为非系统指令。专项 benchmark 当前为 9/9，但持久化摘要仅有 1 个早期事实保持场景，因此不能宣称模型级总结质量、精确 token 控制或大规模并发能力。
