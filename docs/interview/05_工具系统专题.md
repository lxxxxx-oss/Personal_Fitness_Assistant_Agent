# 工具系统与工具注册表设计

这篇文档专门解释三个容易混淆的概念：工具系统、工具注册表和 MCP。

## 1. 先用一个生活化例子理解

把智能体想象成一家公司：

- **LangGraph** 是工作流程图，决定先做什么、后做什么。
- **工具** 是员工实际使用的搜索器、计算器和资料库。
- **ToolRegistry** 是内部工具台账，登记用途、参数、权限和负责人。
- **MCP** 是外部工具接入公司的统一插头标准。
- **ToolResult** 是使用工具后的统一执行报告。

因此三者不能互相替代：

> LangGraph 管流程，ToolRegistry 管工具执行，MCP 管外部工具协议。

## 2. 为什么不能只是调用几个函数

工具数量少时，直接调用函数很方便。工具增多后会出现：

- 调用者不知道工具适合什么任务。
- 参数类型和范围不一致。
- 网络、文件和子进程工具风险不同。
- 每个工具用不同格式表示失败。
- 有的错误可以重试，有的必须停止。
- 无法回答某次请求用了什么工具、耗时多久、是否降级。

所以工具系统需要解决五个问题：

| 检查维度 | 必须讲清什么 | 项目中的做法 |
|---|---|---|
| 职责 | 工具只负责哪一类任务 | 检索、搜索、动作比较和 MCP 调用分开 |
| 输入 | 需要哪些参数，类型和范围是什么 | schema、Pydantic、验证函数、PoseSequence |
| 输出 | 成功和失败怎样稳定表示 | `ToolResult`、`ErrorCode`、`meta` |
| 权限 | 工具允许做什么，谁可以调用 | ToolSpec permission 与执行上下文 |
| 错误 | 不同失败怎样处理 | 参数错误、权限错误、网络错误、数据为空、内部错误 |

面试中的核心结论：

> 函数只是执行载体，工具系统还要定义契约、权限、错误恢复和审计。

## 3. 当前工具调用链路

```text
用户请求
  -> Router 判断能力域
  -> LangGraph 进入对应子工作流
  -> 子工作流通过 ToolRegistry 选择具体工具
  -> Registry 校验参数和权限
  -> 执行工具，必要时有限重试或 fallback
  -> 返回统一 ToolResult
  -> 子工作流把结果写回共享状态
  -> 大模型解释结构化结果
```

当前已经通过 Registry 执行的主路径：

- `knowledge.retrieve`：Knowledge 和 Diet 的领域知识检索。
- `search.tavily`：联网搜索。
- `mcp.call_tool`：MCP 工具调用。

当前已登记但没有全面接管媒体链路的代表工具：

- `motion.compare_pose`：比较两个兼容的姿态序列。

## 4. ToolSpec 是什么

ToolSpec 是一张工具说明卡。最小字段包括：

```python
ToolSpec(
    name="knowledge.retrieve",
    description="检索本地健身知识",
    input_schema={...},
    permission="read_knowledge",
    executor=...,
    timeout_seconds=10.0,
    max_retries=0,
    fallback_tool=None,
)
```

字段含义：

| 字段 | 普通话解释 | 解决的问题 |
|---|---|---|
| `name` | 工具唯一名称 | 避免调用错误工具 |
| `description` | 工具适合做什么 | 帮助调用者理解职责边界 |
| `input_schema` | 参数结构和限制 | 避免模型猜测隐式格式 |
| `permission` | 需要什么权限 | 区分只读、网络、文件和子进程风险 |
| `executor` | 真正执行任务的函数 | 连接元数据和实现 |
| `timeout_seconds` | 预期超时策略 | 为工具自身或后续执行器提供策略信息 |
| `max_retries` | 最多重试几次 | 防止无限重试 |
| `fallback_tool` | 失败后可用的备用工具 | 提高可用性 |

重要边界：当前 `timeout_seconds` 是策略字段，Registry 本身不会强制中断正在运行的执行器。真实硬超时需要工具自身或后续独立 ToolExecutor 实现。

## 5. 工具注册表执行过程

```text
execute(name, args, context)
  -> 查找 ToolSpec
  -> 校验参数
  -> 检查权限
  -> 调用 executor
  -> 判断错误是否允许重试
  -> 需要时调用 fallback
  -> 补充 execution_id、duration_ms、attempts
  -> 写入 audit log
  -> 返回 ToolResult
```

Registry 不决定业务下一步。比如“先搜索还是先给饮食建议”由 Router 和 LangGraph 决定；Registry 只保证被选中的工具怎样受控执行。

## 6. 统一结果为什么重要

成功结果：

```text
ok = true
data = 工具数据
meta = 执行编号、耗时、尝试次数等
```

失败结果：

```text
ok = false
error_code = 可识别错误类型
error_message = 用户或程序能理解的说明
meta = 已执行步骤和降级信息
```

主要错误类型：

| 错误 | 能否重试 | 应该怎样处理 |
|---|---|---|
| 参数错误 | 修正参数后可以 | 不要用相同参数盲目重试 |
| 权限错误 | 不可以 | 停止执行并说明缺少权限 |
| 网络错误 | 可以有限重试 | 超过次数后降级或提示用户 |
| 数据为空 | 通常不重试 | 作为正常业务结果处理 |
| 配置缺失 | 配置修复前不重试 | 指出缺少密钥、模型或服务 |
| 内部错误 | 不应无限重试 | 记录上下文并返回安全错误 |

统一结果让子工作流可以根据错误类型选择“重试、降级、提示用户或终止”。

## 7. MCP 到底是什么

MCP 是模型应用连接外部工具的一种标准协议。它主要解决：

- 怎样初始化连接。
- 怎样发现外部工具。
- 怎样读取工具输入说明。
- 怎样发送工具调用。
- 怎样解析工具结果。

项目中的轻量 MCP Client：

```text
启动本地子进程
  -> 通过标准输入输出发送 JSON-RPC
  -> initialize
  -> tools/list
  -> tools/call
  -> 解析文本结果
  -> 转成 ToolResult
```

MCP 只是外部工具来源之一。知识检索、Tavily 和动作算法即使不使用 MCP，仍然属于内部工具系统。

## 8. 为什么动作模块没有全部迁移

Motion 不只是一个简单函数，它还涉及：

- 图片和视频上传。
- 文件大小与格式检查。
- 视频解码和临时资源。
- MediaPipe 模型文件。
- PoseSequence 和 NumPy 数组。
- 标准动作库兼容性。
- HTTP 错误码和用户提示。

因此当前只把动作比较内核登记为代表工具，媒体上传和姿态估计继续由 API 与 LangGraph 控制。

推荐面试表达：

> 我没有为了形式统一强行迁移整个 Motion 链路。文本检索和 MCP 调用输入输出更清晰，所以先接入 Registry；Motion 只选择职责明确的姿态比较内核作为工具，文件和媒体生命周期仍留在 API 边界。这是对风险和收益的工程取舍。

## 9. 当前实现边界

已经具备：

- 最小 ToolSpec 和 ToolRegistry。
- 参数校验和权限检查。
- 统一 ToolResult 和 ErrorCode。
- 有限重试与 fallback。
- 执行编号、耗时和审计记录。
- Search、Knowledge/RAG、MCP 主执行路径接入。

尚未达到：

- Registry 层硬超时中断。
- 完整租户和用户授权模型。
- 工具级资源沙箱。
- 全部分布式调用追踪。
- MCP inputSchema 深层校验和并发响应编号匹配。
- 所有工具完全由模型动态发现并自由执行。

## 10. 高频追问

### 问题：新增一个工具要做什么？

> 先确定单一职责，再定义结构化输入、稳定输出、权限和错误类型；然后实现 executor，让它返回 ToolResult；最后注册 ToolSpec，在对应 LangGraph 子工作流中受控调用，并补单元测试和审计字段。

### 问题：ToolRegistry 会不会和 LangGraph 重复？

> 不重复。LangGraph 管任务流程，ToolRegistry 管单个工具的执行治理。

### 问题：ToolRegistry 会不会和 MCP 重复？

> 不重复。MCP 是外部工具协议，Registry 是项目内部管理方式。MCP Client 本身可以作为一个工具被 Registry 管理。

### 问题：为什么不让模型自由选择所有工具？

> 健康场景更重视稳定和可测试。当前由 Router 和子工作流限制工具范围，模型只在受控边界中参与参数和表达；生产化还要增加更严格权限、沙箱和用户确认。

### 问题：最能体现工具系统技术含量的是什么？

> 不是工具数量，而是契约、权限、错误分类、有限恢复、降级归因和审计。它们让工具调用从“能运行”变成“可治理”。

更详细的迁移记录见[动作与外部工具注册表迁移评估](../technical/tool-registry/工具注册表迁移评估.md)。

## 11. 从“能调用”到“敢上线”

工具可能触发网络、子进程、文件或数据写入，是最能体现工程意识的部分。

下表是生产化治理目标，不代表当前项目已经完整具备限流、人工确认、沙箱或撤销能力。当前实现只覆盖最小参数/权限检查、有限重试、fallback、审计，以及受控的记忆写入。

| 风险级别 | 示例 | 生产化应采用的治理 |
|---|---|---|
| 只读低风险 | 查本地知识 | 参数校验、超时、审计 |
| 只读外部依赖 | Tavily、远程 MCP | 限流、有限重试、来源标记 |
| 可逆写操作 | 保存用户偏好 | 身份、幂等键、审计、撤销 |
| 高风险或不可逆 | 删除、付款、执行命令 | 最小权限、参数预览、人工确认、沙箱 |

当前项目主要覆盖前两类和受控记忆写入。`permission` 与 `allowed_permissions` 打通了最小权限原型，但生产化还要接用户身份、租户策略和真实授权服务。

### 重试必须先回答四个问题

1. 是瞬时网络错误，还是参数、权限或业务拒绝？
2. 操作是否幂等，重复调用会不会产生两次副作用？
3. 整条请求还剩多少时间预算？
4. fallback 是否过期或会误导用户？

当前 Registry 对配置的错误码做有限重试；生产化还需指数退避、随机抖动、请求截止时间、熔断和并发隔离。Registry 的 `timeout_seconds` 目前是策略元数据，不是强制中断，这一点必须如实回答。

### 工具审计应该记录什么

```text
trace_id / execution_id
tool_name + tool_version
caller + tenant + permission
脱敏参数摘要
duration_ms + attempt
result_status + error_code
fallback_used + fallback_source
```

不要把完整 Prompt、密钥、健康隐私或原始参数直接写入日志。可以保留结构化元数据，对敏感内容脱敏、哈希或采样。

### MCP 的信任边界

MCP 统一连接协议，不会自动保证工具安全。外部服务器的工具描述和返回内容都应视为不可信输入；客户端要验证响应 ID、结果结构和大小，限制允许启动的命令与环境变量，并按工具授权。高风险工具应展示参数并等待确认。当前 stdio 原型已打通协议主链，但不等于完成企业级安全体系。

## 12. 白板题：设计课程预约工具

题目：设计一个能查询健身房课程并预约的 MCP 工具。

1. 拆成 `list_classes` 只读工具与 `book_class` 写工具。
2. 定义课程 ID、用户 ID、时间段、幂等键和业务错误码。
3. 查询可自动执行；预约需要身份、scope、库存复查和用户确认。
4. 网络错误可有限重试；库存不足和参数错误不可重试；结果不确定时先查预约状态。
5. 记录 trace、耗时、尝试次数与确认人，隐藏 token 和个人信息。
6. 用成功率、重复预约率、p95 延迟、取消率和越权测试衡量质量。

评分重点是能否主动识别副作用、幂等、权限、错误恢复和审计，而不是 JSON 写得多复杂。
