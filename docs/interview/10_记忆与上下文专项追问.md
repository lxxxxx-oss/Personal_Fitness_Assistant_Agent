# 记忆与上下文专项追问

这篇文档专门帮助你回答简历中的“分层记忆与持久化会话压缩机制”。目标不是背诵复杂名词，而是能够说明：为什么要分层、当前代码怎样工作、评测发现过什么问题、边界在哪里、生产化会怎样升级。

## 1. 先理解四个概念

### 1.1 完整会话

完整会话是 SQLite 中按 `user_id + conversation_id` 保存的全部用户消息和助手回答。它解决服务重启后如何恢复、怎样审计历史的问题，但不会把全部内容直接交给模型。

替代方案是只保存在进程内存中，实现简单，但服务重启后会丢失，也不适合多实例。

### 1.2 最近对话

最近对话是当前 Prompt 中保留的近期原文。原文细节最完整，但数量必须受限。项目默认由 `MEMORY_MAX_TURNS=6` 控制，也就是最多 6 轮、12 条消息。

### 1.3 会话摘要

会话摘要是同一个 `conversation_id` 内较早消息的有界摘录。项目当前使用确定性抽取，不调用额外大模型；`last_compacted_message_id` 记录覆盖边界，下一次只处理新增消息。

可选方案包括模型摘要和向量化摘要。模型摘要更自然，但会增加延迟、成本和事实改写风险；向量检索适合从大量历史中找相关片段，但会增加索引与一致性治理。这个原型优先选择可解释、可测试的抽取式方案。

### 1.4 长期记忆

长期记忆保存跨会话仍然有价值的目标、偏好和约束，例如“不吃香菜”。它与会话摘要不同：摘要描述当前会话进展，长期记忆描述以后仍可能复用的用户事实。敏感健康信息需要先进入候选确认。

## 2. 当前链路

```text
请求到达
  → 按 user_id + conversation_id 恢复 SQLite 会话
  → 读取当前 active summary
  → 读取摘要边界之后的最近原文
  → 检索已确认的长期记忆
  → PromptBuilder 组合摘要、最近原文、长期记忆和当前问题
  → 生成回答
  → 以事务保存完整 user/assistant turn
  → 旧消息超过字符阈值时更新 active summary
  → 摘要失败则保留滑动窗口路径，不影响本轮消息落盘
```

## 3. 一分钟回答

> 我没有把所有聊天记录无限塞进 Prompt，而是分成完整会话、最近原文、会话摘要和长期记忆。SQLite 保存完整消息；最近原文保留当前任务细节；较早消息超过字符阈值后生成抽取式 active summary，并用消息边界保证增量压缩不重复；跨会话仍有价值的偏好和约束进入长期记忆。HTTP、SSE 和 WebSocket 共用保存入口，摘要加载和更新失败时回退到最近窗口。专项评测目前是 14/14，其中 6 个是多轮摘要场景，但它验证的是信息组装、边界、隔离和降级，不代表模型级摘要质量。

## 4. 连续追问与参考回答

### 追问 1：为什么不能直接传全部历史？

> 历史越长，Prompt 成本、延迟和噪声越高，旧信息还可能和新目标冲突。完整历史适合持久化和审计，不适合每轮全部注入，所以要把“保存什么”和“本轮给模型看什么”分开。

### 追问 2：滑动窗口、摘要和长期记忆有什么区别？

> 滑动窗口保存最近原文，强调细节；摘要保存当前会话中较早的进展，强调连续性；长期记忆保存跨会话仍有效的稳定事实，强调复用。三者生命周期和写入规则不同，不能混成一个向量库。

### 追问 3：一次请求具体怎么走？

> 请求先通过用户和会话 ID 恢复会话，再读取 active summary、摘要边界后的近期消息和相关长期记忆。PromptBuilder 统一组装上下文。回答完成后，用户消息和助手回答在同一事务中写入 SQLite，再以 best-effort 方式检查是否需要更新摘要。

### 追问 4：为什么先保存完整 turn，再更新摘要？

> 完整消息是事实来源，摘要只是派生结果。先保证 user/assistant 两条消息原子落盘，即使摘要失败也不会丢失真实历史；下一轮仍可用滑动窗口继续服务。

### 追问 5：怎样避免每次重复总结全部历史？

> `last_compacted_message_id` 是压缩水位线。每次只读取水位线之后的消息，把旧 active summary 和新增候选组合成新摘要，然后原子地将旧摘要标记为 superseded、写入新 active summary 并推进水位线。

### 追问 6：为什么保留 superseded 摘要，不直接删除？

> 原型中保留旧版本有利于审计和排错，可以知道摘要怎样演进。生产环境还需要结合保留期限、隐私删除和存储成本决定是否定期物理清理。

### 追问 7：为什么不用大模型做摘要？

> 当前目标是面试演示和稳定原型。确定性摘录没有额外推理延迟，结果可复现，也不会把用户事实悄悄改写。代价是语义归纳弱、可能保留冗余；如果生产化，会先建立摘要忠实度数据集，再灰度引入模型摘要，而不是直接替换。

### 追问 8：字符阈值和 token 阈值有什么差别？

> 字符数只是低成本近似，不同语言和 tokenizer 下不能精确代表模型 token。当前用它控制原型复杂度；生产化应使用实际模型 tokenizer，并为系统指令、RAG、摘要、长期记忆和当前问题设置分区预算。

### 追问 9：用户修改目标时怎么办？

> 当前抽取式摘要不会自动消解语义冲突。评测里专门加入了“半程马拉松改为恢复五公里”的场景：旧目标在摘要，新目标在最近原文，Prompt 能同时看到两者，而且新信息位置更靠后。但这只能证明冲突可见，不能证明模型一定正确采用新目标。生产化需要目标版本、`valid_from/valid_to` 或冲突替代规则。

### 追问 10：摘要会不会和最近对话重复？

> 摘要水位线之前的消息进入摘要，水位线之后的消息作为最近原文，两部分按边界去重。扩展评测时曾发现一个真实问题：内存配置保留 6 轮，但 PromptBuilder 只取 6 条消息，相当于只注入 3 轮，导致摘要未触发时出现信息断层。现在 PromptBuilder 已改为跟随 `MEMORY_MAX_TURNS`，并加入回归测试。

### 追问 11：怎样保证不同会话和不同用户不串数据？

> SQLite 查询同时带 `user_id` 和 `conversation_id`，active summary 也按这两个字段读取。评测加入了同一用户两个 conversation 的隔离标记，验证另一个会话的乳糖不耐受信息不会进入当前卧推会话。生产级安全还需要真实认证身份，不能信任客户端随意传入的 `user_id`。

### 追问 12：摘要失败会怎样？

> 加载失败时继续使用最近窗口；更新失败时完整 turn 已经落盘，只是不推进摘要边界。执行轨迹会记录 fallback，但不返回摘要正文和原始异常。这样摘要是增强能力，不是聊天主链路的单点故障。

### 追问 13：14/14 能证明什么？

> 它能证明当前固定数据集上的记忆召回、敏感候选、Prompt 注入、结构化 compact、RAG 来源和多轮摘要链路没有回归。摘要的 6 个场景覆盖早期伤病、多约束、目标修改、两次增量压缩、阈值未触发和会话隔离。它不能证明真实用户回答一定更好，也不能代表大规模并发、模型摘要忠实度或长期线上稳定性。

### 追问 14：为什么不用 Redis 或 PostgreSQL？

> SQLite 对单机原型足够，事务和查询能力比纯内存可靠，也便于演示。Redis 更适合共享热会话和 TTL，PostgreSQL 更适合多实例事务、权限和并发。生产化可以把接口保持不变，将事实存储迁移到 PostgreSQL，并用 Redis 做短期缓存。

### 追问 15：生产化最先升级什么？

> 第一优先级是认证授权、隐私删除和数据保留策略；第二是把字符预算换成真实 token 分区预算，并扩大冲突和遗漏评测；第三才是 PostgreSQL、Redis 和多实例并发控制；有了摘要忠实度基线后，再考虑模型摘要。顺序依据是先保证数据安全和可测，再增加复杂组件。

## 5. 六个多轮摘要场景怎样讲

| 场景 | 验证内容 | 不能证明什么 |
|---|---|---|
| 早期膝伤 | 较早安全约束进入摘要且不重复出现在最近原文 | 最终建议一定医学正确 |
| 多个饮食约束 | 过敏、口味和热量目标可以同时保留 | 自动识别所有隐含偏好 |
| 目标修改 | 旧目标和新目标同时对回答阶段可见 | 已完成语义冲突替代 |
| 两次增量压缩 | 旧摘要复用、水位线推进、最近三轮保留 | 多实例并发下无竞态 |
| 未达到阈值 | 不生成空洞摘要，最近窗口仍提供上下文 | 字符阈值等于精确 token |
| 会话隔离 | 同一用户不同 conversation 不串数据 | 已完成登录鉴权和租户安全 |

## 6. 白板题：画出可降级的会话压缩

先画四块：`ConversationStore`、`Summary Compactor`、`PromptBuilder`、`Long-term Memory`。再补两条边界：

```text
SQLite messages ──按水位线读取──> Summary Compactor
       │                                 │
       │完整历史                          └──> active summary
       │
       └──> recent uncompacted messages ───────┐
                                               ▼
Long-term Memory ──按用户和相关性召回──> PromptBuilder ──> LLM
```

讲解顺序：

1. SQLite messages 是事实来源，summary 是派生数据。
2. 水位线避免重复处理，active/superseded 支持演进审计。
3. Prompt 组合摘要、近期原文、长期记忆和当前问题。
4. 摘要失败走 recent fallback。
5. 单机锁只适合原型，多实例需要版本号或乐观锁。

## 7. 容易说错的话

- 不要说“使用大模型生成高质量语义摘要”，当前是确定性原文摘录。
- 不要说“精确控制 token”，当前主要是字符预算。
- 不要说“解决了记忆冲突”，当前只让新旧信息同时可见。
- 不要说“14/14 证明真实效果很好”，它只是小规模持续回归基线。
- 不要说“支持分布式高并发”，当前是单机 SQLite 原型。
- 不要说“用户隐私已经完整治理”，认证、授权和物理删除仍是生产化工作。
