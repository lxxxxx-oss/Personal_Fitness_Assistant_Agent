# 面试模拟

本文档用于记录模拟面试中的高压追问与回答。后续每轮模拟都按同一格式追加，方便复盘、背诵和修正项目口径。

## 使用方式

- 面试官问题：记录用户模拟面试官提出的原始问题。
- 回答目标：说明这道题想考察什么。
- 推荐回答：给出面试时可以直接说出口的回答。
- 回答拆解：解释为什么这样答，避免只背结论不理解。
- 有价值追问：只列出和本轮主问题相关、能考察项目理解或边界的追问。
- 当前边界：明确哪些是已实现，哪些是原型边界或后续规划。

## 记录原则

- 一切回答围绕简历和项目事实，不主动制造代码与文档矛盾。
- 先讲整体思路，再讲技术选型和工程取舍，最后说明边界与后续优化。
- 遇到未完全实现的能力，用“当前原型做到哪里、为什么这样分阶段做、生产化如何补齐”来回答。
- 不把工具名和框架名堆在一起，第一次出现关键技术点时尽量解释清楚作用和选择原因。
- 不逐字记录所有聊天内容，只沉淀有面试价值的问题。
- 无关追问、重复追问、纯口误纠偏不进入正式模拟记录；必要时只在回答拆解里简要说明。
- 每组追问必须能回扣同一个主问题，例如“完整链路”“Router 必要性”“多意图边界”，避免文档变成散点问答。
- 文档层级采用“主问题 `###` + 追问链 `####`”。同一轮面试里，后续追问如果都围绕同一个主问题展开，就放在该主问题下面，不再作为平级主问题。

## 模拟记录

### 001. 个人健身助手的完整链路

**面试官问题**

> 先整体介绍一下你的本地项目，它整个完整的链路是怎样的？

**回答目标**

这道题是在考察我能不能从端到端视角讲清项目，而不是只零散介绍 LangGraph、RAG、Motion、MCP。回答要从用户输入开始，讲到 Router、子链路执行、结果合成、记忆更新和失败降级。

**推荐回答**

> 我的项目是一个本地运行的个人健身助手 Agent。用户可以通过 Web UI、小程序或 API 输入问题，比如“减脂期间怎么吃”“分析我的深蹲动作”“查一下最近的健身研究”。请求进入后端后，系统会先构造统一状态，包含用户输入、`user_id`、历史对话、当前结果、错误信息等。
>
> 第一步是 Router，也就是意图路由。Router 判断这句话属于知识问答、饮食建议、联网搜索、动作分析，还是外部菜谱工具调用。我没有完全交给大模型路由，而是以加权规则、语义样例和歧义信号为主，可选本地 LLM classifier 作为低置信兜底。
>
> 第二步是进入对应执行链路。稳定健身知识和饮食建议走 RAG/Knowledge 链路，先从本地知识库或 Milvus 检索资料，再交给 LLM 生成回答；最新研究或实时信息走 Search 链路；图片、视频或 `.npz` 姿态数据走 Motion 链路，转换为 PoseSequence 后做归一化、FastDTW、余弦相似度和关节距离等分析；具体菜谱或外部服务走 MCPTool 链路。
>
> 第三步是结果写回统一状态并生成最终回答。每个子链路会把检索片段、搜索结果、动作指标或工具返回数据写回状态，LLM 再把结构化结果组织成用户能看懂的自然语言。如果 Tavily、Milvus 或 MCP Server 不可用，会走 mock 或 fallback，不让整个主流程崩掉。
>
> 第四步是记忆更新和返回。回答生成后，系统会把本轮问答写入按 `user_id` 隔离的滑动窗口记忆，默认保留最近 6 轮。下次用户继续问“那晚餐呢”，系统可以利用近期上下文。
>
> 所以这个项目不是“用户输入 -> 大模型回答”，而是“用户输入 -> Router 判断任务 -> RAG/Search/Motion/MCP 等子链路执行 -> 结果合成 -> 记忆更新 -> 返回”。核心价值是把健身场景里不同性质的能力组织成一个可控、可评测、可降级的 Agent 系统。

**回答拆解**

- 先用一句话定位项目：本地个人健身助手 Agent。
- 按链路讲，而不是按技术名词堆砌。
- 每个子链路都说明“为什么需要它”：RAG 管稳定知识，Search 管最新信息，Motion 管姿态算法，MCP 管外部工具。
- 主动讲 fallback 和记忆，体现工程完整度。

**有价值追问**

- 为什么不直接把所有问题交给一个 Prompt？
- Router 判断错了怎么办？
- RAG 和 Search 的边界是什么？
- Motion 为什么不能只靠 LLM 判断？
- 这个项目哪些能力是当前已实现，哪些是演示或规划？

**当前边界**

- 当前是本地面试原型，不是线上生产服务。
- 媒体 Motion 和对话 Router 仍有入口差异，媒体上传主要通过独立接口。
- 真实模型回答质量、真实 Milvus 召回率、Motion 标准动作库仍需要更多真实数据验证。

#### 追问 001-1. 为什么需要意图路由

**面试官问题**

> 为什么会想要做这个意图路由？有必要吗？

**回答目标**

这道题是在质疑 Router 是否过度设计。回答要说明不是为了炫技，而是因为项目包含异构能力，不同任务需要不同数据源、算法和工具。

**推荐回答**

> 有必要，但我不会说所有大模型项目都必须做 Router。这个项目需要 Router，是因为它不是单一问答，而是多能力健身助手。
>
> 比如“蛋白质有什么用”适合走 RAG 问答；“查一下最近减脂研究”需要联网搜索；“分析我的深蹲视频”需要 Motion 姿态算法；“推荐一道高蛋白晚餐”可能需要 MCP 菜谱工具；“我减脂怎么吃”则需要饮食画像和营养建议。它们表面都是自然语言，但背后依赖完全不同。
>
> 如果没有 Router，所有问题都会进入同一条 LLM 链路。模型可能用本地知识回答最新信息，导致过时；也可能凭语言描述判断动作标准性，而动作分析其实需要关键点和数值算法；外部工具失败、检索失败也会混在一起，难以排查。
>
> Router 的价值主要有三个：第一是准确性，不同任务调用不同工具和数据源；第二是可控性，每个子链路可以单独降级；第三是可评测性，Router 可以用离线样本评测，而不是靠感觉调 Prompt。
>
> 当然 Router 会增加复杂度，也可能误路由。所以我没有默认使用完全开放的 LLM Router，而是用加权规则、语义样例和歧义信号保证稳定；本地 Qwen classifier 做过 A/B，收益不明显且延迟高，所以默认关闭。

**回答拆解**

- 先承认不是所有项目都需要 Router，避免绝对化。
- 用具体用户问题解释不同能力边界。
- 把 Router 的价值落到准确性、可控性、可评测性。
- 主动承认 Router 的代价和误路由风险。

**有价值追问**

- 如果 Router 判断错了会怎样？
- 为什么不用 LLM 直接判断意图？
- Router 的规则是不是太死？
- 你怎么证明 Router 有效果？

**当前边界**

- Router 是受控规则和样例驱动，不是完全智能的自然语言 Planner。
- 当前评测集是项目自建样本，不能代表真实线上分布。
- 低置信、混合表达和新型用户说法仍需要持续补样本。

#### 追问 001-2. 本地 LLM Router 如何服务网站和小程序用户

**面试官问题**

> 你这个 LLM Router 是你本地的模型，别人访问你的网站或者小程序的话，它怎么发挥作用？

**回答目标**

这道题是在考察部署边界和前后端职责。回答要说明模型不跑在用户手机或浏览器，而是在服务端后端运行；同时强调当前默认主路径不是 LLM Router。

**推荐回答**

> 这里要区分“用户在哪里访问”和“模型在哪里运行”。别人访问网站或小程序时，LLM Router 不是跑在用户手机或浏览器上的，而是跑在后端服务里。
>
> 前端只负责把用户问题发到 FastAPI 后端，比如 `/chat` 接口。后端收到请求后，先执行 Router，再决定走 RAG、Search、Motion、MCP 哪条链路。如果后端部署在一台有本地 Qwen 模型的机器上，LLM Router 就可以在这台服务器上运行，前端用户不需要安装模型。
>
> 但我这里要明确边界：当前项目默认工作的不是 LLM Router，而是确定性 Router，也就是加权规则、语义样例和歧义信号。LLM Router 是可选兜底，默认关闭。原因是本地 Qwen classifier 延迟较高，在当前评测样本上没有明显提升准确率。
>
> 如果生产化部署，我会把本地模型从 Web 后端里拆出来，做成独立 model server。前端请求仍然进后端，后端再调用推理服务做分类或生成。这样可以统一管理模型加载、并发队列、超时、限流和资源隔离。

**回答拆解**

- 先讲部署位置：模型在服务端，不在用户端。
- 再讲当前主路径：确定性 Router 默认开启，LLM Router 默认关闭。
- 最后讲生产化：独立推理服务，而不是 Web 进程里硬扛模型。

**有价值追问**

- 小程序端有没有模型能力？
- 如果服务器没有 GPU 怎么办？
- 本地模型延迟高怎么处理？
- 为什么不直接用云端模型 API？

**当前边界**

- 当前项目默认不依赖 LLM Router，因此前端访问不受本地 Qwen 是否开启影响。
- 本地推理适合面试演示和原型，生产化要拆 model server。
- 前端只负责请求和展示，不承担模型推理。

#### 追问 001-3. 如何判断 Router 是否正确

**面试官问题**

> 你怎么判断路由是否正确？

**回答目标**

这道题是在考察评测意识。回答要说明不是主观判断，而是通过离线评测集、困难样本、混淆分析和多意图字段验证。

**推荐回答**

> 我判断 Router 是否正确，不是靠“看着像对”，而是做离线评测集。
>
> 首先我定义每类 intent 的边界：`chat` 负责普通健身知识和概念解释，`diet` 负责减脂、增肌、热量和饮食安排，`search` 负责最新信息和需要联网查证的问题，`motion` 负责动作姿势和媒体姿态分析，`mcp` 负责具体菜谱和外部工具调用。
>
> 然后我人工构造样本并标注期望 intent，比如“蛋白质有什么作用”是 `chat`，“一天吃多少蛋白质比较合适”是 `diet`，“查一下最近减脂研究”是 `search`，“帮我看看深蹲哪里不对”是 `motion`。评测脚本会逐条送进 Router，对比实际输出和标注。
>
> 我还分两类样本：绿色回归集用于保证常见场景不退化；challenge set 专门放困难样本，比如低置信表达、隐式意图、多意图请求、diet 和 mcp 边界、chat 和 diet 边界。多意图场景不只看 primary intent，还看 secondary intents 和 route plan 是否符合预期。
>
> 所以我的判断标准包括：单条样本是否命中标注，不同 intent 的 precision/recall 和混淆情况是否合理，困难样本和多意图样本是否符合预设边界。
>
> 当然这个评测集是自建的，不能代表真实线上准确率。它的价值是让 Router 策略可回归、可解释、可持续优化。生产化还需要收集真实用户 query，人工标注后持续评测。

**回答拆解**

- 先强调“离线评测”，体现工程意识。
- 说明标注边界，避免样本凭感觉。
- 区分绿色集和 challenge set。
- 主动承认自建评测集不等于线上准确率。

**有价值追问**

- 样本是谁标注的？有没有主观性？
- 只看 accuracy 够不够？
- Router eval 里的 100% 是否代表线上准确？
- 多意图样本怎么评估？
- 低置信样本应该算错还是澄清？

**当前边界**

- 当前评测集规模有限，主要服务项目回归和面试展示。
- 评测数字不能夸大成线上准确率。
- 后续需要真实用户 query、人工标注和分场景指标。

#### 追问 001-4. 多意图问题如何处理

**面试官问题**

> 你的多意图问题是怎么处理的？

**回答目标**

这道题是在考察 Router 是否只能处理单一 intent，以及多任务请求是否有状态和执行设计。回答要强调“受控 multi-intent routing”，不是开放式自主 Planner。

**推荐回答**

> 多意图我没有做成完全开放的“让模型自由规划很多步”，而是做成受控的 multi-intent routing。
>
> 健身助手里确实会出现一句话包含多个任务，比如“查一下最近减脂研究，然后告诉我饮食怎么调整”“先分析深蹲动作，再给我纠正建议”。如果只输出一个 intent，就会丢掉一部分用户需求。
>
> 我的处理分三步。第一步，Router 先判断主意图，也就是最优先执行的任务。比如用户说“先分析深蹲动作，然后给建议”，主意图就是 `motion`。
>
> 第二步，Router 会识别次意图和 route plan。状态里不只是 `intent`，还会记录 `_primary_intent`、`_secondary_intents`、`_route_plan`、`_multi_intent_reason` 和 `_needs_clarification`。
>
> 第三步，执行层不是所有组合都执行，而是只允许少数白名单组合，比如 `search -> diet`、`search -> chat`、`motion -> chat`、`motion -> diet`。这样做是为了控制状态传递、失败范围和延迟。
>
> 执行时，每个子任务会单独记录结果。如果某一步失败，系统会保留成功结果，并在最终合成时说明边界。最后由 synthesis 节点把多个子任务结果合成为一个回答。
>
> 所以它不是 AutoGPT 式自主规划，而是“识别主意图、记录次意图、只执行高价值白名单组合、最后统一合成”的受控多任务编排。

**回答拆解**

- 明确多意图不是无限规划，而是受控路由。
- 讲清内部状态字段，体现不是口头概念。
- 解释为什么要白名单：稳定、可测、可降级。
- 说明部分失败时如何处理。

**有价值追问**

- 白名单组合为什么只有这几种？
- 多意图执行的中间结果怎么传递？
- 如果第一步失败，第二步还执行吗？
- 多结果合成会不会产生幻觉？
- 为什么不直接用 Planner？

**当前边界**

- 当前只支持少数两步组合，不支持任意多步计划。
- 多意图结果合成依赖 LLM，仍需要提示约束和测试样本保护。
- 更复杂组合需要明确状态依赖和失败策略后再开放。

#### 追问 001-5. 多意图最多两个，更多意图如何处理

**面试官问题**

> 所以你的多意图其实也最多实现两个意图，如果用户同一个问题有更多意图的话，你的路由会怎么处理？

**回答目标**

这道题是在追问项目边界。面试官想确认我是不是把“多意图”包装得过大，也想看我能不能解释为什么没有做无限多步 Planner。回答时要主动承认当前执行层最多支持两步组合，再说明这是稳定性、延迟、状态复杂度和健康场景安全性的取舍。

**推荐回答**

> 对，这个问题问得很准。当前项目里的多意图不是无限多步 Planner，执行层最多只支持两个意图的受控组合。
>
> 如果用户一句话里有三个甚至更多意图，比如“先分析我的深蹲视频，再查一下最近纠正方法，然后给我安排饮食和训练计划”，这句话可能包含 `motion -> search -> diet/chat`。当前系统不会把这几个任务全部串起来执行。
>
> 我的处理策略是：第一，Router 会先识别主意图，也就是最应该优先处理的任务。像这个例子里，“先分析深蹲视频”通常会被作为主意图 `motion`。第二，Router 会尽量记录次意图和 route plan，但执行层只接受白名单里的两步组合。如果 route plan 超过两步，或者不是白名单组合，就不会强行执行完整链路。第三，如果组合不支持，系统会退回到主意图执行，并记录 warning，比如 `unsupported_route_plan`。最终回答应该围绕主任务给结果，同时提示用户后续可以继续问饮食、搜索或训练计划部分。
>
> 我这样设计是有意取舍。多意图不是越多越好。开放任意多步会带来几个问题：状态传递会复杂，比如 Search 的结果要不要传给 Diet，Motion 的分析结果要不要传给训练建议；失败处理会复杂，多个子任务里只要有一个失败，最终回答怎么合成和提示边界都要设计；延迟也会不可控，每多一个子图就可能多一次搜索、检索、工具调用或 LLM 生成；另外健身健康场景不能让模型自由规划太多步，否则容易生成看似完整但没有验证的建议。
>
> 所以我当前策略是：能识别多意图，但只执行少数高价值、可测试的两步组合。更多意图先收敛到主意图，或者引导用户拆分问题。后续生产化可以升级成受控 Planner，但我不会直接上开放式 AutoGPT，而是定义最大步数、白名单 DAG、超时、失败降级和评测样本，再逐步扩展。

**回答拆解**

- 先承认“最多两个意图”这个事实，不回避边界。
- 把“没有支持更多意图”解释为工程取舍，而不是能力缺失。
- 明确当前策略：识别更多信号，但执行只走主意图或白名单两步组合。
- 用具体例子说明三意图输入会如何降级，避免回答太抽象。
- 最后给生产化路线：受控 Planner、最大步数、白名单 DAG、超时和评测，而不是开放式自由规划。

**有价值追问**

- 如果用户强烈要求一次完成多个任务，为什么不直接全部执行？
- route plan 超过两步时，系统会不会丢失用户需求？
- 多意图结果合成怎么避免前后矛盾？
- 为什么不用 LLM 做 Planner，直接让它自己决定执行顺序？
- 如果后续要支持三步以上，你会优先开放哪些组合？
- unsupported route plan 的 warning 当前是否会暴露给用户？

**当前边界**

- 当前执行层只支持少数两步白名单组合，不是完整多步 Agent Planner。
- 超过两步或非白名单组合时，以主意图执行为主，并保留内部 warning。
- 当前项目更强调“可控、可测、可降级”，不追求任意多步自动化。
- 生产化扩展前必须补 route plan 评测、状态依赖定义、失败合成策略和最大执行步数限制。

### 002. 项目内部工具系统的定义与执行链路

**面试官问题**

> 你项目里的工具系统是怎么做的？比如一个新工具从定义、注册、被 Agent 发现，到最后被调用，完整流程是什么？

**回答目标**

这道题不是考察外部 MCP 怎么调，而是考察我是否理解自己项目里的工具抽象：工具职责、输入 schema、权限边界、执行器、结果回传，以及为什么当前没有过早抽象成完整 `ToolRegistry`。

**推荐回答**

> 我自己定义工具时，不是简单写一个函数让子图随便调，而是尽量把每个工具设计成一个有明确契约的能力单元。核心是五个部分：工具职责、输入 schema、权限边界、执行函数和统一结果。
>
> 第一是工具职责。每个工具只解决一个明确问题，比如 SearchTool 负责搜索，Retriever 负责知识库检索，MotionTool 负责姿态序列分析和相似度计算，PoseEstimator 负责从图片或视频里提取人体关键点。工具只做确定性执行，不负责最终话术总结；总结和表达交给子图里的 LLM 节点。
>
> 第二是输入 schema。当前项目没有做一个独立 JSON Schema Registry，这是边界。但每个工具在代码层面都有输入契约：函数签名定义参数，`check_str_nonempty`、`check_int_range`、`check_float_range` 做基础校验，复杂输入用 `PoseSequence` 这类结构化模型，API 层用 Pydantic 约束请求字段和上传文件。比如姿态分析工具不是随便传文本，而是要求 `.npz` 或 PoseSequence 这种可计算数据。
>
> 第三是权限边界。工具不能让 LLM 任意执行系统命令或访问任意文件。SearchTool 只调用 Tavily 或 mock 搜索，MotionTool 只处理上传文件和指定标准动作目录，Retriever 只检索项目知识库，MCPClient 也只允许连接配置中的 server command。LLM 不能直接越过子图去操作文件系统或网络。
>
> 第四是执行器。工具不是自己决定什么时候运行，而是由 LangGraph 子图节点调用。Search 子图先改写 query，再调用搜索工具；Motion 子图先解析意图，再调用姿态加载和相似度计算；Knowledge 子图先构造 retrieval query，再调用 Retriever。也就是说，工具是底层能力，执行顺序和状态流由子图控制。
>
> 第五是结果回传。我给工具统一了 `ToolResult` 返回结构，成功时 `ok=True`，数据放在 `data`；失败时带 `error_code` 和 `error_message`。上层子图可以统一判断成功失败，成功就把结果写入 `RouterState`，比如 `_retrieved`、`_search_results`、Motion metrics；失败就写入错误或降级原因。最后 LLM 只负责把这些结构化结果解释给用户，而不是替代工具执行。
>
> 所以我项目里的工具定义方式可以概括为：工具是一个有明确输入、输出、权限和错误契约的确定性能力单元。LLM 不直接执行工具，而是由 LangGraph 节点在受控流程中调用工具；工具结果通过 `ToolResult` 和 `RouterState` 回传给后续节点。

**回答拆解**

- 这个问题不要优先讲 MCP，要先讲自己项目内部工具的定义方式。
- 重点围绕五个词：职责、schema、权限、executor、结果回传。
- 要承认当前没有独立 JSON Schema Registry，而是用函数签名、validator、结构化模型、Pydantic 和 `ToolResult` 共同承担契约。
- 把 LLM 和工具的边界讲清：LLM 做理解、规划和解释，确定性工具做执行。

**有价值追问**

- 为什么不统一抽象成一个 `ToolRegistry` 类？
- 如果工具调用失败，比如文件不存在、命令超时、权限不够，Agent 是直接报错还是降级？
- 恢复和重试有没有边界，比如重试几次后放弃？
- 工具参数是谁校验的？LLM 传错参数怎么办？
- 权限边界如何防止 LLM 调用危险操作？

**当前边界**

- 当前是“约定式工具系统”，还不是生产级显式 Tool Registry。
- 工具 schema 分散在函数签名、validator、Pydantic 和数据模型里，没有统一元数据中心。
- 权限主要靠固定子图边界、配置项和工具函数约束，缺少完整鉴权、审计和沙箱。
- 当前设计适合面试原型和本地演示；生产化需要把工具元数据、权限、超时、重试和审计集中管理。

#### 追问 002-1. 为什么不统一抽象成 ToolRegistry

**面试官问题**

> 为什么不统一抽象成一个 ToolRegistry 类？

**回答目标**

这道题是在考察工程取舍。回答要说明不是不知道可以抽象，而是当前工具数量、调用方式和项目阶段不一定值得过早引入统一 registry；同时要讲清生产化时会怎么设计。

**推荐回答**

> 我没有一开始统一抽象成 `ToolRegistry`，主要是因为当前项目的工具数量和调用方式还没有复杂到必须引入这层抽象。这里是工程取舍，不是我不知道可以这么做。
>
> 当前工具差异比较大。Retriever 是长期共享实例，启动时要加载知识库；SearchTool 依赖 API key 和 mock fallback；MotionTool 处理 `.npz`、图片、视频和 NumPy 数组；MCPClient 涉及进程启动、JSON-RPC 和外部协议。如果为了统一而强行塞进一个 `ToolRegistry`，早期会引入很多适配代码，反而影响主链路稳定。
>
> 所以我当前采用的是轻量方式：每个子图直接调用自己领域内的工具，但统一 `ToolResult` 和 `ErrorCode`。也就是说，我先统一了结果契约和错误处理，而不是过早统一注册层。
>
> 这样做适合个人项目和面试原型，因为最重要的是链路可运行、边界能讲清、核心能力有验证。如果一开始就做完整 ToolRegistry，可能显得工程化，但不一定提升核心功能，还可能变成为了架构而架构。
>
> 不过随着工具数量增加，ToolRegistry 是合理的治理层。现在我已经补了最小原型：每个工具可以抽象成 `ToolSpec`，包含 `name`、`description`、`input_schema`、`permission`、`executor`、`timeout_seconds`、`retry_policy`。Registry 负责注册、列出工具、校验参数、检查权限、执行工具和记录审计。当前 Search 与 Knowledge/RAG 已经接入 Registry，每次执行会记录 `execution_id`、`duration_ms`、attempts 和 fallback 归因；其他子图还没有全面替换，但后续新增或迁移工具就可以先注册 spec，再由执行器统一处理。

**回答拆解**

- 先承认 `ToolRegistry` 是合理方向。
- 再解释当前没做是因为工具异构、数量有限、项目阶段不需要过早抽象。
- 说清当前已经统一的部分：`ToolResult`、`ErrorCode`、子图边界。
- 最后给出生产化设计，证明不是没有工程意识。

**当前边界**

- 当前没有统一 `ToolRegistry` 类。
- 新增内部工具仍需要在对应子图中显式 import 和调用。
- 工具权限、超时、重试、审计没有集中在 registry 层。
- 后续生产化应引入 `ToolSpec + ToolRegistry + ToolExecutor`。

#### 追问 002-2. 工具调用失败时如何处理

**面试官问题**

> 如果工具调用失败，比如文件不存在、命令超时、权限不够，你的 Agent 是直接报错，还是会做处理？

**回答目标**

这道题是在考察错误处理和降级策略。回答要说明工具失败不是异常终点，而是状态流中可处理的结构化结果。

**推荐回答**

> 不会直接把底层错误原样丢给用户。我会分层处理：工具失败不是异常终点，而是 Agent 状态流里的一种可处理结果。
>
> 项目里工具调用一般返回统一 `ToolResult`，包含 `ok`、`data`、`error_code`、`error_message`、`meta`。如果成功，子图继续用 `data` 生成回答；如果失败，错误会归类为 `DATA_NOT_FOUND`、`NETWORK_ERROR`、`CONFIG_MISSING`、`PERMISSION_DENIED`、`INVALID_PARAM`、`INTERNAL_ERROR` 等。
>
> 比如文件不存在，Motion 工具或接口层会返回 `DATA_NOT_FOUND`、404 或 422，告诉用户缺少标准动作文件或上传格式不对。比如命令超时，MCP Client 读取 stdout 有 timeout，外部 Server 不响应会返回 `NETWORK_ERROR`，真实 Server 不可用时可以 fallback 到 mock。比如权限不足，Agent 不会让 LLM 绕过权限执行别的命令，而是把它当成工具不可用处理。
>
> 我把失败分三类：可恢复失败，比如 Tavily 没配置、MCP Server 没启动、Milvus 不可用，会 fallback 到 mock、MemoryRetriever 或通用回答；用户输入问题，比如文件格式不对、参数缺失，会返回明确提示；不可安全恢复的问题，比如权限不足、关键模型文件缺失，会明确说明当前能力不可用，不强行继续。
>
> 所以完整链路是：工具失败 -> 结构化错误 -> 子图判断能否降级 -> 能降级就 fallback -> 不能降级就明确说明边界。统一 `ToolResult` 的价值就在这里，它让失败变成可观察、可处理、可测试的一部分。

**回答拆解**

- 不要说“失败就 try/except 打印错误”，要讲结构化错误。
- 用具体场景说明：文件不存在、命令超时、权限不够。
- 区分可恢复、用户输入错误、不可安全恢复三类。
- 说明失败结果会写回状态，由子图决定降级或中止。

**当前边界**

- 不是所有工具都有同等完整的降级策略。
- 部分 API 层错误会直接转成 HTTP 4xx/5xx，而不是自然语言回答。
- 权限处理是原型级配置边界，不是完整安全沙箱。

#### 追问 002-3. 恢复和重试是否有边界

**面试官问题**

> 你的恢复和重试有边界吗？比如重试几次后会放弃？

**回答目标**

这道题是在考察是否会无限重试、无限等待，以及是否理解 retry policy 应该按错误类型区分。回答要诚实说明当前有超时和 fallback，但还没有统一 retry policy。

**推荐回答**

> 有边界，而且我不会设计成无限重试。当前项目更准确的说法是：已经有 fallback 和超时边界，但还没有做完整统一的 retry policy。
>
> 比如 MCP Client 调外部工具时有超时控制，Server 如果 stdout 一直不返回，读取线程会在 timeout 后断开，不会一直卡住。真实 MCP Server 连接失败时，也不会无限重试，而是 fallback 到 mock 模式，保证演示链路可用。
>
> Milvus 也是类似思路。启用 Milvus 后，如果连接、建表或搜索失败，系统会 fallback 到 MemoryRetriever，而不是反复重连卡住用户请求。Tavily 没配置或调用失败，也会走 mock 或错误返回。
>
> 所以当前边界是：单次工具调用有超时，外部依赖失败后走 fallback，不做无限重试，不让一个外部依赖拖死整个请求。
>
> 但我也承认，当前项目没有抽象出统一 retry policy，比如 `max_retries=2`、指数退避、哪些错误可重试、哪些错误不可重试。生产化时我会把 retry policy 做成 ToolRegistry 的一部分，每个工具配置 `timeout_seconds`、`max_retries`、`retryable_error_codes`、`backoff_strategy` 和 `fallback_strategy`。执行器统一处理：成功就返回，失败且可重试就有限重试，仍失败就 fallback；参数错误、权限不足、文件不存在这类不可重试错误则直接返回结构化错误。

**回答拆解**

- 先明确不会无限重试。
- 当前已有的是超时和 fallback，不要夸大成完整 retry framework。
- 讲清哪些错误适合重试：网络波动、临时服务不可用。
- 讲清哪些错误不该重试：参数错误、权限不足、文件不存在。
- 把生产化方案和 ToolRegistry 关联起来。

**当前边界**

- 当前没有统一 `max_retries/backoff/retry_on` 配置。
- 外部依赖主要靠一次调用失败后 fallback，而不是多次重试。
- 更完整的策略应放到未来 `ToolRegistry` 或 `ToolExecutor` 中统一管理。

### 003. 上下文过长与记忆压缩策略

**面试官问题**

> 你项目中如果遇到了上下文过长是怎么处理的？有做上下文的压缩相关的工作吗？

**回答目标**

这道题是在考察上下文管理能力。回答要区分“已经做的上下文规模控制”和“尚未实现的自动摘要压缩”，不能把滑动窗口包装成完整长期记忆系统。

**推荐回答**

> 我项目里确实考虑了上下文过长的问题，但当前没有做复杂的自动摘要压缩，更准确地说是做了几层上下文裁剪和边界控制。
>
> 第一层是会话记忆窗口控制。项目里用滑动窗口记忆，按 `user_id` 隔离，只保留最近 6 轮对话。这样用户连续对话时，系统能保留短期上下文，但不会把历史全部塞进 Prompt。比如用户前面说“我想减脂”，后面问“那晚餐呢”，最近几轮上下文是有价值的；但几十轮以前的内容通常噪声更大，也会增加 token 成本。
>
> 第二层是 RAG 检索控制。知识库不会整篇注入 Prompt，而是先分块、embedding 检索、Top-K 和 threshold 过滤，只把相关 chunk 放进去。`retriever_top_k` 和 `retriever_threshold` 都可以配置，用来控制进入 Prompt 的资料数量和质量。
>
> 第三层是搜索结果控制。Search 子图不会把所有网页原文塞进上下文，而是使用标题、摘要、URL 等结构化结果，再交给 LLM 合成回答。
>
> 第四层是 Prompt 长度边界。本地 LLM Loader 有最大 Prompt 字符数限制，避免超长输入拖垮本地模型。
>
> 所以如果说自动摘要式压缩，比如把历史对话总结成 memory summary，当前没有完整实现；但如果说工程上的上下文控制，已经有滑动窗口、RAG Top-K、threshold 过滤、搜索摘要化输入和 Prompt 最大长度限制。
>
> 我没有一开始做复杂摘要压缩，是因为项目是本地面试原型，主链路更需要稳定。摘要压缩本身也有风险：健身场景里身高、体重、伤病、忌口这些关键信息如果被摘要漏掉或总结错，会影响建议安全性。生产化时，我会升级为“短期原文窗口 + 结构化用户画像 + 可追溯摘要记忆”的分层上下文管理。

**回答拆解**

- 先承认没有完整自动摘要压缩，避免夸大。
- 把已有能力说成“上下文控制”：滑动窗口、RAG 检索过滤、搜索摘要输入、Prompt 长度限制。
- 解释为什么不急着做 summary：摘要可能丢关键信息，健身健康场景有安全边界。
- 给出生产化方向：短期窗口、长期画像、摘要记忆分层。

**有价值追问**

- 为什么不做上下文压缩，滑动窗口会不会太粗暴？
- 6 轮窗口为什么是 6，不是 10 或 20？
- 如果用户早期说过伤病或忌口，后面窗口丢掉了怎么办？
- RAG 检索本身也会带来上下文，如何控制进入 Prompt 的 chunk？
- 自动摘要会不会引入幻觉或错误记忆？

**当前边界**

- 当前没有完整 memory summary 或长期记忆压缩模块。
- 滑动窗口只能保证短期连续性，不能保证长期用户画像不丢。
- Prompt 长度限制是保护性边界，不是智能压缩策略。
- 后续需要结构化画像抽取、摘要记忆和可追溯更新策略。

#### 追问 003-1. 为什么不做上下文压缩，滑动窗口是否太粗暴

**面试官问题**

> 为什么不做上下文压缩，这种滑动窗口的方式会不会太粗暴了？很多信息都会被直接丢掉，你是怎么考虑这一部分的问题的？

**回答目标**

这道题是在追问取舍。回答要主动承认滑动窗口粗糙，但说明它是原型阶段的稳定方案；真正长期有价值的信息应该结构化沉淀，而不是简单做自由文本摘要。

**推荐回答**

> 会，滑动窗口确实是比较粗的方案，我不会把它说成最优解。我的考虑是：当前阶段先用滑动窗口保证系统稳定，再把真正需要长期保留的信息结构化沉淀，而不是一上来做自由文本摘要压缩。
>
> 我没有优先做复杂上下文压缩，主要有三个原因。第一，当前项目是本地面试原型，典型对话轮次不会特别长，相比复杂 memory summary，我更需要先保证 Router、RAG、Motion、Search、MCP 主链路稳定可回归。
>
> 第二，摘要压缩本身有信息损失风险。健身场景里身高、体重、训练目标、伤病、忌口、过敏都很关键。如果用 LLM 随意总结旧对话，可能把“膝盖偶尔不舒服”这种安全相关信息漏掉，后续深蹲建议就可能不合适。
>
> 第三，并不是所有历史都值得保留。很多对话只是一次性解释或寒暄，长期保存反而污染上下文。所以我的策略不是把所有旧上下文都总结，而是区分短期上下文和长期稳定信息。
>
> 当前滑动窗口的作用，是保留最近几轮对话连续性，避免 Prompt 无限膨胀。它确实会丢早期信息，所以我认为它只是第一阶段方案，不是最终记忆系统。
>
> 如果要改进，我不会简单把超过 6 轮的内容丢给 LLM 做 summary，而是做分层记忆：第一层是短期窗口，保留最近 N 轮原文；第二层是用户画像，把长期稳定且重要的信息结构化存起来，比如目标、身高、体重、伤病、忌口、过敏；第三层才是摘要记忆，用于保存较长训练计划讨论或阶段性反馈，并带上时间戳、来源轮次和覆盖规则。
>
> 所以我承认滑动窗口粗暴，但它是可控的原型阶段选择。它的优点是简单、稳定、不会引入摘要幻觉；缺点是会丢长期信息。后续优化不是单纯扩大窗口，而是把真正重要的信息抽取成结构化长期状态，再配合可追溯摘要记忆。

**回答拆解**

- 先承认滑动窗口粗糙，态度不能防御。
- 把“不做摘要”解释为风险控制，而不是没想到。
- 重点强调健身场景的信息安全：伤病、过敏、忌口、目标不能被摘要丢失。
- 改进方案不能只是“窗口变大”，而是分层记忆。

**当前边界**

- 当前滑动窗口会丢掉窗口外的早期对话。
- 项目还没有结构化长期用户画像存储。
- 还没有摘要版本管理、来源轮次和冲突更新策略。
- 生产化需要把用户长期信息从对话流中抽取出来，形成可校验状态。

### 004. RAG、Memory 与 Context Engineering 的关系

**面试官问题**

> 你怎么看 RAG、Memory 和 Context Engineering？这三者分别解决什么问题？在你的个人健身助手项目里它们是怎么配合的？

**回答目标**

这道题是在考察我是否能区分概念层级。回答要说明 RAG、Memory 和 Context Engineering 不是同一个东西：RAG 解决外部知识，Memory 解决用户历史和个性化状态，Context Engineering 解决如何组织所有上下文。

**推荐回答**

> 我理解这三个概念不是同一层东西，它们解决的问题不一样。
>
> RAG 解决的是“外部知识怎么进入模型”。大模型本身的知识可能过时，也可能不包含项目里的私有知识。RAG 的做法是先把知识库文档切成 chunk，转成 embedding，用户提问时检索相关片段，再把片段放进 Prompt，让模型基于证据回答。在我的项目里，RAG 主要用于健身知识和营养知识，比如训练原则、动作注意事项、蛋白质摄入、减脂饮食原则。它关注的是“这个问题需要哪些资料作为依据”。
>
> Memory 解决的是“用户历史和个性化状态怎么延续”。RAG 检索的是知识库，不一定知道这个用户是谁、之前说过什么。Memory 关注的是对话历史和用户上下文。我的项目里当前是滑动窗口记忆，按 `user_id` 保存最近 6 轮对话，让用户连续追问时系统能承接上下文，比如用户先说自己想减脂，后面问“那晚餐呢”，系统可以从近期记忆里知道“晚餐”是围绕减脂目标展开的。
>
> Context Engineering 解决的是“把哪些信息、以什么结构、在什么时候放进 Prompt”。它比 RAG 和 Memory 更上层。RAG 和 Memory 都只是上下文来源，Context Engineering 负责组织这些来源。比如一轮回答里，可能同时有用户当前问题、最近对话、RAG 检索片段、搜索结果、工具返回结果、系统规则、安全边界。Context Engineering 要决定这些内容的优先级、顺序、长度限制和格式。
>
> 所以三者的关系可以概括为：RAG 从外部知识库取证据，Memory 从用户历史里取上下文，Context Engineering 组织所有上下文，让模型能稳定使用。
>
> 在我的个人健身助手项目里，用户提问后 Router 先判断任务类型。如果是稳定健身知识或饮食建议，就进入 Knowledge/RAG 链路，从本地知识库或 Milvus 检索相关 chunk；同时按 `user_id` 拿最近几轮 Memory；然后 Prompt 会把角色规则、安全边界、RAG 片段、对话历史和当前问题组织起来，再交给 LLM 生成回答。
>
> 比如用户第一轮说“我 170cm，80kg，想减脂”，后面问“那晚餐怎么吃”。Memory 负责提供“用户想减脂”这个近期上下文；RAG 负责提供减脂饮食、蛋白质、热量控制等通用知识；Context Engineering 负责把它们组织成一个 Prompt：先给角色和回答规则，再给参考资料，再给对话历史，最后给当前问题。这样模型不是凭空回答，而是同时考虑知识证据和用户上下文。

**回答拆解**

- 先明确三者不是同一层概念。
- RAG 强调“外部知识证据”，Memory 强调“用户历史和个性化”，Context Engineering 强调“组织输入”。
- 用项目里的具体例子说明配合，而不是只背概念。
- 最后主动说明当前边界，避免把轻量实现说成完整上下文系统。

**有价值追问**

- Memory 和 RAG 都是“取上下文”，区别到底在哪里？
- Context Engineering 是不是只是 Prompt Engineering 换个名字？
- 如果 RAG 片段和 Memory 中的用户偏好冲突，谁优先？
- 当前滑动窗口 Memory 能不能支撑长期个性化？
- RAG 检索出来的内容一定要放进 Prompt 吗？

**当前边界**

- 当前 Memory 是最近 6 轮滑动窗口，不是完整长期记忆系统。
- RAG 已有 Memory/Milvus 双后端和 Top-K/threshold 控制，并完成了真实链路效果评测；Recall@K、MRR、忠实度等更系统的离线指标仍是后续扩展项。
- Context Engineering 当前主要体现在 Prompt 结构、检索过滤、来源约束和安全提示，还不是完整自动上下文规划器。
- 生产化方向是把短期 Memory、结构化用户画像、RAG 证据、工具结果和安全策略统一成更显式的上下文组装层。

### 005. Agent 评测体系与任务成功判断

**面试官问题**

> 你的 Agent 有没有评测体系？你怎么设计 benchmark、case、指标？怎么判断一次改动真的让 Agent 变强了？

**回答目标**

这道题是在考察我是否有评测意识。回答要说明项目不是只靠 demo case 和主观感觉调 Prompt，而是按模块、路由、工具、链路和回答质量分层评测；同时不能夸大成完整线上 Agent 评测平台。

**推荐回答**

> 我的项目有评测体系，但我不会把它包装成完整线上级评测平台。当前更准确地说，是围绕面试原型做了几类离线评测和回归测试，用来证明系统不是只靠感觉调 Prompt。
>
> 第一层是单模块单元测试，验证确定性逻辑。比如 Motion 的姿态数据校验、FastDTW 相似度计算、PoseSequence 转换、MCP Client 的参数和 fallback、Retriever 的搜索结果格式、API 输入边界等。这一层主要回答“代码有没有坏，工具输出结构是否稳定”。
>
> 第二层是 Router eval，这是目前最接近 benchmark 的部分。我把用户问题做成 JSONL 样本，每条包含 query 和期望 intent，有些 challenge 样本还包含 primary intent、secondary intents、route plan 和失败边界。指标上，单意图看 accuracy、per-intent precision/recall/F1、confusion matrix；多意图看 primary intent、secondary intents 和 route plan 是否匹配。
>
> 第三层是链路级验收，比如 `/chat`、SSE、WebSocket、Motion 图片/视频上传、Milvus fallback、Tavily mock、MCP mock 等。这一层主要回答“完整流程能不能跑通，失败时有没有降级”。
>
> 判断一次改动是否让 Agent 变强，我不会只看某个 demo case 是否变好，而是看回归集是否不退化、challenge set 是否改善、错误类型是否减少、链路失败是否更可控。比如改 Router 权重后，不能只看一个新样本对了，还要看旧样本有没有被破坏，chat/diet、diet/mcp、多意图这些 slice 有没有更稳定。
>
> 但我也承认，当前还没有完整的人类偏好评测或线上 A/B，也没有真实大规模用户日志。RAG 虽然完成了真实链路效果评测，但 Recall@K、MRR、回答忠实度等系统指标还可以继续扩展；Motion 也可以继续补充更多教练标注样本，最终回答的安全性和有用性也需要生产化 benchmark。

**回答拆解**

- 先说有评测，但不是线上级评测平台。
- 按三层讲：单元测试、Router eval、链路验收。
- “变强”的判断不能只看单个 demo，而要看回归不退化、困难样本改善、错误类型减少。
- 主动承认最终自然语言回答质量还没有完整自动评测。

**有价值追问**

- Agent 有没有办法自动判断任务是否成功？依靠什么指标？
- Router eval 的 accuracy 能不能代表整个 Agent 质量？
- 最终回答质量怎么评测？
- RAG 效果应该看什么指标？
- 工具调用成功是否等于任务成功？
- 如果某次改动让一个 case 变好、另一个 case 变坏，怎么判断是否值得合并？

**当前边界**

- 当前评测更偏原型回归和面试展示，不是完整线上评测平台。
- Router 和工具调用可自动评估较多，最终回答质量自动评估较弱。
- 还缺真实用户 query、人工标注、RAG 证据标注和人类偏好评测。
- 生产化应按 Intent、Retrieval、Tool、Answer 四类 benchmark 分层建设。

#### 追问 005-1. Agent 是否能自动判断任务成功

**面试官问题**

> 简单来说，你的 Agent 有没有办法判断任务是否成功？如果可以，是依靠什么指标来判断？

**回答目标**

这道题是在要求把评测讲得更直白。回答要明确哪些任务可以自动判断成功，哪些任务不能完全自动判断成功，避免用 benchmark 术语绕开问题。

**推荐回答**

> 简单说，有一部分可以判断，有一部分现在还不能完全自动判断。因为 Agent 里的任务类型不一样，成功标准也不一样。
>
> Router 任务是可以比较明确判断成功的。用户输入一句话，Router 输出 intent。我的 eval 样本里有人为标注的标准答案，所以可以判断预测 intent 是否等于标注 intent。如果是多意图，还可以判断 primary intent、secondary intents 和 route plan 是否正确。所以 Router 成功依靠的是标注样本，以及 accuracy、precision、recall、route plan match 这些指标。
>
> 工具调用任务也比较容易判断。搜索工具、MCP 工具、Motion 工具、Retriever 都会返回结构化结果。判断标准包括 `ToolResult.ok` 是否为 True，是否有 data，错误码是不是预期错误，fallback 是否生效，返回数据结构是否符合预期。
>
> API 或链路任务也能判断一部分，比如 `/chat` 是否返回 200，SSE 是否能流式返回，图片上传是不是返回预期 200/422，Motion 分析有没有 frames、joints、metrics。这类依靠 HTTP 状态码、响应字段和异常场景的预期错误。
>
> 但最终自然语言回答质量就不能完全自动判断。比如用户问“减脂晚餐怎么吃”，回答好不好，不只是看有没有输出文字，还要看是否回答了问题，是否基于 RAG 证据，有没有胡编来源，有没有危险建议，是否结合用户目标，表达是否清楚。当前项目还没有完整自动打分体系，更多依靠手工验收、测试样例和文档边界。
>
> 所以一句话说：我的 Agent 对“路由是否正确、工具是否成功、接口链路是否跑通”可以自动判断，依靠标注 eval、结构化 `ToolResult`、HTTP 状态码和响应字段；但对“最终自然语言回答是否足够好”，当前还不能完全自动判断，后续需要建设回答质量 benchmark。

**回答拆解**

- 直接回答“部分可以，部分不可以”，不要含糊。
- 把可自动判断的任务列清：Router、工具、API/链路。
- 把难自动判断的任务列清：自然语言回答质量、专业性、安全性、个性化。
- 指标要落到具体字段：`intent`、`ToolResult.ok`、`error_code`、HTTP status、response schema。

**当前边界**

- 工具成功不等于用户任务一定成功，只能说明执行层没有失败。
- Router 正确也不等于最终回答一定好。
- 当前最终回答质量缺少完整自动评测，需要人工标注、参考答案、证据片段和安全评分维度。
