# 高频技术问答

这篇文档用于准备面试官最可能追问的问题。每个问题都分成三部分：

- **先这样回答**：面试现场先说结论。
- **你需要理解**：帮助你真正理解答案。
- **继续追问**：准备下一轮深入问题。

不要逐字背诵。先记住结论和逻辑，再换成自己的语言。

## 一、项目与智能体架构

### 问题 1：为什么这个项目算智能体，不是普通聊天机器人？

**先这样回答**

> 因为系统不会把所有问题直接交给大模型生成答案。它会先判断任务类型，再选择知识检索、联网搜索、动作算法或外部工具，并把执行结果、来源和错误写回状态，最后生成回答。大模型只是系统中的语言能力，不是唯一执行者。

**你需要理解**

普通聊天通常是“一次输入、一次生成”。智能体至少增加了任务选择、状态、工具和执行流程。本项目的证据包括 Router、LangGraph 子工作流、ToolRegistry、Memory 和执行轨迹。

**继续追问**

如果面试官问“用了工具就一定算智能体吗”，回答：工具调用本身不够，关键还在于系统是否能够根据任务选择工具、管理状态、处理失败并形成可验证的执行闭环。

### 问题 2：为什么使用 LangGraph，不直接写条件判断？

**先这样回答**

> 条件判断适合少量固定分支，但项目还有共享状态、多个子工作流、组合任务、错误隔离和结果合成。LangGraph 用节点、边和状态明确表示这些关系，使流程更容易解释、测试和扩展。

**你需要理解**

LangGraph 不是为了让代码显得复杂，而是让“当前执行到哪里、状态里有什么、下一步去哪里”变得明确。它特别适合需要受控流程的智能体。

**继续追问**

如果只有两个简单分支，手写条件判断更合适。选择 LangGraph 的理由必须来自流程复杂度，而不是框架流行度。

### 问题 3：系统的总体架构是什么？

**先这样回答**

> 我把系统分为接入层、控制层、能力层和保障层。FastAPI 负责接收请求；Router 和 LangGraph 负责选择并编排流程；Knowledge、Search、Motion 和 MCP 负责具体任务；Memory、工具结果、降级和评测负责上下文与稳定性。

**你需要理解**

四层之间是职责关系，不是四个独立程序。分层的价值是面试官追问某个问题时，你能迅速指出它属于哪一层。

**继续追问**

如果问“ToolRegistry 属于哪一层”，可以说它位于能力与保障的交界处：登记具体工具，同时治理参数、权限、失败和审计。

## 二、路由与多意图

### 问题 4：Router 是不是只靠关键词？

**先这样回答**

> 不是。Router 以加权规则为稳定基础，同时使用组合信号、歧义识别和字符 n-gram/词面样例；embedding 样例匹配和本地 Qwen 都是可选路径。Qwen 完成过对比验证，但没有带来足够准确率收益且延迟更高，因此默认关闭；embedding 也要等更大评测集证明净收益后再启用。

**你需要理解**

完全关键词容易误判同义表达，完全大模型路由又可能慢且不稳定。项目使用混合方案，把常见明确任务留给确定性策略，把困难样本单独评测。

**继续追问**

如果问“模型更智能，为什么不用”，回答：技术选择要看评测收益。模型如果没有改善困难样本，却增加延迟和不确定性，就不应该为了技术名词强行启用。

### 问题 5：一句话包含两个任务时怎么办？

**先这样回答**

> Router 会记录主意图和次意图，并生成路由计划；当前只允许经过验证的两步白名单组合，例如先搜索最新资料，再给出饮食建议。其他复杂组合降级为主意图单路由，避免执行顺序和结果合成失控。

**你需要理解**

多意图不等于完全开放 Planner。受控组合更容易测试每一步成功与否，也适合面试演示稳定性。

**继续追问**

如果未来开放三步以上规划，需要补步骤依赖、预算、循环上限、部分失败策略和任务成功判定。

## 三、知识检索与联网搜索

### 问题 6：RAG 的完整流程是什么？

**先这样回答**

> 离线阶段把知识文档按句子边界切块，生成向量并连同内容和来源写入检索后端；在线阶段把问题也转成向量，召回相近片段，经过阈值过滤、去重和来源编号后注入 Prompt，再让大模型基于证据回答。

**你需要理解**

RAG 不是一个单独工具，而是一条从数据准备、检索到回答生成的链路。任何环节都可能影响效果。

**继续追问**

常见追问包括怎样切块、Top-K 怎么选、没有相关资料时怎么办、如何防止模型忽略证据。

### 问题 7：Milvus 在项目里具体做什么？

**先这样回答**

> Milvus 负责向量持久化、索引和相似度搜索。项目保存内容、向量和来源，使用稳定主键保证重复加载幂等，并通过 IVF_FLAT、余弦距离、`nlist`、`nprobe` 和 Top-K 在召回与延迟之间取舍。

**你需要理解**

Milvus 不负责生成回答，也不自动保证检索质量。它只是高效找到候选资料的基础设施。

**继续追问**

如果问“为什么不用 FAISS”，可以回答：FAISS 很适合本地向量检索，Milvus 更方便演示持久化、结构、索引管理和服务化扩展；上层 Retriever 接口保持可替换。

### 问题 8：RAG 怎样评测？

**先这样回答**

> 我把检索和生成分开看。当前项目已经建立第一版检索黄金集，检查 Recall@K、MRR、来源命中、无答案拒绝和延迟；生成侧应该看忠实度、相关性和引用正确性，但这部分尚未形成当前基线，属于下一步评测。

**你需要理解**

“接入 Milvus”只能证明链路存在，不能证明效果好。评测必须从真实问题、期望证据和回答标准出发。

**继续追问**

如果问当前证据，说明项目已经完成真实 Milvus 链路验证和 15 条检索黄金集；不要把计划中的生成忠实度评测说成已完成。

### 问题 9：RAG 和 Tavily 搜索有什么区别？

**先这样回答**

> RAG 负责本地、稳定、可控的领域知识；Tavily 负责最新研究和外部信息。搜索链路还会改写查询、整理标题摘要网址，并基于来源合成回答。

**你需要理解**

区别主要在数据来源和时效性，不在于两者是否都叫“检索”。

**继续追问**

如果问来源是否完全可靠，回答：当前实现了来源透传和证据约束，生产化还需要来源可信度分级、逐句引用验证和缓存。

## 四、动作分析

### 问题 10：动作分析为什么不能直接交给大模型？

**先这样回答**

> 动作标准性包含关键点坐标、时间轨迹和相似度计算。大模型适合理解和解释，但精确计算应该交给姿态模型和数值算法。因此项目用 MediaPipe 提取姿态、PoseSequence 统一数据、FastDTW 对齐时间，再由大模型解释结果。

**你需要理解**

这是项目中“大模型与确定性算法分工”最直观的例子。

**继续追问**

如果问 3D 是否绝对准确，说明当前是姿态模型估计的三维关键点，会受到视角、遮挡和模型精度影响。

### 问题 11：FastDTW、余弦相似度和逐关节距离分别解决什么？

**先这样回答**

> FastDTW 解决两段动作速度和帧数不同的问题；余弦相似度衡量整体姿态特征方向；时间对齐后的逐关节平均距离衡量人体结构偏差。多指标结合比单一分数更容易解释。

**你需要理解**

时间对齐是比较动作序列的前提。两个人做同一动作，一个快一个慢，不能按原始帧位置直接比较。

**继续追问**

这些指标仍不能自动指出某个关节的专业错误。进一步纠错需要动作类型、关键关节、周期阶段、标准范围和教练标注。

### 问题 12：视频能上传，是否意味着动作纠错已经准确？

**先这样回答**

> 不完全等同。当前已经打通视频到姿态序列、标准样本比较和结构化反馈，可以说明与参考动作有多接近；专业级纠错还需要更大的标准动作库、动作周期切分、关键点平滑、关节级规则和真实教练评测。

**你需要理解**

这是一种有层次的回答：先说明已完成的价值，再说明准确性的条件，而不是先自我否定。

## 五、工具系统与 MCP

### 问题 13：工具系统是不是只写了几个函数？

**先这样回答**

> 函数只是执行器。工具系统还需要说明工具职责、输入结构、权限、结果、错误、重试、降级和审计。项目使用 ToolSpec 和 ToolRegistry 集中描述这些信息，并用 ToolResult 与 ErrorCode 统一返回。

**你需要理解**

工具治理关注“怎样安全稳定地执行”，不仅是“能否调用”。

**继续追问**

当前 Search、Knowledge/RAG 和 MCP 执行路径已接入 Registry；Motion 媒体链路因为文件和数组边界更复杂，没有强行全面迁移。

### 问题 14：LangGraph、ToolRegistry 和 MCP 有什么区别？

**先这样回答**

> LangGraph 管业务流程，决定先后步骤；ToolRegistry 管具体工具执行，负责参数、权限、失败和审计；MCP 是连接外部工具的一种协议。MCP Client 可以作为一个工具被 Registry 管理，但三者不能互相替代。

**你需要理解**

最简单的类比：LangGraph 是流程图，ToolRegistry 是工具台账，MCP 是外部插头标准。

### 问题 15：自己实现的 MCP Client 做了什么？

**先这样回答**

> Client 通过子进程启动本地 Server，使用标准输入输出发送 JSON-RPC，覆盖 initialize、tools/list、tools/call 和文本结果解析，并将失败转换成统一工具结果。默认 mock 用于保证演示稳定。

**你需要理解**

项目验证的是协议主链路，不是完整生产级 MCP 平台。

**继续追问**

当前还需要增强响应编号匹配、inputSchema 深层校验、并发请求、进程生命周期、超时隔离和真实 Server 兼容性。

### 问题 16：让大模型调用工具安全吗？

**先这样回答**

> 不能默认安全。工具必须来自允许列表，参数必须校验，权限要由系统而不是模型决定，执行要有超时、资源隔离和审计。项目已经实现最小 schema、权限和审计框架，但生产化仍需更强沙箱与授权。

## 六、记忆、服务与质量

### 问题 17：短期记忆和长期记忆有什么区别？

**先这样回答**

> 短期记忆保存当前对话需要的最近消息，长期记忆保存跨会话仍有价值的目标、偏好和约束。PromptBuilder 再根据当前任务和预算选择哪些信息进入本轮上下文。

**你需要理解**

记忆系统不是“保存得越多越好”。重点是写什么、何时召回、如何确认敏感信息、怎样删除和控制长度。

**继续追问**

项目以 SQLite 作为长期记忆事实来源，Milvus 只是可选语义增强；伤病、过敏等敏感信息先进入候选确认。

### 问题 18：为什么不能把所有历史都放进 Prompt？

**先这样回答**

> 历史越长，成本和长度越高，噪声也越多，重要信息反而可能被淹没。项目使用近期窗口、长期记忆检索和 compact 分层管理，再按 Prompt 预算注入。

### 问题 19：HTTP、SSE 和 WebSocket 怎么选择？

**先这样回答**

> 普通 HTTP 适合一次返回完整结果；SSE 适合服务器持续向浏览器推送文本；WebSocket 适合需要双向长连接的客户端。项目同时支持三种方式，共享同一智能体主链路。

**继续追问**

同步 LangGraph 和模型生成通过线程桥接到异步队列，避免阻塞事件循环；本地模型仍由生成锁控制同进程并发。

### 问题 20：自动化测试数字能证明什么？

**先这样回答**

> 它能证明接口、状态流、工具契约、错误处理和算法没有回归，但不能直接证明真实模型回答质量。Router、RAG、Memory 和 Motion 还需要各自的效果评测。

**你需要理解**

测试回答“系统有没有坏”，评测回答“策略效果好不好”，线上数据回答“真实用户场景能不能泛化”。

### 问题 21：项目能直接部署到公网吗？

**先这样回答**

> 当前定位为本地可运行的面试原型，不能直接暴露公网。生产化前要补认证授权、租户隔离、限流、HTTPS、CORS 白名单、隐私治理、依赖就绪检查、模型服务化和多实例存储。

### 问题 22：如何证明项目不是技术名词堆砌？

**先这样回答**

> 每项技术都对应明确问题：LangGraph 管流程，RAG 管稳定知识，Tavily 管最新信息，动作算法管确定性计算，ToolRegistry 管工具执行，MCP 管外部协议，Memory 管跨轮信息，评测管质量。它们由统一状态、错误和执行轨迹连接，不是互相孤立的演示模块。

## 七、最后的回答原则

面试回答始终遵循：

```text
先说结论
  -> 再解释问题
  -> 再说明方案
  -> 再讲取舍
  -> 最后补边界和升级
```

如果一个问题暂时不懂，不要用更多技术名词掩盖。回到[简历技术点总表](./03_简历技术点总表.md)，先找到它属于控制层、能力层还是保障层，再重新理解。

## 八、真实连续追问：从概念到工程设计

### 追问链 1：Agent 到底比固定工作流多了什么

**基础级：什么是 Agent？**

> Agent 是能根据目标和当前状态选择下一步动作，并调用工具获得新信息的系统。大模型只是理解与决策组件；状态、工具、控制流程和反馈共同构成 Agent。

**项目级：你的项目是真 Agent，还是几个接口套了 LangGraph？**

> 判断标准不是框架，而是系统是否形成执行计划、保存中间状态、选择不同能力并合成结果。本项目在 `search`、`motion`、`diet`、`chat`、`mcp` 间路由，允许四种验证过的双任务组合，子工作流将来源、错误和轨迹写回共享状态。因此它是受控工作流型 Agent，不是完全自治的规划 Agent。

**工程级：为什么不开放任意工具循环？**

> 健康场景更重视可预测性。开放循环会带来调用次数失控、成本放大、间接提示注入和高权限误操作。生产化应增加最大步数、总时间和 token 预算，按工具分级授权，高风险写操作人工确认，并记录每一步 trace。只有离线评测证明开放规划确有收益，才逐步扩大自由度。

### 追问链 2：RAG 回答错误怎样归因

**基础级：RAG 为什么仍会幻觉？**

> RAG 只提供资料，不保证检索到正确资料，也不保证模型忠实使用资料。错误可能来自切块、向量表示、召回、过滤、上下文组装或生成。

**项目级：怎样做一次错误分析？**

> 先检查正确片段是否在库中，再看是否进入 Top-K；如果进入，检查阈值、去重和 Prompt 是否保留；最后检查关键结论能否被证据支持。项目中的检索结果带内容、分数和来源，可把检索失败与生成失败分开。

**工程级：怎样证明调整 `nprobe` 或 Top-K 有效？**

> 固定评测集，同时比较 Recall@K、MRR、无关上下文比例、忠实度、任务成功率和 p95 延迟。扩大召回可能增加延迟与噪声，所以不能只追求 Recall；参数修改要有基线、版本和回滚方案。

### 追问链 3：工具失败如何设计

**基础级：超时、重试、降级分别是什么？**

> 超时限制等待；重试处理短暂故障；降级在主要能力不可用时提供较弱但诚实的结果。三者解决不同问题。

**项目级：当前 ToolRegistry 做到了什么？**

> `ToolSpec` 描述参数、权限、重试错误码、fallback 和超时策略；Registry 做参数与权限检查、有限重试、统一 `ToolResult` 和审计。必须诚实说明：`timeout_seconds` 当前主要是策略和审计元数据，Registry 没有强制中断执行器。

**工程级：为什么不能所有失败都重试三次？**

> 参数错误和权限拒绝重试无意义；非幂等写操作可能造成重复副作用；持续故障时同步重试会放大流量。应按错误类型和幂等性决定重试，使用指数退避与随机抖动，并设置整条请求的截止时间；高错误率时用熔断或限流保护依赖。

### 追问链 4：Memory 怎样既有用又安全

**基础级：为什么不保存所有对话？**

> 并非所有内容都有长期价值，全量保存会增加噪声、成本、隐私风险和错误记忆。

**项目级：本项目怎样存和取？**

> SQLite 保存可修改、可删除、可审计的事实；FTS5/LIKE 做基础检索，Milvus 可作语义增强；PromptBuilder 只注入近期窗口和相关长期记忆。显式“记住”优先写入，伤病、过敏等敏感候选需要确认。

**工程级：多租户最危险的问题是什么？**

> 是跨用户召回和敏感信息泄露。生产化必须在查询层强制用户或租户过滤，不能只靠 Prompt；还要加密、最小化保存、保留期限、删除传播、审计和越权测试，评测中加入“绝不能召回”的负样本。

### 追问链 5：怎样定义 Agent 成功

**基础级：为什么 HTTP 200 不等于成功？**

> 200 只说明服务完成响应，不说明路由正确、工具有效、回答有依据或用户目标达成。

**项目级：本项目应拆哪些指标？**

> 路由看意图和计划准确率；RAG 看召回与忠实度；工具看成功率、错误码和 fallback；Memory 看写入、召回和误注入；Motion 看关键点质量、对齐和教练标注；整体看任务成功率、延迟和成本。

**工程级：自动评测能直接决定上线吗？**

> 只能作为门槛的一部分。LLM-as-judge 需要固定 rubric、校准样本和人工抽检；离线集也可能不代表线上分布。单元测试、专项评测、红队集、灰度指标和用户反馈应共同形成质量闸门。

## 九、工程词汇必须能落到措施

下面是生产化分析模板，不是当前代码已经全部落地的能力。面试时应先说当前基础，再用这一列回答“如果上线怎样补”。

| 术语 | 普通话解释 | 生产化落地方式（当前未全部实现） |
|---|---|---|
| 幂等 | 同一请求重复执行不产生额外副作用 | 写记忆使用业务唯一键，重试前判断工具是否幂等 |
| 截止时间 | 整条请求最多允许花多少时间 | 给路由、检索、工具和生成分配总预算 |
| 熔断 | 依赖持续失败时暂时停止调用 | 外部搜索或 MCP 高错误率时快速失败并诚实降级 |
| 背压 | 下游处理不过来时限制上游 | 视频分析用有界队列，满载时排队或拒绝 |
| 可观测性 | 用外部证据解释系统内部发生了什么 | 统一 trace_id，记录节点、工具、延迟、错误和降级 |
| SLO | 给用户体验设可衡量目标 | 任务成功率、p95 首 token、错误率和泄露事件 |
| 灰度 | 少量流量先用新版本 | 新 Router、Prompt 或索引先小流量验证并可回滚 |

完整工程回答通常包括“触发条件、采取措施、监控指标和恢复方式”。只说加缓存、加重试、加监控仍然不够。
