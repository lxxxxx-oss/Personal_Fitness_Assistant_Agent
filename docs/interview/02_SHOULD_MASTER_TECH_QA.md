# P1 最好掌握：围绕简历的技术问答

这份文档按简历技术点组织。面试官通常看不到代码，他会根据简历追问“为什么这样做”“你具体做了什么”“这个方案有什么边界”。回答时不要总说“代码里怎样”，而要把项目思路讲清楚。

统一回答结构：

```text
先给结论
  -> 结合简历项目讲实现思路
  -> 说明为什么这样选型
  -> 被追问时再讲边界和生产化补齐
```

## 1. LangGraph 与 Agent 架构

### Q1：这个项目为什么算 Agent，不是普通大模型套壳？

**面试官想确认**

你是否理解 Agent 的核心价值，而不是把所有 LLM 应用都叫 Agent。

**推荐回答**

> 普通套壳通常是把用户问题拼进 Prompt，再返回一次模型结果。我的项目先由 Router 判断任务，再进入 RAG、Search、Motion 或 MCPTool 等工作流。不同工作流会调用检索、联网搜索、数值算法或外部工具协议。LLM 主要负责理解、规划和表达，确定性计算和外部数据由工具完成，所以它具备任务路由、工具调用、状态管理和失败降级这些 Agent 特征。

**边界**

> 它是受控工作流型 Agent，不是让模型无限自主规划的开放式 Agent。这样做是为了稳定、可测试，也更适合健身健康场景。

### Q2：为什么使用 LangGraph，而不是自己写 if/else？

**面试官想确认**

你是不是为了用框架而用框架。

**推荐回答**

> 如果只是一个 `intent -> handler`，我不会强行用 LangGraph。这个项目里有统一状态、多条子流程、条件分支、工具失败隔离、多意图组合和结果合成。LangGraph 能把节点、状态和边显式表达出来，让 Search、Motion、Chat、MCPTool 等流程边界更清楚。相比大量 if/else，它更适合展示和维护这种多任务 Agent。

**选型取舍**

> LangGraph 的优势是流程清晰、状态显式、方便扩展；代价是引入框架复杂度。如果项目只是一次模型调用，用普通函数更简单。但这个项目要体现多能力编排，所以 LangGraph 是合理选择。

### Q3：简历写四类功能，怎么解释 Search/Motion/Chat/MCPTool？

**推荐回答**

> 简历里的四类功能是从用户价值归纳的：联网搜索、3D 动作分析、饮食与菜谱、RAG 知识问答。工程实现上可以拆得更细，比如饮食建议和菜谱工具调用属于同一个饮食业务域，但执行方式不同：Diet 更偏用户目标和营养建议，MCPTool 更偏外部工具协议调用。产品层归纳和工程层拆分不矛盾。

## 2. Router 与多意图

### Q4：Router 是不是只靠关键词？

**推荐回答**

> 简历里写关键词规则，是对确定性路由的简化描述。实际设计思路是：先用确定性规则保证稳定和可解释，再通过权重、组合规则、歧义处理、语义样例和可选 LLM classifier 提升边界样本效果。Router 会记录分数、置信度和原因，并用离线样本回归，避免凭感觉调规则。

**为什么不直接用 LLM Router**

> Router 是控制面，最重要的是稳定、低延迟和可解释。LLM 可以作为 fallback，但不适合默认接管所有路由。只有当它在真实样本上带来明确收益，并且延迟可接受时，才值得扩大使用。

### Q5：多意图请求怎么处理？是不是 Planner？

**推荐回答**

> 当前设计不是通用 Planner，而是受控 multi-intent routing。用户明确提出复合任务时，Router 可以识别主意图、次意图和执行顺序，但只开放有限组合，比如先搜索再饮食建议，或先动作分析再解释建议。这样能覆盖真实复合需求，又避免任意组合导致延迟、冲突和不可测试。

**边界**

> 我不会把它包装成完全自主规划 Agent。它的价值在于把常见复合任务做成可控流程。

## 3. 3D 动作分析与 ReAct

### Q6：3D 动作数据是什么格式？

**推荐回答**

> 核心数据是 3D 人体姿态关键点序列，可以理解成 `(T, J, 3)`：T 是帧数，J 是关节点数量，最后一维是 x、y、z 坐标。这样做的好处是把视频或传感器输入转成结构化数据，后续才能计算角度、轨迹和相似度。

### Q7：为什么要做姿态归一化？

**推荐回答**

> 不同用户身高、站位、相机距离不同，如果直接比较原始坐标，站得远近可能比动作本身影响更大。所以要做中心化和尺度归一化，让指标更关注关节之间的相对结构和运动轨迹。归一化之后再计算 FastDTW、余弦相似度和形状差异，结果才更有解释性。

**边界**

> 当前这种归一化不是完整的视角不变算法，摄像机角度、遮挡和关键点噪声仍会影响结果。生产化要结合多视角、置信度过滤和真实动作数据校准。

### Q8：FastDTW、余弦相似度和形状差异分别解决什么？

**推荐回答**

> FastDTW 解决两段动作速度不同、帧数不同的问题，它在时间轴上做近似对齐；余弦相似度更关注归一化后整体姿态方向是否一致；形状差异衡量每帧人体结构变化的平均差距。三个指标关注点不同，组合起来比单一分数更容易解释动作哪里像、哪里不像。

**为什么用 FastDTW**

> 标准 DTW 复杂度较高，FastDTW 更适合原型阶段处理较长序列。生产化可以再比较标准 DTW、Soft-DTW 或动作阶段对齐。

### Q9：Motion 为什么说采用 ReAct 思想？

**推荐回答**

> Motion 子图不是一次函数调用，而是 think -> parse -> tool -> check。think 判断用户想分析什么，parse 把自然语言转成工具参数，tool 调用动作分析工具，check 检查结果并生成反馈。这体现了 ReAct 的思想：先推理，再行动，再观察结果。

**边界**

> 这是受控 ReAct 工作流，不是无限自主循环。最大迭代次数是安全边界，避免模型反复调用工具。

### Q10：图片能不能判断动作标准？

**推荐回答**

> 单张图片只能看到某一个时刻的姿态，可以判断关键点置信度、身体对齐和静态角度，但不能判断完整动作的下降、上升、节奏和重复稳定性。完整动作分析更适合视频或 `.npz` 时序姿态数据。后续路线是视频抽帧、逐帧姿态估计、形成 PoseSequence，再复用 FastDTW 和动作规则。

## 4. RAG、Sentence-Transformers 与 Milvus

### Q11：RAG 的完整数据流是什么？

**推荐回答**

> 数据流是：先把健身知识文档做 sentence-aware 分块，再用 Sentence-Transformer 生成 embedding；用户提问时，把 query 也转成向量，按 COSINE 相似度召回候选片段，再做 threshold 过滤、Top-K、去重和排序，最后把相关片段注入 Prompt 生成回答。

### Q12：简历写 Milvus，面试时怎么讲？

**推荐回答**

> 我会从生产化 RAG 方案讲 Milvus：把 chunk 文本、embedding、source、topic 等元数据写入 Collection，使用 COSINE 度量，索引可以从 IVF_FLAT 开始，根据 nlist、nprobe 平衡召回和延迟。上层 Chat/Diet 不应该感知底层数据库，只依赖 Retriever 的 search 结果。这样即使原型阶段用轻量向量存储，迁移 Milvus 也主要替换底层检索实现。

**如果被问“你真的接了吗”**

> 我会把重点放在“我知道 Milvus 在 RAG 架构里替换哪一层，知道如何设计 Collection、索引和评测”。如果对方追问到当前演示版本，我会说明原型可以先用轻量实现验证链路，生产化再切 Milvus。不要硬编不存在的服务部署细节。

### Q13：为什么选择 COSINE？

**推荐回答**

> 文本 embedding 更关注方向上的语义相似度，而不是向量绝对长度。向量归一化后，点积和 COSINE 等价，计算简单，也和 Milvus 的 COSINE metric 对齐。当然，最终还是要用 RAG 评测集验证，不是凭经验说 COSINE 一定最好。

### Q14：RAG 怎么评测？

**推荐回答**

> RAG 评测要分两层。检索层看 Recall@K、MRR、命中率和无关片段比例；生成层看回答忠实度、完整性、引用正确性和拒答行为。没有标准问答和证据片段标注时，不应该随便报召回率。生产化第一步应该是构建评测集，再谈调参和换数据库。

## 5. MCP 外部工具集成

### Q15：MCP 在项目里解决什么问题？

**推荐回答**

> MCP 的价值是把外部工具发现和调用标准化。Agent 不需要为每个工具写专用协议，而是通过工具发现拿到工具名、描述和 input schema，再通过统一调用接口传参。这个项目里，MCPTool 可以承接具体菜谱或外部工具能力，体现 Agent 的 Tool Use。

### Q16：你自己实现的 MCP Client 做了什么？

**推荐回答**

> 我实现的是轻量 MCP Client：通过 subprocess 启动外部 Server，用 stdin/stdout 传 JSON-RPC；连接时做 initialize 握手，之后支持工具发现和工具调用；请求用 id 匹配响应，并把 content block 解析成结构化结果。错误会包装成统一结果，方便子图降级。

### Q17：MCP 工具由 LLM 调用安全吗？

**推荐回答**

> 不能让 LLM 随便执行任意命令。Server 命令应该来自配置，不接受用户输入拼接；工具名必须来自工具发现结果；参数需要 schema 校验；生产化还要加 allowlist、超时、进程隔离、权限控制和审计日志。我的思路是让 LLM 只在受控工具空间里选择动作。

### Q18：mock / fallback 怎么解释？

**推荐回答**

> mock 或 fallback 不是为了冒充真实工具，而是为了保证面试演示和本地运行稳定。MCP 的核心在于 Client 协议链路和工具调用抽象；真实 Server 不可用时，系统应该降级，而不是让整个 Agent 崩掉。生产化时再补真实 Server 部署、超时和监控。

## 6. Tavily 联网搜索与来源

### Q19：Search 为什么分 Query Understanding、Tavily Search、Answer Synthesis？

**推荐回答**

> 用户问题常常是口语化的，直接搜索可能召回不稳定。Query Understanding 先提取核心搜索词；Tavily Search 获取标题、摘要和 URL；Answer Synthesis 再把搜索结果组织成带来源约束的回答。拆成三阶段后，每一层都能单独优化和降级。

### Q20：为什么不直接把原始问题发给 Tavily？

**推荐回答**

> 原始问题可能包含上下文、口语和无关表达。Query rewrite 能提高搜索词稳定性。但改写也可能丢失约束，所以生产化要记录原 query 和 rewritten query，并评测改写前后的召回质量。

### Q21：来源信息怎么讲才不夸大？

**推荐回答**

> 我会说项目把 Tavily 返回的 title、content、url 进入回答链路，并在合成阶段要求基于来源回答。它体现了 source grounding 思路。但严格逐句 citation 校验是更高一级能力，生产化还需要把来源结构化返回前端，并验证每个引用是否真的支撑结论。

## 7. 滑动窗口记忆

### Q22：为什么用 deque 保存 6 轮？

**推荐回答**

> 滑动窗口需要频繁追加新消息、淘汰旧消息，deque 两端操作是 O(1)，比列表头删更合适。6 轮是上下文连续性、token 成本和噪声之间的折中：太短容易丢目标，太长会增加延迟和无关历史。

### Q23：它如何实现跨任务连续性？

**推荐回答**

> 同一个 user_id 的最近对话会注入统一状态，无论这轮走 Chat、Search、Motion 还是 MCPTool，下一轮都能拿到近期上下文。比如用户先说减脂目标，再问饮食或训练，系统可以承接“刚才的目标”。

**边界**

> 它是短期记忆，不是长期用户画像。生产化要做持久化、用户认证、删除机制、隐私保护和跨实例共享。

## 8. Docker、部署与模型治理

### Q24：Docker 在简历里怎么讲？

**推荐回答**

> Docker 体现的是部署意识：把后端服务、依赖和启动方式容器化，减少环境差异。这个项目里还涉及本地模型、MediaPipe 模型文件、外部 API key 和 MCP Server，所以生产化 Docker 不只是写 Dockerfile，还要处理模型挂载、环境变量、健康检查和日志。

### Q25：本地模型为什么会有内存问题？

**推荐回答**

> 问题不在于模型大小一个数字，而在于是否重复加载、是否并发生成、KV cache 怎么增长。如果多个子图或请求各自加载模型，很容易造成内存峰值。所以我采用共享加载、加载锁和生成控制的思路。生产化应该拆独立推理服务，配合量化、队列、限流和监控。

### Q26：async、SSE、WebSocket 怎么讲？

**推荐回答**

> async 适合 I/O 等待，不会自动让模型推理变快。SSE 适合服务端向客户端单向推送 token，WebSocket 适合双向实时交互。这个项目里主要是文本流式输出，所以 SSE 是更高性价比的主入口；如果后续做实时动作指导或语音交互，WebSocket 会更重要。

## 9. 测试、评测与高压追问

### Q27：Router eval 和测试数字能证明什么？

**推荐回答**

> 它们证明的是当前自建样本和核心逻辑有回归保护，不代表线上准确率。Router eval 能说明路由策略不是凭感觉调；自动化测试能保护接口、工具、检索和错误处理。但真实 LLM 回答质量、RAG 召回质量、Tavily 来源质量、Motion 专业判断，还需要单独评测和真实数据。

### Q28：项目最薄弱的部分是什么？

**推荐回答**

> 最薄弱的是端到端真实质量证据。架构和链路已经能讲清，但 RAG 需要标准问答集，Motion 需要标准动作库和教练标注，来源引用需要 citation 校验，MCP 需要更完整的真实服务治理。下一步不应该继续堆模块，而是补数据、指标和真实环境验证。

### Q29：简历有包装，如何保证可信？

**推荐回答**

> 我会把包装放在项目价值、目标架构和工程演进上，不会在关键事实上乱编。比如 Milvus 我会讲清它在 RAG 中负责向量存储和 ANN 检索，如何替换 Retriever 底层；MCP 我会讲 Client 协议链路和 fallback；Motion 我会讲时序姿态分析和单图边界。面试官可以接受原型与生产化差距，但不能接受逻辑矛盾。

4. RAG 的分块、embedding、COSINE、Milvus 生产化方案。
5. MCP Client 的 initialize、工具发现、工具调用和安全边界。
