# P1 最好掌握：技术追问与项目思路

这份材料不要按“标准答案”背。面试官问技术问题，大多数时候不是想听概念定义，而是在判断三件事：

1. **你为什么这样设计**：有没有比较过其他方案，知道优缺点。
2. **你怎么解决问题**：有没有遇到真实问题，如何定位、修复、验证。
3. **你怎么划定边界**：哪些是当前做不到，为什么先不做，是能力不足、资源限制，还是性价比不高。

回答时优先使用下面四种结构。

## 0. 四种回答框架

### 技术选型类

适合回答：为什么用 LangGraph、为什么不用 LLM Router、为什么用 SSE、为什么不从零训练模型。

```text
我当时有两个选择
  -> A 的优点和问题
  -> B 的优点和问题
  -> 我结合项目目标做了取舍
  -> 当前边界和后续升级方案
```

### 问题解决类

适合回答：本地模型内存、MCP fallback、Router 误判、流式输出。

```text
现象是什么
  -> 我怎么定位
  -> 我怎么改
  -> 怎么验证
  -> 这件事让我做了什么取舍
```

### 能力边界类

适合回答：Motion 目前能不能判断动作标准、RAG 有没有召回率、测试能证明什么。

```text
当前做到哪里
  -> 明确做不到哪里
  -> 为什么先设置这个边界
  -> 后续如果要做完整，需要补什么
```

### 项目价值类

适合回答：是不是 ChatGPT 套壳、是不是 toy、你亲自做了什么。

```text
这个问题背后的担心是什么
  -> 我项目里用什么设计回应这个担心
  -> 有什么代码/测试/文档证据
  -> 我不会夸大的地方是什么
```

## 1. Python、FastAPI 与并发

### Q1：项目为什么使用 `async`？当前实现真的非阻塞吗？

**面试官真正想看**

他不是想听 `async/await` 的语法，而是想看你有没有分清 **I/O 等待** 和 **模型计算**。很多人会说“用了 async，所以性能好”，这是面试里很危险的说法。

**我的思考**

我选择 FastAPI 的 `async`，主要是因为项目有 SSE、WebSocket、文件上传、外部工具调用，这些接口形式天然适合异步框架。但我也发现，仅仅把接口写成 `async def`，并不能让 LangGraph 图执行或本地 Qwen 推理变成非阻塞。

当前项目里，`/chat`、SSE、WebSocket 的入口可以是 async，但内部仍然会有同步调用，比如 `graph.invoke()` 或模型生成。因此这不是生产级并发架构，而是一个可运行原型。

我现在设置这个边界，是因为面试项目的第一目标是稳定演示和讲清架构。如果为了并发直接引入任务队列、独立模型服务、动态 batching，复杂度会明显上升，反而分散主线。

**面试时这样说**

> 我使用 `async` 是因为 FastAPI 的接口层需要支持 SSE、WebSocket、文件上传和外部工具调用，这些场景会有 I/O 等待。但我不会说当前实现已经完全非阻塞，因为 LangGraph 图执行和本地模型推理仍然是同步计算。当前版本先保证单机原型稳定可运行；如果生产化，我会把图执行和模型推理放到线程池、任务队列或独立模型服务，并加超时、取消和限流。

### Q2：SSE 和 WebSocket 有什么区别？项目为什么主要使用 SSE？

**面试官真正想看**

他想看你是不是知道协议选择背后的场景差异，而不是把“能流式输出”都叫 WebSocket。

**我的思考**

我对比过 SSE 和 WebSocket。SSE 是服务端单向推送，前端发一次请求，后端持续返回事件；这和 LLM 文本流式输出很匹配。WebSocket 是双向长连接，适合实时协作、语音控制、游戏这类客户端和服务端都要频繁主动发消息的场景。

这个项目的主场景是用户问一句，后端慢慢生成回答，所以 SSE 性价比更高：实现简单、调试方便、浏览器支持好。

当前 WebSocket 虽然有协议雏形，但还不能夸大成生产级实时系统。后续如果要做语音健身指导、实时动作纠正，WebSocket 才更有必要。

**面试时这样说**

> 我主要选择 SSE，是因为项目当前的核心需求是 LLM 回答的单向增量输出。SSE 比 WebSocket 更简单，适合“用户发问，服务端持续返回 token”的场景。WebSocket 更适合双向实时交互，比如实时语音或视频动作纠正。当前项目的 SSE 已经能支撑原型演示，但生产化还要补客户端断开取消、心跳、代理缓冲和超时处理。

### Q3：FastAPI 在项目里承担什么职责？

**面试官真正想看**

他想知道你的项目有没有分层意识。很多原型项目会把所有业务逻辑堆在 API 函数里，后续很难测试和维护。

**我的思考**

我把 FastAPI 定位成协议层：负责请求进入、参数校验、文件上传、状态码、SSE/WebSocket 和 Web UI。真正的任务分发和业务流程交给 LangGraph Router、子图和工具。

当前确实还有一些原型逻辑留在 API 层，比如图片上传后的姿态估计入口。这是为了让功能快速可演示。后续如果生产化，我会把这些处理继续拆到 service 层。

**面试时这样说**

> FastAPI 在项目里主要是协议层，负责 HTTP 接口、SSE/WebSocket、文件上传、请求校验和静态页面。业务流程不直接堆在 API 里，而是交给 LangGraph Router、子图和工具。当前 Motion 图片上传还保留了一些原型处理逻辑，后续可以拆 service 层，这是我为了先保证可运行做的取舍。

### Q4：本地模型并发和内存问题怎么处理？

**面试官真正想看**

这个问题是在看你有没有遇到过真实运行问题。只说“调用了 Qwen”没有价值，能讲出内存、重复加载和并发取舍才更像真实项目。

**我的思考**

我遇到的核心问题是：本地模型如果反复加载，内存压力会很大。即使是小模型，tokenizer、权重和缓存重复初始化也会让项目不稳定。

所以我做的取舍是优先保证稳定：共享 tokenizer/model，用加载锁避免重复初始化，用生成锁降低并发踩踏。代价是同一进程内吞吐有限，但这符合面试原型的目标。

如果追求生产吞吐，我不会简单开多个 uvicorn worker，因为那可能复制多份模型；更合理的是独立推理服务、请求队列、动态 batching 和显存治理。

**面试时这样说**

> 本地模型最大的问题不是能不能调用，而是内存和并发是否稳定。我做了共享模型加载和锁控制，避免每个请求重复加载 Qwen，也避免多个生成任务同时抢资源。这个方案牺牲了一部分并发，换来单机演示稳定。生产化时我会拆独立推理服务，配合队列、动态 batching 和限流，而不是简单堆 uvicorn worker。

## 2. RAG 与回答可信度

### Q5：当前 RAG 具体怎么实现？为什么这样做？

**面试官真正想看**

他想知道你是不是真的理解 RAG，而不是只会说“我接了向量库”。RAG 的关键是知识怎么来、怎么切、怎么召回、怎么放进回答，以及效果怎么验证。

**我的思考**

我当前没有一上来接 Milvus，是因为项目知识库规模还小。个人面试项目里，先把 RAG 链路和评测意识打通，比先引入重型向量库更重要。

当前做法是读取 `data/knowledge/` 里的健身和饮食文本，做规则切分，再用中文 embedding 模型生成向量，用相似度召回 Top-K 片段。Embedding 不可用时，降级到关键词匹配，保证项目能跑。

这个边界是性价比选择，不是“不知道向量数据库”。等数据量变大、并发上来、需要过滤和索引管理时，再迁移 Milvus 或 pgvector 更合适。

**面试时这样说**

> 当前 RAG 是本地知识库原型。我先从 `data/knowledge/` 读取健身和饮食资料，切成 chunk，再用中文 embedding 做相似度召回，把 Top-K 片段放进 prompt。Embedding 失败时会降级到关键词匹配，保证链路可运行。我没有一开始上 Milvus，是因为当前数据量还小，先验证 RAG 流程、降级和评测意识更重要；生产化再迁移向量数据库、补 reranker 和检索评测。

### Q6：Embedding、Chunk、Top-K、Threshold、Rerank 怎么理解？

**面试官真正想看**

他想确认你知道 RAG 效果不是由一个词决定的，而是多个环节共同影响。

**我的思考**

Chunk 决定知识切得细不细。切太大，召回不精准；切太小，语义断裂。Embedding 负责把文本变成向量，让相似问题能找到相似内容。Top-K 控制拿多少候选，Threshold 控制低相关内容要不要过滤。Rerank 是在初步召回后，用更强模型重新排序。

所以 RAG 优化不是“换个向量库”这么简单。向量库只是存储和检索基础设施，真正影响效果的是知识质量、切分、召回、重排和评测。

**面试时这样说**

> Chunk 影响检索粒度，Embedding 负责语义表示，Top-K 决定候选数量，Threshold 过滤低相关片段，Rerank 对初召回结果做二次排序。RAG 效果不是由向量库单独决定的，而是知识治理、切分策略、召回策略、重排和评测一起决定。当前项目先打通基础链路，后续会重点补知识库治理、混合检索、reranker 和 RAG eval。

### Q7：RAG 怎么评测？当前为什么不给召回率？

**面试官真正想看**

这是典型边界问题。他不是非要你现在有漂亮指标，而是看你会不会编数字。

**我的思考**

RAG 评测至少有两层：检索层和回答层。检索层看 Recall@K、MRR、命中率和噪声比例；回答层看忠实度、完整性、引用正确性和拒答行为。

当前项目还没有“问题 -> 标准证据片段”的冻结标注集，所以我不能给可靠 Recall@K。这个边界不是因为不知道怎么做，而是因为没有足够标注数据，强行报指标反而不可信。

我更愿意在面试中说明评测方法和下一步：先构造标准问答和证据集，再做检索和回答两层评测。

**面试时这样说**

> 当前我不会给 RAG 召回率，因为还没有构建问题到证据片段的冻结标注集。没有标注集就报 Recall@K，是不严谨的。正确做法是先标注一批问题、标准证据片段和期望回答，再分别评估 Recall@K、MRR、无关片段比例，以及回答的忠实度和完整性。当前测试主要证明 Retriever 链路和降级逻辑能工作，生产化才会补系统 RAG eval。

### Q8：当前来源是否真的可追溯？

**面试官真正想看**

他想知道你有没有把“内部用了上下文”和“用户看得到引用”区分开。

**我的思考**

当前项目 prompt 内部会使用检索片段，Search 也会带标题和 URL，这说明内部有 grounding 思路。但 API 响应和 SSE meta 还没有完整返回结构化 citation。

所以当前边界是：系统内部能用来源约束回答，但用户侧来源展示还没做完整。这个边界是因为当前优先级放在 Agent 主链路和 Router eval 上，citation 是下一步产品化体验。

**面试时这样说**

> 当前项目内部会把检索片段或搜索结果放进 prompt，Search 也会带标题和 URL，所以已经有 grounding 思路。但我不会说已经完成生产级 citation，因为 `/chat` 响应和 SSE meta 还没有完整返回结构化 sources。后续会把 source id、标题、URL、证据片段和引用关系作为响应字段返回，并做引用校验。

### Q9：如何降低幻觉和高风险健康建议？

**面试官真正想看**

他想看你有没有安全意识。健身、饮食、动作指导都可能触及健康风险，不能把模型包装成医生。

**我的思考**

幻觉不能完全消除，只能降低风险。我的边界是：项目提供一般健身和饮食信息，不做医疗诊断，不替代医生、康复师或注册营养师。

降低风险的方法包括 RAG grounding、资料不足时拒答、高风险问题提醒咨询专业人士、外部搜索内容不作为高优先级指令、回答后校验等。

**面试时这样说**

> 我不会说项目能完全消除幻觉，也不会把它包装成医生或营养师。当前定位是一般健身和饮食信息助手。工程上可以通过 RAG 上下文约束、资料不足时拒答、来源引用、可信来源过滤和高风险场景提示来降低风险。生产化还要增加安全分类、答案校验和专业咨询提示。

## 3. Agent、Workflow 与 Tool

### Q10：Agent、Workflow、Chain 有什么区别？你的项目是哪一种？

**面试官真正想看**

他想看你是不是盲目把所有 LLM 应用都叫 Agent。

**我的思考**

Chain 是固定步骤，Workflow 是有状态、有分支的流程，Agent 更强调根据任务选择行动。我的项目不是完全开放式 Agent，不会让模型无限循环调用工具。它更准确地说是受控 Agent Workflow。

我这样做是因为健身助手需要稳定、安全、可评测。开放式 Agent 看起来更智能，但更难控、更难测，也更容易出错。

**面试时这样说**

> 这个项目更准确地说是受控 Agent Workflow。它有 Router、状态、子图和工具选择，所以不是普通 Chain；但它也不是完全开放式 Agent，因为行动空间和组合顺序由代码约束。这样设计是为了稳定、可测试和可降级，比较适合面试型原型和健康类场景。

### Q11：为什么使用 LangGraph，而不是普通函数路由？

**面试官真正想看**

他在判断你是不是为了用框架而用框架。

**我的思考**

如果项目只是 `intent -> handler`，普通函数路由完全够用。我使用 LangGraph，是因为项目有统一状态、多节点子图、条件边、多意图组合、错误隔离和最终合成。

我也承认 LangGraph 不是唯一方案。用 Python 状态机也能写，只是节点和边会散在代码里，不利于面试讲解和后续扩展。

当前没有使用 checkpoint 持久化，所以不能夸大成分布式恢复系统。它的价值主要是把受控流程显式化。

**面试时这样说**

> 如果只是简单 `intent -> handler`，我不会强行用 LangGraph。选择它是因为项目里有 RouterState、多节点子图、条件分支、多意图白名单组合和结果合成。LangGraph 能把这些节点和边表达得更清楚，方便测试和讲解。当然它不是唯一方案，普通状态机也能做；当前项目用它主要是为了受控图式编排，不是为了堆框架。

### Q12：子图和工具有什么区别？

**面试官真正想看**

他可能在质疑“你是不是把工具包装成子图，做了冗余设计”。

**我的思考**

工具是一个具体动作，比如搜索、姿态相似度计算、图片关键点提取。子图是一段流程，里面可能包含理解、调用工具、检查结果、降级和生成回答。

如果某个能力只有一次函数调用，确实没必要做子图。但 Search、Diet、Motion 都不是单次函数调用，它们涉及输入理解、工具调用、结果整理和边界处理。

**面试时这样说**

> 工具是一个动作，子图是一段流程。比如搜索工具只负责查结果，但 Search 子图还要理解用户问题、整理 query、调用搜索、筛选结果并合成回答。Motion 工具可以算姿态相似度，但 Motion 子图还要判断任务、解析输入、处理失败和生成反馈。所以子图不是为了包装工具，而是为了管理一类任务的完整流程。

### Q13：为什么决定把 MCP 菜谱模块移出核心架构？

**面试官真正想看**

他想看你有没有架构收敛能力。很多项目会越做越散，模块越来越多，但主线越来越弱。

**我的思考**

MCP 是有价值的工具协议，但菜谱 MCP 和 Diet 的职责重叠。用户问吃什么、怎么吃、菜谱怎么做，本质上都可以归入饮食建议。如果单独保留 MCP 菜谱，会增加 intent、子图、配置、测试和面试解释成本。

我决定把 MCP 从核心架构里移出，是因为项目面试主线应该聚焦健身 Agent 的四个核心能力，而不是展示所有可能的技术。

这不是说 MCP 不好，而是它在当前项目里的性价比不高。

**面试时这样说**

> MCP 本身是有价值的标准化工具协议，但复盘后我发现菜谱 MCP 和 Diet 子图职责重叠，对健身主线贡献有限，却增加了路由、协议和测试成本。为了让项目更聚焦，我决定把菜谱能力并入 Diet，把 MCP 从核心架构中移出。这是一次架构收敛，而不是否定 MCP 技术。

### Q14：代码还没删除，面试时怎么解释？

**面试官真正想看**

这是在看你是否诚实。文档说目标架构收敛，代码里如果还保留 MCP，就不能说“已经完全删除”。

**我的思考**

当前要区分目标架构和代码现状。目标架构已经明确收敛到 Chat、Search、Diet、Motion；但如果仓库仍有 MCP Client、`mcp` intent 和测试，就必须说它是遗留实验。

这个边界要主动讲，避免面试官打开代码后觉得你前后不一致。

**面试时这样说**

> 当前目标架构已经收敛为 Chat、Search、Diet、Motion 四个核心模块，但仓库里如果还保留早期 MCP Client、`mcp` intent 和相关测试，我会把它称为待清理的遗留实验，而不是说已经完成迁移。后续删除代码后，需要重新跑 Router eval 和 pytest，并更新新的基线。

## 4. Router 与 Eval

### Q15：Router 当前怎么实现？为什么不是简单关键词？

**面试官真正想看**

他想看 Router 是否有工程含量，还是一堆 if else。

**我的思考**

我最开始可以用关键词，但很快会遇到边界问题：一句话可能同时包含饮食、搜索、动作，或者用户表达很隐晦。简单 first-match 会让某个关键词抢走路由。

所以我做了加权规则：所有 intent 都能累计分数，再结合组合规则、pattern boost、低置信语义样例和可选 LLM fallback。这样每次路由都有分数、来源和原因。

Phase 4 再增加 primary intent、secondary intents 和 route plan，但只执行白名单组合，避免变成不可控 Planner。

**面试时这样说**

> Router 不是简单 first-match 关键词。我做了加权规则，让不同 intent 都能累积分数，再用组合规则和 pattern boost 处理边界表达；低置信时用字符 n-gram 语义样例补充；LLM classifier 作为 feature flag 控制的 fallback。Phase 4 会记录 primary intent、secondary intents 和 route plan，但只执行白名单组合。这样 Router 可解释、可评测，也不会失控。

### Q16：多意图路由是不是 Planner？

**面试官真正想看**

他想确认你有没有把一个小功能包装成“自主规划 Agent”。

**我的思考**

当前多意图不是通用 Planner。它不是让模型自由决定调用哪些工具，而是当用户明确说“先做 A，再做 B”时，系统识别主意图、次意图和有限 route plan。

我设置这个边界，是因为原型项目最怕开放任意组合。四个模块随便串联，错误会放大，也难以测试。

**面试时这样说**

> 当前多意图路由不是通用 Planner，而是受控 route plan。系统能识别 primary intent、secondary intents 和 route plan，但只有显式顺序表达并命中白名单时才执行两步组合，比如 `search -> diet` 或 `motion -> chat`。其他组合只记录观测信息，然后降级为主意图单路由。这个边界是为了保证可控和可测试。

### Q17：66/66 和 36/36 如何解释？

**面试官真正想看**

他想看你有没有评测意识，同时看你会不会滥用指标。

**我的思考**

66/66 和 36/36 是离线回归指标，不是线上准确率。它们证明的是：在我整理的常规样本和困难样本上，当前 Router 没有回归。

我设置这两套样本，是为了让 Router 优化有证据，而不是凭感觉调规则。绿色集保护常规行为，challenge set 暴露边界冲突和多意图样本。

它们的边界也很明确：样本量小、人工构造多、不能代表真实用户全部表达。生产化要引入冻结留出集和线上日志抽样。

**面试时这样说**

> 66/66 和 36/36 是项目自建 eval 上的回归结果，不是线上泛化准确率。它们的价值是证明 Router 对已知常规样本和困难样本没有回归，也说明我不是凭感觉调规则。但我不会把它夸大成生产准确率，后续还需要冻结留出集、真实日志采样和持续标注。

### Q18：为什么 LLM Router 默认关闭？

**面试官真正想看**

他想看你是不是迷信 LLM。很多人会默认“交给 LLM 更高级”，但控制面不一定适合这么做。

**我的思考**

我不是没有做 LLM Router，而是做过 A/B。结果是本地 Qwen 没有带来净收益，还增加了平均约 6.22 秒延迟，并且早期有覆盖正确规则的风险。

Router 是控制面，稳定、可解释、低延迟很重要。如果 LLM 不能提高准确率，那默认关闭是更合理的工程决策。

**面试时这样说**

> 我没有默认启用 LLM Router，是因为做过本地 Qwen A/B。它在当前 challenge set 上没有增加正确样本，反而增加约 6.22 秒延迟，并且早期出现过覆盖正确规则的风险。Router 是系统控制面，稳定和延迟比“看起来智能”更重要，所以当前默认确定性 Router，LLM classifier 只作为可开关实验能力保留。

## 5. 测试与证据

### Q19：自动化测试能证明什么，不能证明什么？

**面试官真正想看**

他想知道你有没有把测试当成证据，而不是装饰。

**我的思考**

当前测试能证明确定性逻辑、接口契约、错误处理和部分降级路径有回归保护。比如 Router、eval 脚本、Retriever、LLM loader、Motion 数据结构等。

但很多测试使用 mock，所以它不能证明真实 Qwen 回答质量、真实 embedding 召回质量，也不能证明外部服务稳定性。

这就是测试边界：它保证代码行为不随便坏，但不等于模型效果评测。

**面试时这样说**

> 当前自动化测试主要证明接口契约、Router、工具降级、模型加载缓存、Motion 数据结构和错误处理有回归保护。它能说明项目不是只有文档设计，也能支撑后续修改。但我不会说这些测试证明真实模型回答质量或真实搜索稳定性，因为测试里大量使用 mock。真实模型效果需要单独评测和手工验收。

### Q20：Router eval 为什么不直接算普通单元测试？

**面试官真正想看**

他想看你是否理解测试和评测的区别。

**我的思考**

单元测试更像判断某个函数是否符合预期。Eval 更像衡量一个策略在一批样本上的整体表现。Router eval 可以输出 accuracy、每类指标、slice、混淆矩阵、source 分布和 mismatch。

所以我会把 eval 脚本单独保留，同时让测试调用它防止回归。这比把所有样本塞进普通断言里更适合解释策略演进。

**面试时这样说**

> 单元测试适合保护具体函数和边界，Router eval 适合比较整体路由策略。Eval 脚本能输出 accuracy、每类 precision/recall/F1、slice、混淆矩阵、route source 和 mismatch，方便看策略哪里变好了或变差了。测试可以调用 eval 防止回归，但评测报告应该独立保留。

## 6. Memory、可观测性与安全

### Q21：当前 Memory 是什么级别？为什么不做长期记忆？

**面试官真正想看**

他想看你是否知道 memory 涉及身份、隐私、持久化和多实例问题。

**我的思考**

当前 Memory 是进程内 SlidingWindowMemory，只保存最近几轮对话。它适合面试原型，因为实现简单，能支撑上下文连续性。

我没有做长期记忆，是因为长期记忆需要用户认证、隐私治理、持久化、删除机制和跨 worker 共享。健身健康类数据还可能有敏感性，不能随便存。

**面试时这样说**

> 当前 Memory 是进程内滑动窗口，只保存最近几轮对话，主要用于原型演示中的上下文连续性。我没有把它包装成长期记忆，因为长期记忆需要认证、持久化、隐私治理、删除机制和跨实例共享。生产化会把会话迁移到 Redis 或 PostgreSQL，并和用户身份、过期策略、脱敏策略绑定。

### Q22：Router 为什么说可解释？用户看得到吗？

**面试官真正想看**

他想看你是否区分“开发者可解释”和“用户可观测”。

**我的思考**

当前 Router 内部记录 scores、confidence、source、reason、matches 和 route plan，这对开发者定位问题很有用。但普通用户现在主要看到 intent 和最终回答，还没有 trace 面板。

所以准确边界是：内部具备可解释元数据，但生产级可观测性还没完成。

**面试时这样说**

> 当前 Router 的可解释性主要是内部可解释。RouterState 会记录 scores、confidence、source、reason、matches 和 route plan，eval 脚本也能输出 source 分布和 mismatch，方便定位路由错误。但用户侧还没有完整 trace API 或监控面板，所以不能说已经完成生产级可观测性。

### Q23：当前安全边界有哪些？

**面试官真正想看**

他想看你对 LLM 应用风险是否有基本判断。

**我的思考**

当前项目是本地面试原型，没有完整认证、权限、限流和审计。图片上传需要大小和 MIME 校验，搜索结果和工具输出也可能包含 prompt injection。

我设置这个边界，是因为当前项目先证明 Agent 编排和可运行性。安全生产化当然重要，但要和认证、部署、日志、权限一起做。

**面试时这样说**

> 当前安全边界是本地原型级别，不是生产级。主要缺口包括认证、会话隔离、限流、上传大小和 MIME 校验、Prompt Injection 防护、工具权限、日志脱敏和审计。尤其是搜索结果和工具输出都不能当成可信系统指令。生产化时这些会和部署、权限、监控一起补齐。

## 7. 面试官高压追问

### Q24：你到底亲自写了什么，而不是调用库？

**面试官真正想看**

他在判断项目是不是“调库拼接”。你要把自己的工程贡献讲清楚。

**我的思考**

我不会把 LangGraph、Qwen、MediaPipe 说成自己发明的。我的贡献在控制面和工程集成：Router 规则和 eval、状态结构、多意图执行、PoseSequence、工具降级、模型共享加载、UI/API 串联和文档边界。

这也是项目的亮点：不是重新训练基础模型，而是把不同能力组合成可运行、可评测、可解释的 Agent 原型。

**面试时这样说**

> 基础框架和模型当然来自开源生态，我的工作不是重新发明它们。我主要完成的是控制面和工程集成：Router 规则与评测、LangGraph 状态和子图编排、多意图白名单执行、PoseSequence 适配层、错误降级、共享模型加载、接口和 UI 串联，以及面试化文档。项目价值在于把 LLM、RAG、搜索、动作分析组织成可运行、可评测的 Agent 原型。

### Q25：如果去掉 LangGraph，项目还能运行吗？

**面试官真正想看**

他想看你是否有框架依赖迷信。

**我的思考**

答案是能。用普通 Python 状态机也能重写。选择 LangGraph 不是因为没有它做不了，而是因为它让状态、节点、条件边和子图边界更清楚。

这个回答要主动承认替代方案，否则会显得你只会用一个框架。

**面试时这样说**

> 可以运行。这个项目本质上可以用普通 Python 状态机重写，所以 LangGraph 不是业务成立的唯一条件。我选择它，是因为状态、节点、条件边和子图边界更显式，便于展示 Router、子图和多意图组合。它的代价是引入框架复杂度，所以如果只是简单 handler 路由，我不会使用它。

### Q26：项目最薄弱的技术环节是什么？

**面试官真正想看**

他在看你是否能诚实识别风险。

**我的思考**

当前最薄弱的不是缺一个新功能，而是端到端质量证据还不够。Router 有 eval，但 RAG 缺标准问答集，Motion 缺真实标准动作数据集，真实 LLM 回答还需要人工评估。

我不想继续堆模块，而是先收敛主线，补数据和评测。这个回答能体现优先级判断。

**面试时这样说**

> 当前最薄弱的是端到端质量证据。Router 已经有绿色集和 challenge set，但 RAG 还缺标准问答集和召回指标，Motion 还缺真实标准动作数据集，真实 LLM 回答质量也需要更多人工评估。所以下一步我不会盲目加模块，而是先收敛 MCP 菜谱链路，再补数据、评测和真实运行日志闭环。

### Q27：如果只允许你生产化一个模块，先做哪个？

**面试官真正想看**

他想看你有没有产品和工程优先级。

**我的思考**

我会先做 Router + Chat/RAG 主链路，因为它最高频、最容易收集真实问题、最容易建立 RAG eval，也最能带动引用、日志、会话和模型服务改造。

Motion 很有亮点，但对真实数据和专业规则依赖更强，适合保持受控实验入口。

**面试时这样说**

> 如果只能先生产化一个方向，我会选 Router + Chat/RAG 主链路。它是最高频入口，也最容易建立真实数据闭环。具体会先补认证、会话持久化、日志 trace、结构化引用、RAG eval、模型服务和限流。Motion 很有亮点，但依赖标准动作库和真实样本，我会先保留为受控实验入口。

## 8. 容易踩坑的说法

### 不要说

- `async` 已经解决模型推理阻塞。
- WebSocket 已经真正实时逐 token 输出。
- Semantic Examples 是 Embedding Router。
- 多意图 route plan 是通用自主规划。
- 102 条满分证明线上泛化。
- 自动化测试证明真实模型和外部服务质量。
- 图片接口已经完成动作标准性判断。
- MCP 菜谱链路仍然是目标架构中的核心模块。

### 推荐说

- 当前是单机受控原型，异步接口已经打通，但模型服务仍需隔离。
- Router 指标是已知样本回归保障，泛化需要留出集和线上闭环。
- Motion 已完成输入适配和部分分析组件，完整动作判断依赖视频、标准动作库和真实数据。
- MCP 是一次已完成的工具协议实验，但因为业务重叠和面试主线不聚焦，目标架构决定从核心链路中移除。
