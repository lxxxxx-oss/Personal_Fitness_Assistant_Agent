# P1 最好掌握：围绕简历的技术问答

这份文档按简历中的技术点组织，而不是按框架百科组织。每个问题都分成三层：面试官想确认什么、推荐回答、继续追问时的边界。

统一回答结构：

```text
先给结论
  -> 结合项目讲实现
  -> 说明为什么这样选
  -> 被追问时再讲边界和升级方案
```

## 1. 项目与 LangGraph

### Q1：这个项目为什么算 Agent，不是普通大模型套壳？

面试官想确认你是否真的理解 Agent 的价值，而不是堆技术名词。

推荐回答：

> 普通套壳通常是把用户问题直接拼进 Prompt，再返回一次模型结果。我的项目先由 Router 判断任务，再进入 RAG、Search、Diet、Motion 或 MCP 子图。不同子图会调用检索、搜索、数值算法或外部协议工具，并处理状态、错误和结果合成。LLM 不是唯一执行者，主要负责理解、规划和表达；确定性计算与外部数据由工具完成，所以它具备任务路由、工具使用、状态管理和失败降级这些 Agent 特征。

边界：它是受控工作流型 Agent，不是能够任意规划任意工具的通用自主 Agent。

### Q2：简历写四类功能，为什么代码里有五个 intent？

推荐回答：

> 四类是产品能力归纳：联网搜索、3D 动作分析、饮食与菜谱、RAG 知识问答。代码为了让职责和错误边界更清楚，把饮食域拆成 Diet 与 MCP 两条路径：Diet 处理用户画像、营养检索和饮食建议，MCPTool 处理外部菜谱工具发现与调用，所以顶层实际是 Chat、Search、Diet、Motion、MCP 五个 intent。简历按用户价值做压缩，代码按执行职责做拆分。

这是一处可以主动解释的“抽象层级不同”，不需要说成简历写错。

### Q3：为什么使用 LangGraph，而不是自己写 if/else？

推荐回答：

> 最早的单意图分流用普通函数也能做，但项目后来有五个子流程、统一状态、条件边、多意图组合、工具失败隔离和最终结果合成。LangGraph 能把节点、状态和转移关系显式化，每个子图也可以单独测试和演进。如果继续用 if/else，状态字段和异常处理会散落在控制器里。我的选择不是因为 LangGraph 流行，而是因为系统已经出现了图编排真正擅长的问题。

追问“去掉 LangGraph 能不能跑”：

> 能重写成普通 Python workflow，但会失去显式状态图和子图边界。LangGraph 不是业务能力本身，它是降低复杂流程维护成本的编排层。

### Q4：Router 真的是关键词规则吗？

推荐回答：

> 简历里的“关键词规则”是对确定性路由的简化描述。当前版本已经从 first-match 关键词演进为 weighted rules：所有 intent 累积分数，再叠加组合规则、pattern boost、歧义信号、语义样例和可选本地 LLM classifier。Router 会记录 score、confidence、reason 和 source，并用离线样本回归。默认仍优先确定性路由，因为它更快、更可解释，当前真实 Qwen classifier 的 A/B 收益不足，所以保留 feature flag 而没有默认开启。

边界：语义样例目前是轻量 char n-gram Jaccard，不要说成已经使用 Sentence-Transformer 做 Router embedding。

### Q5：多意图请求怎么处理？它是不是 Planner？

推荐回答：

> 当前不是开放式 Planner，而是受控 multi-intent routing。Router 会识别 primary intent、secondary intents 和 route plan，但执行层只开放四种两步白名单：`search -> diet`、`search -> chat`、`motion -> chat`、`motion -> diet`。每个子图异常被隔离，成功结果最后再合成。这样先覆盖真实复合问题，又避免任意组合带来的延迟、冲突和调试复杂度。

## 2. 3D 动作分析与 ReAct

### Q6：3D 动作数据是什么格式？

推荐回答：

> 核心输入是 `.npz` 中的 3D 姿态关键点序列，统一成 `(T, J, 3)`：`T` 是帧数，`J` 是关节数，最后一维是 x、y、z 坐标。我又抽象了 `PoseSequence`，额外保存 source type、pose model、joint schema、fps 和置信度，让 `.npz`、图片和后续视频都能转换为同一个中间结构。

追问为什么不用原始视频直接算：

> 当前先把“姿态数据进入系统后如何分析”做稳定，再接媒体输入。这样算法层不绑定某个姿态模型，后续 MediaPipe、MoveNet 或其他模型只要输出统一 `PoseSequence` 就能复用。

### Q7：姿态归一化怎么做？为什么需要？

推荐回答：

> 不同用户与相机位置会带来平移和尺度差异。如果直接比较坐标，站得远近可能比动作本身影响更大。当前实现把索引 0 约定为中心点，先平移到原点，再用所有关键点到中心点的平均距离做尺度归一化，让指标更关注关节相对结构和运动轨迹。归一化之后再进入 FastDTW、余弦相似度和形状差异计算。

边界：当前归一化不是完整的 3D 视角不变算法，摄像机角度和关键点噪声仍会影响结果。

### Q8：FastDTW、余弦相似度和形状差异各自解决什么？

推荐回答：

> FastDTW 解决两段动作速度不同、帧数不同的问题，它先在时间轴上寻找近似对齐路径，距离越小表示时序轨迹越接近。余弦相似度更关注归一化后整体姿态方向是否一致；形状差异衡量每帧人体结构幅度序列的平均差异。三个指标关注点不同，组合使用比单一阈值更稳，也更方便解释为什么“像”或“不像”。

追问为什么 FastDTW 而不是标准 DTW：

> 标准 DTW 的时间和空间复杂度通常是 `O(T1*T2)`；FastDTW 用近似搜索降低长序列成本，适合原型。生产化会根据序列长度和精度要求比较 Sakoe-Chiba band DTW、Soft-DTW 或关键阶段对齐。

### Q9：关节角度如何计算？

推荐回答：

> 给定三个关键点 A、B、C，以 B 为关节中心，计算向量 `BA` 和 `BC` 的夹角。先做点积，再除以两个向量模长，经过数值截断后做 arccos。角度能表达膝、髋、肘等局部几何关系，比只看全局坐标更适合形成动作反馈。

边界：当前通用工具实现了角度计算能力，但完整“深蹲阶段识别 + 每个阶段的关节规则 + 个体差异阈值”仍需数据标定。

### Q10：Motion 为什么说采用 ReAct 思想？

推荐回答：

> Motion 子图不是一次函数调用，而是 `think -> parse -> tool -> check` 的状态流。think 分析用户想做什么，parse 把自然语言转成工具和参数，tool 执行姿态加载或标准动作对比，check 检查结果并生成解释。状态里保存待调用工具、结果和迭代次数，配置限制最大 5 次，避免模型无限循环。这体现了“推理决定动作、工具返回观察、再检查”的 ReAct 思想。

高压追问“它真的会循环五次吗”：

> 当前主要路径通常是一次解析、一次工具执行、一次检查；最大迭代是安全边界。它是受控 ReAct 风格工作流，不包装成高度自主的开放循环。

### Q11：图片接口能判断深蹲是否标准吗？

推荐回答：

> 单张图片只能提取一个时刻的 33 个关键点、置信度和静态姿态摘要，无法判断下蹲与起身轨迹、节奏、最低点稳定性和重复一致性。完整动作相似度当前依赖 `.npz` 时序输入。下一步是视频抽帧、逐帧姿态估计、置信度过滤和插值，形成 `PoseSequence(T>1)` 后复用现有 FastDTW 链路。

## 3. RAG、Sentence-Transformers 与 Milvus

### Q12：当前 RAG 的完整数据流是什么？

推荐回答：

> 启动时从 `data/knowledge/` 读取健身与营养资料，按中英文标点和换行做 sentence-aware 分块，默认控制单块长度。Sentence-Transformer 对 chunk 编码并做向量归一化；查询也用同一模型编码，归一化向量点积就是 COSINE 相似度。检索后执行 threshold 过滤、降序排序、Top-K 和内容去重，再把片段注入 Chat 或 Diet 的 Prompt。Embedding 加载失败时会降级到关键词匹配，保证主链路仍可运行。

### Q13：简历写 Milvus，为什么当前代码是 NumPy？

这是最重要的防守题。推荐回答：

> 简历把 RAG 按完整目标方案概括成 Milvus，当前这个可公开、可本地演示的仓库版本为了减少服务依赖，存储层使用 NumPy 内存向量。检索链路本身已经包含 sentence-aware chunk、Sentence-Transformer、COSINE、threshold、Top-K、去重和排序，上层只依赖 Retriever 的 `search()` 结果，因此迁移 Milvus 主要替换存储与 ANN 检索实现，不需要重写 Chat/Diet 子图。严格按当前分支说，我不会声称它已经创建 Milvus Collection 或真正执行 IVF_FLAT 查询。

如果面试官追问“那简历是不是写过头了”：

> 这项表述采用的是目标技术栈口径，确实比当前展示仓库更前置。我的重点是我能讲清 Milvus 应该替换哪一层、当前已实现哪些检索环节、哪些数据库调用还没落到该分支；后续我也会把简历改成“基于 Milvus 方案设计 / 可迁移 Milvus”这种更严格的说法。

这比硬说“代码里有”更可信。适当包装可以放在方案与设计深度上，不能捏造 import、Collection 和索引创建代码。

### Q14：如果真正接 Milvus，你会怎么设计？

推荐回答：

> 我会把 chunk 文本、embedding、source、topic、version 等元数据写入 Collection；向量字段维度与 Sentence-Transformer 输出一致，度量使用 COSINE。小中等规模可以从 IVF_FLAT 开始，通过 `nlist` 控制聚类桶数量，查询用 `nprobe` 平衡召回和延迟；如果数据持续增长或实时写入要求更高，会比较 HNSW。检索仍保留 threshold、Top-K、去重与必要的 rerank。迁移前先建立 Recall@K、MRR、延迟和资源占用基线，否则只换数据库没有意义。

追问 IVF_FLAT 原理：

> 建库时先把向量聚成多个倒排桶，查询时只搜索最接近的若干桶。`nlist` 越大分桶越细，`nprobe` 越大召回通常越高但查询更慢。它不是业务质量的保证，只是 ANN 的性能与召回权衡。

### Q15：为什么选择 COSINE？

推荐回答：

> 文本 embedding 更关心向量方向表达的语义，而不是向量绝对模长。当前编码时做归一化，因此点积等价于 COSINE，计算简单，也与未来 Milvus 的 COSINE metric 对齐。生产化仍要用真实问答集验证，不能仅凭经验认定 COSINE 一定最好。

### Q16：sentence-aware 分块比固定长度切分好在哪里？

推荐回答：

> 固定字符切分可能把一个动作原则或营养结论从中间切断。sentence-aware 先在句号、问号、感叹号和换行处分割，再合并到长度上限，能尽量保持语义完整。它仍不是最终方案；长文档还可以按标题层级、段落语义和 overlap 做层次化切分。

### Q17：RAG 怎么评测？为什么当前不能说效果很好？

推荐回答：

> 当前自动化测试主要证明检索流程、参数校验、排序和降级没有回归，还没有足够的领域标准问答集，所以不能给出可靠 Recall@K。生产化我会为每个问题标注相关 chunk，统计 Recall@K、MRR、无答案识别；生成侧再评估 faithfulness、answer relevance 和 citation coverage，并对高风险健康问题人工审核。

## 4. MCP 外部工具集成

### Q18：MCP 在项目里解决什么问题？

推荐回答：

> MCP 的价值是把外部工具发现和调用标准化。Agent 不需要为每个菜谱服务写一套专用协议，而是先通过 `tools/list` 获取工具名、描述和 input schema，再通过 `tools/call` 传入参数。这样工具能力可以由独立 Server 提供，Agent 侧主要维护通用 Client、权限和错误处理。

### Q19：你自己实现的 MCP Client 做了什么？

推荐回答：

> Client 用 subprocess 启动配置的 Server 命令，通过 stdin/stdout 发送 JSON-RPC。连接阶段先发 `initialize`，带 protocol version 和 client info，收到响应后发送 `notifications/initialized`；工具发现调用 `tools/list`，工具执行调用 `tools/call`。请求使用唯一 id 匹配响应，并解析 MCP content block，优先把 `content[0].text` 中的 JSON 转成结构化对象。所有公开方法统一返回 `ToolResult`，把网络、配置、参数和内部错误转成可处理的错误码。

### Q20：MCP 工具由 LLM 随便调用安全吗？

推荐回答：

> 不能让模型直接执行任意命令。Server 命令来自配置，不接受用户动态拼接；工具名必须非空，arguments 必须是字典，真实工具列表由 Server 返回。MCP 子图先做工具发现，再让 LLM 生成受约束 JSON，解析失败时走确定性默认路径。生产化还需要工具 allowlist、schema validation、超时、进程隔离、审计日志和按用户授权。

边界：当前参数校验是原型级，尚未对每个工具完整执行 JSON Schema 校验。

### Q21：默认 mock 还能说 MCP 已实现吗？

推荐回答：

> 可以说 MCP Client 和协议链路已实现，但不能说真实 Server 已经在所有环境稳定运行。默认 `MCP_SERVER_COMMAND=mock` 是为了让离线测试与演示不依赖 Node 全局命令；显式配置 `howtocook-mcp` 时会走真实 subprocess 和协议握手，如果命令缺失或连接失败，子图回退 mock，并记录 configured command、mode 和 fallback reason。mock 是运行策略，不是拿假数据冒充真实联调。

### Q22：MCP 与 Diet 职责是不是重复？

推荐回答：

> 两者都在饮食域，但执行范式不同。Diet 根据用户画像和营养知识生成目标导向建议；MCP 演示标准化外部工具发现与调用，例如查询具体菜谱、分类或餐单。产品化时可以把入口统一成饮食助手，但内部仍可以保留 MCP 作为一种工具适配层。这样既避免用户理解两个入口，又保留协议扩展价值。

这比简单说“删除 MCP”更符合简历主线，也更能解释当前代码。

## 5. Tavily 联网搜索与来源

### Q23：Search 为什么分三阶段？

推荐回答：

> 第一阶段 Query Understanding 把口语问题改写成 1 到 2 组搜索关键词；第二阶段 Tavily Search 获取标题、摘要和 URL；第三阶段 Answer Synthesis 把搜索结果编号后注入 Prompt，要求摘要先行、要点展开并标注来源。拆开后可以分别优化 query rewrite、搜索质量和回答忠实度，也能在搜索失败时明确降级。

### Q24：为什么不直接把原始问题发给 Tavily？

推荐回答：

> 用户问题常包含口语、上下文和无关表达，直接搜索可能降低召回。Query rewrite 提取动作、目标、时间等核心概念，让搜索词更稳定。但改写也可能丢失约束，所以生产化要记录原 query 与 rewritten query，并评测改写前后的结果覆盖率。

### Q25：简历说“带来源信息的结构化回答”，当前真的做到哪一步？

推荐回答：

> Tavily 结果在工具层是结构化的 title、content、url；Search 子图会把 URL 收集到内部 `_sources`，并在 Prompt 中要求用 `[来源N]` 标注。因此“来源数据进入回答链路”已经实现。当前普通 `/chat` 的 `ChatResponse.sources` 还没有把 result state 中的 `_sources` 回填出来，所以用户主要从文本引用看到来源，严格的结构化来源透传还差 API 这一小步。它也不是逐句 citation verifier。

### Q26：Tavily 不可用怎么办？

推荐回答：

> Tool 会区分缺少 API key、网络错误、鉴权错误和内部错误，统一包装为 `ToolResult`。没有 key 时可以返回 mock 结果保证演示；真实调用失败时在 state 记录 warning，Synthesis 会基于当前可用结果保守回答。生产化还要加明确超时、重试、熔断、可信域名分级和缓存。

## 6. 滑动窗口会话记忆

### Q27：为什么用 `deque`？

推荐回答：

> 滑动窗口需要不断在尾部追加新消息、从头部淘汰旧消息，`collections.deque` 两端操作都是 `O(1)`，比列表头删更合适。每轮包含 user 和 assistant 两条消息，默认 `max_turns=6`，所以最多保存约 12 条消息。

### Q28：它如何实现跨任务连续性？

推荐回答：

> FastAPI 层按 `user_id` 获取同一个 `SlidingWindowMemory`，每次调用 Router 前把最近历史放入统一 state。无论本轮走 Chat、Search、Diet、Motion 还是 MCP，下轮都能拿到同一用户的最近上下文，因此用户从训练问题切到饮食问题时仍可以承接“我”“刚才的目标”等信息。

边界：Memory 是统一注入，但不同子图是否充分利用历史仍由各自 Prompt 与逻辑决定，不能把它说成复杂的跨 Agent 共享知识图谱。

### Q29：为什么只保留 6 轮？

推荐回答：

> 这是上下文连续性、推理成本和噪声之间的原型折中。窗口太短容易丢掉目标，太长会增加 token、延迟和无关历史干扰。6 轮通过环境变量可配置。生产化会把短期会话放到 Redis 或数据库，并增加摘要记忆、过期策略、用户删除、脱敏和跨实例共享。

## 7. FastAPI、流式与模型治理

### Q30：项目用了 `async`，是不是所有操作都非阻塞？

推荐回答：

> FastAPI endpoint 使用 async 便于处理网络连接和 SSE，但 LangGraph invoke、本地 Transformers 生成、NumPy 计算和部分工具仍是同步阻塞。当前是单机面试原型，通过单 worker、模型共享和生成串行化控制资源。生产化会把推理放到独立服务或线程/任务队列，不能因为函数写了 async 就说整条链路非阻塞。

### Q31：SSE 与 WebSocket 怎么选？

推荐回答：

> 普通回答主要是服务端向客户端单向推送 token，SSE 基于 HTTP、协议简单、浏览器和代理支持较好，所以作为主流式入口。WebSocket 适合需要持续双向交互的场景，项目也提供了对应接口。当前两者都是先运行图得到最终 Prompt，再流式生成一次，避免子图和接口重复调用模型。

### Q32：本地模型为什么会 OOM，你怎么处理？

推荐回答：

> 问题不是模型只有 0.6B 就一定轻，而是多个子图各自创建 Loader 时可能重复加载权重，再叠加并发生成与 KV cache。当前把 tokenizer/model 做成进程级共享缓存，首次加载加锁，生成阶段串行化，并建议单 worker。生产化会拆独立推理服务，配合量化、批处理、限流和显存监控。

### Q33：Docker 做到什么程度？

推荐回答：

> 仓库提供 Docker 配置，服务端接口和依赖具备容器化入口，但本地模型路径、模型资产体积、MediaPipe task 文件和真实外部服务仍需要环境准备，完整跨机器构建验证还没有形成稳定证据。因此可以说“提供容器化部署方案”，不要说“已经生产部署”。

## 8. 测试、评测与高压追问

### Q34：66/66、36/36 和 114 条分别证明什么？

推荐回答：

> 66 条是 Router 主流绿色回归集，36 条是边界冲突和多意图 challenge set；全部通过说明当前确定性策略对自建样本没有回归。114 条 pytest 覆盖 API、Router、Retriever、Motion、MCP、Memory 和错误处理。它们证明的是代码行为可回归，不是线上准确率，也不代表真实 LLM、Tavily、Embedding 或 MCP Server 的质量。

### Q35：你到底亲自写了什么？

推荐回答：

> 我不是从零训练大模型或姿态网络，但我完成了 Agent 应用层的核心工程：任务拆分与 StateGraph、Router 与 eval、Motion 数值算法和 ReAct 状态流、MCP Client 协议调用、RAG 检索后处理、Tavily 三阶段链路、短期记忆、接口和降级。框架提供基础能力，我负责把它们变成一个可运行、可测试、可解释的系统。

### Q36：项目最薄弱的部分是什么？

推荐回答：

> 最薄弱的是端到端真实质量证据。Router 已有自建 eval，但 RAG 缺标准问答与 Recall@K，Motion 缺足够的标准动作和教练标注，真实 Tavily 来源也缺 citation 校验，MCP 默认演示仍是 mock。下一步优先补数据集、指标和真实联调，而不是继续增加子图数量。

### Q37：简历有包装，如何保证可信？

推荐回答：

> 我会把包装放在项目价值、目标架构和工程演进上，不会在代码事实问题上硬撑。例如简历写 Milvus，我会讲清完整 Milvus 方案和为什么当前仓库替换成 NumPy，同时明确当前分支没有直接创建索引；MCP 则明确 Client 已实现、默认 mock 是演示策略。面试官通常可以接受原型与目标架构的差距，但不能接受把不存在的调用说成已经上线。

### Q38：如果只允许生产化一个模块，先做哪个？

推荐回答：

> 我会先做质量评测与数据闭环，而不是先换更重的框架。具体先为 RAG 建标准问答和来源数据，为 Motion 建标准动作与标注，再采集 Router 真实误判。只有基线建立后，迁移 Milvus、调阈值或增加模型才有可验证的收益。
