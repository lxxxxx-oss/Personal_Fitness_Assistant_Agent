# P1 最好掌握：技术追问问答

## 1. Python 和后端基础

### Q1：项目里为什么会用 `async`？

回答：

> FastAPI 支持异步接口，适合处理网络请求、流式输出和外部工具调用这类 I/O 密集任务。项目里真正重的模型推理不一定因为 `async` 就变快，所以我会区分 I/O 并发和 CPU/GPU 计算。生产化时，阻塞模型调用要放到独立 worker 或队列里，避免卡住事件循环。

项目联系：

- `/chat/stream` 需要持续返回内容。
- MCP、Search、LLM 调用都可能有等待。
- 本地模型推理要注意阻塞和内存。

### Q2：SSE 和 WebSocket 有什么区别？

回答：

> SSE 是服务端单向推送，适合 LLM token streaming；WebSocket 是双向长连接，适合实时协作、游戏、语音控制这类双向交互。这个项目的主要需求是服务端持续把回答片段推给前端，所以 SSE 更简单。

项目联系：

- 流式回答只需要后端不断发事件。
- 前端不需要频繁向后端发送实时控制消息。

### Q3：FastAPI 在项目里承担什么职责？

回答：

> FastAPI 是协议层，负责 HTTP 接口、文件上传、请求校验、响应格式和静态页面。真正的业务判断交给 LangGraph 和子图。这样 API 层不会堆满业务逻辑。

项目联系：

- `/chat` 负责接收用户消息。
- `/motion/analyze-image` 负责图片上传。
- `/ui` 提供 Web UI。

## 2. LLM 和 RAG 基础

### Q4：Embedding、Chunk、Top-K、Rerank 怎么解释？

回答：

> Chunk 是把长文档切成小片段；Embedding 是把文本变成向量；Top-K 是先召回最相似的 K 个片段；Rerank 是再用更强模型对候选片段重新排序。RAG 的核心不是“接一个向量库”，而是让回答有可追溯上下文。

项目联系：

- 当前 RAG 用于健身和饮食知识问答。
- 后续优化是知识库治理、chunk 策略、top_k/threshold、混合检索、reranker 和 RAG eval。

### Q5：RAG 怎么评测？

回答：

> 至少分两层：检索评测和回答评测。检索看 recall@k、命中率、MRR、无关片段比例；回答看是否引用到了正确上下文、有没有幻觉、是否回答完整。项目目前重点是先把链路和评测意识讲清楚，生产化再扩充标准问答集。

面试口径：

> 我不会只说“用了 RAG”，而会说 RAG 效果要靠数据、切分、召回、重排和评测一起保证。

### Q6：如何处理幻觉？

回答：

> 不能承诺完全消除幻觉，只能降低风险。做法包括：约束 prompt 只基于检索上下文回答；没有依据时说不确定；保留来源；对高风险健康问题增加免责声明；必要时引入答案校验。

项目联系：

- 健身建议不是医疗诊断。
- 饮食建议遇到疾病、孕期等要提示咨询专业人士。

## 3. Agent、Tool 和 MCP

### Q7：Agent、Workflow、Chain 有什么区别？

回答：

> Chain 更像固定步骤调用；Workflow 是受控流程，可以有分支和状态；Agent 更强调根据任务选择工具、记忆和行动。这个项目更偏受控 Agent Workflow：不是让模型无限自主循环，而是用 Router 和子图限制行动空间。

项目联系：

- 顶层 Router 决定走哪个子图。
- 子图内部有固定节点和降级策略。
- 这比完全开放式 Agent 更适合面试项目和可演示原型。

### Q8：Tool Schema 为什么重要？

回答：

> Tool Schema 让模型或 Agent 知道工具需要什么输入、返回什么输出、失败时怎么处理。没有清晰 schema，工具调用就会变成字符串拼接，错误难定位。

项目联系：

- MCP 子图通过标准协议调用工具。
- Motion 工具把姿态分析结果结构化返回。
- 项目已有统一错误结构和 fallback 思路。

### Q9：MCP 的价值是什么？

回答：

> MCP 可以理解为把外部工具服务标准化，让 Agent 不需要为每个工具写一套私有协议。项目里的菜谱工具只是例子，重点是展示发现工具、规划参数、执行调用和失败降级这条链路。

面试口径：

> 我不会把菜谱功能当核心亮点，核心是标准化 Tool Use。

## 4. Router 和 Eval

### Q10：Router 当前怎么实现？

回答：

> Router 当前是确定性策略为主：加权规则累积分数，组合规则增强跨域表达，语义样例处理隐式表达，LLM classifier 作为可选 fallback。Phase 4 增加了 primary intent、secondary intents 和 route plan，并只对白名单组合执行多子图。

当前证据：

- Green eval：66/66。
- Challenge eval：36/36。
- Full pytest：114 passed, 1 skipped, 1 warning。

### Q11：为什么 challenge set 满分还不能说泛化完成？

回答：

> 因为 challenge set 是离线小样本，只能证明当前设计覆盖了这些已知困难模式，不能代表真实用户的所有表达。它的价值是防止回归、暴露边界和指导下一步优化，而不是宣称生产级泛化。

项目联系：

- 这体现评测意识。
- 面试时不要过度包装。

### Q12：多意图路由怎么讲？

回答：

> 很多真实问题不是单 intent，比如“先分析深蹲动作，再查最新纠正方法”。Phase 4 不直接开放任意子图串联，而是先记录 primary intent、secondary intents、route plan，再只对白名单里的两步组合执行。这样既能展示组合能力，也避免原型失控。

关键点：

- `intent` 仍兼容主意图。
- 非白名单组合降级为主意图单路由。
- 多步结果最后统一 synthesis。

## 5. 工程稳定性

### Q13：本地 LLM 内存问题怎么讲？

回答：

> 项目曾遇到本地 Qwen 重复加载和内存压力问题。我的处理不是换更大的机器，而是做共享 loader、减少重复初始化、保留 mock 模式、让默认演示路径稳定可跑。这是一个真实工程取舍：面试项目要能跑起来，不能被非必要模型能力拖垮。

### Q14：MCP Server 不存在怎么办？

回答：

> 项目提供 mock fallback。这样面试或本地测试时，即使真实 MCP server 没装好，主流程也不会崩。生产化时可以增加健康检查、超时、重试、熔断和工具权限控制。

### Q15：如何证明项目可维护？

回答：

> 我会从四点讲：模块边界清楚，RouterState 统一状态；接口和工具返回结构可控；Router 和核心接口有测试；文档记录了已实现、边界和后续路线。它不一定生产级，但不是散乱 demo。

## 6. 面试中容易踩坑的说法

不要说：

- 规则 Router 比 LLM 一定更强。
- RAG 可以完全解决幻觉。
- 有 eval 就代表真实线上效果好。
- Motion 已经具备专业教练级识别。

推荐说：

- 当前任务里确定性 Router 更稳、更便宜、更可解释。
- RAG 是降低幻觉风险，不是消除幻觉。
- 离线 eval 是回归保障，线上还需要真实样本闭环。
- Motion 当前验证姿态输入和分析链路，专业化依赖真实动作库和规则集。
