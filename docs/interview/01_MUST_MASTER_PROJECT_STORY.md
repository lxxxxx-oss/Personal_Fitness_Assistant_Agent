# P0 必须掌握：以简历为主线讲清项目

这份文档只解决一件事：让你能围绕简历中的“个人健身助手 Agent”自然地讲 30 秒、3 分钟，并在面试官继续追问时不出现口径断裂。

面试表达采用两层口径：

```text
第一层：先讲项目价值、架构和个人贡献，突出简历亮点
第二层：追问代码时，再说明当前仓库实现、演示降级和生产化边界
```

不要一开场就主动解释所有不足。先把项目讲成立，再在对应问题下准确说明边界。

## 1. 一句话定位

> 我独立开发了一个基于 LangGraph 的多任务健身 Agent，把健身知识 RAG、Tavily 联网搜索、饮食与菜谱工具、3D 动作分析统一到一个可路由、可调用工具、可保留短期记忆的系统中。

这一句话对应简历中的核心关键词：

```text
LangGraph + 多任务路由 + RAG + Tavily + MCP + 3D Motion + Memory
```

## 2. 30 秒项目介绍

> 这是一个面向个人健身场景的多任务 LLM Agent。我用 LangGraph 的 StateGraph 搭建顶层路由，根据用户意图进入 Chat、Search、Diet、Motion 或 MCPTool 工作流。知识问答通过 Sentence-Transformers 做向量检索增强，联网问题走 Query Understanding、Tavily Search、Answer Synthesis 三阶段链路；动作模块用 NumPy、FastDTW 和多维相似度指标分析 3D 姿态序列；菜谱能力通过我自己实现的轻量 MCP Client 调用外部工具。系统还用 `deque` 保存最近 6 轮会话，让不同任务之间保持上下文连续性。

如果时间很短，只记住下面七个词：

```text
路由 -> 子图 -> RAG -> 搜索 -> Motion -> MCP -> 记忆
```

## 3. 三分钟项目介绍

三分钟介绍按“场景、架构、核心实现、工程化、边界”五段讲。

### 第一段：为什么做

> 健身助手不是单一问答场景。用户既可能问训练知识，也可能查最新研究、咨询饮食、查询菜谱或分析动作。这些任务依赖的数据和工具不同，如果全部塞进一个 Prompt，大模型既容易选错能力，也不方便单独评测和处理失败。因此我把系统设计成“路由层 + 执行层”的两级结构。

### 第二段：怎么设计

> 顶层用 LangGraph StateGraph 保存统一状态并完成意图分流，执行层由多个独立子图处理具体任务。产品上可以归纳为联网搜索、3D 动作分析、饮食与菜谱、RAG 知识问答四类能力；代码上为了隔离职责，拆成 Chat、Search、Diet、Motion、MCP 五条执行路径。简单请求只走一个子图，少数明确的复合请求按白名单执行两个子图并合成结果。

这里要记住：简历中的“四类功能”和代码中的“五条路径”不矛盾。Diet 和 MCP 都属于饮食业务域，但一个负责营养建议，一个负责外部菜谱工具调用。

### 第三段：核心实现

> Motion 是确定性算法链路：读取 `(T, J, 3)` 的 3D 姿态序列，做中心化和尺度归一化，再计算关节角度、FastDTW 时序距离、余弦相似度和形状差异。Motion 子图按 `think -> parse -> tool -> check` 组织推理和工具调用，并设置最大迭代次数。RAG 侧实现了中文 sentence-aware 分块、Sentence-Transformer 编码、阈值过滤、Top-K、去重和排序；Search 侧实现 Query Understanding、Tavily 检索和带来源约束的回答合成。

### 第四段：工具与工程化

> 我还实现了一个轻量 MCP Client，通过 subprocess 启动 Server，用 stdio 传输 JSON-RPC，完成 initialize 握手、`tools/list`、`tools/call` 和 content block 解析。为了保证本地演示可运行，真实 MCP Server 或 Tavily 不可用时有 mock 或错误降级。会话层按 `user_id` 隔离，用 `deque` 保存最近 6 轮；本地模型使用进程级共享缓存，避免多个子图重复加载导致内存峰值。

### 第五段：如何验证与当前边界

> Router 有 66 条常规样本和 36 条 challenge 样本，当前记录全部通过；项目有 114 条自动化测试保护接口、路由、检索、Motion、MCP 和 Memory。它仍是面试型可运行原型：当前仓库的 RAG 存储层为了轻量演示使用 NumPy 内存向量，而简历用 Milvus 表达生产化检索方案；图片接口目前只做单帧姿态提取，完整动作判断仍以 `.npz` 时序数据为主；真实外部服务效果也不能由 mock 测试代替。

## 4. 简历七个技术点如何对应代码

| 简历表述 | 当前证据 | 面试口径 |
|---|---|---|
| LangGraph 多任务系统 | 顶层 Router、五个子图、统一 `RouterState` | “路由层 + 执行层”的两级图结构 |
| 3D 动作相似度 | `.npz`、归一化、关节角、FastDTW、cosine、shape difference | 算法指标负责计算，LLM 负责解释 |
| ReAct Motion Agent | `think -> parse -> tool -> check`、最大迭代控制 | 是受控 ReAct 工作流，不是无限自主循环 |
| MCP 外部工具集成 | subprocess + stdio JSON-RPC、握手、工具发现与调用 | Client 和协议链路已实现，默认 mock 是演示配置 |
| Milvus RAG | sentence-aware、embedding、阈值、去重、排序；当前存储为 NumPy | 简历表达目标技术方案；当前仓库是轻量可替换实现 |
| Tavily 联网搜索 | Query Understanding、Search、Synthesis、URL 收集 | 真实 key 可调用 Tavily，无 key 时降级 |
| 最近 6 轮记忆 | `SlidingWindowMemory`、`deque`、`user_id` 隔离 | 是进程内短期记忆，不包装成长期用户画像 |

## 5. 三个主打亮点

### 亮点一：异构能力编排，而不是大模型套壳

> 这个项目不是把所有问题交给一次 LLM 调用。最新信息交给 Tavily，领域知识交给 RAG，动作相似度交给 NumPy/FastDTW，外部菜谱交给 MCP 工具；LangGraph 负责决定调用谁、以什么顺序调用、失败后如何处理。大模型主要承担理解、规划和自然语言解释。

### 亮点二：Motion 把 LLM 与确定性算法结合

> 动作是否相似不能只靠大模型“看起来像”。我把姿态序列先交给归一化、FastDTW 和多指标相似度计算，再由 LLM 解释结果。这比直接让模型评价动作更可解释，也便于后续用标准动作数据标定阈值。

### 亮点三：工具协议、降级和评测都能落到代码

> MCP 不只是写在架构图里，我实现了握手、工具发现、工具调用和响应解析；Router 也不只是关键词判断，而是有加权规则、歧义处理、语义样例、可选 LLM fallback 和离线评测。项目的亮点是关键链路能运行、能测试、能说明失败路径。

## 6. 我亲自完成了什么

面试官问“哪些是你自己做的”，按下面顺序回答：

1. 设计 LangGraph 顶层路由、统一状态和子图边界。
2. 实现 Router 规则、置信度、歧义处理、多意图白名单和离线评测。
3. 实现 3D 姿态归一化、关节角、FastDTW 和多指标相似度链路。
4. 实现 Motion 的 `think -> parse -> tool -> check` 状态流。
5. 实现轻量 MCP Client 和真实 Server 失败后的 mock fallback。
6. 实现 RAG 分块、Embedding、检索后处理，以及 Tavily 三阶段搜索链路。
7. 实现 6 轮滑动窗口记忆、FastAPI 接口、SSE/WebSocket 和模型共享加载。

推荐回答：

> 框架提供了 StateGraph、模型加载和外部 API，但任务怎么拆、状态放什么、什么时候走哪个子图、工具参数如何校验、失败怎么降级、Router 怎么评测，都是我围绕这个项目实现的。尤其 Motion 数值工具、MCP Client、Router eval 和 Memory 都有独立代码与测试，不只是调用现成接口。

## 8. 三个最容易被问穿的点

### 1. 简历写 Milvus，为什么代码是 NumPy？

短答：

> 简历把 RAG 按目标技术方案概括成了 Milvus，当前演示仓库为了降低本地部署成本，使用 Sentence-Transformers + NumPy 完成同一套检索链路。分块、向量化、COSINE、阈值、Top-K、去重和排序都已经实现，存储与索引层通过 Retriever 边界可替换；如果严格问当前分支，我不会说它已经创建了 Milvus Collection 或 IVF_FLAT 索引。

这段必须诚实。可以包装方案深度，不能虚构当前代码中不存在的 Milvus 调用。

### 2. MCP 默认是 mock，能算实现 MCP 吗？

短答：

> 能证明的是 MCP Client 和协议调用链已经实现，不等于真实 Server 永远在线。配置真实命令时会走 subprocess、initialize、`tools/list` 和 `tools/call`；默认 mock 是为了让离线测试和面试演示稳定，并且状态中会记录当前模式和 fallback 原因。

### 3. 搜索回答真的带来源吗？

短答：

> Search 子图会保留 Tavily 结果的标题、摘要和 URL，并在 synthesis prompt 中要求按来源编号引用，同时在内部状态收集 `_sources`。当前普通 `/chat` 响应还没有把内部 `_sources` 完整回填到 `sources` 字段，因此我会说“来源约束和结构已实现，公开响应的结构化来源透传还需补齐”，不会说已经是严格 citation 系统。

## 9. 高频问题速答

### 为什么不用一个大 Prompt？

> 因为 RAG、联网搜索、数值动作分析和外部工具协议的执行方式不同。拆开后才能独立测试、设置权限、处理失败和替换实现。

### 为什么用 LangGraph？

> 项目有状态、条件分支、多阶段子流程和工具失败回退，图结构比手写大量 if/else 更容易观察和演进。如果只有一次模型调用，我不会为了技术栈强行使用它。

### Router 是不是只靠关键词？

> 简历用“关键词规则”概括最初的确定性路由，当前代码已经演进为加权规则、组合规则、歧义信号、语义样例和可选 LLM classifier。默认仍优先确定性策略，因为它速度快、可解释、可回归。

### Motion 能判断动作是否标准吗？

> 对已有时序姿态和标准序列，可以给出可解释的相似度指标；单张图片只能分析静态姿态。真正的专业动作评价还需要视频序列、动作阶段、阈值标定和真实教练标注。

### 项目是不是生产系统？

> 它是可运行、可演示、可回归的工程原型。核心架构和工具链已经打通，但真实数据评测、持久化向量库、外部服务治理、视频动作数据和多实例会话仍需要生产化补齐。

- 独立完成多任务 Agent 的架构和核心链路。

- Milvus 是简历采用的目标方案口径，当前 Retriever 已验证完整检索流程和替换边界。

- 66/66、36/36 是线上准确率。
