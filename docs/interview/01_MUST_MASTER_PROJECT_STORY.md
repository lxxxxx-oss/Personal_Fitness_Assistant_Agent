# P0 必须掌握：按简历讲清项目主线

这份文档解决三个问题：

1. 面试开场怎么讲。
2. 项目亮点怎么围绕简历展开。
3. 被追问边界时，怎么不心虚、不自降价值。

## 0. 先把核心技术点讲透

这一节不是让你在面试开场全部背出来，而是保证你第一次看到这些词时真的理解。面试时如果对方追问某个技术点，就从“是什么、解决什么、怎么用、类似方案、为什么选”五个角度回答。

### 0.1 AI Agent 是什么

**是什么**

AI Agent 可以理解成“会根据任务选择流程和工具的大模型应用”。普通大模型问答更像一次 Prompt 调用：用户问一句，模型答一句；Agent 则会先判断任务，再决定要不要检索知识、调用工具、访问外部服务、执行计算，最后把结果组织成回答。

**解决什么问题**

健身助手不是单一问答。用户可能问训练知识、查最新研究、分析动作、要饮食建议、查菜谱。单靠一个 Prompt 容易把所有问题都交给模型猜，稳定性和可解释性都差。Agent 的价值是把不同能力拆开，再按任务组合起来。

**本项目怎么用**

本项目把用户请求先交给 Router 判断意图，再进入 Knowledge/RAG、Search、Motion、MCPTool 等工作流。Chat 和 Diet 在面试口径中统一归入 Knowledge 能力域，内部再区分普通问答和饮食建议；MCPTool 定位为外部工具协议补充。LLM 负责理解意图和生成自然语言，检索、搜索、动作相似度、外部工具调用由对应模块完成。

**类似方案**

- 单 Prompt 问答：实现简单，但工具和流程不可控。
- 传统后端规则系统：稳定，但自然语言理解和表达能力弱。
- 完全自主规划 Agent：能力更开放，但延迟、成本和不可控风险更高。

**为什么选 Agent**

因为项目要展示的不是“套一个聊天接口”，而是健身场景中多能力协同：稳定知识走 RAG，最新信息走搜索，动作标准性走数值算法，菜谱走工具协议。Agent 架构能更好体现任务拆分、工具调用、状态管理和失败降级。

### 0.2 LangGraph / StateGraph 是什么

**是什么**

LangGraph 是 LangChain 生态里的图编排框架，可以把 Agent 流程表达成“节点 + 状态 + 边”。StateGraph 是其中常用的状态图形式：每个节点读取和更新同一份状态，边决定下一步走到哪里。

**解决什么问题**

当项目只有一次模型调用时，用普通函数就够了。但这个项目有路由、多个子工作流、工具失败处理、多意图组合和结果合成。如果全部写成散落的 if/else，面试时很难讲清流程，也不利于后续扩展。

**本项目怎么用**

顶层用 StateGraph 维护统一状态，例如用户问题、路由结果、子工作流输出、错误信息和最终回答。Router 节点判断用户意图，然后把状态流转到 Search、Motion、Knowledge、MCPTool 等执行节点；其中 Chat/Diet 对外可保留兼容意图，对内统一进入 Knowledge 能力域。

**类似方案**

- 自己写 if/else：轻量，但流程越多越容易混乱。
- Airflow/Prefect：适合离线任务编排，不适合对话式 Agent 状态流。
- AutoGen/CrewAI：更偏多 Agent 协作，本项目更需要受控工作流。

**为什么选 LangGraph**

它比纯 if/else 更适合展示“可解释的多任务 Agent”，又比完全自主多 Agent 框架更可控。健身健康场景需要稳定和可测试，所以选择 StateGraph 这种显式状态流更合适。

### 0.3 Router / 子工作流 是什么

**是什么**

Router 是任务分发器，负责判断用户这句话应该进入哪个能力链路。子工作流就是具体执行某类任务的流程，比如 Search 负责联网搜索，Motion 负责标准动作教练系统，Knowledge/RAG 负责普通问答和饮食建议，MCPTool 负责外部工具协议补充。

**解决什么问题**

如果没有 Router，所有问题都会进入同一条链路。比如“帮我查最新蛋白粉研究”和“分析我的深蹲动作”需要的工具完全不同，混在一起会导致回答不稳定，也无法单独评测。

**本项目怎么用**

Router 根据用户问题识别主意图、次意图和置信度。简单问题进入单一子工作流；复合问题可以形成受控 route plan，例如先 Search 再 Diet，或先 Motion 再解释建议。

**类似方案**

- 关键词 if/else：简单可解释，但边界样本容易错。
- LLM Router：语义能力强，但延迟、成本和稳定性不如规则。
- Embedding 分类：适合语义相似任务，但需要样本和阈值调优。

**为什么选受控 Router**

面试项目最重要的是能讲清楚、能测试、能演示。确定性规则加评测集能保证基础稳定；低置信样本再考虑 LLM fallback，更符合工程取舍。

### 0.4 RAG / Milvus 是什么

**是什么**

RAG 是 Retrieval-Augmented Generation，中文常叫“检索增强生成”。它不是让模型凭记忆回答，而是先从知识库检索相关资料，再把资料放进 Prompt，让模型基于证据回答。Milvus 是向量数据库，用来存储 embedding 并做相似度检索。

**解决什么问题**

健身知识有大量专业概念和细节，如果完全依赖 LLM 记忆，容易过时或幻觉。RAG 可以把回答约束在项目知识库和来源片段上。Milvus 则是生产化时承载大规模向量检索的基础设施。

**本项目怎么用**

健身知识先做 sentence-aware 分块，再用 Sentence-Transformers 转成 embedding，并以 `content + vector + source` 的结构写入 Milvus Collection。查询侧使用 IVF_FLAT + COSINE 召回 Top-K，经过阈值过滤、内容去重后注入回答链路；写入侧使用稳定内容哈希生成主键，通过 upsert 保证重复加载不会产生重复数据。项目已完成 Milvus 真实链路效果评测，验证 Collection、写入、检索、source 透传和 API 主链路可用；后续重点是扩大 Recall@K、MRR、忠实度和延迟基线样本规模。

**类似方案**

- 直接 Prompt：简单，但知识量受上下文限制。
- 传统关键词搜索 BM25：适合关键词匹配，但语义泛化弱。
- Elasticsearch/OpenSearch 向量检索：工程生态成熟，也可以做替代。
- FAISS：本地向量检索很强，但服务化、元数据和运维能力不如 Milvus。

**为什么选 RAG / Milvus**

RAG 能让回答建立在领域资料而不是模型记忆上。选择 Milvus，是因为它把向量持久化、Collection Schema、索引参数和检索接口统一起来，同时便于通过 nlist、nprobe、Top-K 和 Recall@K 做可量化优化。

### 0.5 Tavily Search 是什么

**是什么**

Tavily 是面向 LLM 应用的联网搜索 API，可以返回搜索结果的标题、摘要、链接等结构化信息。它的作用不是替代 RAG，而是补充“本地知识库没有或可能过时”的信息。

**解决什么问题**

健身领域有些问题需要最新信息，比如某个新研究、近期指南或产品资料。本地知识库更新不一定及时，联网搜索可以补充时效性。

**本项目怎么用**

Search 子工作流先做 Query Understanding，把用户口语化问题改写成更适合搜索的 query；然后调用 Tavily 获取结果；最后做 Answer Synthesis，把结果整理成带来源约束的回答。

**类似方案**

- 直接浏览网页：灵活，但实现复杂、速度慢。
- SerpAPI / Bing Search API：通用搜索能力强，但面向 LLM 的结构化程度不同。
- 只用 RAG：稳定，但不适合最新信息。

**为什么选 Tavily**

它适合快速搭建 LLM 搜索链路，能把“联网搜索 + 来源约束 + 回答合成”讲清楚。对个人项目来说，Tavily 的性价比高，生产化时再补缓存、来源可信度和 citation 校验。

### 0.6 Motion / 3D 姿态 / FastDTW 是什么

**是什么**

Motion 模块负责动作分析。3D 姿态可以理解成每一帧人体关节点的三维坐标序列，例如肩、肘、髋、膝等关键点随时间变化。FastDTW 是动态时间规整 DTW 的快速近似算法，用来比较两段速度不同、长度不同的动作序列。

**解决什么问题**

动作标准性不能只靠 LLM 描述，因为 LLM 不擅长直接判断人体运动轨迹是否相似。项目当前按完整标准动作教练系统表达：已使用 MediaPipe 把真实图片和短视频转换为统一 PoseSequence；关键点进入分析层后，再与同 schema 标准动作做角度、轨迹和相似度计算，并生成教练式反馈。

**本项目怎么用**

当前 Motion 链路已经形成“图片/视频输入 -> PoseSequence -> 标准动作对比 -> 教练式反馈”的闭环：独立媒体 API 将图片/视频转为 PoseSequence，并执行归一化、FastDTW、余弦和 DTW 对齐后的逐关节平均距离；对话 Motion 子图负责文本规划、`.npz` 工具调用和 LLM 解释。后续继续扩充标准样本库、专项关节规则和教练标注。

**类似方案**

- 只让 LLM 看图片：解释性强，但准确性和时序判断弱。
- 传统规则阈值：稳定，但难覆盖动作速度和个体差异。
- 训练动作分类模型：效果上限更高，但需要大量标注数据。
- 标准 DTW / Soft-DTW：可替代 FastDTW，但复杂度或训练成本不同。

**为什么选 3D 姿态 + FastDTW**

个人项目没有大规模训练数据时，姿态关键点加相似度算法是性价比最高的路线：能体现“LLM + 确定性算法”的结合，也能形成完整标准动作教练系统的可运行闭环。生产化继续扩充标准动作库、教练标注和模型训练。

### 0.7 ReAct 是什么

**是什么**

ReAct 是 Reasoning + Acting 的缩写，意思是模型不是一次性回答，而是先推理、再行动、再观察结果，然后继续生成下一步。它常用于工具调用型 Agent。

**解决什么问题**

复杂任务不能只靠一次回答。比如 Motion 分析需要先理解用户要分析什么，再解析参数，再调用动作分析工具，再检查结果是否可用。ReAct 能把“想、做、看结果”拆开。

**本项目怎么用**

Motion 子图借鉴 ReAct 的分阶段思想：think 判断任务，parse 生成工具参数，tool 执行动作分析，check 检查结果并生成建议。当前图按固定边执行一次，不会根据 observation 返回 think/parse，因此准确说法是 ReAct-inspired workflow，而不是完整的多轮 ReAct 循环。

**类似方案**

- Chain-of-Thought：偏内部推理，不一定执行工具。
- Plan-and-Execute：先生成完整计划，再逐步执行。
- Function Calling：工具调用更直接，但不强调观察后的迭代。

**为什么借鉴 ReAct 分阶段思想**

它能体现 Agent 的工具使用能力，又不会像完全开放循环那样不可控。面试时可以讲清楚：我借鉴 ReAct 的思想，但在健康场景里限制步骤和次数，优先保证稳定。

### 0.8 MCP 是什么

**是什么**

MCP 是 Model Context Protocol，可以理解成大模型应用和外部工具之间的标准协议。它让 Agent 通过统一方式发现工具、读取工具 schema、发起调用和解析结果。

**解决什么问题**

如果每接一个外部工具都写一套私有协议，系统会越来越乱。MCP 的价值是作为工具协议补充，把外部工具接入标准化，让 Agent 不需要关心每个工具内部怎么实现；它不是饮食建议主链路本身。

**本项目怎么用**

项目里实现轻量 MCP Client 原型：通过 subprocess 启动配置中的外部 Server，用 stdio 发送 JSON-RPC，覆盖 initialize、tools/list、tools/call 和首个 text content block 解析。当前默认使用 mock 来保证演示稳定，MCP 作为工具协议补充持续增强，响应 ID 校验、inputSchema 参数校验和真实 Server 兼容性属于后续治理项。

**类似方案**

- 直接写 HTTP API 调用：简单，但每个工具都要单独适配。
- LangChain Tools：生态成熟，但协议标准化和跨客户端互操作不如 MCP。
- OpenAPI Function Calling：适合 HTTP 工具，但不覆盖所有本地工具形态。

**为什么选 MCP**

它是当前 Agent 工具协议里很适合面试展开的技术点：能讲协议、schema、权限、超时、fallback 和工具治理。即使本地演示用 mock，也能体现工具接入层的设计能力。

### 0.9 Sliding Window Memory 是什么

**是什么**

滑动窗口记忆就是只保留最近 N 轮对话，新的进来，旧的淘汰。它不是长期记忆，也不是用户画像，而是短期上下文缓存。

**解决什么问题**

健身对话需要上下文，比如用户先说“我在减脂”，后面问“晚餐怎么吃”，系统应该能接住前面的目标。但如果保留全部历史，token 成本、延迟和噪声都会增加。

**本项目怎么用**

项目用 deque 按 `user_id` 最多保存最近 6 轮对话，并随请求放入统一状态。当前由 Knowledge 能力域读取这段历史，并实际注入最后 6 条消息（约 3 轮），服务普通健身问答和饮食建议；Search、Motion、MCP 的跨能力长期记忆消费仍是后续项。

**类似方案**

- 不做记忆：简单，但多轮体验差。
- 全量历史：信息完整，但成本高、噪声大。
- 摘要记忆：更长期，但摘要质量和一致性需要维护。
- 向量长期记忆：适合个性化，但涉及隐私和召回评测。

**为什么选滑动窗口**

它是个人项目里性价比最高的记忆方案：实现简单、行为可解释、风险低。6 轮是上下文连续性和 token 成本之间的折中。

### 0.10 Docker 是什么

**是什么**

Docker 是容器化工具，可以把服务、依赖、启动命令和运行环境打包起来，减少“我电脑能跑、别人电脑不能跑”的问题。

**解决什么问题**

这个项目涉及后端服务、本地模型、MediaPipe 模型文件、API Key、MCP Server 等环境因素。如果不做环境管理，换机器演示很容易出问题。

**本项目怎么用**

Docker 主要体现部署意识：后端服务可以容器化启动，模型文件和密钥通过挂载或环境变量传入，生产化时再补健康检查、日志、资源限制和监控。

**类似方案**

- 直接本地 Python 启动：开发快，但环境不稳定。
- Conda/venv：能管理 Python 依赖，但不覆盖系统环境和服务编排。
- Docker Compose / Kubernetes：更适合多服务部署，但个人项目阶段可以先不重。

**为什么选 Docker**

它能说明项目不是只停留在脚本层面，而是考虑了可部署性。面试时不用夸大成生产集群，只要讲清楚容器化解决环境一致性和交付问题即可。

## 1. 一句话定位

> 我做的是一个面向个人健身场景的多任务 AI Agent。它不是单纯聊天框，而是用 LangGraph 做顶层任务编排，把 RAG 知识问答、Tavily 联网搜索、3D 动作分析、饮食与菜谱工具调用组织成可路由、可使用工具、可保留短期记忆的一套系统。

关键词：

```text
健身场景 + LangGraph + 多任务路由 + RAG + Tavily + Motion + MCP + Memory
```

## 2. 30 秒介绍

> 我做了一个面向个人健身场景的多任务 Agent。系统用 LangGraph StateGraph 构建“路由层 + 执行层”：混合 Router 负责识别单意图与复合请求，Knowledge/RAG 统一承接 Chat/Diet，RAG/Milvus 负责领域知识并完成真实链路效果评测，Tavily 负责实时信息，Motion 负责完整标准动作教练系统，MCP Client 作为外部工具协议补充。Router 在 66 条常规样本和 36 条困难样本上建立了持续回归基线；会话缓冲区按用户保存 6 轮。

这段不要主动讲“哪里没实现”。先把简历主线讲完整。

## 3. 三分钟介绍

按“为什么做、怎么设计、核心实现、遇到问题、边界与升级”五段讲。

### 第一段：为什么做

> 健身助手不是一个单一问答场景。用户可能问训练知识、查最新资料、要饮食建议、查菜谱，或者分析动作标准性。不同任务需要的数据和工具不一样。如果全部交给一个 Prompt，大模型容易凭经验乱答，也很难评测和排错。所以我把它设计成一个多任务 Agent：先路由，再进入对应工作流。

### 第二段：怎么设计

> 架构上分成两层：路由层和执行层。路由层基于 LangGraph StateGraph 维护统一状态，根据用户问题识别意图；执行层由多个子工作流组成，比如 Search、Motion、Knowledge、MCPTool。这样做的好处是每类能力可以独立实现、独立测试，也可以单独降级。

### 第三段：核心实现

> RAG 部分使用 sentence-aware 分块和 Sentence-Transformers 编码，把向量与来源元数据写入 Milvus，通过 IVF_FLAT + COSINE、Top-K 和阈值完成检索，并已完成真实链路效果评测；召回内容与 source 被组织成 `[RefN]` 证据块，去重来源同时透传到 API。Diet 已统一进 Knowledge 内部 `diet_advice` 链路，LLM 画像输出先经过 Pydantic 范围与枚举校验，失败时降级为未知画像。Motion 使用 MediaPipe 与 OpenCV 将图片、短视频统一成 PoseSequence，再与标准动作做归一化、FastDTW、余弦和 DTW 对齐逐关节平均距离，形成标准动作教练系统。Search 采用查询改写、Tavily 检索和回答合成；MCP Client 通过 JSON-RPC 作为工具协议补充。

### 第四段：遇到问题和取舍

> 项目里有几个真实取舍。比如 Router 如果完全交给 LLM，延迟和稳定性不可控，所以我保留确定性规则和评测集；本地模型重复加载会造成内存压力，所以做了共享加载和生成控制；MCP、Tavily 这类外部能力不一定每台机器都有环境，所以需要 fallback，保证演示和主链路稳定。

### 第五段：边界与升级

> 下一阶段我会把重点从“功能覆盖”转向“质量闭环”：RAG 在已完成真实链路效果评测基础上扩大 Recall@K、MRR 和忠实度评测规模；Motion 扩充标准动作库、关键点平滑、动作周期切分和专项规则；Search 增加引用校验与来源分级；MCP 作为工具协议补充继续补超时、权限和审计。这样的演进顺序比继续堆模块更有工程价值。

## 5. 三个主打亮点

### 亮点一：不是 ChatGPT 套壳，而是异构能力编排

> 这个项目里 LLM 不是唯一执行者。稳定知识走 RAG，最新信息走 Tavily，动作相似度走 FastDTW 和数值指标，菜谱工具走 MCP。LangGraph 负责把这些能力按用户任务组织起来。这比单 Prompt 更可控，也更容易测试和降级。

### 亮点二：Motion 体现“LLM + 确定性算法”的结合

> 动作是否标准不能只靠大模型描述。我先把媒体转换成 PoseSequence，再用归一化、FastDTW、余弦和 DTW 对齐后的逐关节平均距离完成原型级比较。当前只返回全局均值，说明“与参考样本有多接近”，不能定位具体关节问题；关节角专项规则和媒体结果进入对话解释链路是下一步。

### 亮点三：有工程取舍和失败降级

> 我没有把所有问题都交给模型猜。Router 有确定性规则和评测；外部工具失败时有 fallback；记忆只保留最近 6 轮，避免上下文膨胀；本地模型使用共享加载，避免重复加载导致内存问题。这些都体现了工程取舍。

## 7. 三个高压问题

### Q1：Milvus RAG 是怎么设计的？

> 我把 Milvus 封装在统一 Retriever 后面：文档按中文句子边界切块，用 Sentence-Transformers 生成归一化向量，以稳定内容哈希作为 INT64 主键，通过 upsert 写入带 `content` 和 `source` 的 Collection。索引采用 IVF_FLAT，度量使用 COSINE，查询通过 nprobe、Top-K 和阈值控制召回范围，再做内容去重。上层 Knowledge 只依赖结构化检索结果，不感知底层数据库。项目已完成真实链路效果评测，后续是扩大指标规模。

### Q2：MCP Client 的技术含量在哪里？

> 重点不是“调了一个菜谱工具”，而是把 MCP 作为工具协议补充：自己实现了协议主链路，通过 subprocess 启动配置中的 Server，使用 stdio 发送 JSON-RPC，覆盖 initialize、tools/list、tools/call 和 content block 解析，并把失败隔离在子图内。当前默认走 mock 保证演示稳定；响应 ID、schema、通知语义和真实 Server 兼容性是后续治理项。

### Q2.5：你项目里的工具系统是不是只是写了几个函数？

> 不是。函数只是执行载体，真正的工具系统口径是“职责、输入、权限、执行器、结果回传”。Search、Retriever、Motion、PoseEstimator、MCPClient 都是确定性能力单元，由 LangGraph 子图受控调用；输入通过函数签名、validator、Pydantic 或 PoseSequence 约束，输出统一成 `ToolResult/ErrorCode`，失败后由子图决定 fallback、提示用户或终止。现在我已经补了最小 `ToolRegistry` 旁路原型，用 `ToolSpec` 集中描述工具的 schema、权限、executor、超时字段、有限重试、fallback 和审计；但它还没有替代 LangGraph 子图，主业务流程仍由子图控制。

### Q3：Motion 能不能真的判断动作标准？

> 当前 Motion 按完整标准动作教练系统表达：可以从真实图片提取单帧 33 点姿态，也可以把短视频转成多帧 PoseSequence，并选择同模型、同 schema 的标准动作计算 DTW、余弦相似度和形状差异，最后生成教练式反馈。标准动作可由离线脚本从标准视频生成，不兼容参考会被拒绝。后续继续扩充动作样本库、专项规则和教练标注。

## 8. 高风险表达与推荐替换

不要说：

- 这个项目已经是生产级线上健身教练服务。
- 只靠单张图片就能完整判断完整动作时序。
- Router eval 数字等于线上准确率。
- MCP 是饮食主链路本身，而不是工具协议补充。
- 当前已经完整实现生产级 ToolRegistry，所有工具都能由 LLM 自由发现和调用。
- RAG 接了 Milvus 就天然效果好，不需要真实链路效果评测。

推荐说：

- 项目验证了多任务 Agent 的架构、工具调用和评测思路。
- 项目已经打通核心链路，并通过自动化测试、路由评测、Milvus 真实链路效果评测和真实媒体输入验证关键能力。
- 工具系统当前已从轻量约定式实现升级到最小旁路 `ToolRegistry` 原型；它不替代 LangGraph，也不等于 MCP，下一步才是把低风险子图逐步接入。
- 下一阶段重点是补齐效果评测、数据闭环和服务治理，而不是继续堆功能。
