# P0 必须掌握：先听懂，再背会

这份是整个项目的入门文档。不要从技术名词开始硬背，也不要把答案背成很死的标准稿。面试官更关心的是：你为什么做这个项目、为什么这样设计、遇到问题怎么处理、哪些地方你主动设置了边界。

按下面的顺序学习：

1. 先读“健身服务中心”的比喻，弄懂项目在做什么。
2. 再背 30 秒和 3 分钟介绍。
3. 最后学习证据、边界和高频问题。

如果第一遍只能理解一半，很正常。先抓住主线：**系统先判断用户要做什么，再把任务交给合适的模块。**

回答任何项目问题时，都尽量带上这四层：

```text
为什么这样做
  -> 我做了什么
  -> 我怎么验证
  -> 我知道还差什么
```

## 1. 先用一个比喻理解项目

可以把这个项目想成一家“智能健身服务中心”。

用户进来以后，不是所有问题都交给同一个人处理：

- 前台先判断用户需要什么服务。
- 健身知识问题交给知识问答区。
- 最新资讯交给搜索区。
- 减脂、增肌饮食交给饮食区。
- 动作数据交给动作分析区。
- 在目标架构里，具体菜谱和餐单也统一交给饮食区处理。

在项目里：

- **Router（路由器）**就是前台，负责判断用户要去哪个区域。
- **子图（Subgraph）**就是不同的业务区域，每个区域有自己的处理步骤。
- **Tool（工具）**就是具体设备，例如搜索工具、动作相似度计算工具。
- **RouterState（状态）**像一张任务单，保存用户问题、路由结果、执行过程和最终答案。
- **LangGraph**负责按照任务单，把整个流程连接起来。

这就是项目最核心的逻辑。其他技术都是围绕这条主线展开的。

## 2. 一句话介绍

先把这一句说顺：

> 我做了一个个人健身 AI Agent。系统会先判断用户想问健身知识、搜索最新资料、获得饮食或菜谱建议，还是分析动作，然后把任务交给不同模块处理。

这一句话不炫技，但面试官能立刻听懂项目做什么。

## 3. 30 秒介绍：建议直接背熟

> 这是一个面向个人健身场景的 AI Agent 项目。我用 LangGraph 把健身问答、联网搜索、饮食与菜谱建议、动作分析拆成四个核心模块，再由 Router 判断用户需要哪个模块。这个项目的重点不是做一个普通聊天框，而是让不同能力能够被正确选择、组合、测试和降级。目前项目可以完整运行和演示，但 Motion、RAG 和部署等能力仍然是原型水平，我会明确说明已经完成和还没完成的部分。

背诵时记住四个关键词即可：

```text
四个模块 -> Router 分配 -> 可以测试 -> 边界诚实
```

## 4. 三分钟介绍：按五段讲

不用逐字背整段。记住五个小标题：**为什么做、怎么设计、我做了什么、如何验证、还缺什么。**

### 第一段：为什么做

> 健身场景不只有普通问答。用户可能要查最新资料、制定饮食建议、上传动作数据，或者查询一道菜的做法。这些任务需要的能力不一样。如果全部交给一个大模型和一个 Prompt，系统很难稳定选择工具，也不方便单独测试和定位错误。

### 第二段：怎么设计

> 所以我用 FastAPI 提供接口，用 LangGraph 组织任务流程。目标架构由 Chat、Search、Diet 和 Motion 四个核心模块组成。正常情况只执行一个模块；少数明确的多任务请求，可以按照白名单顺序执行两个模块。当前代码还保留早期 MCP 菜谱路由，后续会把菜谱请求并入 Diet 并删除这条遗留链路。

### 第三段：我做了什么

> 我主要完成了四类工作：第一，设计 Router 和路由评测；第二，搭建 LangGraph 状态和子图流程；第三，完成 PoseSequence、图片关键点提取和 `.npz` 动作相似度链路；第四，处理本地模型共享加载、内存、会话记忆和流式接口等运行问题。

### 第四段：如何验证

> Router 有 66 条常规回归样本和 36 条困难样本，当前代码对这些样本全部通过。项目还有 114 条自动化测试，用来保护接口、路由、工具和错误处理不被后续修改破坏。不过测试中大量使用了 mock，所以这些数字证明的是代码回归稳定性，不代表真实大模型回答质量。

### 第五段：还缺什么

> 当前项目是可运行原型，不是生产系统。图片 Motion 只能提取单帧姿态和置信度，不能判断完整动作节奏；视频分析和真实标准动作库还没完成。RAG 也缺少标准评测集。我的重点是先把架构、工具链、评测方法和失败降级做清楚，再逐步补真实数据和生产能力。

## 5. 我亲自完成了什么

面试官很可能会问：“这些库本来就有，你自己到底做了什么？”

可以按下面五点回答。

### 1. Router 和评测

我设计了意图规则、分数计算、困难样本处理、多意图字段和评测脚本。

简单理解：**我不只让系统做选择，还为这些选择准备了考试题。**

### 2. LangGraph 流程

我把四类核心能力组织成不同模块，定义统一状态，并处理单模块失败和多模块结果合成。

简单理解：**我负责把各个能力连接成一条可控制的工作流。**

### 3. Motion 工具链

我定义了统一姿态格式 PoseSequence，接入图片关键点提取，并保留 `.npz` 序列相似度分析。

简单理解：**我把图片或姿态数据转成程序能够计算的结构。**

### 4. 可运行性

我处理了模型重复加载导致的内存压力、会话记忆、流式接口以及外部服务失败后的降级。

简单理解：**我不只让项目“写出来”，还尽量让它能稳定演示。**

如果项目不是独立完成，要按照真实情况改成“我负责的部分”，不要把团队成果全部说成自己的。

## 6. 四个核心模块分别做什么

### Chat：健身知识问答

```text
用户问题 -> 从本地知识库找资料 -> 大模型根据资料回答
```

这种“先找资料再回答”的方式叫 RAG。

当前边界：知识库规模不大，也没有标准 RAG 评测集。

### Search：联网搜索

```text
用户问题 -> 整理搜索词 -> Tavily 搜索 -> 根据搜索结果回答
```

当前边界：外部搜索可能不可用，也需要继续加强来源筛选。

### Diet：饮食建议

```text
提取身高、体重、目标和偏好 -> 查营养资料 -> 生成建议
```

当前边界：只能提供一般信息，不能代替医生或专业营养师。

具体菜谱、食材搭配和餐单请求也计划统一由 Diet 处理，避免为了菜谱再维护一套独立路由。

### Motion：动作相关能力

目前有两条不同链路：

- `.npz` 姿态序列：可以和标准动作数据计算相似度。
- 图片上传：只提取单帧人体关键点和置信度摘要。

当前边界：图片还不能判断完整动作是否标准，视频链路也没有完成。

## 7. 三个亮点怎么讲

### 亮点一：不是所有事情都让大模型猜

> 我一开始考虑过直接用一个大 Prompt 兜住所有问题，但很快会遇到两个问题：最新资料、动作分析这些任务不适合让模型凭空猜；同时所有能力混在一个 Prompt 里也不好测试。所以我把大模型放在理解和表达的位置，把搜索、RAG、动作计算这些能力拆出去，由 Router 选择。这样牺牲了一点实现简单性，但换来了可控、可测和可降级。

### 亮点二：Router 有评测

> Router 是这个项目的控制面，不能只靠感觉调规则。我给它准备了常规样本和困难样本，每次修改以后都能重新跑 eval。66/66 和 36/36 不代表线上准确率，但能说明当前规则对已知样本没有回归。这个边界我会主动说明，避免把离线小样本夸大成生产指标。

### 亮点三：失败时还能运行

> 我遇到过本地模型加载、外部服务和依赖不稳定的问题，所以项目里保留了 mock、关键词匹配、共享模型加载等降级方案。这样做不是为了假装能力完整，而是为了保证面试演示时核心链路能跑，并且清楚区分真实能力和替代路径。

## 8. 必须记住的三个数字

### 66 条

Router 常规回归样本，用来保护主流路由行为。

### 36 条

Router 困难样本，主要覆盖边界冲突和多意图表达。

### 114 条

自动化测试数量，覆盖接口、Router、工具、Motion、记忆等模块。

这三个数字都不能被解释成“生产准确率”：

- 66/66 和 36/36 只说明当前代码通过项目自己整理的样本。
- 114 条测试大量使用 mock，不代表真实 LLM、Embedding 或搜索效果；其中仍包含待移除的 MCP 遗留测试。

## 9. 六个高频问题

### Q1：为什么不直接用一个 Prompt？

> 因为不同任务需要不同工具和失败处理。拆成模块以后，可以单独测试、替换和降级。

### Q2：为什么使用 LangGraph？

> 因为项目有状态、分支和多个处理步骤。LangGraph 可以把这些节点和流转关系表达清楚。如果只有一次函数调用，我不会强行使用它。

### Q3：子图和工具有什么区别？

> 工具是一个具体动作，子图是一段流程。例如搜索工具只负责查询，Search 子图还要整理搜索词、调用工具并生成最终回答。

### Q4：为什么 Router 不直接交给 LLM？

> 路由决定系统下一步做什么，需要稳定、快速和可解释。我测试过本地 Qwen Router，但没有增加正确样本，反而增加约 6 秒延迟，所以默认关闭。

### Q5：图片能判断深蹲标准吗？

> 目前不能。单张图片只能提取一个时刻的姿态，无法判断动作轨迹、节奏和重复稳定性。当前图片接口也只返回关键点和置信度摘要。

### Q6：怎么证明项目不只是 Demo？

> 我不会说它已经生产可用，但它有清晰模块、路由评测、自动化测试、工具错误处理和真实问题修复记录。它是一个能运行、能演示、能解释、能继续迭代的原型。

## 10. 面试时一定要说清楚的边界

不要说：

- 图片或视频已经可以专业判断所有健身动作。
- Router 的 102 条样本代表真实线上准确率。
- 114 条测试证明真实模型回答质量。
- WebSocket 已经实现真正实时的逐 token 输出。
- MCP 菜谱路由是当前面试主线中的核心能力。
- RAG 可以完全避免幻觉。

可以说：

- 当前是可运行、可演示、可回归验证的 Agent 原型。
- Router 在项目自建样本上表现稳定，真实泛化还需要用户数据。
- Motion 已完成部分输入和分析组件，视频与完整动作判断仍待补。
- 项目重点是任务编排、评测、工具调用和失败降级。
- 当前代码仍保留早期 MCP 菜谱实验，但目标架构会将菜谱并入 Diet；在代码真正清理前要主动说明这一过渡状态。

第一轮只掌握：

- 第 1 节比喻。
- 第 2 节一句话介绍。

- 第 9 节六个高频问题。
- 第 10 节能力边界。

1. 项目是做什么的？
2. 为什么要拆成多个模块？
3. 我亲自完成了什么？
4. 项目目前还做不到什么？

能用自己的话说清这四点，就已经具备继续学习 P1 的基础。
