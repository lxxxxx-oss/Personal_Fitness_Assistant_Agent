# P0 必须掌握：项目主线背诵稿

## 1. 30 秒开场

这是一个面向个人健身场景的 AI Agent 项目。我没有把它做成单纯的 ChatGPT 聊天框，而是用 LangGraph 把健身知识问答、联网搜索、饮食建议、动作分析和菜谱工具调用拆成多个受控子图，再由 Router 根据用户意图选择或组合执行。

项目最想体现的是三件事：第一，Agent 能把大模型、RAG、工具和确定性算法编排起来；第二，Router 不是拍脑袋写 if else，而是有规则分数、语义样例、LLM fallback 和 eval 验证；第三，项目保留了诚实边界，当前是可运行原型，生产化可以继续替换模型、向量库和动作数据集。

## 2. 三分钟项目介绍

可以按“背景、架构、难点、结果、边界”五段讲。

背景：健身问题不是单一问答。用户可能问训练知识、查最新资料、要饮食方案、上传动作图片，甚至问一道菜怎么做。单个大模型可以回答一部分，但它不擅长稳定调用外部工具、记录路由原因、处理动作数据和做可评测的能力拆分。

架构：我用 FastAPI 做后端入口，用 LangGraph 做 Agent 编排。顶层 Router 负责判断 intent，下面拆成 Chat、Search、Diet、Motion、MCP 五个子图。Chat 和 Diet 偏 RAG，Search 偏外部信息检索，Motion 偏姿态分析工具链，MCP 偏标准工具协议调用。这样每个子图内部可以有自己的流程，而顶层保持统一状态和统一降级。

难点：最大难点不是多放几个节点，而是让路由和执行可控。Router 经历了 Weighted Rules、Semantic Examples、LLM Classifier Fallback、Multi-intent Routing 四阶段。现在绿色集 66/66，challenge set 主意图、次意图和 route plan 都是 36/36。LLM Router 做过本地 Qwen A/B，但没有带来净收益且有延迟，所以默认关闭，只保留 feature flag 和观测能力。

结果：项目现在可以完整运行，核心接口、Web UI、图片上传入口、Router eval、MCP mock fallback、流式输出和测试基线都已经打通。当前 full pytest 基线是 `114 passed, 1 skipped, 1 warning`。这个 warning 来自 Starlette TestClient 对 httpx 兼容层的弃用提示，不影响主流程。

边界：这个项目是面试展示型原型，不说自己已经生产可用。比如 Motion 当前打通了图片姿态提取接口和姿态序列格式，但缺少有代表性的真实标准动作数据集；RAG 当前适合本地知识库验证，生产化会迁移向量数据库、补 rerank 和检索评测；小程序端也需要继续联调。我的重点是证明架构、评测和演进路径成立。

## 3. 三个主打亮点

### 亮点一：异构能力编排，而不是 ChatGPT 套壳

回答模板：

> 这个项目不是只把用户问题丢给大模型。大模型主要负责语言理解和最终表达，RAG 负责知识 grounding，MCP 负责标准化工具调用，Motion 工具负责姿态数据计算，Router 负责选择和组合路径。Agent 的价值是把这些能力用受控工作流串起来。

面试官追问“为什么不用一个 prompt 全做”时：

> 一个 prompt 可以做 demo，但很难保证可解释、可测试和可降级。比如动作相似度必须由工具计算，菜谱查询最好走 MCP，最新信息要走 Search，历史健身知识要走 RAG。拆成子图后，每条路径能单独测试，也能单独优化。

### 亮点二：Router 可评测、可解释、可降级

回答模板：

> Router 不是简单关键词匹配。我做了加权规则、组合规则、低置信语义样例、可选 LLM classifier 和多意图 route plan。每次路由会记录 scores、confidence、source、reason 和 matches。这样出错时能知道是规则问题、语义样例问题，还是 LLM fallback 没必要启用。

当前证据：

- 绿色回归集：66/66。
- Challenge set：36/36。
- Phase 4：已支持 primary intent、secondary intents、route plan。
- 多意图执行：只开放四种两步白名单组合，不开放任意子图串联。
- LLM fallback：真实接入过本地 Qwen A/B，但默认关闭。

### 亮点三：真实工程问题和边界意识

回答模板：

> 我在项目里保留了几个真实工程取舍。比如本地 Qwen 重复加载会导致内存压力，所以做了共享 loader 和 mock 模式；MCP server 不可用时走 mock fallback；Motion 不夸大成完整健身教练，而是明确当前只完成姿态输入链路和规则分析原型。

这能体现：

- 不是只写功能，还考虑运行稳定性。
- 不是只写设计，还做了测试和 eval。
- 不是过度包装，而是能说明当前边界和升级路径。

## 4. 为什么使用 LangGraph

一句话：

> LangGraph 适合这种“有状态、多分支、可降级、需要中间节点可观测”的 Agent，而不是一次性 prompt chain。

展开：

- 用户请求先进入 Router，再进入不同子图。
- 每个子图可以有多个节点，例如 Motion 有 think、parse、tool、check。
- `RouterState` 统一保存用户输入、历史、intent、结果、错误、路由元信息和多意图字段。
- 后续要加新子图或替换某个节点，不需要重写整个主流程。

## 5. 子图和工具的区别

必须能讲清楚：

> 工具是一个动作，子图是一段受控流程。搜索工具只能“查一下”，但 Search 子图可以先理解问题、改写 query、调用搜索、过滤结果、合成回答。Motion 工具可以算相似度，但 Motion 子图会先判断任务、解析输入、调用工具、检查是否需要补充信息。子图的价值是把多个步骤、状态和失败处理组织起来。

如果被质疑“子图是不是冗余”：

> 如果某个能力只有一次函数调用，那确实没必要做子图。但这个项目里的子图承载的是不同任务域的流程边界。它让 RAG、Search、Motion、MCP 的输入输出和错误处理分开，也方便 eval 和后续生产化替换。

## 6. 五个子图怎么背

### Chat：健身知识 RAG

作用：回答训练原则、动作常识、健身建议等相对稳定的问题。

讲法：

> Chat 子图用本地知识库做检索增强，先检索相关知识，再把片段放进 prompt 生成回答。当前适合验证 RAG 链路，生产化会补充更高质量知识库、向量数据库、reranker 和 RAG eval。

边界：

- 当前不是大规模医学级知识库。
- 不把 RAG 说成完全避免幻觉，只说能降低无依据回答。

### Search：联网搜索

作用：处理最新信息、权威资料、需要查找来源的问题。

讲法：

> Search 子图解决的是知识时效性问题。RAG 适合稳定知识，Search 适合最新内容。它的价值不只是调用搜索工具，而是把 query 理解、搜索、结果筛选和回答合成组织成流程。

边界：

- 搜索质量依赖外部结果。
- 生产化要加强来源可信度、去噪、引用校验。

### Diet：饮食建议

作用：根据用户目标、饮食偏好和基础营养知识给建议。

讲法：

> Diet 和 Chat 都会用到知识检索，但 Diet 的任务目标更结构化：它要围绕热量、蛋白质、增肌减脂目标和饮食偏好组织答案。所以它不是和 Chat 重复，而是把饮食问题单独封装成可优化的任务域。

边界：

- 当前不是专业医疗营养系统。
- 复杂疾病、孕期、处方级饮食要建议咨询专业人士。

### Motion：动作分析

作用：把用户上传的姿态数据、图片或后续视频输入转成关键点，再做动作质量分析。

讲法：

> Motion 的路线是先用开源姿态估计模型从图片或视频提取人体关键点，再统一成 `PoseSequence` / `.npz` 这样的中间格式，最后由动作分析工具做角度、相似度和规则判断。当前已经打通图片上传入口和姿态序列原型，完整视频动作评估和真实标准动作库是后续重点。

边界：

- 不能说已经有完整训练模型。
- 不能说已经能专业判断所有健身动作。
- 可以说“原型链路已打通，后续替换标准动作库和规则集”。

### MCP：菜谱工具调用

作用：演示 Agent 如何通过标准协议调用外部工具。

讲法：

> MCP 子图不是为了菜谱本身，而是展示 Tool Use 的标准化。它会发现工具、规划参数、执行调用、格式化结果。MCP server 不可用时有 mock fallback，保证项目可以演示。

边界：

- 菜谱只是工具调用示例。
- 重点是协议化工具接入和失败降级。

## 7. 高频问题标准答案

### Q1：这个项目和普通 ChatGPT 套壳有什么区别？

> 普通套壳主要是 prompt 包装。这个项目有 Router、RAG、工具调用、动作分析、MCP、流式输出和 eval。大模型不是唯一执行者，而是被放在一个可控工作流里。能用确定性工具解决的部分不用 LLM 硬猜，需要语言表达的部分再交给 LLM。

### Q2：为什么 Router 里还需要规则，不直接 LLM 分类？

> 因为路由是系统控制面，稳定性比智能感更重要。规则路由可解释、低延迟、可回归测试。LLM classifier 我不是没做，而是做过本地 Qwen A/B，发现准确率没有提升、延迟增加，还出现过覆盖正确规则的风险，所以默认关闭，保留为可开关实验能力。

### Q3：RAG 的知识库从哪里来？

> 当前是项目内置的健身和饮食知识资料，目标是验证 RAG 链路和回答 grounding。生产化会引入更系统的数据来源，比如权威训练指南、营养资料、动作库文档，并做 chunk、embedding、rerank、召回率和答案质量评测。

### Q4：Motion 没有训练模型，为什么还能讲？

> 我没有从零训练姿态模型，而是采用工程上更现实的路线：用开源姿态估计模型提关键点，再把关键点交给规则和相似度算法分析。健身动作分析的关键不一定是训练一个大模型，而是把视觉输入转成结构化姿态序列，再做角度、轨迹和标准动作对比。

### Q5：怎么证明项目不是 toy？

> 我不会说它已经生产级，但它不是纯 toy。因为它有清晰模块边界、统一状态、错误降级、Router eval、测试基线、MCP mock fallback、流式输出和真实内存问题修复记录。它是一个可运行、可演示、可评测、可继续演进的面试型 Agent 原型。

## 8. 当前边界必须诚实

面试时不要说：

- 已经完成生产级动作识别。
- 已经有大规模真实标准动作库。
- RAG 已经达到医疗/专业教练级准确率。
- LLM Router 默认在线上接管所有路由。
- 小程序已经生产可用。

可以说：

- 当前是可运行原型。
- 核心链路、接口、UI、Router eval 和测试已经打通。
- 生产化方向清楚：向量库、reranker、动作数据集、视频分析、线上部署、权限和监控。
- 项目重点是展示 Agent 编排、工具调用、评测和工程演进能力。
