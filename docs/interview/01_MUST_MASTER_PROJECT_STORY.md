# P0 必须掌握：项目主线与证据边界

这份材料要掌握事实和因果，不要逐字背。面试官真正判断的是：你是否知道自己做了什么、为什么这样做、证据能证明什么、当前还缺什么。

## 1. 30 秒开场

> 这是一个面向个人健身场景的、可评测的受控 AI Agent 原型。我用 LangGraph 把知识问答、搜索、饮食建议、动作工具和 MCP 菜谱工具拆成不同任务域，再由 Router 选择单条路径或少量白名单组合。项目重点不是堆功能，而是三件事：异构能力如何受控编排，Router 如何通过离线回归集持续验证，以及外部工具、模型或数据不完整时如何诚实降级。

不要在开场连续堆叠 RAG、MCP、FastDTW、SSE 等名词。先让面试官记住一个定位：**可评测、受控、能降级的多能力 Agent 原型**。

## 2. 三分钟项目介绍

可以按“背景—架构—个人工作—证据—边界”讲。

### 背景

健身场景不是单一问答。稳定知识适合 RAG，最新信息需要搜索，菜谱可以走外部工具，动作数据需要确定性算法。把这些任务全部交给一个 Prompt，难以解释路由原因，也难以单独测试和降级。

### 架构

> 我用 FastAPI 提供 HTTP、SSE、WebSocket 和文件上传接口，用 LangGraph 组织顶层 Router 以及 Chat、Search、Diet、Motion、MCP 五个任务域。RouterState 保存输入、历史、路由元信息、执行计划和结果；执行层默认走单路由，只对四种两步组合开放多子图执行，并对单个子图错误做隔离。

### 我主要完成的工程工作

> 第一，我把 Router 从 first-match 关键词判断演进为加权规则、字符 n-gram/Jaccard 语义样例、可选 LLM classifier 和受控多意图计划，并补了离线评测脚本。第二，我搭建了 LangGraph 子图和统一状态、错误隔离、组合结果合成。第三，我实现了轻量 MCP stdio JSON-RPC Client 的核心工具链路和 mock fallback。第四，我为 Motion 定义了 PoseSequence 中间格式，接入图片姿态估计适配器，并保留 `.npz` 相似度分析链路。第五，我处理了本地模型共享加载、内存压力、会话记忆和流式接口这些可运行性问题。

如果项目不是独立完成，应在现场把“我负责”和“团队已有”重新划清，不要把团队成果都说成个人完成。

### 当前证据

> Router 绿色回归集有 66 条，challenge set 有 36 条；当前代码在这些人工标注样本上全部通过。自动化测试基线是 `114 passed, 1 skipped, 1 warning`。这些测试主要证明确定性逻辑、接口契约和降级路径没有回归，LLM、Embedding、搜索和多数外部依赖使用了 mock，所以它们不是模型回答质量或生产效果指标。

### 当前边界

> Motion 图片接口目前只完成单帧关键点提取和静态摘要，没有对图片直接做完整动作标准性判断；`.npz` 序列接口可以做归一化、DTW 和相似度分析，但标准动作库仍不充分，视频时序入口还没有完成。RAG 也还没有标准问答集和 Recall@K。这个项目的价值是证明架构、评测方法和演进路径成立，不是宣称已经生产可用。

## 3. 能力、证据和边界

| 能力 | 已实现证据 | 面试时必须说明的边界 |
|---|---|---|
| Router | 加权规则、语义样例、可选 LLM classifier、多意图字段、四种组合、66/66 与 36/36 回归结果 | 数据集由项目人工构造并持续调优，不是独立线上样本或泛化证明 |
| LangGraph 编排 | 五个任务域、统一 RouterState、条件边、错误隔离、组合结果合成 | 当前是受控 Workflow，不是开放式自主规划 Agent |

| Motion `.npz` | PoseSequence、归一化、FastDTW、余弦相似度和可选标准动作对比 | 标准动作样本有限；数值相似度不等于专业动作质量判断 |
| Motion 图片 | Web UI 上传、MediaPipe 关键点提取、PoseSequence 和置信度摘要 | 当前绕过 Motion 子图，只做单帧静态摘要；不能判断节奏、轨迹和发力顺序 |
| MCP | 自实现 stdio JSON-RPC Client，覆盖 initialize、`tools/list`、`tools/call`，支持 mock fallback | 是 MCP 核心子集原型，不是完整 SDK、权限系统或生产级生命周期管理 |
| Streaming | SSE 增量输出；WebSocket 协议和 token 消息格式已实现 | SSE 内部仍有同步调用；WebSocket 当前先收集完整 token 再发送，不应说成真实低首字延迟流式 |
| Tests | 114 条自动化测试、Router eval、接口和工具专项记录 | 外部模型和服务大量 mock，证明回归稳定性，不证明真实端到端质量 |

## 4. 三个主打亮点

### 亮点一：受控异构能力编排

> LLM 主要负责语言理解和表达，RAG 负责知识 grounding，搜索和 MCP 负责外部能力，Motion 负责结构化姿态数据计算，Router 负责选择受控路径。能用确定性工具计算的部分不让 LLM 猜。

为什么不使用一个 Prompt：

> 单 Prompt 可以快速演示，但动作相似度、外部搜索和工具协议有不同输入输出、失败模式与测试方式。拆成任务域后，可以分别测试、替换和降级。

### 亮点二：评测驱动的 Router 演进

> Router 会记录 scores、confidence、source、reason、matches、secondary intents 和 route plan。我用绿色回归集保护主流行为，用 challenge set 保存已知困难模式。LLM classifier 真实接入过本地 Qwen，但在当前样本上没有提升准确率且增加约 6 秒延迟，所以默认关闭。

这里最重要的不是“满分”，而是：**修改 Router 之前和之后都有同一套回归证据，且能用评测拒绝一个看起来更智能、实际收益不足的方案。**

### 亮点三：可运行性和诚实边界

> 项目处理过本地模型重复加载导致的内存压力、外部搜索和 MCP 不可用、Embedding 加载失败、Motion 数据不足等真实问题。当前策略优先保证核心链路可运行，并明确哪些是 mock、哪些只完成接口、哪些需要真实数据才能继续。

## 5. 为什么使用 LangGraph

一句话：

> LangGraph 适合这个有状态、多分支、需要错误隔离和受控组合的 Workflow；如果只有一次函数调用，我不会为了使用框架而强行建图。

展开：

- 顶层根据 RouterState 进入不同任务域。
- 子图内部可以有检索、工具调用、检查和生成等多个节点。
- 多意图执行需要保存 route plan、游标、各子任务结果和 warning。
- 某个子图异常时可以记录失败，并让组合计划基于成功结果降级。

当前没有使用 LangGraph checkpoint 做持久化；会话记忆由进程内 SlidingWindowMemory 单独管理。这是原型取舍，不要回答成已经具备分布式状态恢复。

## 6. 子图和工具的区别

> 工具是一个动作，子图是一段受控流程。搜索工具只负责查询；Search 子图还包括 Query 改写、调用搜索、组织上下文和回答合成。Motion 工具负责加载姿态或计算相似度；Motion 子图负责理解请求、解析输入、调用工具和检查结果。

如果被质疑子图冗余：

> 质疑是合理的。只有一次函数调用的能力不需要子图。这个项目使用子图，是因为这些任务域已经有不同状态、步骤和失败处理；收益是隔离和可测试性，不是节点越多越高级。

## 7. 五个任务域速记

### Chat

- 当前：本地健身知识 RAG 和滑动窗口历史注入。
- 边界：知识库规模小，没有 RAG 标注集；引用编号由 Prompt 约束，API 来源字段尚未完整暴露。

### Search

- 当前：Query 改写、Tavily/mock 搜索、答案合成。
- 边界：外部网页内容可信度和 Prompt Injection 防护不足。

### Diet

- 当前：用户画像文本提取、营养知识检索和建议生成。
- 边界：不是医疗营养系统；疾病、孕期和处方饮食必须建议专业咨询。

### Motion

- 当前：`.npz` 序列分析与图片单帧姿态提取是两条不同链路。
- 边界：图片接口没有完成动作质量判断，视频和真实标准动作库待补。

### MCP

- 当前：工具发现、参数规划、调用和结果格式化，默认可用 mock。
- 边界：菜谱只是协议示例；真实 server、权限和完整生命周期需要继续建设。

## 8. 高频问题

### Q1：为什么这不只是一个聊天框？

> 因为系统里存在可独立验证的控制面和执行面：Router 决定任务域，RAG 和搜索提供上下文，MCP 调外部工具，Motion 做确定性数据计算，LangGraph 组织状态和失败路径。我的证据不是 UI 长得复杂，而是路由评测、工具契约、子图执行和专项测试。

“不是 ChatGPT 套壳”只需要说明一次，不要反复使用这个口号。

### Q2：为什么不用 LLM 直接分类？

> 路由是控制面，当前五类任务边界相对固定，所以低延迟、可解释、可回归的确定性策略更合适。我做过本地 Qwen A/B；在当前样本上没有增加正确率，反而增加延迟，因此默认关闭。未来只有在真实困难样本证明有增量时才启用。

### Q3：66/66 和 36/36 能证明什么？

> 能证明当前 Router 对这 102 条人工标注样本没有回归，也能按类别观察已知困难边界。不能证明真实线上泛化，因为它们不是独立随机采样或长期线上数据。生产化需要补留出集、真实日志抽样和人工复核闭环。

### Q4：Motion 图片能判断深蹲标准吗？

> 目前不能。图片接口只提取单帧关键点和置信度摘要；完整动作判断需要视频序列、动作阶段、轨迹、标准动作库和专项规则。当前真正可以计算序列相似度的是 `.npz` 分析链路。

### Q5：114 个测试能证明什么？

> 它们主要证明接口契约、Router、状态流转、工具输入校验、错误降级和确定性算法没有回归。测试为了可重复执行，mock 了本地 LLM、Embedding 和多数外部依赖，因此不能把这个数字解释为回答质量或真实服务稳定性。

### Q6：你认为项目最有价值的个人决策是什么？

> 不是增加一个新模型，而是把 Router 变成可评测控制面，并用 A/B 结果决定关闭收益不足的 LLM classifier。这个决策同时考虑准确率、延迟、可解释性和演示稳定性。

## 9. 绝对不能越过的表述边界

不要说：

- Motion 已经能根据图片或视频专业判断动作质量。
- Router 102 条满分代表真实用户泛化完成。
- 114 个测试覆盖了真实 LLM、真实搜索和真实 MCP 端到端质量。
- WebSocket 已经实现真正逐 token、低首字延迟流式。
- MCP Client 完整支持所有 MCP 能力。
- RAG 已经达到医疗或专业教练级准确率。
- 小程序或 Docker 已经生产可用。

可以说：

- 当前是可运行、可演示、可回归验证的受控 Agent 原型。
- Router 在项目自建回归集上表现稳定，泛化需要真实样本闭环。
- SSE 已有增量输出，WebSocket 协议已打通但发送实现仍可优化。
- Motion 已打通姿态输入和序列分析的原型组件，但图片、视频和动作质量判断尚未完全串联。
- 项目重点是工程拆分、评测意识、失败降级和可解释演进。
