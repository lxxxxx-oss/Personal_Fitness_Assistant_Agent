# 技术深挖与白板讲解

这篇文档用于第二轮或资深面试官追问。它不是入门材料，也不要求逐字背诵。你的目标是能够画出关系、解释数据流，并说明设计取舍。

## 1. 总体架构白板

先画四层，不要直接画所有类和函数：

```text
┌─────────────────────────────────────────────┐
│ 接入层：HTTP / SSE / WebSocket / Web / 小程序 │
└─────────────────────┬───────────────────────┘
                      ↓
┌─────────────────────────────────────────────┐
│ 控制层：Router / LangGraph / 共享状态 / 路由计划 │
└─────────────────────┬───────────────────────┘
                      ↓
┌─────────────────────────────────────────────┐
│ 能力层：Knowledge / Search / Motion / MCP      │
│         Retriever / ToolRegistry / LLM         │
└─────────────────────┬───────────────────────┘
                      ↓
┌─────────────────────────────────────────────┐
│ 保障层：Memory / PromptBuilder / ToolResult    │
│         fallback / audit / 测试与评测           │
└─────────────────────────────────────────────┘
```

讲解顺序：

1. 请求从哪里进入。
2. 谁决定任务路径。
3. 谁执行具体任务。
4. 状态、错误和质量怎样管理。

## 2. LangGraph 与共享状态

### 为什么需要共享状态

多个节点需要看到同一请求中的不同信息：

- 用户原始问题。
- 主意图、次意图和置信度。
- 路由计划。
- 知识来源和工具结果。
- 用户画像和长期记忆。
- 警告、错误和执行轨迹。
- 最终回答。

共享状态像一张随着流程不断补充的任务单。节点不需要通过隐式全局变量互相猜测结果。

### 顶层图与子工作流

```text
顶层图：决定进入哪个能力域
  ├── Knowledge：普通问答与饮食建议
  ├── Search：查询改写、搜索、合成
  ├── Motion：动作任务规划、工具、解释
  └── MCP：工具发现、计划、调用、格式化
```

顶层图负责跨能力调度，子工作流负责领域内部步骤。这样既能保持统一入口，又能单独测试每个能力。

## 3. 路由系统深入分析

### 混合路由思路

```text
规则和组合信号
  -> 歧义检测
  -> 字符 n-gram / 词面样例
  -> 可选 embedding / 本地模型兜底（默认关闭）
  -> 主意图 / 次意图 / 路由计划
```

为什么不完全使用大模型：

- 高频明确意图不需要模型成本。
- 规则得分和原因可解释。
- 模型输出可能格式错误或不稳定。
- 本地 Qwen 对比没有产生足够准确率收益。
- 离线评测可以约束每次策略调整。

### 多意图为什么使用白名单

复合任务不仅要识别两个意图，还要处理：

- 哪个先执行。
- 前一步结果是否是后一步输入。
- 一步失败是否继续。
- 怎样合并来源、警告和回答。
- 怎样判断整个任务成功。

所以当前只开放经过设计和测试的两步组合。开放式 Planner 属于后续演进，而不是当前必须能力。

### 路由系统怎样评测

每个样本至少包含：

- 用户问题。
- 期望主意图。
- 期望次意图。
- 期望路由计划。
- 是否需要澄清。

常规集保护基本行为，困难集专门覆盖否定、跨域、菜谱与饮食混淆、计划与动作分析混淆等边界样本。

## 4. RAG 与 Milvus 深挖

### 离线入库

```text
原始文档
  -> 清洗
  -> 句子感知分块
  -> Sentence-Transformers 向量化
  -> 内容哈希生成稳定主键
  -> content + vector + source 写入 Milvus
  -> 建立 IVF_FLAT / COSINE 索引
```

稳定主键和 upsert 让同一资料重复加载时不会产生大量重复内容。

### 在线检索

```text
用户问题
  -> 查询向量
  -> Milvus 近似最近邻搜索
  -> Top-K 与阈值
  -> 内容去重
  -> 编号证据块
  -> PromptBuilder
  -> 基于证据生成回答
```

### 关键参数怎么讲

| 参数 | 控制什么 | 取舍 |
|---|---|---|
| chunk 大小 | 每段资料包含多少上下文 | 太小信息破碎，太大主题混杂 |
| Top-K | 取多少候选资料 | 太少漏信息，太多增加噪声 |
| threshold | 最低相关程度 | 太高可能无结果，太低引入无关内容 |
| `nlist` | IVF_FLAT 分多少桶 | 影响建索引和候选范围 |
| `nprobe` | 查询扫描多少桶 | 越大召回通常更高，延迟也更高 |

### RAGAS 评测链路

RAGAS 不参与线上回答，也不会替 Retriever 重新检索。它位于离线评测侧：先让项目的真实 RAG 链路跑出上下文和答案，再对不同阶段分别评分。

```text
黄金集 query
  -> 项目真实 Retriever
  -> retrieved_contexts
  -> 项目真实 Chat RAG generate_node
  -> response
  -> RAGAS 离线评分
       ├─ query + retrieved_contexts            -> ContextRelevance
       ├─ query + retrieved_contexts + response -> Faithfulness
       └─ query + response                      -> AnswerRelevancy
```

这里的 Retriever 负责“从知识库找证据”，`generate_node` 负责“根据问题和证据组织回答”。RAGAS 是旁路裁判，负责判断“找得准、说得真、答得对”，不改变用户实际收到的回答。

项目没有搭第二套评测平台，只保留 `scripts/eval_rag.py` 一个入口：15 条黄金集样例中，12 条可回答样例依次经过真实 Retriever、真实生成节点和三个 RAGAS scorer；3 条无答案样例明确排除在三项均分之外，因为正确拒答是一种独立能力，不能简单当成“检索失败”。

### 三个指标的底层逻辑

#### 1. ContextRelevance：资料找得准不准

输入是原始问题和检索上下文，主要评价 Retriever。当前使用的 RAGAS 0.4.3 会让裁判模型做两次独立相关性判断，每次给出 `0`、`1`、`2`，分别表示不相关、部分相关和高度相关。单次结果除以 2 归一化到 `0～1`，再对两次结果取平均，以降低一次判断的偶然波动。

```text
context_relevance = mean(judge_score / 2)
```

它判断的是检索上下文整体是否有助于回答 query，不等同于 `Recall@K`、`Precision@K` 或 `MRR`。后者需要明确标注哪些 chunk 应该被召回；当前项目没有把它们包装成已经完成的 RAGAS 能力。

#### 2. Faithfulness：回答有没有脱离证据

输入是问题、检索上下文和最终回答，主要评价生成阶段的幻觉风险。RAGAS 先让裁判把回答拆成尽量独立的原子事实，再逐条做类似自然语言推理中的蕴含判断：这条陈述能否由给定上下文支持。

```text
faithfulness = 被上下文支持的陈述数 / 全部可判断陈述数
```

例如回答包含 3 个事实，只有 2 个能从上下文推出，忠实度约为 `0.667`。这个指标衡量“是否忠于资料”，不等于“现实世界中一定正确”：如果知识库本身有错误，模型照着错误资料回答，忠实度仍可能很高。

#### 3. AnswerRelevancy：最终有没有答到问题上

输入是原始问题和最终回答，评价端到端有效性。RAGAS 不直接比较疑问句和陈述句，而是让裁判根据答案反向生成若干个可能的问题；当前默认严格度会生成 3 次。随后由 embedding 模型分别编码原问题和反向问题，计算余弦相似度并取平均。

```text
answer_relevance = mean(cosine(query_vector, generated_question_vector))
```

反向问题越接近原问题，说明答案越可能答在点上。如果裁判认为答案只是回避、含糊或没有实质信息，非承诺性判断会进一步把得分压低；全部判断为非承诺性时得分为 0。

### 裁判模型与 embedding 的分工

| 组件 | 在正常 RAG 中的职责 | 在 RAGAS 中的职责 |
|---|---|---|
| 项目 Retriever | 使用 BGE 表示查询并召回知识片段 | 被评测对象 |
| 项目 `generate_node` | 根据 query 和证据生成回答 | 被评测对象 |
| Qwen3-8B 裁判 | 不参与线上 RAG | 相关性判断、事实拆分、证据支持判断、反向生成问题 |
| 本地 BGE | 为知识检索生成向量 | 计算原问题与反向问题的语义相似度 |

正常检索和答案相关性评测可以复用同一个 BGE 物理模型，但角色不同：前者负责选择知识片段，后者负责测量两个问题的语义距离。Qwen3-8B GGUF 由 llama.cpp 暴露 OpenAI 兼容接口，RAGAS 和项目代码继续运行在现有 Conda 环境中；裁判 LLM 与 embedding 解耦，不要求 llama.cpp 同时提供 embedding API。

### 怎样通过三项分数定位问题

| 上下文相关性 | 忠实度 | 答案相关性 | 优先排查方向 |
|---|---|---|---|
| 低 | 低 | 低 | 检索失败且生成失控，先查知识、分块、embedding、Top-K 和阈值 |
| 低 | 高 | 低 | 模型忠实使用了错误资料，主要问题仍在 Retriever |
| 高 | 低 | 高 | 证据找对但回答加入了无依据内容，查 Prompt 和生成模型 |
| 高 | 高 | 低 | 回答有依据但没有正面或完整回答原问题 |
| 高 | 高 | 高 | 固定样例上链路整体表现较好，仍需结合样本量和人工复核解释 |

三个分数用于诊断，不应简单合并成一个“RAG 总准确率”。完整的生产化观察还应包括：

```text
检索层：Recall@K / MRR / 命中率 / 无关片段率
生成层：忠实度 / 相关性 / 来源正确性 / 无证据拒答
系统层：延迟 / 降级率 / 依赖错误 / 成本
```

### 当前边界与可信表达

- 黄金集虽然包含人工 `reference`，但当前三个 scorer 的实际调用没有传入它。因此当前链路评价检索相关性、上下文忠实度和回答相关性，不等于直接验证“答案与标准答案完全一致”。
- LLM-as-a-Judge 是自动化质量信号，不是绝对真理；分数会受裁判模型能力、Prompt、采样波动、中文理解和量化精度影响。生产化可以采用更强裁判、重复评测取平均和人工抽检进行校准。
- 一条样例会触发多次裁判推理：上下文相关性约 2 次、忠实度约 2 次、答案相关性默认约 3 次，外加 embedding 计算。因此 12 条样例在本地运行数分钟是合理现象。
- 当前已完成代码、数据流、单元测试和一条中文合成样例的本地三指标兼容性烟测；12 条真实项目样例的完整裁判基线尚未记录。面试时要区分“评测能力已接入”和“已经取得稳定高分”。

面试中可以这样总结：

> 我把 RAGAS 放在离线评测侧，不让它影响线上回答。黄金集先走真实 Retriever 和生成节点，再由 Qwen 裁判与 BGE 从上下文相关性、忠实度和答案相关性三个维度评分。这样低分时可以定位问题在检索还是生成，而不是只得到一个无法解释的总分。

## 5. 动作分析深入讲解

### 统一数据结构

```text
PoseSequence
  keypoints: (时间帧, 关节点, 坐标维度)
  fps
  source_type
  pose_model
  joint_schema
  confidence
  metadata
```

为什么不直接传 NumPy 数组：数组本身不知道来自图片还是视频、使用什么模型、关节顺序是什么，也无法安全判断两个序列是否兼容。

### 媒体输入链路

```text
图片
  -> 解码
  -> MediaPipe 图片模式
  -> 单帧 PoseSequence

视频
  -> OpenCV 解码和抽帧
  -> MediaPipe 视频模式 + 递增时间戳
  -> 多帧 PoseSequence
```

单张图片只能分析静态姿态，不能判断完整动作节奏和轨迹。

### 相似度链路

```text
用户姿态与标准姿态
  -> 检查模型和关节结构兼容
  -> 髋中心平移归一化
  -> 尺度归一化
  -> FastDTW 时间对齐
  -> 余弦相似度
  -> 对齐后逐关节平均距离
  -> 结构化结果
```

这些指标回答“和参考样本有多接近”，不能自动回答“医学上是否安全”。专业纠错需要动作阶段、关键关节和教练标签。

## 6. 工具系统深挖

### 五项契约

```text
职责：这个工具只做什么
输入：参数类型、必填项、范围
输出：成功数据和元信息
权限：允许访问什么资源
错误：调用方怎样恢复
```

### 工具注册表执行链

```text
ToolSpec
  -> 参数校验
  -> 权限检查
  -> executor
  -> 有限重试
  -> fallback
  -> ToolResult
  -> execution_id / duration / audit
```

重要边界：当前超时是策略字段，Registry 本身不强制中断长执行。生产化需要独立执行器、进程或任务队列提供真正隔离。

### MCP 协议主链路

```text
启动 Server 子进程
  -> initialize
  -> initialized 通知
  -> tools/list
  -> tools/call
  -> 解析 content
  -> ToolResult
```

生产化还需要响应编号匹配、深层 schema 校验、并发请求表、进程池、权限和沙箱。

## 7. 记忆与上下文深入讲解

### 为什么要分四类上下文

```text
近期对话：当前会话最近发生了什么
active summary：跨请求承接同一会话的较早原文摘录
prompt compact：只保护本轮 Prompt 长度，不持久化成用户记忆
长期记忆：跨会话仍有价值的目标、偏好和约束
```

四者生命周期不同，不能全部当作聊天记录或都塞进向量库。完整消息仍保存在 SQLite，是摘要和近期窗口的事实来源。

### 长期记忆链路

```text
用户明确表达需要记住的信息
  -> 判断是否敏感
  -> 普通内容写入 memory_items
  -> 敏感内容进入 candidate 等待确认
  -> SQLite 保存事实
  -> FTS / LIKE 基础检索
  -> 可选 Milvus 语义增强
  -> 按预算注入 Prompt
```

SQLite 是事实来源，因为长期记忆需要可修改、可删除和可审计；Milvus 是派生索引，不应该成为唯一真相。

### PromptBuilder 的作用

PromptBuilder 集中决定本轮模型看到：

- 任务说明。
- 用户原始问题。
- 最近对话。
- 长期记忆。
- RAG 证据。
- 工具结果摘要。
- 压缩后的结构化状态。

这比每个子工作流随意拼接 Prompt 更容易管理长度和调试。

## 8. 流式服务深挖

本地 LangGraph 和模型生成是同步逻辑，而 FastAPI 接口是异步服务。直接调用会阻塞事件循环。

当前桥接思路：

```text
同步模型生成线程
  -> 线程安全地写入 asyncio queue
  -> SSE 或 WebSocket 异步读取 queue
  -> 持续推送 token
```

模型生成锁仍限制同进程并发。生产化应把模型拆成独立推理服务，并增加请求队列、批处理、超时和资源监控。

## 9. 质量体系白板

```text
代码测试
  检查接口、状态、工具、算法是否回归

离线评测
  检查 Router、RAG、Memory、Motion 策略效果

线上观测
  检查真实成功率、延迟、降级率和用户反馈
```

三个层次解决不同问题，不能用 `pytest passed` 代替真实效果。

## 10. 生产化路线

按风险和价值排序：

1. 认证授权、租户隔离、限流、HTTPS 和隐私治理。
2. 模型服务化、请求队列、超时和资源监控。
3. 扩大 Router 与 RAG 真实评测集。
4. 工具硬超时、熔断、沙箱和分布式追踪。
5. Memory 用户确认、删除同步和跨实例存储。
6. Motion 标准动作、周期切分、专项规则和教练标注。
7. 小程序真机、弱网和容器跨机器验收。

## 11. 白板收尾句

> 这套架构的核心不是堆叠框架，而是按问题性质分工：大模型处理语言，Router 和 LangGraph 控制流程，RAG 与搜索提供信息，数值算法完成动作计算，工具系统治理执行，Memory 管理跨轮信息，测试和评测约束质量。

## 12. 工程问题地图

面试官问“上线会有什么问题”时，不要无序罗列高并发、缓存和限流。先按六个维度检查：

| 维度 | 本项目当前基础 | 生产化重点 |
|---|---|---|
| 正确性 | Router/RAG/Memory 专项入口与测试 | 真实黄金集、困难集、回归门禁 |
| 可靠性 | `ToolResult`、错误码、重试、fallback | 硬超时、截止时间、熔断、隔离 |
| 可观测性 | 状态元数据、execution_id、审计 | 全链路 trace、指标、告警、脱敏日志 |
| 安全 | 工具 permission、敏感记忆确认 | 身份、租户隔离、最小权限、人工审批 |
| 性能成本 | 流式接口、部分耗时元数据 | p95/p99、token 成本、队列、缓存 |
| 演进 | 模块接口与自动化测试 | Prompt/模型/索引版本化、灰度、回滚 |

标准故障回答是：先定义用户症状和影响范围，沿 trace 拆解 Router、检索、工具、模型和合成阶段；根据错误是否短暂、操作是否幂等，选择重试、降级、熔断或拒绝；恢复后把样本加入回归集，用任务成功率、错误率和尾延迟验证修复。

## 13. 五道渐进式白板题

### 白板题 1：请求链路与失败点

用户问“查最新增肌研究，再安排晚餐”。画出接入层、Router 的 `search + diet` 计划、两个子工作流、共享状态、合成和流式返回，并在 Tavily、模型生成、客户端断连处标注失败。

优秀答案还会说明：Search 失败时不能伪装成最新信息；Diet 可在明确标注缺少实时依据时提供通用建议；客户端断连应取消下游，避免继续消耗资源。

### 白板题 2：RAG 参数实验

把 Top-K 从 3 改成 8，怎样证明更好？固定问题集和正确证据，比较 Recall@K、MRR、上下文噪声、忠实度、任务成功率和 p95 延迟；保持模型、Prompt、索引版本不变。不能只挑几个答案凭感觉比较。

### 白板题 3：全链路可观测性

根 span 表示一次请求，下设 Router、Retriever、Tool、LLM、Synthesis 子 span；记录版本、耗时、状态和错误码。指标看 p50/p95/p99、错误率、队列时间、首 token、token 数和 fallback 比例；日志通过 trace_id 关联并脱敏。

### 白板题 4：高风险工具

如果 Agent 可以购买课程，应包含读写工具拆分、最小权限、强类型参数、价格二次展示、用户确认、幂等键、金额上限、审计和撤销。外部网页属于不可信数据，不能绕过策略触发购买。

### 白板题 5：从单机扩到服务

100 个用户同时上传视频并聊天时，先区分轻请求与重任务。视频解码、姿态和本地模型推理是 CPU/GPU 密集任务，不能因为用了 `async` 就自动并行；应进入有界队列和独立 worker，配置资源配额、上传限制、取消与背压。SQLite 还需评估并发写，跨实例状态与文件需迁移到共享持久化服务。

## 14. 指标设计不能只说准确率

| 层级 | 建议指标 | 回答的问题 |
|---|---|---|
| Router | 每类 precision/recall/F1、计划准确率 | 有没有走错路 |
| Retrieval | Recall@K、MRR、无关片段率 | 正确证据是否找到且靠前 |
| Generation | 忠实度、相关性、引用正确率 | 是否依据证据回答 |
| Tool | 成功率、错误码、重试和降级率 | 外部能力是否稳定 |
| Memory | 正确召回、错误注入、敏感泄露 | 个性化是否有用且安全 |
| Motion | 关键点可用率、对齐质量、教练标签一致性 | 动作反馈是否可靠 |
| 端到端 | 任务成功率、p95、首 token、单任务成本 | 用户目标是否以合理代价完成 |

RAGAS 将检索上下文与生成回答分维度评估，说明 RAG 不能压缩成一个准确率。本项目已经接入其 collections API：`ContextRelevance` 对应上下文相关性，`Faithfulness` 对应忠实度，`AnswerRelevancy` 对应答案相关性。这里的实现重点是复用真实项目链路并帮助定位问题，不是搭建第二套评测平台。

面试中可以直接回答：

> 我只做了一个轻量评测入口。上下文相关性低，先查检索；忠实度低，查 Prompt 和生成模型；答案相关性低，查最终表达是否答到点上。这样三个分数分别对应三个可行动的问题。当前入口和测试已完成，但还没有把一次临时的裁判结果包装成稳定基线。
