# 面试前十分钟速记

这篇只用于面试前快速唤醒记忆，不承担第一次教学。

## 一句话定位

> 我设计了一个面向个人健身场景的多任务智能体，用 LangGraph 组织路由和子工作流，把知识检索、实时搜索、动作分析、外部工具和记忆系统整合成可解释、可评测、可降级的执行流程。

## 三十秒介绍

> 健身问题可能需要专业知识、最新资料、动作计算或外部工具，如果全部交给一次大模型回答会很难控制。我把系统拆成控制层和能力层：Router 判断任务，LangGraph 编排流程，RAG、Tavily、Motion 和 MCP 分别完成不同任务；Memory、统一错误和评测负责上下文与稳定性。项目主要体现任务拆分、工具编排和工程边界意识。

## 三层架构

| 层次 | 一句话 | 技术 |
|---|---|---|
| 控制层 | 决定任务怎样流转 | LangGraph、StateGraph、Router |
| 能力层 | 完成检索、搜索、计算和工具调用 | RAG、Milvus、Tavily、Motion、ToolRegistry、MCP |
| 保障层 | 管理上下文、失败和质量 | Memory、PromptBuilder、ToolResult、fallback、测试与评测 |

## 三个主打亮点

1. **受控编排**：不同问题进入不同子工作流，多意图只执行白名单组合。
2. **大模型与确定性能力分工**：知识交给检索，最新信息交给搜索，动作交给数值算法。
3. **质量闭环**：路由有评测集，工具有统一错误和审计，RAG 与 Memory 有专项 benchmark。

## 六个高压问题

### 为什么不是聊天套壳？

> 因为系统会判断任务、选择工作流、调用检索或工具、执行数值计算，并根据执行结果回答。

### 为什么 Router 不全部交给大模型？

> 明确任务使用确定性策略更稳定、快速和可解释；模型只有带来评测收益时才值得启用。

### LangGraph、ToolRegistry 和 MCP 有什么区别？

> LangGraph 管业务流程，ToolRegistry 管工具执行，MCP 管外部工具协议。

### Milvus 做了什么？

> 它负责向量持久化、索引和相似度搜索，不负责生成回答，也不自动保证 RAG 质量。

### 视频能上传是否等于专业动作纠错？

> 当前能生成姿态序列、比较参考样本并输出结构化反馈；专业级纠错还需要正式样本、动作周期、关节规则和教练评测。

### 自动化测试数字能证明什么？

> 它证明代码契约和核心链路可回归，不直接等于真实回答质量；策略效果要看专项评测。

## 当前边界

- 本地面试原型，不直接暴露公网。
- 没有完整认证、限流、租户隔离和生产级隐私治理。
- MCP 默认 mock，协议和进程治理仍可增强。
- Motion 相似度不等于医疗或专业教练诊断。
- Docker 和小程序仍需跨机器、真机和弱网完整验收。

## 收尾句

> 这个项目最想证明的是：我知道什么问题该交给大模型，什么问题该交给知识、搜索、算法或工具，并能用受控流程、错误治理和评测把它们组织成一个完整系统。
