# 会话续接与上下文防爆规则

> 目标：避免 Codex 长会话依赖聊天窗口本身，同时尽量不打断当前任务的工作流。默认先在当前任务内做轻量检查点和原生压缩；只有上下文持续逼近失控、压缩卡住或需要跨任务交接时，才用仓库交接文件恢复现场。

## 1. 问题判断

之前会话容易失败，通常不是代码本身的问题，而是协作上下文太长：

- 聊天里混入大量代码、日志、设计解释和中间计划。
- 自动压缩会把细节折叠成摘要，可能漏掉“正在改哪个文件、下一步做什么、哪些改动不是我做的”。
- 如果压缩过程本身卡住，新会话没有稳定入口，只能重新翻历史。

因此，本项目不再把聊天记录当成唯一记忆源，而是采用三道防线：先限制进入会话的内容，再在较低阈值主动压缩，最后把“可恢复现场”写入仓库。这里的新任务续接是兜底方案，不是日常默认动作；能在当前任务内安全继续时，就优先保持当前任务连续性。

## 2. 三道防线

### 2.1 输入限流

项目在 `.codex/config.toml` 中设置：

```toml
tool_output_token_limit = 4000
```

它限制单次工具结果保留到上下文中的 token 数，避免一次完整测试日志、异常栈或大文件输出占满会话。日常命令仍应主动使用 `rg`、`Select-Object -First`、`Get-Content -Tail` 等方式缩小输出，因为配置上限是最后保险，不是鼓励每次都输出到上限。

### 2.2 提前压缩

项目使用动态会话正文预算：

```toml
model_auto_compact_token_limit = 32000
model_auto_compact_token_limit_scope = "body_after_prefix"
experimental_compact_prompt_file = "compact_prompt.md"
```

- `32000` 让压缩在正文仍可控时发生，降低把超长历史一次性交给压缩流程的风险。
- `body_after_prefix` 只按可增长的会话正文判断，避免仓库规则、技能说明等固定前缀刚加载就消耗压缩预算。
- `.codex/compact_prompt.md` 要求摘要保留目标、文件、决定、验证和下一步，并丢弃原始日志、重复说明和已解决探索。

这些是 Codex 客户端的项目级协作设置，不是本项目业务 Agent 的 `PromptBuilder` 或用户聊天接口配置。项目必须处于 trusted 状态；修改后应在新任务中验证是否生效。

### 2.3 仓库检查点

交接文件为：

```text
.codex/SESSION_STATE.md
```

它只记录当前协作状态，不替代正式文档：

| 内容 | 放哪里 |
|---|---|
| 当前任务目标、进度、下一步 | `.codex/SESSION_STATE.md` |
| 项目事实、能力边界、面试口径 | `docs/` |
| 接口字段、运行方式、测试证据 | 对应的 `docs/接口说明.md`、`docs/运行与排错.md`、`docs/项目证据.md` |

## 3. 检查点生命周期

Codex 在以下场景必须更新 `.codex/SESSION_STATE.md`：

- 任务超过一个简单改动，可能持续多轮对话。
- 修改多个文件、共享模块、接口、测试或面试文档。
- 准备运行较长测试、启动服务、提交代码或推送代码。
- 发现上下文已经很长，或者用户明确说“先停一下”“下次继续”。
- 由于报错、环境问题、权限问题暂时无法继续。

如果只是回答一个问题、查看一个命令结果、做一个很小的单文件修改，可以不更新。

状态只有两种：

- `active`：任务尚未完成，`Next Steps` 必须能让新任务直接继续。
- `idle`：没有待交接工作；保留通用恢复提示，不保留上一任务的长历史。

多步骤任务开始时就标记 `active`，而不是等到准备结束才写。每完成一个可验证里程碑，用最新结论覆盖旧内容；任务完成后立即收敛为 `idle`。

### 3.1 用户主动压缩口令

本项目约定以下自然语言为主动压缩控制口令：

```text
提前帮我压缩上下文
```

“先压缩一下上下文”“帮我做上下文压缩”等明确同义表达采用相同行为。只有当压缩是用户消息的主要意图时才触发；引用这句话、询问规则或要求修改规则时不触发。

收到口令后，Codex 必须按顺序执行：

1. 停止新的探索、代码修改和长测试，不顺带继续原任务。
2. 等待当前不可安全中断的工具调用结束。
3. 覆盖式更新 `.codex/SESSION_STATE.md`，保留目标、文件、决定、验证、阻塞、下一步和可复制的 `Resume Prompt`，总大小不超过 4 KB。
4. 向用户输出不超过 600 字的交接摘要，并说明“压缩检查点已完成”。
5. 若当前产品表面暴露了可调用的原生 compact 能力，则立即调用；否则提示用户下一条只输入 `/compact`。
6. `/compact` 成功后优先在当前任务继续，不要求新建任务。只有出现 3.2 节的兜底条件时，才建议把 `Resume Prompt` 带到真正的新任务。

这里必须区分两件事：仓库检查点可由 Codex 根据自然语言口令自动完成；客户端内部上下文压缩目前由 `/compact` 命令执行。当前助手工具集中没有 compact 调用，因此不能在用户尚未执行 `/compact` 时声称压缩已经完成。官方命令的作用是总结当前可见对话并释放上下文容量。

### 3.2 何时留在当前任务，何时新建任务

主动压缩后的默认策略是“保留当前任务，继续最小必要工作”。新建任务只是兜底，不应把它当成每次压缩后的固定步骤。

留在当前任务继续的条件：

- `/compact` 已成功完成，`/status` 显示上下文压力明显下降。
- 当前任务只剩少量明确步骤，例如补一个测试、改一个函数、完成一次复验。
- 没有新的大范围探索需求，不需要重新读取大量文档或日志。
- 当前任务的聊天记录仍然可响应，没有卡死、明显延迟或命令长时间不返回。

建议新建任务的条件：

- `/compact` 卡住、失败，或成功后 `/status` 仍显示上下文接近上限。
- 当前任务还要继续大范围排查、跑长测试、读大量文件或处理多个模块。
- 已经发生过一次自动压缩，随后上下文又快速膨胀。
- 当前任务响应变慢、工具调用结果频繁截断，或出现明显“记忆混乱”。
- 需要把一个阶段明确交接给下一轮工作，例如从排查转入重构、从实现转入提交/推送。

如果只是担心上下文偏高，先做检查点和 `/compact`，不要立刻换任务。换任务的成本是固定前缀、项目规则、技能说明和交接文件会重新加载；频繁换任务反而会增加摩擦。

## 4. 写法要求

交接内容要短，不要把聊天重新复制一遍：

- `Goal` 写一句话，说明当前任务。
- `Touched Files` 只列真正读写过、后续需要关注的文件。
- `Important Decisions` 只写会影响后续判断的决定。
- `Commands Run` 只记关键命令和结论，不粘贴长日志。
- `Next Steps` 写成下一位 Codex 可以直接执行的清单。
- `Resume Prompt` 保持可复制，让新会话能快速接上。
- 整个文件不超过 4 KB；它是状态快照，不是工作日志。
- 不记录密码、Token、完整用户级配置或环境变量全集。

## 5. 输出预算规则

为了从源头减小上下文：

1. 先用 `rg`/文件列表定位，再读命中的必要片段。
2. 大型测试先用安静模式；失败时只取失败用例、摘要和日志尾部。
3. `git diff` 优先限定到本轮文件，不重复打印整棵工作区的 diff。
4. 长日志需要保留时写入临时文件或测试产物，在聊天中只报告路径和结论。
5. 技能规则要求完整读取 `SKILL.md` 时照常完整读取，但不顺带加载无关参考资料。
6. 排查 Codex 配置只读取明确需要的键，不整份输出可能含认证信息的用户级配置。

## 6. 新会话恢复流程

新会话开始时按这个顺序做：

```text
1. 读 AGENTS.md
2. 读 .codex/SESSION_STATE.md
3. 运行 git status --short
4. 只检查 Touched Files 和 Next Steps 指向的文件
5. 继续未完成任务，避免重新扫描整个项目
```

如果 `.codex/SESSION_STATE.md` 的状态是 `idle`，说明没有正在交接的任务，可以按用户的新请求正常开始。

如果自动压缩已经发生过且会话仍快速增长，不要继续堆积第二轮大范围探索。先完成当前最小可交付闭环、更新检查点；如果 3.2 节的兜底条件成立，再把 `Resume Prompt` 复制到新任务继续。

如果用户使用主动压缩口令，则优先走 3.1 节流程，不必等到自动阈值触发。

## 7. 生效与验证

- `.codex/config.toml` 仅在 trusted 项目中加载。
- 当前已经打开的任务不一定重新加载启动期配置；修改后新建任务最稳妥。
- 使用 TOML 解析器检查语法，并确认 `.codex/SESSION_STATE.md` 小于 4096 bytes。
- 若仍发生卡死，保留任务发生前的会话规模、最后一个工具输出类型和是否已 compact 三项信息，再针对性调低 `tool_output_token_limit` 或 `model_auto_compact_token_limit`，不要盲目扩大上下文窗口。

## 8. 当前边界

这套方案能显著降低输入膨胀并保证可恢复，但不能承诺消除 Codex 客户端或模型服务本身的卡死。仓库无法接管客户端内部压缩实现；因此工程目标是“更早、更小地压缩，并且失败后不丢现场”。

## 9. 面试表达

这个规则也可以作为项目工程能力的一个回答点：

> 我把长会话协作拆成三层：工具输出先做 token 限流，动态会话正文达到安全阈值时提前 compact，仓库检查点负责跨任务恢复。这样即使客户端压缩失败，后续智能体仍能从目标、已改文件、验证状态和下一步恢复，而不是重新扫描整个项目。

它体现的不只是“多写一个文档”，而是智能体工程里的预算控制与可恢复性设计：限制噪声输入，把压缩负载前移并缩小，再把易丢的运行态变成版本化、可检查、可交接的项目状态。
