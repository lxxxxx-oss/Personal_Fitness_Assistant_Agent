# 记忆系统设计

> **定位说明**：本文档记录持久化记忆系统的目标设计和当前落地。表结构中既有已实现字段，也有生产化预留；判断当前事实时优先看本节“当前状态与边界”和[项目总览](../项目总览.md)。
>
> 记忆系统 ≠ 上下文压缩。记忆系统决定存什么、存多久、怎么检索；上下文压缩决定进 prompt 时带什么、怎么压缩、怎么拼接。

## 1. 架构总览

```
                          ┌──────────────────────────┐
                          │     Prompt Builder        │
                          │  (上下文压缩 + 拼接策略)   │
                          └──────────┬───────────────┘
                                     │
          ┌──────────────────────────┼──────────────────────────┐
          │                          │                          │
          ▼                          ▼                          ▼
┌─────────────────┐    ┌───────────────────────┐    ┌──────────────────┐
│SlidingWindowMemory│    │  Memory Retrieval     │    │ compact_summary  │
│ (hot cache)      │    │  SQLite + Milvus      │    │ (active only)    │
│ 最近 6 轮原文     │    │  轻量检索分数 top-k    │    │ SQLite summaries │
└────────┬────────┘    └───────────┬───────────┘    └────────┬─────────┘
         │                         │                         │
         ▼                         ▼                         ▼
┌──────────────────────────────────────────────────────────────────┐
│                        SQLite (source of truth)                  │
│  conversations │ messages │ memory_items │ summaries │ task_states│
│  memory_sources │ memory_relations │ access_count / embedding_jobs│
│  candidate_memories │ FTS5                                       │
└──────────────────────────────┬───────────────────────────────────┘
                               │ embedding_jobs（当前由接口触发）
                               ▼
┌──────────────────────────────────────────────────────────────────┐
│                        Milvus (embedding retrieval only)          │
│  memory_items embedding                                           │
└──────────────────────────────────────────────────────────────────┘
```

### 1.1 核心原则

- **SQLite 是 source of truth**，不是 Milvus
- **SlidingWindowMemory 只是 hot cache**，服务重启后从 SQLite 恢复
- **写入 SQLite 是主路径**，Milvus 异步同步，写入不阻塞请求链路
- **记忆注入 ≠ 全部注入**：当前按检索分数取 top-k，并用 1200 字符上限控制长期记忆块；更细的分层 token 预算属于目标设计
- **`_structured_state` 不是长期记忆**：当前只服务本轮路由和 Prompt 构建，尚未自动抽取进 `memory_items`

### 1.2 当前状态与边界

当前代码已经完成持久化记忆的最小闭环，但还没有实现本文设计中的全部生产化治理能力。

| 层级 | 当前状态 |
|---|---|
| 已实现 | SQLite 会话与消息持久化、会话恢复、抽取式 active summary、增量压缩边界、HTTP/SSE/WebSocket 摘要接入、长期记忆 CRUD、显式 Memory Writer、敏感 candidate 确认、FTS5/LIKE 检索、Prompt 预算注入、embedding jobs 和可选 Milvus 用户记忆增强 |
| 部分实现 | SQLite 是 source of truth，Milvus 和摘要更新失败不影响聊天主链路；Memory + Context + RAG benchmark 已覆盖 14 条链路，其中摘要 6 个场景；embedding job 可由接口同步处理 |
| 生产化补齐 | 登录鉴权、用户授权、加密脱敏、常驻后台 worker、过期清理、跨实例共享、删除同步、更大规模记忆质量与隐私评测 |

面试时应表达为：

> 当前项目已经完成原型级分层记忆闭环：SQLite 保存完整会话和长期事实，抽取式 active summary 承接较早对话，最近原文保留可读细节；Memory Writer 控制显式写入，敏感内容先确认，Milvus 只做可选语义召回。当前摘要按字符阈值生成，不宣称模型级语义总结；下一步重点是授权、后台任务、跨实例和隐私治理。

## 2. 数据模型

### 2.1 核心表：`memory_items`

当前代码中的 `memory_items` 是原型级事实表，字段以 `app/memory/memory_store.py` 为准：

```sql
CREATE TABLE memory_items (
    id              TEXT PRIMARY KEY,
    user_id         TEXT NOT NULL,
    kind            TEXT NOT NULL CHECK (
                      kind IN ('rule','fact','preference','todo',
                               'goal','constraint','decision','note')
                    ),
    content         TEXT NOT NULL,
    scope           TEXT NOT NULL DEFAULT 'global',
    source_type     TEXT NOT NULL CHECK (
                      source_type IN (
                        'code_rule','user_explicit_remember','llm_candidate',
                        'compact_extraction','project_file','manual_import'
                      )
                    ),
    importance      REAL NOT NULL DEFAULT 0.5,
    status          TEXT NOT NULL DEFAULT 'active'
                      CHECK (status IN ('active','deleted')),
    access_count    INTEGER NOT NULL DEFAULT 0,
    last_accessed_at TEXT,
    memory_key      TEXT NOT NULL,
    metadata        TEXT NOT NULL DEFAULT '{}',
    created_at      TEXT NOT NULL,
    updated_at      TEXT NOT NULL
);
```

生产化设计中还可以增加 `pinned`、`project_id`、`expires_at`、更细状态和访问日志，但这些不应在面试中说成当前已经完整落地。

目标设计可扩展为：

```sql
CREATE TABLE memory_items (
    id            TEXT PRIMARY KEY,          -- UUID
    user_id       TEXT NOT NULL,             -- 隔离键
    kind          TEXT NOT NULL CHECK (kind IN (
                      'rule','fact','preference','todo',
                      'goal','constraint','decision','note'
                  )),
    content       TEXT NOT NULL,             -- 记忆正文
    importance    REAL NOT NULL DEFAULT 0.5, -- 0.0-1.0，归一化后的重要度
    pinned        INTEGER NOT NULL DEFAULT 0,-- 1 = 永不衰减，始终优先
    scope         TEXT DEFAULT 'global',     -- global / project / conversation
    project_id    TEXT,                      -- 关联项目
    metadata      TEXT DEFAULT '{}',         -- JSON 扩展字段
    access_count  INTEGER DEFAULT 0,         -- 被检索/注入次数
    last_accessed_at TEXT,                   -- ISO timestamp
    created_at    TEXT NOT NULL,
    expires_at    TEXT,                      -- NULL = 永不过期
    status        TEXT NOT NULL DEFAULT 'active'
                      CHECK (status IN ('active','superseded','archived','deleted'))
);

CREATE INDEX idx_memory_user ON memory_items(user_id, status);
CREATE INDEX idx_memory_kind ON memory_items(user_id, kind, status);
CREATE INDEX idx_memory_pinned ON memory_items(user_id, pinned, status);
```

### 2.2 `kind` 枚举语义

当前 `POST /memory` 的默认 importance 是 0.5，也允许调用方显式传入；聊天中的显式“记住...”统一写 0.8，并用关键词把内容粗分为 goal、constraint、preference 或 note。下表中的按 kind 初始值是更细的目标策略，不是当前默认值。

| kind | 含义 | 目标初始 importance | 示例 |
|---|---|---|---|
| `rule` | 项目规则、代码规范、必须遵守的长期规则 | 0.9 | "不能修改 generated 文件" |
| `constraint` | 限制条件、安全边界 | 0.9 | "用户膝盖有旧伤" |
| `goal` | 当前目标 | 0.7 | "训练目标是减脂" |
| `fact` | 稳定事实 | 0.5 | "用户是素食者" |
| `preference` | 用户偏好 | 0.5 | "喜欢详细解释" |
| `decision` | 已做出的设计决策 | 0.6 | "记忆系统选用 SQLite + Milvus" |
| `todo` | 待办事项 | 0.4（按紧急程度浮动） | "完成 memory_items 表设计" |
| `note` | 一般长期笔记、调试经验 | 0.2 | "Qwen3-0.6B 在 CPU 上推理约 X tokens/s" |

`memory_items` 只保存跨会话仍有价值的长期记忆。`compact_summary` 和 `session_summary` 不进入 `memory_items`，统一放在 `summaries` 表中，避免把“当前会话状态快照”和“长期可复用记忆”混在一起。

### 2.3 importance 计算规则

当前代码使用较轻量的分数：搜索结果合并后按关键词命中、可选 embedding 分数、`importance` 和 `access_count` 加权排序。`access_count` 只在检索命中后增加，访问次数加成有上限，避免某条记忆因为早期频繁命中而永久霸榜。

下面的时间衰减、pinned 规则和 recency 加权是生产化目标设计，不是当前已完整实现的在线策略。

#### 初始值

由 `kind` 决定初始 importance（见上表），pinned 规则的 importance 始终为 1.0。

#### 动态调整

```python
importance = clamp(
    initial_importance
    + 0.10 * min(access_count, 20) / 20  # 频繁引用加权重（最多 +0.1）
    + 0.10 * recency_bonus(last_accessed) # 最近使用加权重
    - 0.02 * weeks_since_creation,        # 时间衰减，按周计算，避免短期记忆过快掉权重
    0.0, 1.0
)
```

- pinned=true 的记忆 **不参与衰减**，importance 保持初始值
- 被 superseded 或 archived 的记忆 **不再注入**
- 如果后续希望更慢衰减，可把 `weeks_since_creation` 换成 `months_since_creation`；不要按天直接衰减，否则常规用户偏好会在短时间内被错误降权。

### 2.4 写入路径

| 来源 | source_type | confidence | 需要确认 |
|---|---|---|---|
| 代码规则自动写入 | `code_rule` | 1.0 | 否 |
| 用户明确说“记住...” | `user_explicit_remember` | 1.0 | 普通内容否；当前健康敏感内容进入候选确认 |
| LLM 从对话推测 | `llm_candidate` | < 1.0 | **目标能力**：需要确认，当前未自动提取 |
| 压缩流程提取 | `compact_extraction` | 0.7 | **预留来源**：当前未自动写入 |
| 项目文件导入 | `project_file` | 0.8-1.0 | **预留来源**：当前无自动导入链路 |
| 人工批量导入 | `manual_import` | 1.0 | 通过 API/代码显式写入时可用 |

- 用户明确说“记住...”时，当前代码会调用显式 Memory Writer：普通内容直接写 `memory_items`；命中健康敏感词时先写 `candidate_memories`。
- `llm_candidate`、`compact_extraction` 等枚举已经为演进预留，但当前聊天主链路不会自动生成这些候选。

### 2.5 去重策略

#### 主路径：memory_key 优先

每条记忆写入前，先计算 `memory_key = hash(user_id + kind + normalized_content)`。
同 `memory_key` 已存在且 `status='active'` → 返回已有记录并标记 `deduplicated=true`，不创建新记录，也不会因为重复写入自动增加 `access_count`；访问次数只在检索命中后更新。

#### 目标辅助路径：embedding 相似度

当前写入只做 `memory_key` 精确去重。生产化后可在同 user + 同 kind 但不同 memory_key 时增加 embedding 相似度检查：

| 阈值 | 行为 |
|---|---|
| ≥ **0.95** | **自动合并**：更新已有记录 content，不创建新记录 |
| **0.92-0.95** | **候选去重**：写入 `candidate_memories` 标记 `duplicate_of_memory_id`，不自动合并，等下一次确认 |
| < 0.92 | 视为独立新记忆，正常写入 |

这组阈值尚未接入当前写路径；落地前需要基于中文记忆样本重新标定，不能把 0.92/0.95 当作已经验证的项目指标。

### 2.6 `_structured_state` 与 `memory_items` 的关系

#### 当前事实与设计原则

`_structured_state` 和 `memory_items` 不是同一个东西，也不是完全独立：

- **`_structured_state`**（进程内存）：当前会话的轻量工作态。实际默认字段是 `task`、`decisions`、`tool_results_summary`、`profile`、`knowledge_sources`、`user_context`；其中 `user_context` 目前只是预留空对象。
- **`memory_items`**（SQLite + Milvus）：长期可检索记忆，只保存跨会话仍有价值的规则、事实、偏好、目标、约束、长期决策、项目知识。
- **当前关系**：两者没有自动提升链路。长期记忆只来自用户“记住...”或 Memory API 的显式写入。
- **目标关系**：未来可把 `_structured_state` 作为候选来源，由 Memory Writer 判断长期价值后再提升。

#### 字段提升规则

| `_structured_state` 字段 | 提升到 memory_items？| 规则 |
|---|---|---|
| `task` | 当前任务留在会话状态；长期目标可提升为 `kind='goal'` | 目标规则，尚未自动提升 |
| `decisions` | 长期架构决策 → `kind='decision'`，同步写入 | 选择性提升 |
| `profile` / `user_context` | 长期偏好/事实/约束 → `kind='preference'/'fact'/'constraint'` | 选择性提升 |
| `knowledge_sources` | 不直接提升，作为 `memory_sources` 的引用证据 | 不提升 |
| `tool_results_summary` | 默认不进 memory_items | 不提升 |

上表是演进规则，不代表当前已运行自动抽取。`files`、`errors`、`todos` 也不是当前 `_structured_state` 的正式字段；如果未来需要，应先更新状态结构和测试，再补文档。

#### 完整数据流

```
当前会话运行
  → 子图写入 _structured_state（进程内临时态）
  → 当前 active summary 由 conversation compactor 单独持久化到 summaries（SQLite）
  → 当前没有 _structured_state → memory_items 的自动提升
  → 目标：Memory Writer 判断长期价值 → 候选确认/去重 → memory_items
```

### 2.7 Memory Writer：长期记忆写入决策

当前 Memory Writer 是一个明确、可解释的最小写入器：只处理用户“记住...”和 Memory API 的显式写入。下文更完整的自动筛选器是目标设计，尚未接入聊天主链路。

#### 输入

| 输入 | 来源 | 用途 |
|---|---|---|
| 当前用户消息 | `messages` | 判断是否有明确的“记住...”请求 |
| 当前助手回复 | `messages` | 目标输入：识别长期承诺或设计决策，当前未启用 |
| `_structured_state` | 当前会话工作态 | 目标输入：提供 profile、user_context、decisions 等候选来源，当前未启用 |
| compact_summary | `summaries` | 目标输入：长对话压缩后补充上下文，当前未启用 |
| 工具结果摘要 | `tool_results_summary` | 目标输入：形成可复用事实或决策时进入候选，当前未启用 |

#### 输出

| 输出 | 说明 |
|---|---|
| `memory_items` | 已确认的长期记忆，可参与 FTS5 / Milvus 检索和 prompt 注入 |
| `candidate_memories` | LLM 推测、敏感、冲突或低置信的候选记忆，需要用户确认或后续规则确认 |
| `memory_sources` | 记录记忆来源，方便审计“这条记忆为什么被写入” |
| `memory_relations` | 目标输出：记录 supersedes / contradicts / supports / derives_from 等关系；当前未自动生成 |

#### 写入决策规则

| 场景 | 处理 |
|---|---|
| 用户明确说“记住...” | **当前已实现**：普通内容写 `memory_items`；健康敏感内容写 `candidate_memories` |
| Memory API 显式写入 | **当前已实现**：按调用参数写入，适合演示 CRUD 和人工维护 |
| 用户长期画像、助手承诺、项目决策、跨会话 TODO | **目标能力**：由自动 Writer 识别后写正式记忆或候选 |
| 一次性闲聊、临时工具结果、普通搜索摘要 | **当前不会自动写入**，因此不会由聊天链路污染长期记忆 |
| LLM 推测出的偏好/事实 | **目标能力**：先进入 candidate，当前没有 LLM 自动抽取器 |
| 与已有记忆冲突 | **目标能力**：建立 conflict/relation 并确认；当前只有精确 `memory_key` 去重，没有语义冲突处理 |

#### 防记忆污染规则

- **少写优于乱写**：没有长期价值或置信度不足时，不进入 `memory_items`。
- **敏感信息谨慎写入**：健康、身体状况、隐私偏好等内容默认进入候选确认流程，除非用户明确要求记住。
- **工具结果默认不长期保存**：只有工具结果沉淀成稳定事实、用户偏好、设计决策或可复用经验时才提升。
- **冲突不直接覆盖**：这是目标规则；当前尚未自动识别语义冲突。
- **写入前先去重**：当前只走 `memory_key` 精确去重；embedding 辅助去重是后续能力。

#### 面试口径

可以这样解释：

> 我先落地了规则驱动的显式 Memory Writer：只有用户明确说“记住”或调用 Memory API 才写入，健康敏感内容先进入候选，所以普通闲聊和工具结果不会自动污染记忆库。更完整的 LLM 候选提取、语义冲突识别和结构化状态提升已经有数据模型设计，但我不会把它们说成当前已实现。

### 2.8 辅助表

#### candidate_memories

当前表主要承接显式“记住...”触发的健康敏感内容，等待用户确认。候选记忆不参与普通检索、FTS5、Milvus embedding 和 prompt 注入。未来才会扩展为 LLM 推测候选。

```sql
CREATE TABLE candidate_memories (
    id            TEXT PRIMARY KEY,
    user_id       TEXT NOT NULL,
    kind          TEXT NOT NULL,
    content       TEXT NOT NULL,
    scope         TEXT NOT NULL DEFAULT 'global',
    source_type   TEXT NOT NULL,
    importance    REAL NOT NULL DEFAULT 0.5,
    privacy_level TEXT NOT NULL DEFAULT 'normal',
    status        TEXT NOT NULL DEFAULT 'pending'
                    CHECK (status IN ('pending','confirmed','rejected')),
    metadata      TEXT NOT NULL DEFAULT '{}',
    created_at    TEXT NOT NULL,
    updated_at    TEXT NOT NULL
);
```

**当前流程**：显式健康敏感内容写入 pending candidate；用户通过确认接口将其转为正式记忆，或通过拒绝接口标记 rejected。`project_id`、置信度、冲突指针和原文证据等更细字段仍是生产化扩展项。

#### FTS5 全文索引

FTS5 表使用真正独立的虚拟表，不使用 `content_rowid='id'`。原因是 `memory_items.id` 是 TEXT UUID，不能直接和 SQLite FTS5 的整数 `rowid` 对齐。

因此 FTS5 表显式保存 `memory_id UNINDEXED`，全文检索先命中 FTS5，再用 `memory_id` 回查 `memory_items`：

```sql
CREATE VIRTUAL TABLE memory_items_fts USING fts5(
    memory_id UNINDEXED,
    content
);
```

当前代码没有使用 SQLite trigger，而是在 `MemoryStore.create_memory()`、`update_memory()`、`delete_memory()` 的 Python 写路径里同步维护 FTS5。这样做更适合原型演示：行为集中在应用层，测试更容易 mock；生产化时可以再把索引维护下沉到触发器或后台任务。

下面是生产化可选的触发器方案：

```sql

-- 触发器：INSERT
CREATE TRIGGER memory_items_fts_insert AFTER INSERT ON memory_items
WHEN NEW.status = 'active'
BEGIN
    INSERT INTO memory_items_fts(memory_id, content)
    VALUES (NEW.id, NEW.content);
END;

-- 触发器：UPDATE
CREATE TRIGGER memory_items_fts_update AFTER UPDATE OF content, status ON memory_items
BEGIN
    DELETE FROM memory_items_fts WHERE memory_id = OLD.id;
    INSERT INTO memory_items_fts(memory_id, content)
    SELECT NEW.id, NEW.content
    WHERE NEW.status = 'active';
END;

-- 触发器：软删除
CREATE TRIGGER memory_items_fts_delete AFTER UPDATE ON memory_items
WHEN OLD.status = 'active' AND NEW.status != 'active'
BEGIN
    DELETE FROM memory_items_fts WHERE memory_id = OLD.id;
END;
```

keyword_score 计算时查询 FTS5：

```sql
SELECT mi.id AS memory_id,
       snippet(memory_items_fts, 1, '<b>', '</b>', '...', 40) AS snippet,
       mi.content,
       mi.kind,
       mi.importance
FROM memory_items_fts
JOIN memory_items mi ON mi.id = memory_items_fts.memory_id
WHERE memory_items_fts MATCH ?
  AND mi.user_id = ?
  AND mi.status = 'active'
ORDER BY rank
LIMIT 20;
```

### 2.9 并发控制

#### conversations 版本号 + Transaction

当前 `save_compact_summary()` 使用进程内锁和单个 SQLite transaction，原子完成旧摘要 supersede、新摘要插入和压缩边界更新；`version` 会递增，但当前 SQL 没有 `WHERE version = ?` 条件，因此不能把它说成已实现跨进程乐观锁。

生产化目标可在现有 transaction 上增加真正的乐观锁：

```
1. 压缩开始前读取 conversation：
   - conversation_id
   - current_version
   - active compact_summary
   - last_compacted_message_id
   - 新增 messages

2. 基于这些内容生成新的 compact_summary

3. 写入时开启 SQLite transaction

4. 更新 conversations 时带版本条件：
   UPDATE conversations
   SET version = version + 1,
       updated_at = ?,
       last_compacted_message_id = ?
   WHERE id = ? AND version = ?   -- 乐观锁检查

5. 如果 affected_rows = 1（无并发冲突）：
   - 将旧 compact_summary 标记为 superseded
   - 插入新的 compact_summary（status='active'）
   - 更新 last_compacted_message_id
   - commit

6. 如果 affected_rows = 0（有并发冲突）：
   - 当前压缩结果丢弃
   - 重新读取最新 state
   - 最多重试 1 次
   - 仍冲突则终止本次压缩，不阻塞请求
```

- **当前 transaction**：保证一次摘要替换和边界更新在本进程内原子提交；当前没有同时更新 `task_states`。
- **目标乐观锁**：用于发现多实例并发冲突，避免后提交的旧上下文摘要覆盖更完整的新摘要。

## 3. 检索与注入

### 3.1 检索流程

当前已实现流程较轻：

```text
用户 query
  -> SQLite FTS5 检索
  -> FTS5 不可用或无结果时走 LIKE 兜底
  -> MEMORY_MILVUS_ENABLED=1 时合并 Milvus 语义召回
  -> score = keyword / embedding / importance / access_count 加权
  -> PromptBuilder 按字符预算注入最多 5 条长期记忆
```

下面 §3.2 至 §3.5 的 pinned 优先、metadata_score、recency、固定权重和分层 token 预算都是目标设计，用于说明生产化升级方向；它们不是当前在线打分公式。

每次 builder 构建 prompt 时执行一次 Memory Retrieval：

```
1. 先取 pinned rule + constraint 候选
   SELECT * FROM memory_items
   WHERE user_id = ? AND status = 'active' AND pinned = 1
     AND kind IN ('rule','constraint')
   ORDER BY
     CASE WHEN scope = 'project' AND project_id = ? THEN 0 ELSE 1 END,
     importance DESC,
     last_accessed_at DESC

2. 当前 query 做 hybrid relevance 检索
   - embedding_score: Milvus 向量相似度（COSINE）
   - keyword_score: SQLite FTS5 全文搜索
   - metadata_score: scope / project_id / kind 匹配加分

3. 计算 final_score
   final_score = 0.35 * importance
               + 0.45 * relevance
               + 0.10 * recency
               + 0.10 * pinned_bonus

4. 按分层预算取 top-k 注入 prompt
```

### 3.2 relevance 计算

```python
relevance = 0.6 * embedding_score   # Milvus COSINE (0-1)
          + 0.3 * keyword_score     # FTS5 rank-position 归一化 (0-1)
          + 0.1 * metadata_score    # scope / project_id / kind match (0-1)
```

#### embedding_score

直接使用 Milvus COSINE 相似度，天然 0-1 范围。Milvus 不可用时降级为 0。

#### keyword_score

FTS5 `ORDER BY rank ASC` 取 top-k，Python 层按排名位置归一化，不直接使用 FTS5 raw rank 数值：

```python
# FTS5 返回的排名位置：第 1 名 position=1，第 2 名 position=2，...
keyword_score = 1.0 - (rank_position - 1) / k
# 第 1 名 → 1.0，最后一名 → 1/k
```

#### metadata_score

```python
metadata_score = 0.0
+ (0.4 if scope == 'project' and query_project_id == memory.project_id else 0.0)
+ (0.3 if kind matches query context intent else 0.0)   # 运动类问题优先 kind='fact/preference/goal'
+ (0.3 if category matches query domain else 0.0)        # category='nutrition' 匹配饮食类问题
```

最大值 1.0，无匹配为 0。

### 3.3 final_score 公式

所有因子归一化到 0-1 范围：

| 因子 | 权重 | 归一化方式 | 说明 |
|---|---|---|---|
| `importance` | 0.35 | 直接使用（0.0-1.0） | 从 memory_items.importance 读取 |
| `relevance` | 0.45 | embedding + keyword + metadata 加权 | 见 §3.2 |
| `recency` | 0.10 | `1.0 / (1.0 + weeks_since_last_access)` | 本周访问 → 1.0；约 30 天前 → 0.19 |
| `pinned_bonus` | 0.10 | `0.10 if pinned else 0.0` | pinned 记忆始终获得额外加分 |

### 3.4 注入规则矩阵

以下矩阵是目标方案。当前实现统一取最多 5 条检索结果，长期记忆块最多 1200 **字符**，尚未按 kind 划分 token 子预算，也没有 `pinned` 字段。

| 类型 | 注入规则 |
|---|---|
| `rule` / `constraint`（pinned=true）| **优先注入但不是无限注入**，走自己的子预算（最多 800 tokens）；超出时按 scope / project_id / importance / last_accessed_at 排序，必要时摘要化 |
| `fact` / `preference` / `goal` | 按 `final_score` 取 top-k，共享子预算（最多 700 tokens） |
| `decision` / `todo` / `note` | **不走常驻**，仅当 relevance 命中阈值时注入，共享子预算（最多 500 tokens） |
| compact_summary | 压缩后持续注入当前会话，直到被新 summary 替代，单独预算（800-1200 tokens） |

### 3.5 分层预算模型

以下是生产化目标预算，不是当前配置：

```
memory_total_budget = 2000 tokens (硬上限)

子预算（可弹性溢出）：
  pinned rule / constraint    → 最多 800 tokens
  fact / preference / goal    → 最多 700 tokens
  decision / todo / note      → 最多 500 tokens

弹性规则：
  - 某类记忆没用满预算时，剩余额度释放给高 final_score 的其他记忆
  - 超过总预算时，按 final_score 从低到高截断
  - pinned 规则优先进入候选，但仍受 800 tokens 子预算限制
  - pinned 超出预算时，先按 scope / project_id / importance / last_accessed_at 排序；仍超出时生成短摘要，不把全文无限塞进 prompt

排序优先级（同分时）：
  pinned_rule > compact_summary > task_state > 高 final_score 其他记忆
```

### 3.6 compact_summary 生命周期

- **生成**：当前输入 = 上一 active compact_summary + 尚未压缩的 messages；采用确定性摘录，不读取 `task_state`
- **替换**：新 summary 标记 `active`，旧 summary 标记 `superseded`（保留在 summaries 表用于审计）
- **注入**：仅注入当前 active compact_summary，不注入历史版本
- **不入 memory_items**：compact_summary 是会话状态快照，不是长期记忆；当前也没有自动抽取到长期记忆
- **会话结束时**：`session_summary` 是表结构预留类型，当前没有自动生成流程

## 4. 会话生命周期

### 4.1 三者关系

| 组件 | 职责 | 持久化 |
|---|---|---|
| SQLite `conversations` / `messages` | 完整历史，source of truth | ✅ |
| `_sessions` 字典 | 运行时活跃会话对象 | ❌ 进程内存 |
| `SlidingWindowMemory` | 最近 6 轮原文 hot cache | ❌ 进程内存 |

### 4.2 会话创建

- 请求开始时先解析会话：有合法 `conversation_id` 就续接；不传时优先复用该用户最近的 active conversation，没有才新建。
- 一轮回答完成后，user/assistant 两条消息通过 `add_turn()` 在同一 transaction 中写入；不是收到用户字节后就单独落一条 user message。

### 4.3 清空历史

- `DELETE /chat/{user_id}/history` → 清空该用户全部 hot cache，并把 SQLite 中该用户所有 active conversations 标记为 archived；响应中的 `conversation_id` 只是清理前最近的一条
- 当前没有公开的“只清 hot cache、不动 SQLite”接口；如需该语义，应另行设计并明确用户预期

### 4.4 服务重启恢复

```
1. _sessions 为空（正常）
2. 用户继续某个 conversation_id
   → 从 SQLite messages 读取最近 N 条
   → 恢复 SlidingWindowMemory 最近 6 轮
   → 读取 active compact_summary
   → 当前不读取 task_state；该表是后续任务恢复能力的预留
3. 用户不指定 conversation_id
   → 查找该用户最近 active conversation
   → 恢复或新建
```

### 4.5 conversation 状态

| status | 含义 |
|---|---|
| `active` | 当前可继续 |
| `idle` | 超过 `idle_timeout_minutes` 未活跃 |
| `archived` | 用户清空或系统归档 |
| `deleted` | 逻辑删除 |

表结构支持四种状态；当前主链路主要创建/恢复 `active`，清空历史时转为 `archived`，尚未实现自动 idle 检测和 conversation deleted 接口。

## 5. Milvus 同步策略

### 5.1 异步同步

当前已实现的是 `embedding_jobs` 表和 `/memory/embedding-jobs/process` 本地处理接口，适合演示和测试；它不是常驻后台 worker。生产化时可以把这个同步处理迁移到独立 worker 或任务队列。

```
写入 memory_items (SQLite)
  → 创建 embedding_jobs 记录 (status='pending')
  → worker 或本地处理接口读取 pending jobs
  → Sentence-Transformers 编码
  → Milvus 写入（`source = memory_id`，主键为通用 Retriever 生成的 INT64 chunk id）
  → 更新 embedding_jobs status='completed'
  → 失败时标记 status='failed'，不阻塞记忆写入
```

### 5.2 Milvus Collection 结构

用户记忆当前复用通用 `MilvusRetriever` schema，不单独保存用户画像字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | INT64 (PK) | 通用 Retriever 生成的稳定 chunk id |
| `vector` | FLOAT_VECTOR(512) | 当前默认 `BAAI/bge-small-zh-v1.5` 维度；切换模型时新建 Collection，不能复用不同维度索引 |
| `content` | VARCHAR | 记忆正文 |
| `source` | VARCHAR | 当前写入 `memory_items.id`，检索后据此回查 SQLite |

当前语义检索会先从 Collection 取候选，再用 `get_memory(user_id, memory_id)` 做用户过滤。它能满足本地原型隔离，但用户字段没有进入 Milvus filter，跨用户数据增多时会挤占 top-k，也不等同于生产级租户隔离；生产化应把 `user_id` 加入向量侧 schema 和过滤表达式。

### 5.3 降级策略

- Milvus 不可用时 → embedding_score 退化为 0，继续使用 SQLite FTS5/LIKE、importance 和 access_count 排序
- embedding_jobs 积压 → 新记忆仍然可被 FTS5 检索到，只是 embedding 检索暂时缺失

## 7. 实现阶段

实现顺序遵循一个原则：先把短期会话“可恢复”，再把长期记忆“可写可删”，最后再引入 Milvus 和复杂压缩联动。这样能保证每一步都可演示、可测试，不会一开始就被分布式检索和后台 worker 拖住。

### 阶段 1：会话持久化最小闭环

状态：**已实现**。
- 创建 SQLite 数据库文件（`data/memory/memory.db`）。
- 先建 `conversations`、`messages`、`summaries`、`task_states` 四张表。
- `/chat` 支持可选 `conversation_id`。
- 服务重启后可以恢复最近 conversation 和最近消息。
- 验收标准：重启服务后，用户继续同一 `conversation_id`，仍能拿到最近上下文。

### 阶段 2：长期记忆基础表与 CRUD

状态：**已实现原型闭环**；Milvus 更新/删除同步尚未完成。
- 建 `memory_items`、`memory_sources`、`memory_relations`。
- 实现 `MemoryStore` 基础 CRUD。
- 实现 `GET /memory`、`POST /memory`、`PATCH /memory/{id}`、`DELETE /memory/{id}`。
- 删除先做 SQLite 逻辑删除，后续再同步派生索引。
- 验收标准：用户可以显式新增、查看、修改、删除一条长期记忆。

### 阶段 3：最小 Memory Writer

状态：**部分实现**。显式“记住...”和精确 `memory_key` 去重已完成；结构化状态自动提升尚未实现。
- 支持用户明确说“记住...”时写入正式记忆。
- 目标：支持从 `_structured_state.profile/user_context/decisions` 中提升长期事实。
- 实现 `memory_key` 精确去重。
- 暂不依赖 LLM 自动推测，先用规则保证稳定。
- 验收标准：显式记忆写入准确率高，一次性信息不会被写入。

### 阶段 4：候选记忆与隐私确认

状态：**部分实现**。候选表与确认/拒绝接口已完成，自动隐私识别目前只覆盖一组健康关键词。
- 建 `candidate_memories`。
- 实现候选确认/拒绝接口。
- 当前 health 关键词命中时进入候选；security 拦截仍是目标能力。
- 目标：冲突记忆先进入 candidate 或 relation，不直接覆盖。
- 验收标准：敏感信息候选拦截率达标，用户能确认或拒绝候选记忆。

### 阶段 5：SQLite 检索与注入

状态：**已实现轻量版**。当前是 FTS5/LIKE + importance/access_count + 可选 embedding 的合并排序，并按 1200 字符预算注入；metadata_score、recency、pinned 和分层 token 预算属于目标设计。
- 建独立 FTS5 表 `memory_items_fts(memory_id UNINDEXED, content)`。
- 已实现 keyword/LIKE 基础检索；metadata_score 是后续项。
- 已实现轻量加权分数和字符预算注入；完整 final_score 与分层预算是后续项。
- 先不依赖 Milvus，也能完成基础个性化召回。
- 验收标准：Recall@5 达到 MVP 基线，prompt 不超预算。

### 阶段 6：Milvus 异步增强

状态：**已实现本地任务处理闭环**，不是常驻异步 worker；尚无证据证明 Milvus 一定优于 SQLite 基线。
- 建 `embedding_jobs`。
- 当前已实现本地处理接口：Sentence-Transformers 编码 → Milvus upsert。
- 后续生产化再改为常驻后台 worker 或任务队列。
- 实现指数退避重试和失败降级。
- Milvus 不可用时退回 FTS5 + metadata。
- 当前已证明不可用时主流程可降级；“Milvus 可用时召回更好”仍需扩大记忆检索评测证明。

### 阶段 7：Prompt Builder 与上下文压缩联动

状态：**已实现轻量版**。长期记忆、active summary、最近对话已经组合注入；按 kind/pinned 划分 token 子预算尚未实现。
- 在统一 Prompt Builder 中注入长期 memory、active compact_summary 和最近对话。
- 目标：按预算控制 pinned、fact/preference/goal、decision/todo/note。
- compact_summary 只作为会话状态，不直接进入 `memory_items`。
- 验收标准：长对话、长期记忆、RAG 证据能同时进入 prompt 且不超限。

### 阶段 8：评测、审计与生产化增强
- 建立 memory eval case：写入准确率、误写率、Recall@K、候选拦截率。
- 增加 memory_access_logs 清理策略。
- 补充登录鉴权、用户授权、脱敏和删除同步。
- 验收标准：一次记忆系统改动可以通过指标判断是否变强。

## 8. 风险与边界

下表同时包含当前措施和生产化目标，已用标签区分：

| 风险 | 处理 |
|---|---|
| Milvus 不可用导致检索质量下降 | **当前**：FTS5/LIKE 兜底；metadata_score 是目标项 |
| embedding_jobs 积压 | **当前**：新记忆仍可被 FTS5 检索，接口触发有限重试；**目标**：常驻 worker 与积压告警 |
| 记忆表膨胀 | **当前**：查询按 active status 过滤；**目标**：过期字段和定期清理 |
| 重复注入相同记忆 | **当前**：检索合并按 id 去重、字符预算截断；**目标**：访问日志与连续注入降权 |
| LLM candidate 记忆质量低 | **当前**：没有 LLM 自动提取；**目标**：候选确认、来源与置信度治理 |
| SQLite 并发写入瓶颈 | **当前**：单机 WAL + 进程内锁；**目标**：压测后决定是否迁移服务化存储 |
| 同一用户并发请求 race condition | **当前**：单进程 transaction；**目标**：真正的 version 乐观锁和冲突重试 |
| 冷启动无初始记忆 | **当前**：首次对话注入空记忆，安全规则直接写在各 Prompt 模板中 |
| final_score 各因子量级不一致 | **目标**：importance/relevance/recency 统一归一化并重新标定权重 |
| Token 计数偏差 | **当前**：按字符预算；**目标**：使用实际模型 tokenizer 精确计数 |
| content 相似记忆重复写入 | **当前**：`memory_key` 精确去重；**目标**：embedding 相似度去重，阈值需用评测集标定 |

## 9. 评测体系

记忆系统的评测目标不是“写入的记忆越多越好”，而是证明：该记的能记住，不该记的不污染，检索时能找回，注入时不撑爆 prompt。

### 9.1 评测样本类型

| Case 类型 | 测什么 | 示例 |
|---|---|---|
| 显式记忆写入 | 用户明确要求记住时是否进入 `memory_items` | “记住，我是素食者” |
| 隐式长期偏好识别 | 用户没有说“记住”时，是否能识别长期价值 | “我平时不吃牛肉，之后推荐饮食注意一下” |
| 敏感信息候选确认 | 健康/隐私类信息是否先进 `candidate_memories` | “我膝盖有旧伤” |
| 一次性信息过滤 | 临时闲聊和短期任务是否不会污染长期记忆 | “今天下午我想喝咖啡” |
| 冲突检测 | 新旧记忆矛盾时是否不直接覆盖 | 旧记忆“素食者”，新输入“我现在开始吃肉了” |
| 检索命中 | 给定 query 是否召回正确 memory | “给我推荐晚餐”应召回素食偏好 |
| Milvus 降级 | Milvus 不可用时 FTS5 + metadata 是否可用 | 关闭 Milvus mock 后仍能召回关键词记忆 |
| 注入预算 | 高分记忆是否进入 prompt，低分记忆是否被截断 | 多条记忆同时命中时检查 top-k |

### 9.2 核心指标

| 指标 | 判断方式 | 目标 |
|---|---|---|
| 记忆写入准确率 | 应写入的 case 是否进入正确 kind | MVP ≥ 80% |
| 误写率 | 不该长期保存的内容是否被写入 `memory_items` | 越低越好，重点观察 |
| 候选拦截率 | 敏感/低置信内容是否进入 candidate 而非正式 memory | MVP ≥ 90% |
| 检索 Recall@K | golden memory 是否出现在 top-k | Recall@5 ≥ 80% |
| 冲突识别率 | 矛盾记忆是否进入 conflict 流程 | MVP ≥ 80% |
| 去重成功率 | 重复表达是否合并或候选去重 | MVP ≥ 80% |
| 注入预算稳定性 | memory 注入后 prompt 是否超预算 | 0 超限 |

### 9.3 最小评测集

MVP 阶段建议维护 30-50 条 case，覆盖写入、检索和注入三条链路：

- 10 条显式记忆写入 case
- 10 条隐式长期偏好/事实识别 case
- 8 条敏感信息 candidate case
- 8 条一次性信息过滤 case
- 6 条冲突与去重 case
- 10 条检索 Recall@K case

每条 case 至少包含：用户输入、预期动作（写入/候选/忽略/冲突）、预期 kind、预期 source_type、检索 query、golden memory id 或内容片段。

### 9.4 变强的判定

一次记忆系统改动只有满足以下条件，才认为是正向优化：

- Recall@K 上升，或在不下降的情况下显著降低误写率。
- 敏感信息不会绕过 candidate confirmation。
- 一次性信息过滤能力不下降。
- Milvus 不可用时仍能通过 FTS5 / metadata 完成基础召回。
- 注入预算和 prompt 长度不出现回归。

面试时可以这样讲：

> 我对 Memory 的评测不只看“有没有记住”，还看“有没有乱记”。因为 Agent 的长期记忆一旦污染，后续回答会持续受影响。所以我会同时评测写入准确率、误写率、敏感信息候选拦截、Recall@K 和注入预算稳定性。这样能证明 Memory 是可控增强，而不是把历史对话简单塞进向量库。

## 10. 隐私与可删除性

个人健身助手会涉及饮食偏好、训练目标、身体限制、伤病风险等信息。记忆系统必须把“能记住用户”与“用户能控制记忆”同时讲清楚，否则长期记忆会变成隐私风险。

### 10.1 敏感信息分级

下表是应达到的隐私策略。当前自动识别只覆盖一组健康相关关键词；`personal` 和 `security` 的完整识别、拒绝或脱敏尚未实现。因此演示时不要输入账号、密码、token、联系方式等真实敏感信息。

| 级别 | 示例 | 默认处理 |
|---|---|---|
| normal | “喜欢详细解释”“偏好在家训练” | 可写入正式记忆 |
| personal | 身高、体重、作息、饮食偏好 | 用户明确表达长期有效时写入；LLM 推测时进入 candidate |
| health | 膝盖旧伤、腰痛、疾病、过敏 | 当前显式“记住...”命中健康词时进入 `candidate_memories`；覆盖范围仍有限 |
| security | 账号、联系方式、隐私凭据 | 目标是拒绝或脱敏；当前没有可靠拦截，禁止用真实凭据测试 |

### 10.2 用户控制能力

| 能力 | 接口/位置 | 说明 |
|---|---|---|
| 查看记忆 | `GET /memory` / `GET /memory/{id}` | 用户可以看到系统记住了什么 |
| 删除记忆 | `DELETE /memory/{id}` | 将 `memory_items.status` 标记为 `deleted` 并删除 FTS5 项；Milvus 删除同步待补齐 |
| 更新记忆 | `PATCH /memory/{id}` | 用户可以修正错误偏好、目标或限制 |
| 查看候选记忆 | `GET /memory/candidates` | 用户可以看到待确认的敏感/低置信记忆 |
| 确认候选 | `POST /memory/candidates/{id}/confirm` | 确认后才写入正式 `memory_items` |
| 拒绝候选 | `POST /memory/candidates/{id}/reject` | 拒绝后不参与检索、注入和向量同步 |
| 清空会话历史 | `DELETE /chat/{user_id}/history` | 归档当前 conversation，不等同于删除长期记忆 |

### 10.3 删除语义

删除默认采用逻辑删除：

```text
memory_items.status = 'deleted'
  → 当前 Python 写路径删除 FTS5 中对应 memory_id
  → 后续生产化再创建 Milvus delete job 或立即按 vector_id 删除
  → 后续生产化再用 memory_access_logs 保留最小审计记录
  → 后续 retrieval / prompt injection 不再读取该记忆
```

这样做的原因是：SQLite 是 source of truth，系统需要先在主存储中标记删除，再清理派生索引。Milvus 和 FTS5 都只是检索加速层，不能成为删除真相的唯一依据。

### 10.4 候选确认策略

| 来源 | 是否需要确认 | 说明 |
|---|---|---|
| `user_explicit_remember` | 视隐私等级而定 | 普通内容直接写；当前命中健康敏感词时先进入候选 |
| `code_rule` / `project_file` / `manual_import` | 否 | 项目规则或人工导入，来源确定 |
| `llm_candidate` | 是 | 目标策略；当前聊天链路尚未自动生成 LLM 候选 |
| `compact_extraction` | 视内容而定 | 预留来源；当前压缩流程不会自动写长期记忆 |

### 10.5 面试口径

可以这样解释：

> 这个项目是健身助手，长期记忆会涉及训练目标、饮食偏好甚至身体限制，所以当前只开放显式写入，普通闲聊不会自动沉淀；命中健康敏感词时先进入候选确认。SQLite 是 source of truth，用户可以查看、修改、删除记忆；删除已经覆盖 SQLite 和 FTS5，Milvus 删除同步与更完整的凭据拦截是我明确保留的生产化边界。
