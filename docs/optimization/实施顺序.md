# 上下文压缩与记忆系统实施状态

本文负责说明两套设计怎样落地、现在走到哪里。具体结构见[上下文压缩设计](./上下文压缩设计.md)和[记忆系统设计](./记忆系统设计.md)，测试结论见[项目证据](../项目证据.md)。

## 1. 为什么按这个顺序做

落地顺序是：

```text
会话可恢复
  -> Prompt 入口统一
  -> 长期记忆可管理
  -> 敏感信息可确认
  -> SQLite 基础召回
  -> 上下文 compact
  -> Milvus 可选增强
  -> 统一 benchmark
```

先保证主链路可运行和可降级，再增加语义召回与治理能力；SQLite 始终是事实源，Milvus 不是聊天链路的单点依赖。

## 2. 当前实现

| 能力 | 当前状态 | 真实边界 |
|---|---|---|
| 会话持久化 | 已实现 | SQLite 保存完整 turn，可按 `conversation_id` 恢复 |
| 近期上下文 | 已实现 | `SlidingWindowMemory` 只作为热缓存 |
| 会话摘要 | 已实现原型 | 确定性原文摘录，使用字符阈值，不是模型语义摘要 |
| PromptBuilder | 已实现 | 统一主要文本 Prompt，并记录长度、组成和 compact 信息 |
| 结构化状态 | 已实现 | 多意图步骤可共享任务、工具摘要、画像和来源 |
| 长期记忆 | 已实现原型 | 支持 CRUD、显式“记住”写入和精确去重 |
| 敏感记忆 | 已实现后端闭环 | 健康/安全信息先进 candidate；完整端侧确认界面未完成 |
| 基础召回 | 已实现 | SQLite FTS/LIKE 可独立工作 |
| Milvus 增强 | 已实现可选链路 | 通过本地 job 处理接口演示，不是常驻 worker |
| 上下文 compact | 已实现 | 超预算时保留近期轮次并裁剪旧信息，失败不阻塞聊天 |
| 联合评测 | 已实现 | Memory + Context + RAG benchmark 已建立 |

HTTP、SSE 和 WebSocket 共用持久化、摘要更新和 Prompt 构建逻辑，避免三条入口出现不同记忆行为。

## 3. 关键边界

- `compact_summary` 只表示当前会话状态，不直接写入长期记忆。
- 长期事实必须经过 Memory Writer；健康和安全信息仍需候选确认。
- Milvus 是 SQLite 之上的派生语义索引，加载或连接失败时降级到 SQLite。
- 当前按字符控制预算，不宣称精确 token 管理。
- 摘要器不具备模型级归纳和冲突消解能力。
- 用户授权、加密、多实例并发、常驻 worker 和完整审计属于生产化方向。

## 4. 下一步只做验证和治理

当前不需要继续扩展功能。若恢复这一专题，优先级为：

1. 扩大长会话样本，验证目标修改、无关信息和更长对话下的关键事实保留。
2. 为 candidate 确认、删除同步和 Milvus 降级补端到端证据。
3. 根据真实误差决定是否引入模型摘要或精确 token 预算，而不是预先增加复杂度。

## 5. 面试回答

> 我没有把全部历史直接塞进向量库，而是把完整会话、近期窗口、会话摘要和长期记忆分层管理。PromptBuilder 再按任务和预算注入长期记忆、近期对话和 RAG 证据；超阈值时执行确定性 compact。SQLite 是事实源，Milvus 只是可降级的语义增强，敏感记忆还要经过候选确认。当前是可运行、可评测的原型，模型摘要和生产级隐私治理没有夸大为已完成。
