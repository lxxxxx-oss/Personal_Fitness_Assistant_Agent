## 核心关系

```text
近期对话 + 长期记忆 + RAG 证据
  -> PromptBuilder 按任务和预算组合
  -> 上下文过长时执行 compact
  -> 大模型获得本轮真正需要的信息
```

## 当前已经完成

- 会话与长期记忆使用 SQLite 持久化。
- `summaries`、active/superseded 状态和 `last_compacted_message_id` 已投入存储层，完整一轮问答使用事务写入。
- 长期记忆支持基本管理和检索。
- 敏感内容支持候选确认流程。
- SQLite FTS/LIKE 提供基础检索。
- Milvus 可以作为长期记忆语义召回增强。
- PromptBuilder 统一主要文本 Prompt。
- 记录 Prompt 长度、组成部分和 compact 轨迹。
- Memory + Context + RAG 专项 benchmark 已建立。

## 仍属于后续优化

- 确定性抽取式摘要生成、自动触发和 Prompt 注入；模型摘要仅保留为生产化方向。
- 完整候选确认界面。
- 常驻异步 embedding worker。
- 用户授权、加密、过期和审计清理。
- 多实例共享和删除同步。
- 更大规模记忆准确率与隐私评测。

## 面试表达

> 长对话的问题不只是上下文窗口不够，而是历史越长噪声越多，用户偏好容易丢，敏感信息也需要治理。所以我把近期对话、长期记忆和 RAG 证据分开管理，再由 PromptBuilder 按任务和预算注入，超过阈值时进行 compact，并用专项 benchmark 验证组合链路。

当前事实以[项目总览](../项目总览.md)为准，面试学习以[简历技术点总表](../interview/03_简历技术点总表.md)为准。
