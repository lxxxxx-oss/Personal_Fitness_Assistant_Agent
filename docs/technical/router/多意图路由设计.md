# 多意图路由设计

本文只说明当前已经落地的受控多意图路由。单意图评分、评测数字和 LLM/embedding A/B 结论见[路由优化状态](./路由优化状态.md)。

## 1. 解决的问题

单一 `intent` 无法完整表达“先分析动作，再查纠正方法”这类请求。当前 Router 因此同时记录主意图、次意图和执行顺序，但不会开放任意子图组合。

代码层保留五条执行路径：

```text
chat | search | diet | motion | mcp
```

其中 `diet` 负责营养和饮食规划，`mcp` 负责外部菜谱工具调用；产品层可以把二者合并介绍为“饮食与菜谱”。

## 2. 当前状态字段

Router 在内部 `RouterState` 中记录：

```python
_primary_intent: str
_secondary_intents: list[str]
_route_plan: list[str]
_multi_intent_reason: str
_needs_clarification: bool
```

- `_primary_intent`：本轮优先任务，公开响应中的 `intent` 与它一致。
- `_secondary_intents`：同一句话中识别出的次要任务。
- `_route_plan`：允许执行时的子图顺序。
- `_multi_intent_reason`：供日志、调试和评测解释。
- `_needs_clarification`：信息不足时避免强行组合执行。

这些是内部执行字段，不属于当前公开 API 响应协议。

## 3. 识别策略

主意图优先级由确定性策略决定：

- “先/再”等顺序词优先遵守用户顺序。
- 上传动作数据、`.npz` 或姿势判断优先 `motion`。
- “最新、最近、研究、权威说法”优先 `search`。
- 具体菜名、食材和做法进入 `mcp`；摄入量和饮食规划进入 `diet`。
- “不需要具体做法”等否定约束会抑制 `mcp`。
- 信息不足时退回 `chat` 或澄清，不凭空扩展任务。

多意图识别不额外调用模型；本地 Qwen classifier 和 embedding fallback 都是默认关闭的实验能力。

## 4. 受控执行

简单请求仍只执行一个子图。当前只允许四种两步组合：

```text
search -> diet
search -> chat
motion -> chat
motion -> diet
```

执行过程为：

```text
route_plan
  -> 逐个执行白名单子图
  -> 单独记录每步结果或错误
  -> final synthesis 统一生成最终回答
```

单步计划不增加合成调用；多步计划只在末尾生成一次。某一步失败时保留其他成功结果并返回可解释的降级信息。非白名单组合只执行主意图，避免状态传递和错误范围失控。

## 5. 评测与边界

`data/eval/router_challenge_eval.jsonl` 同时标注 `primary_intent`、`secondary_intents` 和 `route_plan`，用于检查识别和规划，不只检查公开 `intent`。当前基线统一见[项目证据](../../项目证据.md)，避免在设计文档复制易过期数字。

当前没有证明生产流量上的泛化能力，也没有开放三步以上或任意组合。后续只有在真实请求显示明确收益时，才扩大样本和组合范围。

## 6. 面试回答

> 我先把单一 intent 扩展为 primary、secondary 和 route plan，再只对白名单中的四种两步组合串联执行。简单请求仍走单子图，多步请求逐步记录结果并在末尾统一合成；部分失败时保留成功步骤。这样解决了真实多任务输入的信息丢失，又没有把 LangGraph 变成不可控的任意工作流。
