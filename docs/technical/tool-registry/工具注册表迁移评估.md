# 工具注册表接入状态

本文只记录 ToolRegistry 已接管哪些真实链路，以及为什么 Motion 没有被整体包装成工具。概念学习和面试主线见[工具系统专题](../../interview/05_工具系统专题.md)。

## 1. 当前状态

| 能力 | ToolSpec | 主链路通过 Registry | 结论 |
|---|---|---|---|
| Search | `search.tavily` | 是 | 已接入 |
| Knowledge/RAG | `knowledge.retrieve` | 是 | Chat/Diet 检索已接入 |
| MCP | `mcp.call_tool` | 是 | execute 节点已接入，发现和规划仍由 MCP 子图负责 |
| Motion | `motion.compare_pose` | 否 | 只注册代表性算法契约，媒体主链路未迁移 |

“存在 ToolSpec”不等于“整个能力已由 Registry 接管”。Registry 负责统一契约、权限、执行元数据和错误结构，不替代 LangGraph 编排、MCP 协议或 FastAPI 上传边界。

## 2. MCP 的分工

MCP 子图保持：

```text
discover -> plan -> Registry.execute("mcp.call_tool") -> format
```

Registry 对调用参数做最小校验，并记录 `execution_id`、耗时和 `subprocess` 权限；MCPClient 继续负责真实 server 调用和 mock fallback。`timeout_seconds` 当前主要是策略元数据，不能表述为完整进程隔离。

## 3. Motion 为什么不整体迁移

Motion 主链路包含上传校验、临时文件、MediaPipe 模型、视频解码、`PoseSequence`、标准动作兼容检查和数值比较。把它们包装成一个大工具会模糊 HTTP 错误和资源清理边界。

最适合 Registry 的部分是纯算法内核：

```text
PoseSequence + reference PoseSequence -> similarity result
```

当前已为它注册 `motion.compare_pose` 契约，但主 API 仍直接调用现有算法。除非后续确实需要统一审计这段算法执行，否则不为了形式统一改动稳定链路。

## 4. 当前边界与面试回答

- Registry 的参数校验是轻量 schema 校验，不是完整生产级沙箱。
- MCP 工具名仍需来自发现结果，外部 server 的安全性不能只靠 Registry 保证。
- Motion 上传和姿态估计继续由 API 层控制是明确边界，不是遗漏。

> 我把输入输出清晰的 Search、RAG 和 MCP execute 先纳入 Registry，统一参数、权限、错误和审计元数据。Motion 涉及上传、模型和临时文件，所以没有把整个 API 包成一个大工具；只为纯姿态比较注册了代表性契约。这样工具治理有真实落点，又不会为了架构统一破坏资源和错误边界。
