# Codex 项目协作约定

本文件是 Codex 在本项目中的主要协作入口。后续修改代码、配置、测试或文档时，应优先遵守这里的规则。

本文件承接了原 `.claude/` 配置的核心逻辑：

- 原 `auto-document` 技能已迁移到 `.codex/skills/auto-document/SKILL.md`。
- 原 `check-tool-spec` 技能已迁移到 `.codex/skills/check-tool-spec/SKILL.md`。
- 原 Claude 本地权限白名单已整理为 `.codex/COMMANDS.md`，仅作为常用命令参考；实际执行权限由 Codex 当前运行环境控制。

## 面试导向总原则

本项目是为了面试展示而生的项目。后续所有代码、文档、讲解和优化，都应服务于“让面试官觉得项目新颖、有意义，同时用户能够清楚应对追问”这个目标。

具体要求：

- 代码层面可以保持原型项目的务实取舍，不必为了展示而过度工程化。
- 文档层面必须清晰、准确、无逻辑矛盾、无低级错误，不能把规划项说成已落地能力。
- 技术描述必须区分“当前已实现”“当前边界”“后续优化方向”，方便面试时诚实且有层次地回答。
- 所有关键技术点都要尽量用用户容易理解的话解释，同时保留能应对面试官深入追问的工程细节。
- 当代码实现较轻量时，文档要准确说明它的工程取舍、为什么这样做、如果生产化会如何升级。
- 面试表达优先强调项目的新颖性、实际意义、架构思路、可观测性、可评测性和演进路径。
- 若发现文档表述与代码现状不一致，应优先修正文档，避免面试时出现逻辑矛盾。
- interview 学习文档第一次提到某个技术点或知识点时，必须解释清楚“是什么、解决什么问题、在项目中怎么用、类似方案有哪些、为什么选择它”，避免只堆名词。
- 简历上出现过的技术点必须在 `docs/interview/` 中有可学习、可背诵的总结，优先维护 `docs/interview/03_简历技术点总表.md`。

## 可运行版本与面试表达边界

当前项目必须始终保持为一个可以完整运行、可以演示、可以解释的版本。后续优化时优先保证核心链路能启动、能调用、能测试、能支撑面试讲解；没有实现的功能可以作为锦上添花的后续规划，不应阻塞当前可运行版本。

具体执行要求：

- 必要功能优先真实补齐：如果某个能力会影响项目成立、接口演示、核心流程或面试可信度，应优先实现到可运行的原型水平。
- 非必要功能可以规划化表达：如生产级模型训练、真实标准动作数据集、线上向量库、大规模部署等，可以说明为“架构已预留、原型链路已打通、后续可替换或扩展”。
- 允许面试友好地强化表达，但不能制造低级矛盾：不要把完全未实现的能力写成已完整落地；可以强调已完成的接口、数据结构、评测方法、降级策略和工程演进路径。
- 代码细节不为完美主义服务：只要不影响运行、演示和核心讲解，不必为了生产级细节过度消耗时间。
- 文档必须服务面试防守：每个关键能力都要能回答“当前做到哪里、为什么这样做、边界是什么、生产化如何升级”。

## 简历口径优先规则

`docs/interview/agent.json` 是用户本地简历源文件，包含个人信息，不提交到 GitHub。后续上传、提交和推送时必须忽略该文件。

面试资料应围绕简历中对项目的描述来组织，优先服务于“简历写了什么，面试官会追问什么，用户如何回答”。当简历表述领先于当前代码实现时，`docs/interview/` 中的材料应帮助用户形成可防守口径：

- 对外表达以简历主线为准，突出项目亮点、技术价值和岗位匹配度。
- 对代码尚未完整落地的能力，要准备好“当前代码状态、为什么这样分阶段做、如果被追问如何解释、后续如何补齐”的回答。
- 不主动弱化简历亮点，但要避免出现低级逻辑矛盾；必要时用“目标架构”“原型链路”“核心思路已验证”“后续生产化补齐”等方式承接。
- 后续修改 `docs/interview/` 时，应优先检查是否支撑简历中的项目描述和技能关键词。

## 文档维护规则

后续修改代码时，必须同步检查并维护 `docs/` 目录中的相关文档。

维护原则：

- `docs/README.md` 是文档总导航，只维护阅读顺序、区域职责和事实优先级。
- 修改项目定位、模块状态、当前边界或下一步计划时，更新 `docs/项目总览.md`。
- 修改后端接口、请求响应字段或流式协议时，更新 `docs/接口说明.md`。
- 修改安装、配置、测试命令、部署或故障处理方式时，更新 `docs/运行与排错.md`。
- 完成自动化测试、专项评测或真实链路验证时，更新 `docs/项目证据.md`。
- 修改面试口径或简历技术点时，更新 `docs/interview/` 对应学习材料。
- 修改微信小程序端时，同步检查 `docs/miniprogram/` 下的实现状态、设计说明和实施计划。
- 路由、动作、工具系统等深入设计维护在 `docs/technical/`；记忆和上下文演进维护在 `docs/optimization/`。
- 早期方案和规格文档保留在 `docs/superpowers/`，旧版面试资料保留在 `docs/technical/interview-archive/`，归档内容不作为当前事实。

判断标准：

如果一次代码修改会改变“项目是做什么的、现在做到哪里、如何运行、接口怎么调、遇到什么问题、问题如何处理”中的任意一项，就必须同步更新文档。

## 长会话与上下文防爆规则

为避免 Codex 长会话上下文爆掉、自动压缩卡死或新会话难以续接，本项目同时使用项目级上下文预算和 `.codex/SESSION_STATE.md` 固定交接文件。`.codex/config.toml` 负责提前压缩并限制单次工具输出，交接文件负责恢复现场；两者缺一不可。新建任务是兜底方案，不是每次主动压缩后的默认动作；`/compact` 成功且当前任务仍轻量可控时，应优先在当前任务继续。

执行要求：

- 新会话开始处理本项目任务时，先读 `AGENTS.md`，再读 `.codex/SESSION_STATE.md`，再运行 `git status --short`。
- 当任务可能跨多轮对话、修改多个文件、运行较长验证、准备暂停/交接/提交/推送，或用户明确表示“下次继续”时，必须更新 `.codex/SESSION_STATE.md`。
- `.codex/SESSION_STATE.md` 只记录短交接信息：当前目标、进度、触碰文件、关键决定、命令结论、验证状态、下一步和可复制的 resume prompt。
- 不要把长日志、大段代码、完整聊天记录复制进交接文件；正式项目事实仍然维护在 `docs/` 对应文档中。
- 如果 `.codex/SESSION_STATE.md` 标记为 `active`，后续 Codex 应优先从其中的 `Next Steps` 继续，而不是重新扫描整个项目或依赖自动压缩摘要。
- 多步骤任务开始时立即把状态标记为 `active`；每完成一个可独立验证的里程碑、准备运行长测试、准备提交/推送或即将暂停时，都要覆盖式更新一次。完成任务后改回 `idle`，不要累积历史流水账。
- 交接文件保持在 4 KB 以内。只记录结论，不粘贴 diff、完整异常栈、测试日志或敏感配置。
- 读取和执行命令时主动限制输出：优先用 `rg` 定位，再读取必要片段；测试输出优先保留失败摘要和末尾结果。除技能明确要求完整读取外，不把超长文件或日志整份送入会话。
- 同一轮出现大量探索结果时，先写检查点再继续；不要等自动压缩触发后才整理现场。如果 `/compact` 成功且剩余工作明确，应留在当前任务完成最小闭环；只有压缩卡住、压缩后仍接近上限、后续还要大范围探索或当前任务明显变慢时，才更新 `Resume Prompt` 并建议新任务续接。
- 不要打印用户级 `~/.codex/config.toml`、环境变量全集或认证文件；排查配置时只查询所需键，防止密钥进入工具输出和压缩摘要。

用户主动压缩口令：

- 当用户消息的主要意图是“提前帮我压缩上下文”（包括“先压缩一下上下文”“帮我做上下文压缩”等明确同义表达）时，把它视为高优先级控制命令，而不是普通咨询。用户只是在引用、讨论或修改这条规则时，不触发。
- 收到口令后立即停止新的探索、代码修改和长测试；允许当前不可安全中断的工具调用结束，然后更新 `.codex/SESSION_STATE.md`，确保目标、已改文件、关键决定、验证状态、阻塞项、下一步和 `Resume Prompt` 完整且总文件小于 4 KB。
- 完成检查点后，只输出一份不超过 600 字的压缩交接摘要，不重复长日志、代码或历史解释；摘要应说明默认可以在当前任务继续，除非出现压缩失败、上下文仍高或后续工作范围明显扩大的情况。
- 如果当前会话提供可调用的原生 compact 能力，立即调用；如果没有，则明确提示用户下一条只需输入 `/compact`。在 `/compact` 真正执行前，只能说“压缩检查点已完成”，不能声称客户端上下文已经压缩。
- 处理口令时不顺带继续原任务；待检查点和 `/compact` 完成后，优先在当前任务从 `Next Steps` 继续。只有满足兜底条件时，才改为新任务恢复。

详细规则见 `docs/optimization/会话续接与上下文防爆规则.md`。

## 工具/函数设计规则

当新增、重构或审查工具、函数接口、MCP 工具定义、Tool Schema 时，使用 `.codex/skills/check-tool-spec/SKILL.md` 中的五项标准检查：

- 职责清晰。
- 输入清晰。
- 输出清晰。
- 权限清晰。
- 错误可处理。

本项目已有统一工具返回结构 `ToolResult` 和 `ErrorCode`，新增工具应优先复用这套模式。

## 常用命令

运维、测试和联调命令见 `.codex/COMMANDS.md`。

重要提醒：

- 不要把 Claude 专用权限配置当作 Codex 权限配置使用。
- 修改接口后同步 `docs/接口说明.md`。
- 修改运行方式或部署配置后同步 `docs/运行与排错.md`；修改项目状态或边界后同步 `docs/项目总览.md`。
