# Codex 项目协作约定

本文件是 Codex 在本项目中的主要协作入口。后续修改代码、配置、测试或文档时，应优先遵守这里的规则。

本文件承接了原 `.claude/` 配置的核心逻辑：

- 原 `auto-document` 技能已迁移到 `.codex/skills/auto-document/SKILL.md`。
- 原 `check-tool-spec` 技能已迁移到 `.codex/skills/check-tool-spec/SKILL.md`。
- 原 Claude 本地权限白名单已整理为 `.codex/COMMANDS.md`，仅作为常用命令参考；实际执行权限由 Codex 当前运行环境控制。

## 面试导向总原则

本项目是为了面试展示而生的项目。后续所有代码、文档、讲解和优化，都应服务于“让面试官觉得项目新颖、有意义，同时用户能够清楚应对追问”这个目标。

具体要求：

- 代码层面可以保持原型项目的务实取舍，不必为了展示而过度工程化。
- 文档层面必须清晰、准确、无逻辑矛盾、无低级错误，不能把规划项说成已落地能力。
- 技术描述必须区分“当前已实现”“当前边界”“后续优化方向”，方便面试时诚实且有层次地回答。
- 所有关键技术点都要尽量用用户容易理解的话解释，同时保留能应对面试官深入追问的工程细节。
- 当代码实现较轻量时，文档要准确说明它的工程取舍、为什么这样做、如果生产化会如何升级。
- 面试表达优先强调项目的新颖性、实际意义、架构思路、可观测性、可评测性和演进路径。
- 若发现文档表述与代码现状不一致，应优先修正文档，避免面试时出现逻辑矛盾。

## 文档维护规则

后续修改代码时，必须同步检查并维护 `docs/` 目录中的相关文档。

维护原则：

- 修改后端接口、请求/响应字段、流式协议时，更新 `docs/API.md`。
- 修改项目能力、模块状态、部署方式、配置项、已知问题或下一步计划时，更新 `docs/README.md`。
- 修改微信小程序端时，同步检查 `docs/miniprogram/` 下的设计、计划和完成度记录。
- 做阶段性重构、问题修复或重要实现步骤时，可在 `docs/progress/` 追加记录。
- 做手工验收、冒烟测试或分级测试时，可在 `docs/tests/` 追加记录。
- 早期方案和规格文档保留在 `docs/superpowers/`，除非明确需要迁移或归档，不随意删除。
- 顶层文档保持精简：`docs/README.md` 负责项目说明、进度、问题处理和运维方式；`docs/API.md` 负责接口细节。

判断标准：

如果一次代码修改会改变“项目是做什么的、现在做到哪里、如何运行、接口怎么调、遇到什么问题、问题如何处理”中的任意一项，就必须同步更新文档。

## 工具/函数设计规则

当新增、重构或审查工具、函数接口、MCP 工具定义、Tool Schema 时，使用 `.codex/skills/check-tool-spec/SKILL.md` 中的五项标准检查：

- 职责清晰。
- 输入清晰。
- 输出清晰。
- 权限清晰。
- 错误可处理。

本项目已有统一工具返回结构 `ToolResult` 和 `ErrorCode`，新增工具应优先复用这套模式。

运维、测试和联调命令见 `.codex/COMMANDS.md`。

- 修改运行方式、部署配置、项目进度或已知问题后同步 `docs/README.md`。
